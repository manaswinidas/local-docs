自Spark 2.4发布以来， Spark SQL提供了内置支持，用于读写Apache Avro数据。

部署中

的spark-avro模块是外部的，不包含在spark-submit要么spark-shell默认。

与任何Spark应用程序一样， spark-submit用于启动您的应用程序。spark-avro_2.12它的依赖关系可以直接添加到spark-submit使用--packages ， 如，

./bin/spark-submit --packages org.apache.spark:spark-avro_2.12:3.0.0-preview ...

为了实验spark-shell ，您也可以使用--packages添加org.apache.spark:spark-avro_2.12直接依赖

./bin/spark-shell --packages org.apache.spark:spark-avro_2.12:3.0.0-preview ...

有关提交具有外部依赖关系的应用程序的更多详细信息，请参见《 应用程序提交指南》 。

加载和保存功能

以来spark-avro模块是外部的，没有.avro API在DataFrameReader要么DataFrameWriter 。

要以Avro格式加载/保存数据，您需要指定数据源选项format如avro （要么org.apache.spark.sql.avro ）。

to_avro（）和from_avro（）

Avro软件包提供功能to_avro将列以Avro格式编码为二进制，并且from_avro()将Avro二进制数据解码为一列。这两个函数都将一列转换为另一列，并且输入/输出SQL数据类型可以是复杂类型或原始类型。

从诸如Kafka的流媒体源读取或写入时，将Avro记录用作列很有用。每条卡夫卡键值记录都将添加一些元数据，例如，卡夫卡的提取时间戳，卡夫卡的偏移量等。

  • 如果包含数据的“值”字段位于Avro中，则可以使用from_avro()提取数据，充实数据，清理数据，然后将其再次向下游推送到Kafka或写出到文件中。
  • to_avro()可用于将结构转换为Avro记录。当您想在将数据写到Kafka时将多列重新编码为一个列时，此方法特别有用。

这两个函数当前仅在Scala和Java中可用。

与Databricks spark-avro的兼容性

该Avro数据源模块最初来自Databricks的开源存储库spark-avro并与之兼容。

默认情况下使用SQL配置spark.sql.legacy.replaceDatabricksSparkAvro.enabled启用，数据源提供者com.databricks.spark.avro映射到此内置Avro模块。对于使用创建的Spark表Provider作为财产com.databricks.spark.avro在目录元存储中，如果使用此内置Avro模块，则映射对于加载这些表至关重要。

请注意Databricks的spark-avro隐式类AvroDataFrameWriter和AvroDataFrameReader是为快捷功能创建的.avro() 。在此内置但外部模块中，两个隐式类均被删除。请用.format("avro")在DataFrameWriter要么DataFrameReader相反，它应该干净而且足够好。

如果您喜欢使用自己的版本spark-avro jar文件，您只需禁用配置spark.sql.legacy.replaceDatabricksSparkAvro.enabled ，并使用该选项--jars部署您的应用程序。有关更多详细信息，请阅读《应用程序提交指南》中的“ 高级依赖关系管理”部分。

除了上面列出的类型，它还支持阅读union类型。以下三种类型被认为是基本类型union类型：

  1. union(int, long)将被映射到LongType。
  2. union(float, double)将被映射到DoubleType。
  3. union(something, null) ，其中任何受支持的Avro类型。这将被映射为与某物相同的Spark SQL类型，并将nullable设置为true。所有其他联合类型都被认为是复杂的。根据联合的成员，它们将被映射到字段名称为member0，member1等的StructType。这与在Avro和Parquet之间转换时的行为一致。

目前，它会忽略Avro文件中存在的文档，别名和其他属性。
