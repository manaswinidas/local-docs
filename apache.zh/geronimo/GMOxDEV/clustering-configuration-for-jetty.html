标准和特定于Geronimo的部署描述符更新

为了集群Web应用程序，必须按以下方式更新标准和Geronimo特定的部署描述符：

  • 标准部署描述符，即web.xml：必须将Web应用程序标记为可分发。换句话说，必须定义可选的“可分发”元素。
  • Geronimo特定的部署描述符，即geronimo-web.xml：必须定义“集群-wadi”替换组。此元素提供特定于Web应用程序的群集配置，该配置将覆盖默认的群集配置。默认的clusering配置由org.apache.geronimo.configs / jetty6-clustering-builder-wadi // car模块声明的GBean WADIJettyClusteringBuilder定义。

群集配置概述

我们感兴趣的是四种配置：

  • org.apache.geronimo.configs / clustering // car ：它定义了集群Web应用程序必须具有的运行时依赖项。例如，它将geronimo群集JAR导入到此类Web应用程序的类路径中。另外，它定义了BasicNode GBean，它为集群中的这个Geronimo实例定义了唯一的名称。
    为了给Geronimo实例指定特定的名称，可以通过更新在var / config / config-substitutions.properties中定义的clusterNodeName属性来重新配置此模块： clusterNodeName = MyOwnNodeName
  • org.apache.geronimo.configs / jetty6-clustering-wadi // car ：它定义了WADI的运行时依赖性，WADI是实际的集群提供程序。此外，它还定义了Jetty6集群合同的WADI实现。
  • org.apache.geronimo.configs / jetty6-clustering-builder-wadi // car ：它定义了构建时间依赖性以及WADIJettyClusteringBuilder GBean，该GBean声明了默认的集群配置并知道如何处理“ clustering-wadi”元素。
  • org.apache.geronimo.configs / wadi-clustering // car ：它定义了WADI特定的GBean，用于进一步配置集群应用程序的行为。在此模块中，通过重写DefaultBackingStrategyFactory GBean，可以配置给定会话在整个群集中要维护的默认副本数（设置为开箱即用的2个）。

创建一个新的Geronimo实例

创建新Geronimo实例的最简单方法是：

  • 在Geronimo安装目录下创建一个新目录，例如node2；
  • 将var目录复制到其中；和
  • 更改node2 / var / config / config-substitutions.properties中的PortOffset属性。例如，可以像下面这样来创建名为node2的新实例： cd $ GERONIMO_HOME mkdir node2 cp -r var node2 perl -pi -e's / PortOffset = 0 / PortOffset = 1 /'node2 / var / config / config-substitutions.properties
  • 通过更新node2 / var / config / config-substitutions.properties中的clusterNodeName属性为该实例赋予唯一名称： clusterNodeName = NODE2为了启动此新实例，您需要设置org.apache.geronimo.server.name将系统属性更改为新实例的名称。在上面的示例中，可以从gshell提示符像这样启动node2： geronimo / start-server -G server.name = node2 -b
  • 确保在启动Geronimo实例时将系统属性java.net.preferIPv4Stack设置为true。取消注释etc / rc.d / start-server，default.groovy中的以下行： //取消注释此属性，以便Tribes在Mac OS X上运行command.properties ['java.net.preferIPv4Stack'] = true 系统属性java。 net.preferIPv4Stack

    必须将系统属性java.net.preferIPv4Stack设置为true，否则，WADI使用的组通信Tribes将会失败（至少在MAC OS X上如此）。

测试群集

以上两个实例现在正在运行演示Web应用程序的群集版本。该应用程序实现了一种可视化的方式来观察会话的更新位置和访问位置。

  • 让我们通过访问以下URL 在即用型实例（红色节点）上创建会话： http : //127.0.0.1 : 8080/wadi-webapp/index.jsp
    您应该看到：
    • 红色1表示会话已在红色节点上更新。每帧的红色背景表示该会话已被红色节点访问。
    • 在浏览器中，您可以验证JSESSIONID cookie以NODE结尾。负载平衡器可以利用此信息来提供粘性负载平衡。
  • 现在让我们从黄色的节点node2请求相同的会话： http : //127.0.0.1 : 8081/ wadi-webapp/ index.jsp
    您应该看到：
    • 黄色的2表示会话已在黄色节点上更新。会话状态（即红色的1）已从红色迁移到黄色节点。每一帧的黄色背景表示该会话已被黄色节点访问。
    • 在浏览器中，您可以验证/ wadi路径的JSESSIONID cookie以NODE2结尾。正如预期的那样，会话cookie已更新为反映会话的新位置。 这没有按预期工作

      如果看到的是1而不是2，则节点之间不会看到彼此。验证的第一件事是系统属性java.net.preferIPv4Stack确实设置为true。如果将其设置为true，则多播在运行节点的盒子上可能无法正常工作。查看防火墙和网络配置，以确保允许多播。

  • 如果将这两个实例放在Apache mod_proxy后面并访问相同的URL： http : //127.0.0.1 : 8100/wadi-webapp/index.jsp
    您应该看到类似以下内容：
    • 3可能是黄色或红色，并且每帧的背景色是黄色和红色的混合色。这种颜色的混合表明，会话已根据mod_proxy为每个帧选择的目标实例在红色和黄色实例之间迁移。
  • 您可以重复访问反向代理URL，以确认会话迁移实施的可靠性。所有请求都成功，即没有任何状态丢失。例如，您应该看到类似的内容：
  • 为了确认复制工作正常，您可以杀死-9个节点并访问其余节点的URL，并观察到状态已正确还原。
