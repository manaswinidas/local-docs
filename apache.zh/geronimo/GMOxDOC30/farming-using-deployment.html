排队

可以使用单个逻辑部署步骤将配置部署到Geronimo服务器集群。

将配置部署到群集后，可以在所有群集成员之间透明地启动，停止或取消部署此配置。
这两个功能极大地简化了集群中正在运行的应用程序的维护，因为Geronimo会为您处理集群范围的应用程序分发和管理。

集群成员的配置

  1. 应该为您的Geronimo实例指定一个特定的名称。这可以通过更新clusterNodeName内定义的属性/var/config/config-substitutions.properties ： 虚线

    clusterNodeName = NODE1
    RemoteDeployHostname = NODE1_IP

  2. 集群成员是在Geronimo服务器上配置的，该服务器可以但不是集群的成员。这可以通过添加org.apache.geronimo.farm.config来实现。每个群集成员的BasicNodeInfo GBean都可以进行耕种配置config.xml 。该GBean如下所示： NODE2 系统 管理器 rmi NODE2_IP 1100 JMXConnector false它定义了用于通过JMX连接到集群成员的网络地址（主机，端口，urlPath）和凭据（用户名和密码）。此类声明应包含在config.xml文件，Geronimo服务器的耕种模块。
    默认情况下，服务器场配置将本地服务器定义为群集成员。要排除它，可以阻止GBean NodeInfo启动。

场部署

要部署配置以配置成员，只需将其部署到服务器场配置定义的MasterConfigurationStore存储库中。此配置已被立即停用，您可能必须使用以下GShell命令首次启动它：

部署/启动org.apache.geronimo.configs / farming // car

农业将两个新的存储库添加到服务器： MasterConfigurationStore和ClusterStore 。MasterConfigurationStore是您应该经常使用的存储库（如果不是经常的话）。ClusterStore是一个存储库，您可能需要在特定和不常见的情况下使用它。
要部署到MasterConfigurationStore，请将--targets标志传递给如下的distribute或deploy命令：

部署/分发--targets XXX，name = MasterConfigurationStore

部署完成后，

  • MasterConfigurationStore包含一种以更改名称命名的虚拟配置，在配置名称后附加“ _G_MASTER”，以及
  • ClusterStore包含与您的部署同名的实际配置，并定义了GBean，用于控制整个群集内实际配置的远程启动和停止。

群集范围内的配置管理

若要跨服务器场取消部署，用户需要在用于跨服务器场部署/分发应用程序的同一节点上执行undeploy命令。请注意，此节点充当管理服务器的一种，因此它是唯一需要维护其服务器场配置以反映服务器场成员的节点，并且针对服务器场模块的所有生命周期操作（例如，启动/停止）都将传播到其他节点。要在整个群集中启动，停止或取消部署配置，请针对虚拟配置执行相关的部署任务。

部署/取消部署_G_MASTER

初始部署时配置的所有群集成员必须正在运行。否则，这些操作将部分执行。例如，如果取消部署虚拟配置时配置了三个成员，而其中只有两个正在运行，则将从两个运行的服务器的ClusterStore中删除该配置。第三台服务器的ClusterStore仍包含配置，并且必须在该服务器上手动取消部署。
