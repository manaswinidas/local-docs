起搏器

介绍

Pacemaker是一个风暴守护程序，旨在处理工人的心跳。随着Storm规模的扩大，由于执行心跳的工作人员的大量写入操作，ZooKeeper开始成为瓶颈。当ZooKeeper试图保持一致性时，会大量写入磁盘，并在网络上产生过多的流量。

由于心跳具有短暂性，因此不需要将它们保留在磁盘上或跨节点同步。内存中的商店就可以。这就是起搏器的作用。Pacemaker充当简单的内存键/值存储，具有类似于ZooKeeper的目录样式键和字节数组值。

相应的Pacemaker客户端是ClusterState接口， org.apache.storm.cluster.PaceMakerStateStorageFactory 。心跳电话由ClusterState由。。。生产pacemaker_state_factory进入Pacemaker守护程序，同时将其他设置/获取操作转发到ZooKeeper。

组态

  • pacemaker.servers ：运行Pacemaker守护程序的主机
  • pacemaker.port ：Pacemaker将侦听的端口
  • pacemaker.max.threads ：Pacemaker守护程序将用于处理请求的最大线程数。
  • pacemaker.childopts ：需要进入Pacemaker的所有JVM参数。
  • pacemaker.auth.method ：使用的身份验证方法（下面的更多信息）

例

要启动并运行Pacemaker，请在所有节点上的集群配置中设置以下选项：storm.cluster.state.store: "org.apache.storm.cluster.PaceMakerStateStorageFactory"

还需要在所有节点上设置Pacemaker服务器：pacemaker.servers: - somehost.mycompany.com - someotherhost.mycompany.com

然后启动所有守护程序

（包括起搏器）：$ storm pacemaker

现在，Storm集群应该通过Pacemaker推动所有工作人员的心跳。

安全

当前支持摘要（基于密码）和Kerberos安全。当前，安全性仅围绕读取而不是写入。写入可由任何人执行，而读取只能由授权和认证的用户执行。这是一个未来的发展领域，因为它使群集不受DoS攻击，但它可以防止任何敏感信息进入未经授权的眼睛，这是主要目标。

消化

要配置摘要身份验证，请设置pacemaker.auth.method: DIGEST在托管Nimbus和Pacemaker的节点上的群集配置中。节点还必须具有java.security.auth.login.config设置为指向包含以下结构的JAAS配置文件：PacemakerDigest { username="some username" password="some password"; };

配置了这些设置的任何节点都可以从Pacemaker中读取。工作节点不需要设置这些配置，并且可以保留pacemaker.auth.method: NONE设置，因为它们不需要从Pacemaker守护程序读取。

的Kerberos

要配置Kerberos身份验证，请设置pacemaker.auth.method: KERBEROS在托管Nimbus和Pacemaker的节点上的群集配置中。节点还必须具有java.security.auth.login.config设置为指向JAAS配置。

Nimbus上的JAAS配置必须看起来像这样：```PacemakerClient {com.sun.security.auth.module。Krb5LoginModule必需useKeyTab = true keyTab =“ / etc / keytabs / nimbus.keytab” storeKey = true useTicketCache = false serviceName =“ pacemaker”主体=“ nimbus@MY.COMPANY.COM ”; };


The JAAS config on Pacemaker must look something like this:

PacemakerServer {com.sun.security.auth.module。Krb5LoginModule需要useKeyTab =真密钥表= “的/ etc / keytabs / pacemaker.keytab” storeKey =真useTicketCache =假主要= “ pacemaker@MY.COMPANY.COM ”; }; ```

  • 客户端的用户主体PacemakerClient Nimbus主机上的“部分”必须与nimbus.daemon.user风暴集群配置值。
  • 客户的serviceName值必须与中的服务器的用户主体匹配PacemakerServer在“ Pacemaker主机”上。

容错能力

Pacemaker作为单个守护程序实例运行，使其成为潜在的单点故障。

如果Nimbus通过崩溃或网络分区无法访问Pacemaker，则工作进程将继续运行，并且Nimbus将反复尝试重新连接。Nimbus功能将被破坏，但是拓扑本身将继续运行。如果群集的分区中Nimbus和Pacemaker位于分区的同一侧，则位于分区另一侧的工作人员将无法心跳，并且Nimbus将在其他地方重新安排任务。无论如何，这可能是我们想要发生的事情。

ZooKeeper比较

与ZooKeeper相比，Pacemaker使用更少的CPU，更少的内存，并且对于相同的负载，当然没有磁盘，这是由于维护节点之间的一致性而导致的开销不足。在千兆网络上，理论上大约有6000个节点的限制。但是，实际限制可能在2000-3000个节点左右。这些限制尚未经过测试。在270个管理器群集中，该群集完全按拓扑进行调度，Pacemaker资源利用率是一核的70％，一台装有4核的计算机的RAM接近1GiB。 Intel(R) Xeon(R) CPU E5530 @ 2.40GHz和24GiB的RAM。

Pacemaker现在支持HA。风暴群集中可以一次使用多个Pacemaker实例，以实现大规模的可伸缩性。只需在起搏器中包括起搏器主机的名称即可。servers配置中，worker和Nimbus将开始与它们进行通信。它们也是容错的。只要至少有一个起搏器处于运行状态，该系统就可以继续工作-只要它能够处理负载即可。
