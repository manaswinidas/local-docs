CGroup执法

风暴中的CGroups

Storm使用CGroup来限制工作人员的资源使用，以确保公平和QOS。

请注意：CGroups当前仅在Linux平台（内核版本2.6.24及更高版本）上受支持。

设定

要使用CGroup，请确保正确安装cgroup并配置cgroup。有关设置和配置的更多信息，请访问：

https://access.redhat.com/documentation/zh-CN/Red_Hat_Enterprise_Linux/6/html/Resource_Management_Guide/ch-Using_Control_Groups.html

/ conf目录中提供了样本/默认cgconfig.conf文件。内容如下：

mount {
    cpuset  = /cgroup/cpuset;
    cpu = /cgroup/storm_resources;
    cpuacct = /cgroup/storm_resources;
    memory  = /cgroup/storm_resources;
    devices = /cgroup/devices;
    freezer = /cgroup/freezer;
    net_cls = /cgroup/net_cls;
    blkio   = /cgroup/blkio;
}

group storm {
       perm {
               task {
                      uid = 500;
                      gid = 500;
               }
               admin {
                      uid = 500;
                      gid = 500;
               }
       }
       cpu {
       }
       memory {
       }
       cpuacct {
       }
}

有关cgconfig.conf文件的格式和配置的更详细说明，请访问：

https://access.redhat.com/documentation/zh-CN/Red_Hat_Enterprise_Linux/6/html/Resource_Management_Guide/ch-Using_Control_Groups.html#The_cgconfig.conf_File

为了让Storm通过单个工作人员管理cgroup，您需要确保将要控制的资源安装在与上例相同的目录下。如果它们不在同一目录中，那么主管将引发异常。

需要配置perm部分，以便主管运行的用户可以修改组。

如果启用了“以用户身份运行”，以便主管以启动拓扑的用户身份生成其他进程，请确保权限使得单个用户具有读取访问权限，而没有写入访问权限。

与Storm中的CGroups相关的设置

设置 功能
storm.resource.isolation.plugin.enable 此配置用于设置是否使用cgroup。设置为“ true”以启用cgroups。将“ false”设置为不使用cgroups。当此配置设置为false时，与cgroup相关的单元测试将被跳过。默认设置为“假”
storm.cgroup.hierarchy.dir Storm将使用的cgroup层次结构的路径。默认设置为“ / cgroup / storm_resources”
storm.cgroup.resources 由CGroups监管的子系统列表。默认设置为cpu和内存。当前仅支持cpu和内存
storm.supervisor.cgroup.rootdir 主管使用的根cgroup。cgroup的路径为/ 。默认设置为“风暴”
storm.cgroup.cgexec.cmd 用于在cgroup中启动工作程序的cgexec命令的绝对路径。默认设置为“ / bin / cgexec”
storm.worker.cgroup.memory.mb.limit 每个工作程序的内存限制（以MB为单位）。可以在每个主管节点的基础上进行设置。此配置用于设置cgroup配置memory.limit_in_bytes。有关memory.limit_in_bytes的更多详细信息，请访问： https ://access.redhat.com/documentation/zh-CN/Red_Hat_Enterprise_Linux/6/html/Resource_Management_Guide/sec-memory.html。请注意，如果您使用的是Resource Aware Scheduler，请不要设置此配置，因为此配置将覆盖Resource Aware Scheduler所计算的值
storm.worker.cgroup.cpu.limit 每个工人的cpu份额。可以在每个主管节点的基础上进行设置。此配置用于设置cgroup配置cpu.share。有关cpu.share的更多详细信息，请访问： https ://access.redhat.com/documentation/en-US/Red_Hat_Enterprise_Linux/6/html/Resource_Management_Guide/sec-cpu.html。请注意，如果您使用的是Resource Aware Scheduler，请不要设置此配置，因为此配置将覆盖Resource Aware Scheduler计算的值。

由于通过cpu.shares限制CPU使用率仅会限制进程的比例CPU使用率，因此，要限制管理程序节点上所有工作进程的CPU使用率，请设置config administrator.cpu.capacity。其中每个增量代表一个核心的1％，因此，如果用户将supervisor.cpu.capacity：200设置为用户，则表示正在使用2个核心。

与Resource Aware Scheduler集成

CGroup可以与Resource Aware Scheduler结合使用。然后，CGroup将强制执行由Resource Aware Scheduler分配的工作程序的资源使用。要将cgroup与Resource Aware Scheduler一起使用，只需启用cgroup，并确保不要设置storm.worker.cgroup.memory.mb.limit和storm.worker.cgroup.cpu.limit配置。

CGroup指标

CGroups不仅可以限制工作人员可以访问的资源数量，而且还可以帮助监视工作人员的资源消耗。默认情况下，启用了多个度量标准，这些度量标准将检查工作进程是否是CGroup的一部分并报告相应的度量标准。

CGroupCPU

org.apache.storm.metric.cgroup。CGroupCPU返回与org.apache.storm.metrics.sigar类似的指标。CPUMetric，但CGroup中的所有内容除外。它以毫秒为单位报告用户和系统的CPU使用情况（以毫秒为单位）

{
   "user-ms": number
   "sys-ms": number
}

CGroup将这些报告为CLK_TCK计数，而不是毫秒数，因此精度由CLK_TCK设置为什么来确定。在大多数系统上，它是每秒100次，因此精度最高为10毫秒。

为了使此度量标准有效，必须安装cpuacct。

CGroupCpuGuarantee

org.apache.storm.metric.cgroup。CGroupCpuGuarantee报告此工作线程保证获得的大约CPU时间毫秒数。这是根据该给定工作程序中任务所请求的资源来计算的。

CGroupMemory

org.apache.storm.metric.cgroup。CGroupMemoryUsage以字节为单位报告cgroup中所有进程的当前内存使用情况

CGroupMemoryLimit

org.apache.storm.metric.cgroup。CGroupMemoryLimit以字节为单位报告cgroup中所有进程的当前限制。如果在风暴中启用了CGroups的情况下运行，则这是工作程序内所有任务的堆上请求+堆外请求+分配给工作程序的任何额外倾斜空间。

在拓扑中使用/调试CGroup

当代码在CGroup下运行时，这些指标对于调试代码已经发生或正在发生的工作非常有用。

中央处理器

风暴中的CPU保证很软。这意味着，如果有可用的CPU，工人可以轻松地履行其保证。要检测到您的工作人员正在使用更多的CPU，然后请求您可以对CGroupCPU中的值求和并将其与CGroupCpuGuarantee进行比较。
如果CGroupCPU始终高于或等于CGroupCpuGuarantee，则您可能希望查看请求更多的CPU，因为如果集群上有更多负载，您的工作线程可能会饿死。等于CGroupCpuGuarantee意味着您的工作人员可能已经受到限制。如果使用的CPU比CGroupCpuGuarantee小得多，那么您可能正在浪费资源，并且可能希望减少CPU的要求。

如果您确实有很高的CPU，您可能还想检查一下您工人的GC指标和/或GC日志。随着垃圾回收的发生，堆上的内存压力可能会导致CPU增加。

记忆

由于多个原因，在cgroup下对Java进行内存调试可能很困难。

  1. JVM内存管理很复杂
  2. 在撰写本文档时，只有JVM提供了对cgroups的实验性支持
  3. JNI和其他进程可能会耗尽JVM并不总是知道的cgroup中的内存。
  4. 堆中的内存压力可能导致增加的CPU负载，而不是增加的内存分配。

默认情况下，storm提供了一些指标，这些指标可以帮助您了解工作人员中正在发生的事情。

如果CGroupMemory接近CGroupMemoryLimit，那么您知道此工作程序可能很快就会开始发生坏事。内存不是像CPU这样的软保证。如果您翻阅Linux上的OOM杀手，将开始与您的工作人员一起射击进程。请注意这些指标。如果您运行的是cgroup感知的Java版本，那么超出限制通常意味着您将需要增加堆外请求。如果不是这样，可能是因为您需要更多的堆外内存，或者可能是Java分配了更多的内存，然后作为垃圾收集过程的一部分。找出最好的方法是反复试验（对不起）。

Storm还分别通过“内存/堆”和“内存/非堆”指标报告JVM的堆上和堆外使用情况。这些可以用来给您提示增加的提示。查看每个下面的“ usedBytes”字段可以帮助您了解JVM当前正在使用多少内存。虽然，就像我说的那样，堆外部分并不总是准确的，并且当堆增长时，它可能导致未记录的堆外内存，这将导致cgroup射击进程。

GC指标的名称因所使用的垃圾收集器而异，但都以“ GC /”开头。如果汇总给定工作对/窗口对的所有“ GC / *。timeMs”度量标准，您应该能够看到有多少CPU保证用于GC。默认情况下，java在抛出OutOfMemoryError之前允许98％的cpu时间进入GC。对于近实时流系统而言，这远非理想，因此请注意该比率。

如果该比率处于相当稳定的状态，并且您的内存使用量还没有接近极限，那么您可能需要考虑减少内存请求。这也可能很复杂。
