三叉戟喷口

就像在香草Storm API中一样，喷口是Trident拓扑中流的源。在香草Storm喷口上，Trident公开了更多API，用于更复杂的喷口。

在您如何获取数据流和如何基于这些数据流更新状态（例如数据库）之间有着千丝万缕的联系。有关此说明，请参见Trident状态文档 -了解此链接对于了解可用的喷口选项非常重要。

常规Storm喷口在Trident拓扑中将是非事务性喷口。要使用常规的Storm IRichSpout，请在TridentTopology中创建如下流：

TridentTopology topology = new TridentTopology();
topology.newStream("myspoutid", new MyRichSpout());

必须为Trident拓扑中的所有喷口赋予流唯一的标识符-该标识符在集群上运行的所有拓扑中必须是唯一的。Trident将使用该标识符存储有关Zookeeper消耗的喷口的元数据，包括txid和与喷口相关的任何元数据。

您可以通过以下配置选项配置喷口元数据的Zookeeper存储：

  1. transactional.zookeeper.servers ：Zookeeper主机名列表
  2. transactional.zookeeper.port ：Zookeeper集群的端口
  3. transactional.zookeeper.root ：Zookeeper中存储元数据的根目录。元数据将存储在路径/

流水线

默认情况下，Trident一次处理一个批次，在尝试另一个批次之前，等待该批次成功或失败。通过流水线处理批次，可以显着提高吞吐量，并降低处理延迟。使用“ topology.max.spout.pending”属性配置要同时处理的最大批处理数量。

即使在同时处理多个批次时，Trident也会命令批次之间拓扑中发生的任何状态更新。例如，假设您正在将全局计数聚合到数据库中。这个想法是，当您更新数据库中批次1的计数时，仍然可以计算批次2至10的部分计数。在成功完成第1批的状态更新之前，Trident不会继续进行第2批的状态更新。如Trident state doc中所述 ，这对于实现一次精确的处理语义至关重要。

三叉戟喷嘴类型

以下是可用的喷口API：

  1. ITridentSpout ：可以支持事务性或不透明事务性语义的最通用的API。通常，您将使用此API的分区风味之一，而不是直接使用该风味。
  2. IBatchSpout ：一次发出一批元组的非事务性喷嘴
  3. IPartitionedTridentSpout ：从分区数据源（如Kafka服务器群集）读取的事务处理喷嘴
  4. IOpaquePartitionedTridentSpout ：从分区数据源读取的不透明事务喷嘴

而且，就像本教程开头提到的那样，您也可以使用常规的IRichSpout。

关于Apache Storm

Apache Storm与任何排队系统和任何数据库系统集成。Apache Storm的喷口抽象使集成新排队系统变得容易。同样，将Apache Storm与数据库系统集成很容易。
