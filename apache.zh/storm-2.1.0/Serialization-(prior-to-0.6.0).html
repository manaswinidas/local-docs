元组可以由任何类型的对象组成。由于Storm是一个分布式系统，因此当任务之间传递对象时，它需要知道如何序列化和反序列化对象。默认情况下，Storm可以序列化int，shorts，longs，floats，doubles，bool，bytes，string和byte数组，但是如果要在元组中使用其他类型，则需要实现自定义序列化器。

动态打字

元组中没有字段的类型声明。您将对象放在字段中，Storm会动态计算出序列化。在进入序列化接口之前，让我们花点时间来理解为什么Storm的元组是动态类型的。

将静态类型添加到元组字段将为Storm的API增加大量复杂性。例如，Hadoop静态地键入其键和值，但用户方面需要大量注释。Hadoop的API使用负担重，“类型安全”是不值得的。动态类型简单易用。

除此之外，不可能以任何合理的方式静态地键入Storm的元组。假设一个Bolt订阅了多个流。所有这些流中的元组在字段之间可能具有不同的类型。当螺栓收到Tuple在execute ，该元组可能来自任何流，因此可以具有任何类型的组合。对于螺栓所订阅的每个元组流，可能有一些反射魔术可以声明一个不同的方法，但是Storm选择了更简单，直接的动态类型化方法。

最后，使用动态类型的另一个原因是，可以从Clojure和JRuby之类的动态类型语言中直接使用Storm。

自定义序列化

让我们深入了解定义自定义序列化的Storm API。用户需要执行两个步骤来创建自定义序列化：实现序列化程序，并将序列化程序注册到Storm。

创建一个序列化器

自定义序列化程序实现ISerialization接口。实现指定如何将类型序列化和反序列化为二进制格式。

该界面如下所示：

public interface ISerialization<T> {
    public boolean accept(Class c);
    public void serialize(T object, DataOutputStream stream) throws IOException;
    public T deserialize(DataInputStream stream) throws IOException;
}

风暴使用accept确定类型是否可以通过此序列化器序列化的方法。请记住，Storm的元组是动态键入的，因此Storm可以确定在运行时使用哪种序列化器。

serialize以二进制格式将对象写出到输出流。必须以一种可以稍后反序列化的方式编写该字段。例如，如果要写出对象列表，则需要先写出列表的大小，以便知道要反序列化多少个元素。

deserialize从流中读取序列化的对象并返回它。

您可以在SerializationFactory的源代码中看到示例序列化实现。

注册序列化器

创建序列化程序后，您需要告诉Storm它存在。这是通过Storm配置完成的（有关配置如何在Storm中工作的信息，请参阅概念 ）。您可以通过提交拓扑时给定的配置注册序列化，也可以在整个集群的storm.yaml文件中注册序列化。

序列化器注册是通过Config完成的。TOPOLOGY_SERIALIZATIONS配置，并且只是序列化类名称的列表。

Storm提供了用于在拓扑配置中注册序列化程序的助手。Config类有一个称为addSerialization它需要一个序列化器类来添加到配置中。

有一个称为Config的高级配置。TOPOLOGY_SKIP_MISSING_SERIALIZATIONS。如果将其设置为true，Storm将忽略所有已注册但在类路径中没有可用代码的序列化。否则，Storm在找不到序列号时将引发错误。如果您在一个群集上运行许多拓扑，并且每个拓扑都有不同的序列化，但是您想在该拓扑中的所有拓扑之间声明所有序列化，这将很有用。 storm.yaml文件。
