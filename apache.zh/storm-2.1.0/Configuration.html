组态

Storm具有多种配置，可用于调整灵气，管理程序和运行拓扑的行为。某些配置是系统配置，不能逐个拓扑地在拓扑上进行修改，而其他配置可以针对每个拓扑进行修改。

每个配置都有一个默认值，该值在Storm代码库的defaults.yaml中定义。您可以通过在Nimbus和主管的类路径中定义storm.yaml来覆盖这些配置。最后，您可以定义使用StormSubmitter时随拓扑一起提交的特定于拓扑的配置。但是，特定于拓扑的配置只能覆盖以“ TOPOLOGY”为前缀的配置。

Storm 0.7.0及更高版本允许您基于每个螺栓/每个喷口来覆盖配置。可以通过这种方式覆盖的唯一配置是：

  1. “ topology.debug”
  2. “ topology.max.spout.pending”
  3. “拓扑最大任务并行性”
  4. “ topology.kryo.register”：这与其他方法略有不同，因为序列化将可用于拓扑中的所有组件。有关序列化的更多详细信息。

Java API使您可以通过两种方式指定组件特定的配置：

  1. 内部：覆盖getComponentConfiguration在任何喷嘴或螺栓中返回特定于组件的配置图。
  2. 外部： setSpout和setBolt在TopologyBuilder用方法返回一个对象addConfiguration和addConfigurations可用于覆盖组件的配置。

配置值的优先顺序为defaults.yaml

螺栓，喷口和插件

在几乎所有情况下，应该通过螺栓或喷嘴实现的设置器（而不是拓扑配置）来完成螺栓或喷嘴的配置。在极少数情况下，可能会暴露出当前不属于Config或DaemonConfig的拓扑范围的配置，例如将自定义调度程序或插件编写到Storm的某些部分时。在这种情况下，您可以创建自己的类，例如Config，但是实现Validated 。任何public static final String在此类中声明的字段将被视为配置和来自org.apache.storm.validation.ConfigValidationAnnotations类可用于强制执行该配置中存储的内容。要让验证者知道此类，您需要将该类当作一种服务，该服务将通过ServiceLoader为Validated类加载，并包括一个META-INF/services/org.apache.storm.validation.Validated jar中的文件，其中包含Config类的名称。

关于Apache Storm

Apache Storm与任何排队系统和任何数据库系统集成。Apache Storm的喷口抽象使集成新排队系统变得容易。同样，将Apache Storm与数据库系统集成很容易。
