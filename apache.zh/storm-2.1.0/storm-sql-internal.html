Storm SQL的内部

此页面描述了Storm SQL集成的设计和实现。

总览

SQL是一种被广泛采用但又很复杂的标准。包括Drill，Hive，Phoenix和Spark在内的几个项目已在其SQL层上进行了大量投资。StormSQL的主要设计目标之一是利用这些项目的现有投资。StormSQL利用Apache Calcite实施SQL标准。StormSQL专注于将SQL语句编译为Storm拓扑，以便可以在Storm集群中执行它们。

图1描述了在StormSQL中执行SQL查询的工作流程。首先，用户提供了一系列SQL语句。StormSQL解析SQL语句，并将其转换为Calcite逻辑计划。逻辑计划由一系列SQL逻辑运算符组成，这些逻辑运算符描述了如何与基础执行引擎无关地执行查询。逻辑运算符的一些示例包括TableScan ， Filter ， Projection和GroupBy 。

图1：StormSQL的工作流程。

下一步是将逻辑执行计划编译为物理执行计划。物理计划由物理操作员组成，这些操作员描述了如何在StormSQL中执行SQL查询。物理操作员，例如Filter ， Projection和GroupBy直接映射到Storm拓扑中的操作。StormSQL还将SQL语句中的表达式编译为Java代码块，并将其插入Storm Streams API函数中，该函数将被编译一次并在运行时执行。

最后，StormSQL将创建的具有空打包JAR的Storm拓扑提交到Storm集群。Storm以与执行其他Storm拓扑相同的方式调度和执行Storm拓扑。

以下代码块显示了一个示例查询，该查询可以过滤和投影来自Kafka流的结果。

CREATE EXTERNAL TABLE ORDERS (ID INT PRIMARY KEY, UNIT_PRICE INT, QUANTITY INT) LOCATION 'kafka://...' ...

CREATE EXTERNAL TABLE LARGE_ORDERS (ID INT PRIMARY KEY, TOTAL INT) 'kafka://...' ...

INSERT INTO LARGE_ORDERS SELECT ID, UNIT_PRICE * QUANTITY AS TOTAL FROM ORDERS WHERE UNIT_PRICE * QUANTITY > 50

前两个SQL语句定义了外部数据的输入和输出。图2描述了StormSQL如何获取最后一个的过程SELECT查询并将其编译为Storm拓扑。

图2：将示例查询编译为Storm拓扑。

查询流表的约束

查询代表实时数据流的表时，有几个约束条件：

  • 的ORDER BY子句不能应用于流。
  • 至少有一个单调场GROUP BY允许StormSQL的子句限制缓冲区的大小。

有关更多信息，请参阅http://calcite.apache.org/docs/stream.html 。

相依性

Storm会考虑Storm SQL的必要依赖关系，但由EXTERNAL TABLE 。您可以使用--jars要么--artifacts选择storm sql这样就可以将数据源JAR包含到Storm SQL Runner和Storm Topology运行时类路径中。（使用--artifacts如果您的数据源JAR在Maven存储库中可用，因为它可以处理传递依赖项。）

请参考Storm SQL集成页面以了解操作方法。

关于Apache Storm

Apache Storm与任何排队系统和任何数据库系统集成。Apache Storm的喷口抽象使集成新排队系统变得容易。同样，将Apache Storm与数据库系统集成很容易。
