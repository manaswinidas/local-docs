在Storm Core中加入流

Storm core支持借助以下功能将多个数据流合并为一个JoinBolt 。 JoinBolt是一个窗口式螺栓，即它等待配置的窗口持续时间以匹配要加入的流中的元组。这有助于在Window边界内对齐流。

每个JoinBolt的传入数据流必须是在单个字段上分组的字段。流只能使用已被FieldsGrouped的字段与其他流合并。
知道这一点将有助于理解以下描述的联接语法。

执行加入

考虑以下涉及4个表的SQL连接：

select  userId, key4, key2, key3
from        table1
inner join  table2  on table2.userId =  table1.key1
inner join  table3  on table3.key3   =  table2.userId
left join   table4  on table4.key4   =  table3.key3

类似的联接可以表示为使用4个喷嘴生成的元组JoinBolt ：

JoinBolt jbolt =  new JoinBolt("spout1", "key1")                   // from        spout1  
                    .join     ("spout2", "userId",  "spout1")      // inner join  spout2  on spout2.userId = spout1.key1
                    .join     ("spout3", "key3",    "spout2")      // inner join  spout3  on spout3.key3   = spout2.userId   
                    .leftJoin ("spout4", "key4",    "spout3")      // left join   spout4  on spout4.key4   = spout3.key3
                    .select  ("userId, key4, key2, spout3:key3")   // chose output fields
                    .withTumblingWindow( new Duration(10, TimeUnit.MINUTES) ) ;

topoBuilder.setBolt("joiner", jbolt, 1)
            .fieldsGrouping("spout1", new Fields("key1") )
            .fieldsGrouping("spout2", new Fields("userId") )
            .fieldsGrouping("spout3", new Fields("key3") )
            .fieldsGrouping("spout4", new Fields("key4") );

螺栓构造函数带有两个参数。第一个参数介绍来自spout1成为第一个流，并指定它将始终使用field key1与其他流一起加入时。指定的组件名称必须引用直接连接到连接螺栓的喷嘴或螺栓。从这里接收到的数据spout1必须是分组的字段key1 。同样，每个leftJoin()和join()方法调用会引入一个新的流以及用于联接的字段。从上面的示例中可以看出，相同的FieldsGrouping要求也适用于这些流。join方法的第三个参数引用另一个要加入的流。

的select()方法用于指定输出字段。争论select是用逗号分隔的字段列表。各个字段名称可以使用流名称前缀，以消除出现在多个流中的相同字段名称之间的歧义，如下所示： .select("spout3:key3, spout4:key3") 。如果已使用嵌套完成，则支持嵌套元组类型Map s。例如outer.inner.innermost指嵌套在三层深处的字段outer和inner是类型Map 。

任何join（）参数中的字段均不允许使用流名称前缀，但是支持嵌套字段。

致电withTumblingWindow()以上，将连接窗口配置为10分钟的滚动窗口。以来JoinBolt是开窗螺栓，我们也可以使用withWindow将其配置为滑动窗口的方法（请参见下面的提示部分）。

流名称和加入顺序

  • 在引用流名称之前（在连接方法的第3个参数中），必须先引入流名称（在构造函数中或作为各种连接方法的1st arg）。不允许向前引用流名称，如下所示：
new JoinBolt( "spout1", "key1")                 
  .join     ( "spout2", "userId",  "spout3") //not allowed. spout3 not yet introduced
  .join     ( "spout3", "key3",    "spout1")
  • 在内部，将按照用户表示的顺序执行连接。

根据流名称加入

为简单起见，Storm拓扑通常使用default流。拓扑也可以使用命名流代替default流。为了支持这样的拓扑， JoinBolt可以通过构造函数的第一个参数配置为使用流名称，而不使用源组件（喷口/螺栓）名称：

new JoinBolt(JoinBolt.Selector.STREAM,  "stream1", "key1")
                                  .join("stream2", "key2")
    ...

第一个论点JoinBolt.Selector.STREAM通知螺栓stream1/2/3/4指的是命名流（与上游喷口/螺栓的名称相反）。

下面的示例将来自四个喷口的两个命名流连接起来：

new JoinBolt(JoinBolt.Selector.STREAM,  "stream1", "key1") 
                             .join     ("stream2", "userId",  "stream1" )
                             .select ("userId, key1, key2")
                             .withTumblingWindow( new Duration(10, TimeUnit.MINUTES) ) ;

topoBuilder.setBolt("joiner", jbolt, 1)
            .fieldsGrouping("bolt1", "stream1", new Fields("key1") )
            .fieldsGrouping("bolt2", "stream1", new Fields("key1") )
            .fieldsGrouping("bolt3", "stream2", new Fields("userId") )
            .fieldsGrouping("bolt4", "stream1", new Fields("key1") );

在上面的示例中， bolt1 ，例如，还发出其他流。但是连接螺栓只订阅stream1和stream2从不同的螺栓。 stream1从bolt1 ， bolt2和bolt4被视为单个流并加入stream2从bolt3 。

局限性：

  1. 当前仅支持INNER和LEFT联接。

  2. 与SQL允许将不同键上的同一表连接到不同表的SQL不同，此处必须在流上使用相同的一个字段。字段分组可确保将正确的元组路由到连接螺栓的正确实例。因此，为了获得正确的结果，FieldsGrouping字段必须与join字段相同。要对多个字段执行联接，可以将这些字段合并为一个字段，然后发送到“联接”螺栓。

提示：

  1. 连接可能会占用大量CPU和内存。当前窗口中累积的数据越大（与窗口长度成比例），执行连接所需的时间越长。滑动间隔短（例如几秒钟）会触发频繁的联接。因此，如果使用较大的窗口长度或较小的滑动间隔或同时使用两者，则会降低性能。

  2. 使用滑动Windows时，可以跨窗口复制联接的记录。这是因为在使用“滑动窗口”时，元组继续跨多个窗口存在。

  3. 如果启用了消息超时，请确保超时设置（topology.message.timeout.secs）足够大，可以舒适地容纳窗口大小，以及其他喷嘴和螺栓的额外处理。

  4. 在最坏的情况下 ，将两个包含M和N个元素的两个流的窗口合并在一起，可以生成MxN个元素，每个输出元组都从每个输入流锚定到一个元组。这可能意味着JoinBolt有很多输出元组，甚至还有更多的ACK用于下游螺栓发出。如果不小心，这可能会对消息传递系统造成很大压力，并极大地降低拓扑速度。为了管理消息传递子系统上的负载，建议执行以下操作：

    • 增加工作人员的堆（topology.worker.max.heap.size.mb）。
    • 如果您的拓扑不需要ACKing，请禁用ACKers（topology.acker.executors = 0）。
    • 禁用事件记录器（topology.eventlogger.executors = 0）。
    • 轮到拓扑调试（topology.debug = false）。
    • 将topology.max.spout.pending设置为足够大的值，以容纳估计的完整窗口元组值以及更多的净空值。当消息传递子系统承受过多负载时，这有助于减少喷口发出过多元组的可能性。当其值设置为null时，可能会发生这种情况。
    • 最后，将窗口大小保持为解决当前问题所需的最小值。
