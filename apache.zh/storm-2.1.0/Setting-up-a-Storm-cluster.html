设置风暴集群

该页面概述了启动和运行Storm集群的步骤。

如果您在使用Storm群集时遇到困难，请首先在“ 疑难解答”页面中检查解决方案。否则，通过电子邮件发送邮件列表。

以下是设置Storm集群的步骤的摘要：

  1. 设置Zookeeper集群
  2. 在Nimbus和辅助计算机上安装依赖项
  3. 将Storm发行版下载并解压缩到Nimbus和辅助计算机
  4. 将必需的配置填写到storm.yaml中
  5. 使用“风暴”脚本和您选择的主管在监督下启动守护程序
  6. 设置DRPC服务器（可选）

设置Zookeeper集群

Storm使用Zookeeper协调群集。Zookeeper 不用于消息传递，因此Storm在Zookeeper上的负载非常低。在大多数情况下，单节点Zookeeper集群就足够了，但是如果您要进行故障转移或部署大型Storm集群，则可能需要更大的Zookeeper集群。有关部署Zookeeper的说明，请参见此处 。

有关Zookeeper部署的一些注意事项：

  1. 至关重要的是，您必须在监督下运行Zookeeper，因为Zookeeper的运行速度极快，如果遇到任何错误情况，它将退出该过程。有关更多详细信息，请参见此处 。
  2. 设置cron来压缩Zookeeper的数据和事务日志至关重要。Zookeeper守护程序不会自行执行此操作，如果您未设置cron，Zookeeper将很快用完磁盘空间。有关更多详细信息，请参见此处 。

在Nimbus和辅助计算机上安装依赖项

接下来，您需要在Nimbus和辅助计算机上安装Storm的依赖项。这些是：

  1. Java 8+（针对Java 8 JDK通过travis ci测试了Apache Storm 2.x）
  2. Python 2.6.6（Python 3.x也应该工作，但未作为CI环境的一部分进行测试）

这些是经过Storm测试的依赖项版本。Storm可能适用于Java和/或Python的不同版本。

将Storm发行版下载并解压缩到Nimbus和辅助计算机

接下来，下载Storm发行版并将zip文件解压缩到Nimbus和每台辅助计算机上的某个位置。可以从此处下载Storm版本。

将必需的配置填写到storm.yaml中

Storm版本包含位于的文件conf/storm.yaml配置Storm守护程序。您可以在此处查看默认配置值。 storm.yaml会覆盖defaults.yaml中的任何内容。要使集群正常工作，必须执行一些配置：

1） storm.zookeeper.servers ：这是Storm集群的Zookeeper集群中主机的列表。它看起来应该像这样：

storm.zookeeper.servers:
  - "111.222.333.444"
  - "555.666.777.888"

如果Zookeeper群集使用的端口与默认端口不同，则还应设置storm.zookeeper.port 。

2） storm.local.dir ：Nimbus和Supervisor守护程序需要本地磁盘上的目录来存储少量状态（如jar，conf和类似的东西）。您应该在每台计算机上创建该目录，为其赋予适当的权限，然后使用此配置填写目录位置。例如：

storm.local.dir: "/mnt/storm"

如果您在Windows上运行Storm，则可能是：

storm.local.dir: "C:\\storm-local"

如果使用相对路径，则该路径将相对于安装storm（STORM_HOME）的位置。您可以将其保留为默认值$STORM_HOME/storm-local

3） nimbus.seeds ：工作节点需要知道哪些机器是主节点的候选者，才能下载拓扑jar和conf。例如：

nimbus.seeds: ["111.222.333.44"]

建议您将值填写到计算机的FQDN列表中。如果要设置Nimbus H / A，则必须解决所有运行nimbus的计算机的FQDN。当您只想设置“伪分布式”集群时，您可能希望将其保留为默认值，但是仍然建议您填写FQDN。

4） Supervisor.slots.ports ：使用此配置，为每台工作计算机配置在该计算机上运行的工作计算机数量。每个工作程序使用一个端口来接收消息，并且此设置定义哪些端口可以打开使用。如果您在此处定义五个端口，那么Storm将最多分配五个工作程序在此计算机上运行。如果定义三个端口，Storm最多只能运行三个。默认情况下，此设置配置为在端口6700、6701、6702和6703上运行4个工作进程。例如：

supervisor.slots.ports:
    - 6700
    - 6701
    - 6702
    - 6703

5） drpc.servers ：如果要设置DRPC服务器，则需要指定它们，以便工作人员可以找到它们。这应该是DRPC服务器的列表。例如：

drpc.servers: ["111.222.333.44"]

监控主管的健康状况

Storm提供了一种机制，管理员可以通过该机制配置主管以定期运行管理员提供的脚本来确定节点是否正常。管理员可以通过在storm.health.check.dir中的脚本中执行自己选择的任何检查，来使主管确定节点是否处于健康状态。如果脚本检测到该节点处于不正常状态，则它必须返回非零的退出代码。在Storm 2.x之前的发行版中，一个错误将脚本退出值0视为失败。现在，此问题已得到解决。主管将定期在运行状况检查目录中运行脚本并检查输出。如上所述，如果脚本的输出包含字符串ERROR，则主管将关闭所有工作程序并退出。

如果主管正在监督下运行，则可以调用“ / bin / storm node-health-check”来确定是否应启动主管或该节点是否不正常。

运行状况检查目录位置可以配置为：

storm.health.check.dir: "healthchecks"

脚本必须具有执行权限。可以使用以下命令配置允许任何给定的运行状况检查脚本在被标记为由于超时而失败之前运行的时间：

storm.health.check.timeout.ms: 5000

配置外部库和环境变量（可选）

如果您需要外部库或自定义插件的支持，则可以将此类jar放入extlib /和extlib-daemon /目录中。请注意，extlib-daemon /目录存储仅由守护程序（Nimbus，Supervisor，DRPC，UI，Logviewer）使用的jar，例如HDFS和自定义的调度库。因此，用户可以配置两个环境变量STORM_EXT_CLASSPATH和STORM_EXT_CLASSPATH_DAEMON，以包括外部类路径和仅守护程序的外部类路径。有关使用外部库的更多详细信息，请参见类路径处理 。

使用“风暴”脚本和您选择的主管在监督下启动守护程序

最后一步是启动所有Storm守护程序。至关重要的是，您必须在监督下运行这些守护程序。Storm是一个快速运行的系统，这意味着一旦遇到意外错误，进程就会停止。Storm的设计使其可以在任何时候安全地停止并在重新启动进程时正确恢复。这就是为什么Storm不会保持进程状态的原因-如果Nimbus或Supervisor重新启动，则正在运行的拓扑不会受到影响。这是运行Storm守护程序的方法：

  1. Nimbus ：运行命令bin/storm nimbus在主机上进行监督。
  2. 主管 ：运行命令bin/storm supervisor在每台工人机器的监督下。主管守护程序负责在该计算机上启动和停止工作进程。
  3. UI ：通过在监视下运行命令“ bin / storm ui”来运行Storm UI（可以从浏览器访问的站点，该站点可提供有关群集和拓扑的诊断信息）。可以通过将Web浏览器导航到http：// {ui主机}：8080来访问UI。

如您所见，运行守护程序非常简单。守护程序将在您提取Storm发行版的任何位置登录到logs /目录。

设置DRPC服务器（可选）

就像使用nimbus或主管一样，您将需要启动drpc服务器。为此，请运行命令bin/storm drpc在您配置为其中一部分的每台计算机上drpc.servers配置。

DRPC Http设置

DRPC还可以选择提供REST API。启用此设置drpc.http.port到要在启动DRPC服务器之前运行的端口。有关如何使用它的更多信息，请参见REST文档 。

通过设置还支持SSL drpc.https.port以及密钥库和可选信任库，与您配置UI的方式类似。
