在生产集群上运行拓扑

在生产集群上运行拓扑类似于在本地模式下运行。步骤如下：

1）定义拓扑（如果使用Java定义，则使用TopologyBuilder ）

2）使用StormSubmitter将拓扑提交到集群。 StormSubmitter输入拓扑名称，拓扑配置以及拓扑本身作为输入。例如：

Config conf = new Config();
conf.setNumWorkers(20);
conf.setMaxSpoutPending(5000);
StormSubmitter.submitTopology("mytopology", conf, topology);

3）创建一个包含您的拓扑代码的JAR。您可以选择将代码的所有依赖项捆绑到该JAR中（Storm除外-Storm JAR将添加到工作节点上的类路径中），也可以利用Storm中的Classpath处理功能来使用外部库，而无需将其捆绑到拓扑JAR中。

如果您使用的是Maven，则Maven Assembly Plugin可以为您打包。只需将其添加到您的pom.xml中：

  <plugin>
    <artifactId>maven-assembly-plugin</artifactId>
    <configuration>
      <descriptorRefs>  
        <descriptorRef>jar-with-dependencies</descriptorRef>
      </descriptorRefs>
      <archive>
        <manifest>
          <mainClass>com.path.to.main.Class</mainClass>
        </manifest>
      </archive>
    </configuration>
  </plugin>

然后运行mvn assembly：assembly以获取适当包装的jar。确保您排除了Storm jar，因为集群在类路径上已经有Storm。

4）使用以下命令将拓扑提交到集群storm客户端，指定jar的路径，要运行的类名以及它将使用的所有参数：

storm jar path/to/allmycode.jar org.me.MyTopology arg1 arg2 arg3

storm jar将jar提交给集群并配置StormSubmitter与正确的集群交谈。在此示例中，上传罐子后storm jar调用主函数org.me.MyTopology带有参数“ arg1”，“ arg2”和“ arg3”。

您可以了解如何配置自己的storm客户与Storm集群谈论建立开发环境 。

常用配置

您可以为每个拓扑设置多种配置。您可以在此处找到可以设置的所有配置的列表。可以在特定于拓扑的基础上覆盖带有“ TOPOLOGY”的前缀（其他是集群配置，不能覆盖）。以下是为拓扑设置的一些常见设置：

  1. 配置TOPOLOGY_WORKERS ：这设置了用于执行拓扑的工作进程数。例如，如果将其设置为25，则集群中将有25个Java进程执行所有任务。如果您在拓扑中的所有组件上组合了150个并行性，则每个工作进程中将有6个任务作为线程在其中运行。
  2. 配置TOPOLOGY_ACKER_EXECUTORS ：这设置了将跟踪元组树并检测何时已完全处理喷嘴元组的执行程序的数量。确认是Storm可靠性模型不可或缺的一部分，您可以在保证消息处理中阅读有关它们的更多信息 。通过不设置此变量或将其设置为null，Storm会将确认执行程序的数量设置为等于为此拓扑配置的工作程序的数量。如果将此变量设置为0，则Storm将在元组脱离喷嘴后立即对其进行确认，从而有效地降低了可靠性。
  3. 配置TOPOLOGY_MAX_SPOUT_PENDING ：这设置一次可以在单个喷管任务上挂起的喷管元组的最大数量（挂起表示该元组尚未被确认或失败）。强烈建议您设置此配置以防止队列爆炸。
  4. 配置TOPOLOGY_MESSAGE_TIMEOUT_SECS ：这是在将喷嘴元组视为失败之前必须完全完成的最长时间。该值默认为30秒，对于大多数拓扑来说已经足够。有关Storm可靠性模型如何工作的更多信息，请参阅保证消息处理 。
  5. 配置TOPOLOGY_SERIALIZATIONS ：您可以使用此配置向Storm注册更多序列化程序，以便可以在元组中使用自定义类型。

杀死拓扑

要终止拓扑，只需运行：

storm kill {stormname}

给同一个名字storm kill就像您在提交拓扑时使用的一样。

Storm不会立即杀死拓扑。取而代之的是，它禁用所有喷口，以便它们不再发出任何元组，然后Storm等待Config。销毁所有工作人员前TOPOLOGY_MESSAGE_TIMEOUT_SECS秒。这为拓扑提供了足够的时间来完成它被杀死时正在处理的任何元组。

更新正在运行的拓扑

要更新正在运行的拓扑，当前唯一的选择是终止当前拓扑并重新提交新拓扑。计划中的功能是实施storm swap该命令将正在运行的拓扑替换为新的拓扑，从而确保最少的停机时间，并且两个拓扑都不会同时处理元组。

监控拓扑

监视拓扑的最佳位置是使用Storm UI。 Storm UI提供有关任务中发生的错误的信息以及有关每个运行拓扑的每个组件的吞吐量和延迟性能的细粒度统计信息。

您还可以查看集群计算机上的工作日志。

关于Apache Storm

Apache Storm与任何排队系统和任何数据库系统集成。Apache Storm的喷口抽象使集成新排队系统变得容易。同样，将Apache Storm与数据库系统集成很容易。
