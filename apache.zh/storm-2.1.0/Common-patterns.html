常见的拓扑模式

此页面列出了Storm拓扑中的各种常见模式。

  1. 批处理
  2. 基本螺栓
  3. 内存中缓存+字段分组组合
  4. 流顶N
  5. TimeCacheMap用于有效地保留最近更新的内容的缓存
  6. 分布式RPC的CoordinatedBolt和KeyedFairBolt

批处理

经常出于效率原因或其他原因，您希望分批处理一组元组，而不是单独处理。例如，您可能要批量更新数据库或进行某种流式聚合。

如果您希望数据处理具有可靠性，那么正确的方法是在螺栓等待进行批处理的同时保留实例变量中的元组。一旦完成了批处理操作，就可以确认所有持有的元组。

如果螺栓发出元组，则您可能需要使用多锚定来确保可靠性。这完全取决于特定的应用。有关可靠性如何工作的更多详细信息，请参见保证消息处理 。

基本螺栓

许多螺栓遵循类似的模式：读取输入元组，基于该输入元组发出零个或多个元组，然后在execute方法结束时立即确认该输入元组。匹配此模式的螺栓是函数和过滤器之类的东西。这是一种常见的模式，Storm公开了一个名为IBasicBolt的接口，该接口可以为您自动执行该模式。有关更多信息，请参见保证消息处理 。

内存中缓存+字段分组组合

将缓存保留在Storm螺栓中是很常见的。将其与字段分组结合使用时，缓存将变得特别强大。例如，假设您有一个螺栓可以将短网址（如bit.ly，t.co等）扩展为长网址。通过将短URL的LRU缓存保留为长URL扩展，可以避免重复执行相同的HTTP请求，从而提高性能。假设组件“ urls”发出短URL，而组件“ expand”将短URL扩展为长URL，并在内部保留缓存。考虑以下两个代码片段之间的区别：

builder.setBolt("expand", new ExpandUrl(), parallelism)
  .shuffleGrouping(1);

builder.setBolt("expand", new ExpandUrl(), parallelism)
  .fieldsGrouping("urls", new Fields("url"));

第二种方法将具有更有效的缓存，因为相同的URL将始终执行相同的任务。这避免了在任务中的任何缓存之间进行重复，并且使短URL更有可能命中缓存。

流顶N

在Storm上完成的常见连续计算是某种“流式前N”。假设您有一个螺栓，该螺栓发出[[value]，“ count”]形式的元组，并且您希望一个螺栓根据计数发出前N个元组。最简单的方法是使用螺栓对流进行全局分组，并在内存中保留前N个项目的列表。

这种方法显然无法扩展到大数据流，因为整个数据流必须完成一项任务。进行计算的一种更好的方法是在流的各个分区上并行执行许多前N个，然后将这些前N个合并在一起以获得全局前N个。该模式如下所示：

builder.setBolt("rank", new RankObjects(), parallelism)
  .fieldsGrouping("objects", new Fields("value"));
builder.setBolt("merge", new MergeObjects())
  .globalGrouping("rank");

此模式之所以有效，是因为第一个螺栓完成了字段分组，从而使您需要分区在语义上是正确的。你可以看到在风暴启动这种模式的一个例子在这里 。

但是，如果在处理的数据中存在已知的偏斜，则使用partialKeyGrouping而不是fieldsGrouping可能是有利的。这样会将每个键的负载分配在两个下游螺栓之间，而不是一个。

builder.setBolt("count", new CountObjects(), parallelism)
  .partialKeyGrouping("objects", new Fields("value"));
builder.setBolt("rank" new AggregateCountsAndRank(), parallelism)
  .fieldsGrouping("count", new Fields("key"))
builder.setBolt("merge", new MergeRanksObjects())
  .globalGrouping("rank");

拓扑需要额外的处理层来聚合上游螺栓的部分计数，但是现在只处理聚合的值，因此，螺栓不会受到数据偏斜导致的负载的影响。你可以看到在风暴启动这种模式的一个例子在这里 。

TimeCacheMap用于有效地保留最近更新的内容的缓存

有时您可能希望在内存中保留最近“活动”的项目的缓存，并且使一段时间不活动的项目自动过期。TimeCacheMap是执行此操作的有效数据结构，并提供了挂钩，因此您可以在项目过期时插入回调。

分布式RPC的CoordinatedBolt和KeyedFairBolt

在Storm之上构建分布式RPC应用程序时，通常需要两种常见的模式。这些由CoordinatedBolt和KeyedFairBolt封装，它们是Storm代码库附带的“标准库”的一部分。

CoordinatedBolt包装包含您的逻辑的螺栓，并弄清楚何时螺栓已收到任何给定请求的所有元组。它大量使用直接流来做到这一点。

KeyedFairBolt还包装了包含您的逻辑的螺栓，并确保您的拓扑同时处理多个DRPC调用，而不是一次连续地进行一次。

有关更多详细信息，请参见分布式RPC 。
