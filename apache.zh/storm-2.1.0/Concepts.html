拓扑结构

实时应用程序的逻辑被打包到Storm拓扑中。Storm拓扑类似于MapReduce作业。一个主要区别是MapReduce作业最终完成，而拓扑将永远运行（或者，直到您杀死它）。拓扑是与流分组连接的喷口和螺栓的图形。这些概念如下所述。

  • 本地模式 ：阅读本文以了解如何在本地模式下开发和测试拓扑。

流

流是Storm中的核心抽象。流是无限制的元组序列，它以分布式方式并行处理和创建。流使用命名流元组中字段的模式定义。默认情况下，元组可以包含整数，多头，短裤，字节，字符串，双精度数，浮点数，布尔值和字节数组。您还可以定义自己的序列化器，以便可以在元组中本地使用自定义类型。

声明时，每个流都有一个ID。由于单流喷口和螺栓非常常见，因此OutputFieldsDeclarer具有方便的方法来声明单流而无需指定id。在这种情况下，流的默认ID为“默认”。

  • 元组 ：流由元组组成
  • OutputFieldsDeclarer ：用于声明流及其模式
  • 序列化 ：有关Storm动态元组输入和声明自定义序列化的信息

喷口

喷口是拓扑中流的源。通常，喷口将从外部源中读取元组，并将它们发送到拓扑中（例如，Kestrel队列或Twitter API）。喷嘴可能是可靠的也可能是不可靠的 。如果Storm无法对其进行处理，则可靠的喷嘴可以重播元组，而不可靠的喷嘴会在元组发出后立即将其忘记。

喷口可散发出不止一个流。为此，请使用declareStream OutputFieldsDeclarer的方法，并指定使用emit SpoutOutputCollector中的方法。

壶嘴的主要方法是nextTuple 。 nextTuple要么在拓扑中发出新的元组，要么在没有新元组发出时简单地返回。当务之急是nextTuple不会阻止任何spout的实现，因为Storm会在同一线程上调用所有spout方法。

壶嘴的其他主要方法是ack和fail 。当Storm检测到从喷嘴发出的元组通过拓扑成功完成或失败时，将调用这些方法。 ack和fail只要求可靠的喷嘴。有关更多信息，请参见Javadoc 。

资源：

  • IRichSpout ：这是spout必须实现的接口。

螺栓

拓扑中的所有处理均通过螺栓完成。螺栓可以执行任何操作，包括过滤，函数，聚合，联接，与数据库对话等等。

螺栓可以进行简单的流转换。进行复杂的流转换通常需要多个步骤，因此需要多个螺栓。例如，将一连串推文转换为趋势图像流至少需要两个步骤：一个螺栓对每个图像进行滚动式推文滚动计数，以及一个或多个螺栓将最前面的X张图像流化（您可以执行此操作使用三个螺栓比使用两个螺栓更具可扩展性的方式进行特定的流转换）。

螺栓可以发射不止一个流。为此，请使用declareStream OutputFieldsDeclarer的方法，并指定使用emit OutputCollector上的方法。

声明螺栓的输入流时，您始终订阅另一个组件的特定流。如果要订阅另一个组件的所有流，则必须分别订阅每个组件。InputDeclarer具有语法糖，用于订阅在默认流ID上声明的流。说declarer.shuffleGrouping("1")订阅组件“ 1”上的默认流，等效于declarer.shuffleGrouping("1", DEFAULT_STREAM_ID) 。

螺栓的主要方法是execute新的元组作为输入的方法。螺栓使用OutputCollector对象发出新的元组。螺栓必须ack上的方法OutputCollector对于它们处理的每个元组，Storm会知道元组何时完成（并最终确定对原始喷嘴元组的确认是安全的）。对于处理输入元组，基于该元组发出0个或多个元组然后确认输入元组的常见情况，Storm提供了一个IBasicBolt接口，该接口会自动进行确认。

在螺栓中启动异步处理的新线程非常好。OutputCollector是线程安全的，可以随时调用。

  • OutputCollector ：使用此类的实例，螺栓将元组发射到其输出流

流分组

定义拓扑的一部分是为每个螺栓指定应接收的流作为输入。流分组定义了如何在螺栓任务之间分配该流。

Storm中有八个内置的流分组，您可以通过实现CustomStreamGrouping接口来实现自定义流分组：

  1. 随机分组 ：元组以一定方式随机分布在螺栓任务上，以确保每个螺栓都具有相等数量的元组。
  2. 字段分组 ：按分组中指定的字段对流进行分区。例如，如果按“用户ID”字段对流进行分组，则具有相同“用户ID”的元组将始终执行相同的任务，但是具有“用户ID”不同的元组可能会执行不同的任务。
  3. 部分密钥分组 ：流按分组中指定的字段进行分区（如“字段”分组），但在两个下游螺栓之间进行负载平衡，当输入数据倾斜时，可以更好地利用资源。本文对它的工作方式及其优点提供了很好的解释。
  4. 全部分组 ：流在螺栓的所有任务之间复制。请谨慎使用此分组。
  5. 全局分组 ：整个流进入螺栓任务的其中一项。具体来说，它将转到ID最低的任务。
  6. 无分组 ：此分组指定您不关心流的分组方式。当前，没有分组等效于随机分组。不过，最终，Storm会将没有分组的螺栓向下推，以与其所预订的螺栓或喷口在同一线程中执行（如果可能）。
  7. 直接分组 ：这是一种特殊的分组。以这种方式分组的流意味着元组的生产者决定消费者的哪个任务将接收此元组。直接分组只能在已经声明为直接流的流上声明。发出直接流的元组必须使用[emitDirect]（javadocs / org / apache / storm / task / OutputCollector.html＃emitDirect（int，int，java.util。清单）方法。螺栓可以使用提供的TopologyContext或通过跟踪销的输出来获取其使用者的任务ID。 emit OutputCollector中的方法（返回元组发送到的任务ID）。
  8. 本地或随机分组 ：如果目标螺栓在同一工作进程中具有一个或多个任务，则元组将被随机混合为那些正在进行的任务。否则，这就像常规的随机分组。

资源：

  • TopologyBuilder ：使用此类定义拓扑
  • InputDeclarer ：每当返回此对象时setBolt被调用TopologyBuilder用于声明螺栓的输入流以及应如何对这些流进行分组

可靠性

Storm保证每个喷嘴元组将被拓扑完全处理。它通过跟踪每个喷口元组触发的元组树并确定何时成功完成该元组树来做到这一点。每个拓扑都有一个与之关联的“消息超时”。如果Storm无法检测到在该超时时间内已完成喷嘴元组，则它将使元组失败并稍后重播。

要利用Storm的可靠性功能，必须在创建元组树中的新边时告诉Storm，并在完成单个元组的处理后告诉Storm。这些是使用螺栓用来发出元组的OutputCollector对象完成的。锚固在emit方法，然后使用声明已完成一个元组ack方法。

所有这些在“ 保证消息处理”中都有更详细的说明。

任务

每个喷嘴或螺栓在整个集群中执行尽可能多的任务。每个任务对应一个执行线程，流分组定义如何将元组从一组任务发送到另一组任务。您可以为每个喷口或螺栓设置平行度setSpout和setBolt TopologyBuilder的方法。

工作人员

拓扑跨一个或多个工作进程执行。每个工作进程都是一个物理JVM，并执行拓扑所有任务的子集。例如，如果拓扑的组合并行度为300，并分配了50个工作程序，则每个工作程序将执行6个任务（作为工作程序中的线程）。Storm试图将任务平均分配给所有工人。
