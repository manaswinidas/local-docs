摘要

该模块不提供其自身的任何配置指令。它需要mod_proxy_balancer的服务，并提供byrequests负载平衡方法。

请求计数算法

通过lbmethod=byrequests启用，此调度程序背后的思想是我们将请求分配给各个工作人员，以确保每个工作人员都获得了请求数量的已配置份额。其工作方式如下：

lbfactor是我们期望这个工人工作多少 ，或者该工人的工作配额 。这是一个标准化的值，表示他们在工作量中的“份额”。

lbstatus是该工人必须多么紧急的工作以完成其工作任务 。

工作服务器是负载平衡器的成员，通常是服务于受支持协议之一的远程主机。

我们将每个工人的工作配额分配给该工人，然后查看其中哪个工人最需要紧急工作（最大lbstatus）。然后选择该工人进行工作，其lbstatus减去我们分配给所有工人的总工作配额。因此，所有lbstatus的总和不会改变（*），我们将根据需要分配请求。

如果某些工人被禁用，其他工人仍将被正确安排。

for each worker in workers
    worker lbstatus += worker lbfactor
    total factor    += worker lbfactor
    if worker lbstatus > candidate lbstatus
        candidate = worker

candidate lbstatus -= total factor

如果平衡器配置如下：

工人 一种 b C d
磅因子 25 25 25 25
磅状态 0 0 0 0

并且b被禁用，产生以下时间表：

工人 一种 b C d
磅状态 -50 0 25 25
磅状态 -25 0 -25 50
磅状态 0 0 0 0
（重复）

这就是时间表： a c d a c d a c d ...请注意：

工人 一种 b C d
磅因子 25 25 25 25

具有与以下行为完全相同的行为：

工人 一种 b C d
磅因子 1个 1个 1个 1个

这是因为所有lbfactor值都相对于其他值归一化。对于：

工人 一种 b C
磅因子 1个 4 1个

工人b的平均请求量是a和c的 4倍。

以下非对称配置可以正常工作：

工人 一种 b
磅因子 70 30
 
磅状态 -30 30
磅状态 40 -40
磅状态 10 -10
磅状态 -20 20
磅状态 -50 50
磅状态 20 -20
磅状态 -10 10
磅状态 -40 40
磅状态 30 -30
磅状态 0 0
（重复）
