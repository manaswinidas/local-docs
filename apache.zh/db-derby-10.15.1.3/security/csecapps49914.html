在不使用SSL / TLS的情况下配置网络服务器身份验证

如果不使用SSL / TLS加密网络通信，则可以使用属性在网络服务器端指定用户名和密码的加密。

重要提示：强烈建议在生产应用程序中使用SSL / TLS。仅在异常情况下使用这些属性。

在嵌入式模式下运行Derby或使用Derby Network Server时，可以启用或禁用服务器端用户身份验证。（强烈建议启用用户身份验证。）但是，当您使用网络服务器时，默认的安全机制（ CLEAR_TEXT_PASSWORD_SECURITY ）要求您同时提供用户名和密码。

除了默认的用户名和密码安全机制， org.apache.derby.jdbc.ClientDataSource.CLEAR_TEXT_PASSWORD_SECURITY ， Derby Network Server支持以下安全属性：

  • 用户身份： org.apache.derby.jdbc.ClientDataSource.USER_ONLY_SECURITY

    使用此机制时，必须仅指定user属性。所有其他机制都要求您同时指定用户名和密码。

  • 加密的用户ID和加密的密码： org.apache.derby.jdbc.ClientDataSource.ENCRYPTED_USER_AND_PASSWORD_SECURITY

    使用此机制时，密码和用户ID均被加密。

如果存在具有该名称的架构，则在连接时指定的用户名是该连接的默认架构。有关模式和用户名的更多信息，请参见《 Derby开发人员指南》 。

如果您指定任何其他安全机制，您将收到一个例外。

要更改默认值，您可以将其他安全机制指定为属性或URL（使用securityMechanism=value属性）。有关详细信息，请参阅《 Derby参考手册 》中的配置不使用SSL / TLS的网络客户端身份验证和“ securityMechanism = value属性”。

您为客户端指定的安全性机制是否真正生效取决于设置的derby.drda.securityMechanism网络服务器的属性。如果derby.drda.securityMechanism如果设置了属性，则网络服务器仅接受使用属性设置指定的安全性机制的连接。如果derby.drda.securityMechanism如果未设置属性，则客户端可以使用任何有效的安全机制。有关详细信息，请参阅《 Derby服务器和管理指南 》中的“ derby.drda.securityMechanism属性”。

在网络服务器上启用用户身份验证时的安全性机制选项

在Derby中启用用户身份验证后，可以使用以下两种安全机制之一。

  • 明文用户名和密码安全性，默认
  • 加密的用户名和密码安全

在网络服务器上禁用用户身份验证时的安全性机制选项

在Derby中关闭用户身份验证后，您可以使用任何安全性机制选项。

您必须为所有安全机制提供用户名和密码，但USER_ONLY_SECURITY 。但是，由于在Derby服务器中禁用了用户认证，因此您提供的用户名和密码不必在Derby识别为有效的用户名和密码中。

启用加密的用户ID和密码安全机制

要使用加密的用户ID和密码安全性机制，您需要具有JCE（Java密码学扩展）的Java环境，该环境支持256位公共素数的Diffie-Hellman算法。

Java平台标准版（Java SE）需要512位或更多的公共素数。

要在使用网络客户端的JDBC连接期间使用加密的用户ID和密码安全性机制，请指定securityMechanism=value连接属性。
注意：如果在网络服务器中启动了加密数据库，则用户可以连接到数据库而无需提供bootPassword 。与数据库的第一个连接必须提供bootPassword ，但所有后续连接都不需要提供它。要从加密数据库中删除访问权限，请使用shutdown=true关闭数据库的选项。有关更多信息，请参见配置数据库加密 。
