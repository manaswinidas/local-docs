在不使用SSL / TLS的情况下配置网络客户端身份验证

如果不使用SSL / TLS加密网络流量，则可以使用属性在客户端上指定用户名和密码的加密。

重要提示：强烈建议在生产应用程序中使用SSL / TLS。仅在异常情况下使用这些属性。

的securityMechanism=value属性指定Derby Network Client的安全性机制。有关此属性的详细信息，请参见《 Derby参考手册 》。

您可以设置securityMechanism通过以下方式之一设置属性：

  • 当您使用java.sql.DriverManager类，集securityMechanism=value在一个java.util.Properties对象，然后再调用DriverManager.getConnection包括java.util.Properties参数。
  • 当您使用ClientDataSource界面来创建和部署自己的DataSource对象，调用ClientDataSource.setSecurityMechanism创建ClientDataSource对象后的方法。

下表列出了Derby Network Client支持的安全机制，以及为获得此安全机制而指定的相应属性值。默认的安全机制是仅在未设置密码的情况下的用户ID。如果设置了密码，则默认的安全机制是用户标识和密码。如果未指定其他用户，则默认用户为APP。

表格1。Derby网络客户端支持的安全机制
安全机制 securityMechanism属性值 评论
用户名和密码 ClientDataSource.CLEAR_TEXT_PASSWORD_SECURITY （0x03） 如果设置了密码，则为默认值
仅用户ID ClientDataSource.USER_ONLY_SECURITY （0x04） 如果未设置密码，则为默认值
加密的用户标识和加密的密码 ClientDataSource.ENCRYPTED_USER_AND_PASSWORD_SECURITY （0x09） 加密需要一个JCE实现，该实现支持具有256位公共素数的Diffie-Hellman算法。

Derby提供了两种ClientDataSource实现。使用org.apache.derby.jdbc.ClientDataSource除Java SE 8 Compact Profile 2以外的所有受支持Java SE版本上的类。在Java SE 8 Compact Profile 2上，使用org.apache.derby.jdbc.BasicClientDataSource40类。
