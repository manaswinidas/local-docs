关于java.sql映射的注释。Blob和java.sql。Clob接口

常用的Derby锁定机制（共享锁）可防止其他事务更新或删除java.sql所在的数据库项目。Blob或java.sql。Clob对象是一个指针。

但是，在某些情况下， Derby的瞬时锁定机制可能会在一段时间内保留java.sql所在的列。Blob或java.sql。Clob未受保护。对getBlob / getClob或java.sql的后续调用。Blob / java.sql。Clob方法，可能导致未定义的行为。

此外，没有什么可以阻止持有java.sql的事务。Blob / java.sql。Clob （相对于另一个事务）不更新基础行。（ getXXXStream方法也存在相同的问题。）对应用程序进行编程，以防止在使用java.sql时更新基础对象。Blob / java.sql。Clob在上面打开；否则可能导致未定义的行为。

如果方法之一是下列之一，则不要在同一列上调用多个ResultSet getXXX方法：
  • getBlob
  • getClob
  • getAsciiStream
  • getBinaryStream
  • getCharacterStream
这些方法共享相同的基础流。在同一列上调用这些方法中的多个可能导致未定义的行为。例如：
    ResultSet rs = s.executeQuery("SELECT text FROM CLOBS WHERE i = 1");
    while (rs.next()) {
        aclob = rs.getClob(1);
        ip = rs.getAsciiStream(1);
    }

处理长列的流不是线程安全的。这意味着，如果用户选择打开多个线程并从每个线程访问流，则结果是不确定的。

Clob不区分语言环境。
