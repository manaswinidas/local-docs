介绍

请参阅《 MIT Kerberos教程》以获得有关Kerberos的良好介绍。
Windows指南以及此Wikipedia页面也值得检查。

设定

Unix系统

1。安装软件包

> sudo apt-get install krb5-kdc krb5-admin-server

在安装过程中，输入“ localhost”作为Kerberos服务器的主机名（除非您要输入更具体的主机名），并设置一个默认域，例如“ MYCOMPANY.COM”。请遵循此博客条目中的1.2步骤，以正确设置此默认领域。

2。创建主体

从此博客条目的步骤1.3开始：

2.1创建主密钥：
> sudo kdb5_util创建-s

2.2创建用户和服务主体

>须藤kadmin.local

其次是

> addprinc爱丽丝
> addprinc HTTP /本地主机

其中“ HTTP / localhost”是协商方案中使用的典型服务主体名称，如果需要，请替换为“ localhost”。
根据需要也添加更多的用户和服务主体。

3启动KDC

>须藤krb5kdc

4。创建可选的票证缓存

> klist

返回空响应

> kinit爱丽丝

> klist

确认“ alice”的TGT在缓存中。

2.4创建密钥表

当密钥表可用时，不必在登录配置中指定主体密码。
请按照此博客条目中的步骤1.4进行操作。

请注意，创建密钥表实际上会重置原始的主体密码，例如，在为“ alice”创建密钥表后，一个人将无法使用原始密码（TODO：显然可以恢复-了解如何）。因此，如果您想尝试使用密钥表，那么您可能希望创建很少的用户和服务主体，仅使用密钥表选择选定的主体。

客户端配置

HTTP管道

请参阅此页面以获取有关Spnego / Kerberos HTTPConduit客户端支持的信息。

拦截器

org.apache.cxf.jaxrs.security。KerberosAuthOutInterceptor可以用作配置HTTPConduit的替代方法。

KerberosAuthOutInterceptor和HTTPConduit Spnego处理程序共享相同的基本代码。在许多情况下，具有HTTPConduit配置就足够了
特别是在管道级别也要设置SSL时。在测试以及设置很少的额外属性时，使用拦截器会很方便，这在通用HTTP管道授权策略级别很难设置。

拦截器属性将在以下小节中介绍

授权政策

如本页所述 ，授权策略通常需要将其类型设置为“协商”，并将其“授权”属性设置为JAAS上下文的名称。AuthorizationPolicy在拦截器上设置为“ policy”属性，例如：

WebClient wc = WebClient.create("http://localhost:" + PORT + "/bookstore/books/123");
        
KerberosAuthOutInterceptor kbInterceptor = new KerberosAuthOutInterceptor();
        
AuthorizationPolicy policy = new AuthorizationPolicy();
policy.setAuthorizationType(HttpAuthHeader.AUTH_TYPE_NEGOTIATE);
policy.setAuthorization("KerberosClientKeyTab");
        
kbInterceptor.setPolicy(policy);
WebClient.getConfig(wc).getOutInterceptors().add(kbInterceptor);
        
Book b = wc.get(Book.class);

在此示例中，使用了KerberosClientKeyTab策略，该策略链接到可用的密钥表。否则，很可能也必须设置AuthorizationPolicy的'UserName'和'Password'属性（Windows上可能存在例外）

配置服务主体名称

服务主体标识目标服务。

默认情况下，服务主体名称是通过将“ HTTP”，“ /”和目标主机的名称连接在一起来计算的，例如，在“ http：// localhost：8080 / services”上调用时，服务主体名称设置为“ HTTP /本地主机”。

可以使用“ servicePrincipalName”和“ realm”属性来自定义它，例如，将“ servicePrincipalName”设置为“ HTTP / www.mycompany.com”，将领域设置为“ services.org”将导致“ HTTP / www.mycompany” “ .com @ services.org”服务主体名称。

如果未指定“ servicePrincipalName”，则使用提供的端点URL中的目标主机按原样构造一个主机。要对此主机名进行规范化（例如，如果CNAME记录host.example.com指向A记录host-x.example.com ，则在构造servicePrincipalName时使用“ host-x.example.com ”），即“ useCanonicalHostname” ”属性可以设置为“ true”。

使用JAAS配置

HTTPConduit和拦截器处理程序都需要设置“ java.security.auth.login.config”系统属性。该属性需要指向包含特定Kerberos登录模块配置的文件。

与其设置此系统属性和维护配置文件，不如使用javax.security.auth.login的实现。配置并在拦截器上将其设置为“ loginConfig”属性。

如何避免设置用户名和密码属性

通常，可能必须为Kerberos登录模块设置AuthorizationPolicy UserName和Password属性，以对用户进行身份验证。

下一个选项是按照“设置”部分所述创建一个密钥表，这样可以避免指定密码属性。
最后，如果用户实际上拥有运行代码的Java进程，则无需提供用户名和密码属性，并假设Kerberos登录配置的“ useTicketCache”和“ renewTGT”属性均设置为“ true”

服务器配置

org.apache.cxf.jaxrs.security。KerberosAuthenticationFilter可用于保护JAX-RS端点并强制客户端使用协商身份验证方案，例如：

<bean id="kerberosFilter" class="org.apache.cxf.jaxrs.security.KerberosAuthenticationFilter">
   <property name="loginContextName" value="KerberosServiceKeyTab"/>
</bean>

<jaxrs:server>
  <jaxrs:serviceBeans>
    <bean class="org.mycompany.MyCompanyResource"/>
  </jaxrs:serviceBeans>
  <jaxrs:providers>
    <ref bean="kerberosFilter"/>
  </jaxrs:providers>
</jaxrs:server>

如果身份验证成功，则KerberosAuthenticationFilter将在当前消息上设置CXF SecurityContext 。此SecurityContext将返回KerberosAuthenticationFilter $ KerberosPrincipal的实例，此Principal将返回“简单”和“ kerberos”源主体名称，例如，给定“ HTTP/localhost@myrealm.com”，Principal＃getName将返回“ HTTP / localhost” ，并且KerberosPrincipal＃getKerberosName将返回“ HTTP/localhost@myrealm.com”。

服务主体名称和JAAS配置

可以选择使用“ servicePrincipalName” +“ realm”和“ loginConfig”属性使用KerberosAuthOutInterceptor设置服务主体名称和JAAS配置。

回调处理程序

javax.security.auth.callback。如果不使用Kerberos密钥选项卡，则需要注册CallbackHandler，这是从Java进行设置的示例：

public class TestResource {
 public static void main(String[] args) {
   JAXRSServerFactoryBean sf = new JAXRSServerFactoryBean();
   sf.setResourceClasses(BookStore.class);
   KerberosAuthenticationFilter filter = new KerberosAuthenticationFilter();
   filter.setLoginContextName("KerberosServer");
   
   CallbackHandler handler = 
     new org.apache.cxf.interceptor.security.NamePasswordCallbackHandler("HTTP/localhost", "http"); 
   filter.setCallbackHandler(handler);

   //filter.setLoginContextName("KerberosServerKeyTab");
   //filter.setServicePrincipalName("HTTP/ktab");
   sf.setProvider(filter);
   sf.setAddress("http://localhost:" + PORT + "/");
      
   sf.create();
 }
}

在此示例中，使用了KerberosServer策略。

凭证委派

请参阅本节 ，了解如何在HTTP管道级别启用和实现客户端凭据委派。

请注意，如果您有一个保护端点的JAX-RS KerberosAuthenticationFilter，那么该过滤器将具有org.ietf.jgss。通过其KerberosAuthenticationFilter $ KerberosSecurityContext实现在当前CXF SecurityContext中可用的GSSContext实例，该实例可用于获取org.ietf.jgss。GSSCredential（如果给定的源主体支持凭据委派）。接下来可以将当前凭据（如果有）设置为客户端属性，例如：

import org.ietf.jgss.GSSCredential;

import org.apache.cxf.jaxrs.security.KerberosAuthenticationFilter;
import org.apache.cxf.jaxrs.security.KerberosAuthenticationFilter.KerberosSecurityContext;
import org.apache.cxf.phase.PhaseInterceptorChain;
import org.apache.cxf.security.SecurityContext;

@Path("service")
public class MyResource {

   @GET
   public Book getBookFromKerberosProtectedStore() {
       WebClient wc = webClient.create("http://internal.com/store");
       SecurityContext securityContext = PhaseInterceptorChain.getCurrentMessage().get(SecurityContext.class);

       if (securityContext instanceof KerberosSecurityContext) {
           KerberosSecurityContext ksc = (KerberosSecurityContext)securityContext;
           GSSCredential cred = ksc.getGSSContext().getDelegCred();
           if (cred != null) {
               WebClient.getConfig(wc).getRequestContext().put(GSSCredential.class.getName(), cred);
           } 
       }
       return wc.get(Book.class); 
   }

}

HTTPConduit或KerberosAuthOutInterceptor处理程序将使用可用的GSSCredential。

还要注意，如果在最初启用委托时使用KerberosAuthOutInterceptor而不是客户端上的HTTPConduit，则可以将其“ credDelegation”属性设置为“ true”。
