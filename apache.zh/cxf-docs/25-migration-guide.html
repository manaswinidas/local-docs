新的功能

  • WS-RM正在更新以支持1.1。还包括一些功能改进，例如WS-RM客户端进行的序列划分。
  • WS-RM组件已完全通过JMX进行检测。可以通过JMX监视有关WS-RM处理的各种状态信息，例如序列，重试，确认。
  • WS-MetadataExchange支持服务器端。CXF服务现在将正确响应WS-MEX请求。
  • 配置以配置CXF使用的工作队列。
  • 卡拉夫命令
    • cxf：list-buses
    • cxf：list-endpoints [busid]
    • cxf：stop-endpoint busid端点名称
    • cxf：start-endpoint busid端点名称
  • 新的WS-Notification服务和API（请参阅ws_notification示例）
  • 新的安全令牌服务（请参阅sts示例）
  • 对JAX-RS的初始OAuth支持（请参阅oauth示例）
  • 对JAX-RS的初始XML安全性和SAML支持。

卸下的模块

  • 所有与JBI相关的模块都已从CXF中删除。如果将CXF与JBI一起使用，建议使用ServiceMix CXF JBI组件。过去CXF版本所需的那些模块的所有代码都将移入那些ServiceMix组件。

API变更

  • 现在，用于选择要使用的策略替代方案的AlternativeSelector.selectAlternative方法需要一个新的List >参数，该参数在服务器端是传入请求匹配的替代方案的列表。AlternativeSelector可以使用它来帮助选择要使用的最佳响应替代方案。
  • 现在，PolicyEngine.getEffectiveServerResponsePolicy会采用匹配的传入策略替代方案的附加参数。对于先前的行为，它可以为null。

运行时更改

  • 如果MBeanServer在Spring上下文中可用或作为OSGi服务器可用（在OSGi中运行），则InstrumentationManger将自动启用并将使用该MBeanServer和CXF MBeans将被注册。
  • 现在，在OSGi中运行时，用于Bus MBean服务器的默认总线ID包含捆绑包上下文SymbolicName，以使识别适当的Bus更加容易。
  • 现在，许多可能导致安全检查的调用通过AccessControll.doPrivileged调用进行包装，这些调用可以允许对CXF所需的权限进行更精细的控制。参见https://issues.apache.org/jira/browse/CXF-2683 。
  • 空的部分响应消息将作为不带内容的HTTP 202响应返回，而不是带有空的SOAP正文消息的HTTP 200。
  • 现在，日志记录拦截器使用服务特定的类别/记录器记录日志，而不仅仅是LoggingInInterceptor / LoggingOutInterceptor。所使用的记录器的名称为org.apache.cxf.services。服务名称。端口名。PortTypeName。这允许用户在其日志记录配置中配置特定的每个服务筛选器和格式化程序。
