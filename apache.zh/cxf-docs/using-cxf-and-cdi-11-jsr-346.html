介绍

JAX-RS 2.0规范（JSR-339）要求对CDI 1.1（JSR-346）的支持，而从版本3.0开始的Apache CXF引入了对该功能的最初支持。首先，重点是支持将嵌入式Jety 8/9和Tomcat 7/8容器作为主要部署（尽管将来将支持其他应用程序服务器）。

建筑与设计

目前，Apache CXF和CDI的集成围绕两个关键组件展开，这两个组件位于名为cxf-integration-cdi的新模块中。

  • CXFCdiServlet servlet
  • JAXRSCdiResourceExtension便携式CDI扩展

包括cxf-integration- cdi作为依赖项的事实允许CDI容器发现JAXRSCdiResourceExtension便携式CDI扩展。JAXRSCdiResourceExtension创建总线的实例并将其注册到BeanManager 。从这一点来说， 总线实例是一个普通的CDI豆（用@ 适用范围）可用于注入。一旦由servlet容器初始化了总线，就将其插入CXFCdiServlet servlet中。

根据设计， JAXRSCdiResourceExtension可能使用基于零的配置方法或依赖于特定的JAX-RS 应用程序实例。org.apache.cxf.cdi。还应该配置CXFCdiServlet （下面的程序化和WAR方案更多示例）。

从零开始的配置

如果在Apache CXF申请包含为了要么没有JAX-RS 的应用的实例或仅一个JAX-RS 的应用 （带@ApplicationPath注解），没有单身和类定义，下面的规则正在由JAXRSCdiResourceExtension施加的单个实例来配置和发布配置的JAX-RS资源：

  • 正在创建JAX-RS 应用程序实例（用@ApplicationPath注释）并向BeanManager注册
  • 正在创建发现的JAX-RS 提供程序的所有实例（使用@Provider注释）并向BeanManager注册
  • 正在创建发现的JAX-RS 资源的所有实例（用@Path注释）并向BeanManager注册

最后，将使用之前发现的所有服务Bean和提供程序来创建和配置JAXRSServerFactoryBean的实例。此外，提供程序还丰富了javax.ws.rs.ext的服务。MessageBodyReader和javax.ws.rs.ext。通过ServiceLoader加载的MessageBodyWriter 。从这一刻起，Apache CXF应用程序已准备就绪，可以处理请求。快速示例如下所示。

空的JAX-RS 应用程序 ：

@ApplicationPath("/api")
public class BookStoreApplication extends Application {
}

还有一个JAX-RS资源：

@Path("/bookstore/")
public class BookStore {
    @Inject private BookStoreService service;
    
    @GET
    @Path("/books/{bookId}")
    @Produces(MediaType.APPLICATION_JSON)
    public Book getBook(@PathParam("bookId") String id) {
        return service.get(id);
    }
}

定制配置

如果Apache CXF应用程序包含一个或多个定义了单例或类的JAX-RS 应用程序实例（用@ApplicationPath注释），则JAXRSCdiResourceExtension将应用以下规则以配置和发布已配置的JAX-RS资源：

  • 正在创建每个JAX-RS 应用程序的实例（用@ApplicationPath注释）并向BeanManager注册
  • 对于之前创建的每个JAX-RS Application实例，将以以下方式创建和配置JAXRSServerFactoryBean实例：将应用程序的单例/类拆分为提供程序（用@Provider注释），服务bean（用@Path注释）和功能。 （ org.apache.cxf.feature的实现。功能 ）

从这一刻起，Apache CXF应用程序已准备就绪，可以处理请求。快速示例如下所示。

配置的JAX-RS 应用程序 ：

@ApplicationPath("/custom")
public class BookStoreCustomApplication extends Application {
    @Inject private BookStore bookStore;
    
    @Override
    public Set< Object > getSingletons() {
        return Sets.< Object >newHashSet(
            bookStore, 
            new JacksonJsonProvider(),
            new ValidationExceptionMapper(),
            new JAXRSBeanValidationFeature());
    }
}

还有一个JAX-RS资源：

@Path("/bookstore/")
public class BookStore {
    @Inject private BookStoreService service;
    
    @GET
    @Path("/books/{bookId}")
    @Produces(MediaType.APPLICATION_JSON)
    public Book getBook(@PathParam("bookId") String id) {
        return service.get(id);
    }
}

其他配置功能

如果你有需要，你可以通过实现界面JAXRSServerFactoryCustomizationExtension配置潜在JAXRSServerFactoryBean。可以通过as CDI bean的ServiceLoader定位此扩展的实例。您可以使用它来以编程方式添加新的提供程序，或者在创建服务器之前操纵运行时。在SseTransportCustomizationExtension上可以看到如何使用此方法的示例。

用于JAX-RS上下文对象的CDi生命周期

CDI扩展还支持@Context对象的注入（通过CDI）。可以通过CDI注入任何已知对象。使用CDI扩展名并将@Context声明为资源类上的字段时，会自动发生这种情况。请注意，这的确允许您将@Context对象也注入非JAX-RS组件中。逻辑分为两部分：

  1. 重写注入点
    1. 当发现包含@Context的注入点时，将添加两个附加注释
    2. 这些注释是@Inject和@ContextResolved
    3. @Inject的使用将强制CDI运行时接管Bean值的注入，而@ContextResolved只是我们应用于注入点以避免歧义的限定符
  2. 提供豆
    1. 对于每种内置的@Context类型，我们注册一个bean，该bean提供一个@RequestScoped bean，通过查找内部上下文值来解析该值。
    2. 另外，用户或集成者可以实现ContextClassProvider来将其他类注册为要解析的上下文类型。

 

使用嵌入式Jetty 8/9进行部署（编程配置）

使用Jetty 8/9 ，可以创建完全可嵌入的REST / JAX-RS服务器，而无需使用web.xml或WAR文件。对于使用CDI 1.1的Apache CXF应用程序， 应将CXFCdiServlet servlet用作起点。以下示例演示了创建嵌入式Jetty 8/9实例所需的配置点。作为CDI 1.1的实现，正在使用JBoss Weld 2.0 。

System.setProperty("java.naming.factory.url", "org.eclipse.jetty.jndi");
System.setProperty("java.naming.factory.initial", "org.eclipse.jetty.jndi.InitialContextFactory");

// Register and map the dispatcher servlet
final Server server = new Server(<port>);
final ServletHolder servletHolder = new ServletHolder(new CXFCdiServlet());
final ServletContextHandler context = new ServletContextHandler();         
context.setContextPath(<context path>);          
context.addEventListener(new Listener());         
context.addEventListener(new BeanManagerResourceBindingListener());
context.addServlet(servletHolder, "/rest/*");
server.setHandler(context);
server.start();

此代码段足以触发CDI便携式扩展发现，执行配置（取决于存在的JAX-RS 应用程序 ）并将所有已定义的依赖项连接在一起。

使用嵌入式Jetty 8/9进行部署（基于WAR的部署）

部署具有CDI 1.1支持和嵌入式Jetty 8/9的 Apache CXF应用程序的另一种方法是使用web.xml描述符和类似WAR的部署结构。在这种情况下，Apache CXF应用程序需要声明CXFCdiServlet servlet（及其映射），如果需要，还需要声明CDI特定的侦听器（在下面的示例中， JBoss Weld 2.0被用作CDI 1.1容器）。

<web-app version="3.0" xmlns="http://java.sun.com/xml/ns/javaee"
    xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance">

    <listener>
        <listener-class>org.jboss.weld.environment.servlet.Listener</listener-class>
    </listener>

    <servlet>
        <servlet-name>CXFServlet</servlet-name>
        <display-name>CXF Servlet</display-name>
        <servlet-class>org.apache.cxf.cdi.CXFCdiServlet</servlet-class>    
        <load-on-startup>1</load-on-startup>
    </servlet>

    <servlet-mapping>
        <servlet-name>CXFServlet</servlet-name>
        <url-pattern>/rest/*</url-pattern>
    </servlet-mapping>

    <resource-env-ref>
        <resource-env-ref-name>BeanManager</resource-env-ref-name>
        <resource-env-ref-type>javax.enterprise.inject.spi.BeanManager
        </resource-env-ref-type>
    </resource-env-ref>
</web-app>

在这种情况下，服务器初始化看起来更简单。

System.setProperty("java.naming.factory.url", "org.eclipse.jetty.jndi");
System.setProperty("java.naming.factory.initial", "org.eclipse.jetty.jndi.InitialContextFactory");

final Server server = new Server(<port>);
final WebAppContext context = new WebAppContext();
context.setContextPath(<context path>);
context.setWar(<path to WAR folder/file>);
context.setServerClasses(new String[] { "org.eclipse.jetty.servlet.ServletContextHandler.Decorator" });
        
HandlerCollection handlers = new HandlerCollection();
handlers.setHandlers(new Handler[] {context, new DefaultHandler()});
server.setHandler(handlers);
server.start();

请注意，使用特定于Jetty的服务器类（“ org.eclipse.jetty.servlet。ServletContextHandler。装饰器”）对于允许CDI 1.1注入（由JBoss Weld 2.0支持）在Servlet /侦听器/过滤器之间无缝工作非常重要。对于Apache CXF来说，这并不是绝对必要的（一切都会按预期进行），但是复杂的应用程序肯定会从中受益。

使用嵌入式Tomcat 7/8进行部署（基于WAR的部署）

如果是嵌入式Tomcat 7/8，则可以使用web.xml描述符和类似WAR的部署结构来部署具有CDI 1.1支持的Apache CXF应用程序，这与基于Jetty 8/9 WAR的部署类似。Apache CXF应用程序需要声明CXFCdiServlet servlet（及其映射），如果需要，还需要声明CDI特定的侦听器（在下面的示例中， JBoss Weld 2.0被用作CDI 1.1容器）。

<web-app version="3.0" xmlns="http://java.sun.com/xml/ns/javaee"
    xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance">

    <listener>
        <listener-class>org.jboss.weld.environment.servlet.Listener</listener-class>
    </listener>

    <servlet>
        <servlet-name>CXFServlet</servlet-name>
        <display-name>CXF Servlet</display-name>
        <servlet-class>org.apache.cxf.cdi.CXFCdiServlet</servlet-class>    
        <load-on-startup>1</load-on-startup>
    </servlet>

    <servlet-mapping>
        <servlet-name>CXFServlet</servlet-name>
        <url-pattern>/rest/*</url-pattern>
    </servlet-mapping>

    <resource-env-ref>
        <resource-env-ref-name>BeanManager</resource-env-ref-name>
        <resource-env-ref-type>javax.enterprise.inject.spi.BeanManager
        </resource-env-ref-type>
    </resource-env-ref>
</web-app>

Tomcat 7/8服务器采用了非常简单的初始化过程，具有不同的API。

final Tomcat server = new Tomcat();
server.setPort(<port>);

final File base = createTemporaryDirectory();
server.setBaseDir(base.getAbsolutePath());    
    
server.getHost().setAppBase(base.getAbsolutePath());
server.getHost().setAutoDeploy(true);
server.getHost().setDeployOnStartup(true);
            
server.addWebapp(<context path>, <path to WAR folder/file>);   
server.start();

未来的工作

Apache CXF团队致力于改善CDI集成，并涵盖更多的部署方案以及广泛的应用程序配置和需求。但是，总体而言，CDI集成与实现无关，并且可以在JBoss WildFly或其他EE容器中使用。
