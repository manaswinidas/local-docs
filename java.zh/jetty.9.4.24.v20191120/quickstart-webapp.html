Servlet规范的自动发现功能会使部署缓慢且不确定。Web应用程序配置的自动发现在开发Webapp时可能很有用，因为它允许通过简单地放入jar文件来启用新功能和框架。但是，对于部署而言，需要扫描许多jar的内容可能会对Webapp的启动时间产生重大影响。

在Jetty 9.2发行版中，包含了一个快速启动模块，该模块允许对webapp进行预扫描和预配置。这意味着所有扫描都在部署之前完成，并且所有配置都被编码为有效的web.xml ，称为WEB-INF/quickstart-web.xml ，可以在部署前对其进行检查以了解将要部署的内容。不仅quickstart-web.xml包含所有发现的Servlet，过滤器和约束，但也将所有发现的编码为上下文参数：

  • ServletContainerInitializers
  • HandlesTypes类
  • Taglib描述符

借助快速启动机制，Jetty能够完全绕开所有扫描和发现模式，并以可预测的快速方式启动Web应用程序。测试表明，花费了数秒钟来扫描和部署的Web应用程序现在可以在几百毫秒内完成部署。

设置快速入门

先决条件

码头分布

在标准Jetty发行版中，可以使用以下命令配置快速入门模块：

$ java -jar $JETTY_HOME/start.jar --add-to-start=quickstart
嵌入式的

在Maven项目中，您添加了对工件的依赖jetty-quickstart 。

<dependency>
    <groupId>org.eclipse.jetty</groupId>
    <artifactId>jetty-quickstart</artifactId>
    <version>9.4.24.v20191120</version>
</dependency>

组态

Webapp必须是org.eclipse.jetty.quickstart.QuickStartWebApp而不是正常的org.eclipse.jetty.webapp.WebAppContext 。

org.eclipse.jetty.quickstart.QuickStartWebApp实例提供与熟悉的设置器相同的设置器org.eclipse.jetty.webapp.WebAppContext ，此外：

自动预配置
（真假）。如果为true，则首次运行Web应用程序时，将在部署Web应用程序之前生成WEB-INF / quickstart-web.xml。后续运行使用先前生成的快速入门文件。
originAttribute
要插入到生成的元素的属性名称，该属性将在quickstart-web.xml中给出元素的来源。默认情况下是origin 。
generateOrigin
（真假）。默认情况下是false 。如果为true，则origin属性将插入quickstart-web.xml中的每个元素中。请注意，如果启用了调试日志级别，也会生成origin属性。

注意

如果您使用的是Spring-Boot，则必须设置generateOrigin真实。

源是描述符（例如，web.xml，web-fragment.xml，override-web.xml文件）或注释（例如，@ WebServlet）。对于xml验证，每个属性必须唯一，因此，将整数计数器附加到每个值。具有原始属性信息的元素的一些示例是：

<listener origin="DefaultsDescriptor(file:///path/to/distro/etc/webdefault.xml):21">
<listener origin="WebDescriptor(file:///path/to/base/webapps/test-spec/WEB-INF/web.xml):22">
<servlet-class origin="FragmentDescriptor(jar:file:///path/to/base/webapps/test-spec/WEB-INF/lib/test-web-fragment.jar!/META-INF/web-fragment.xml):23">
<servlet-class origin="@WebServlet(com.acme.test.TestServlet):24">
在XML中

如果Web应用程序已经具有上下文xml文件，例如webapps/myapp.xml文件，只需在Configure元件。否则，使用以下信息创建上下文XML文件（除了contextPath，war等的常规设置之外）：

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE Configure PUBLIC "-//Jetty//Configure//EN" "http://www.eclipse.org/jetty/configure_9_3.dtd">
<Configure class="org.eclipse.jetty.quickstart.QuickStartWebApp">
  <Set name="autoPreconfigure">true</Set>
</Configure>
在代码中

创建一个实例org.eclipse.jetty.quickstart.QuickStartWebApp而不是正常的org.eclipse.jetty.webapp.WebAppContext 。然后，以与WebAppContext完全相同的方式使用QuickStartWebApp实例。

这是一个片段：

    QuickStartWebApp webapp = new QuickStartWebApp();
    webapp.setAutoPreconfigure(true);
预先生成quickstart-web.xml文件

而不是使用autoPreconfigure QuickStartWebApp的功能-懒惰地生成quickstart-web.xml文件-您可以通过将其作为主要类别来积极为现有战争预先生成文件org.eclipse.jetty.quickstart.PreconfigureQuickStartWar 。请注意，您将需要在命令行类路径上提供所有必要的码头罐子。如有必要，这将打开战争包装，并创建quickstart-web.xml第一次部署之前：

$ java -cp [jetty classpath] org.eclipse.jetty.quickstart.PreconfigureQuickStartWar myapp.war

运行不带任何参数的类以查看其他运行时选项。

或者，您可以使用Jetty Maven插件目标jetty:effective-web-xml ：这将生成快速入门信息，但将其打印到stderr。该目标提供了一个配置选项，可将输出保存到文件中，然后您可以将其复制到Webapp的WEB-INF目录中。请注意，由于Jetty Maven插件是运行Web应用程序的通用工具，因此它的类路径上的jar可能比应用程序所需的jar还要多，因此可能会生成额外的快速入门信息：我们建议您仅将此目标用作快速指南。快速入门生成的信息类型。

避免使用预编译的JSP进行TLD扫描

当然，预编译JSP是改善Web应用程序启动时间的绝佳方法。从Jetty 9.2开始，已使用Apache Jasper JSP实现，并对其进行了增强，以允许跳过TLD扫描。这可以通过添加一个context-param到web.xml文件（这是由Jetty Maven JSPC插件自动完成的）：

<context-param>
  <param-name>org.eclipse.jetty.jsp.precompiled</param-name>
  <param-value>true</param-value>
</context-param>

绕过start.jar

码头start.jar机制是一种非常强大且灵活的机制classpath并执行以Jetty XML格式编码的配置。但是，此机制确实需要一些时间来构建classpath 。可以使用以下命令绕过start.jar机制–dry-run用于生成并重用完整命令行以在以后启动Jetty的选项：

$ RUN=$(java -jar $JETTY_HOME/start.jar --dry-run)
$ eval $RUN

注意--dry-run可以在temp目录中创建属性文件，并将其包含在生成的命令行中。如果是这样，则应获取临时属性文件的副本，并使用新的持久位置更新命令行。
