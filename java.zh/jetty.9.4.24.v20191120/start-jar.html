使用start.jar

执行start.jar

带阴影的Start.jar

启动Jetty独立服务器的最基本方法是执行start.jar ，这是使用所需配置启动Jetty的引导程序。

[jetty-distribution-9.4.24.v20191120]$ java -jar start.jar
2013-09-23 11:27:06.654:INFO:oejs.Server:main: jetty-9.4.24.v20191120
...

Jetty是高度模块化的Web服务器容器。强制性和必需性很少，大多数组件是可选的。您可以根据环境的需要启用或禁用它们。

最基本的是，您可以从两个元素配置Jetty：

  1. 组成服务器类路径的一组库和目录。
  2. 一组Jetty XML配置文件（IoC样式），用于确定如何构建Jetty服务器及其组件。

Jetty 9.1并没有直接编辑它们，而是引入了有关如何配置Jetty的更多选项（这些只是语法糖，最终可以分解为两个基本配置组件）。

Jetty启动功能包括：

  • Jetty发行二进制文件的分离${jetty.home}以及特定于环境的配置（和二进制文件） ${jetty.base} （有关管理Jetty基地和Jetty家庭的详细信息。）
  • 您可以通过新引入的模块系统启用一组库和XML配置文件。
  • 现在，Jetty附带的所有预构建XML配置文件都可以使用您可以在${jetty.base}/start.ini （在“ 快速入门配置”中演示）。

这些强大的新功能可支持各种样式的Jetty配置，从基于属性的简单配置，到服务器上的多个安装处理，再到Jetty之上的定制技术堆栈，甚至是经典的定制XML配置，老。

例如，如果您使用${jetty.base}正确地使用概念，您可以升级Jetty发行版本，而不必重新构建对Jetty的整个修改树。只需将您对${jetty.base} ，以后再升级${jetty.home}具有新Jetty发行版的目录。

执行start.jar

执行时start.jar执行以下操作：

  • 加载并解析在中找到的所有INI ${jetty.base}/start.d/*.ini作为命令行参数。

  • 加载并解析${jetty.base}/start.ini作为命令行参数。

    • 有关两者之间的区别，请参阅Start.ini与Start.d 。
  • 解析用于执行的实际命令行参数start.jar本身。

  • 使用基本解析步骤和原始解析步骤来解析所有XML配置文件，模块和库：

    1. 检查文件是否存在作为相对引用${jetty.base}.
    2. 检查文件是否存在作为相对引用${jetty.home}.
    3. 使用以下行为java.io.File （关系到System.getProperty （“ user.dir”），然后作为绝对文件系统路径）。
  • 加载任何从属模块（将XXNK，库和属性结果与活动命令行合并）。
  • 构建服务器类路径。

  • 确定运行模式：

    • 显示参考命令行选项并退出。
    • 正常执行Jetty，等待Jetty停止。
    • 执行派生的JVM以运行Jetty，等待派生的JVM退出。

start.jar命令行选项

命令行选项

- 救命
获取当前命令行选项列表和一些基本用法帮助。
- 版
显示服务器类路径条目的列表，并打印为每个条目找到的版本信息。
--list-classpath
与--version类似，显示服务器类路径。
--list-config

列出将启动Jetty的已解决配置。

  • Java环境
  • Jetty环境
  • JVM参数
  • 性质
  • 服务器类路径
  • 服务器XML配置文件
-空运行
打印已解析的命令行start.jar应该用于启动Jetty的派生实例。
-执行
启动Jetty的派生实例。
--exec-properties =
为用于将属性传输到子流程的文件分配一个固定的名称。这样可以保存并重用生成的属性文件。如果没有此选项，将使用一个临时文件。
--commands =
指示start.jar使用指定文件的每一行作为命令行上的参数。

调试并开始记录

-调试

启用启动过程的调试输出。

注意 ：这不会为Jetty本身设置调试日志记录。有关日志记录的信息，请参阅“ 配置Jetty记录 ”一节。 ]

--start-log-file =
将所有启动输出发送到指定的文件名。文件名是相对于${jetty.base} 。这对于捕获由于可能的启动配置错误而尚未启动特定于Jetty的记录器的启动问题很有用。

模块管理

--list-modules
列出系统定义的所有模块。使用常规查找模块文件${jetty.base}和${jetty.home}解析逻辑 。还根据命令行上显示的信息以及所有活动的启动INI文件列出启用状态。
--list-modules = （， ）*
按标签列出模块。对所有标签使用* 。在标签前面加上-以排除标签。除非明确包含特殊标签“内部”，否则始终将其排除。
--list-all-modules
列出所有模块。
--module = ，（ ）*
按名称启用一个或多个模块（使用--list-modules查看可用模块列表）。这也使模块系统中的所有传递（相关）模块成为可能。如果从shell命令行使用此命令，则将其视为临时效果，对于测试场景很有用。如果您希望始终启用此模块，请将此命令添加到${jetty.base}/start.ini.
--add-to-start = ，（ ）*

通过向以下行添加行来启用模块${jetty.base}/start.ini文件。添加的行由模块定义的INI模板提供。注意：还附带了传递模块。如果模块包含具有属性的.ini模板，则在激活模块时也可以编辑这些属性。为此，只需在属性之后列出属性及其值-add-to-start命令，例如以下示例：

$ java -jar start.jar --add-to-start=http jetty.http.port=8379 jetty.http.host=1.2.3.4

这样做将取消注释关联的.ini文件中的属性并将其设置为指定的值。

--update-ini

用于更新指定的属性或现有.ini文件中存在的属性。Jetty扫描命令行， ${jetty.base}和${jetty.home}具有指定属性的.ini文件，并进行相应更新。

$ java -jar ../start.jar --update-ini jetty.http.port=8417
ConfigSource <command-line>
ConfigSource ${jetty.base}
INFO  : http            property updated jetty.http.port=8417
INFO  : http            updated ${jetty.base}/start.d/http.ini
ConfigSource ${jetty.home}
--create-startd
创建一个${jetty.base}/start.d/目录。如果一个${jetty.base}/start.ini文件已存在，已复制到${jetty.base}/start.d目录。

注意

关于start.ini和start.d/*.ini文件，则应仅实现这些方法之一 。混合一个start.ini与模块中特定的ini文件{$jetty.base}/start.d除非格外小心，否则目录可能会导致服务器问题。有关更多信息，请参见Start.ini与Start.d 。

--write-module-graph =

先进的功能：创建模块图的graphviz 点文件 （存在于活动图中） ${jetty.base} 。

# generate module.dot
$ java -jar start.jar --module=websocket --write-module-graph=modules.dot

# post process to a PNG file
$ dot -Tpng -o modules.png modules.dot

有关如何将此dotty文件后处理为最适合您的输出的详细信息，请参见graphviz.org 。

-创建文件
创建初始化模块所需的所有丢失的文件。如果模块提供URL，则可以从网络下载文件。
--skip-file-validation = （，< /span=""><>
禁用[文件]部分中内容的验证${jetty.base}特定模块的目录。对于具有可下载内容的模块非常有用，这些模块已被替代内容覆盖${jetty.base}`目录。

警告

此高级选项适用于完全了解其配置的管理员${jetty.base}并且愿意放弃Jetty启动机制中内置的一些安全检查。

-批准所有许可
批准所有许可证问题。对于从不需要用户交互的脚本启用模块很有用。

启动/关闭命令行

- 停

向正在运行的Jetty实例发送停止信号。

注意：必须已使用各种停止属性启动服务器，此服务器才能起作用。

STOP.PORT =

用于停止正在运行的Jetty服务器的端口。这是在本地主机上打开的内部端口，仅用于停止正在运行的Jetty服务器。选择您不用于提供网络流量的端口。

需要--stop发挥作用。

STOP.KEY =

定义的密码短语来停止服务器。

需要--stop发挥作用。

STOP.WAIT =

等待确认正在运行的Jetty服务器已停止的时间（以秒为单位）。如果未指定，则停止器将无限期等待服务器停止。

如果经过了指定的时间，但未确认服务器已停止，则--stop命令以非零返回码退出。

您可以为Jetty配置端口号，以侦听stop命令，因此您可以从其他终端将其停止。这要求使用“秘密”密钥，以防止恶意或意外终止。使用STOP.PORT和STOP.KEY （要么-DSTOP.PORT=和-DSTOP.KEY= ，分别将其设置为系统参数）参数作为start.jar ：

> java -jar ${JETTY_HOME}/start.jar STOP.PORT=1234 STOP.KEY=secretpassword

然后，要从其他终端停止Jetty，您需要提供此端口和密钥信息。您可以使用Jetty发行版的副本， jetty-maven-plugin ， jetty-ant插件或自定义类来完成此操作。充分利用Jetty发行版的方法start.jar ，以执行停止操作：

> java -jar start.jar STOP.PORT=8181 STOP.KEY=abc123 --stop

注意

要正常关闭Jetty， stats 必须启用模块 。

高级命令

--lib =
将任意类路径条目添加到服务器类路径。
--include-jetty-dir =
包括一个额外的Jetty目录，以用作配置详细信息的来源。此目录的行为类似于${jetty.base}但位于两者之间的一层${jetty.base}和${jetty.home} 。这允许一些复杂的配置详细信息层次结构。
--download = |
如果文件在给定位置不存在，请从给定的http URI下载。注意：位置始终是相对于${jetty.base} 。您可能需要转义斜杠“ \ |”在某些环境中使用它。
maven.repo.uri = [URL]
用于下载Maven依赖项的URL。默认值为https://repo1.maven.org/maven2/ 。

带阴影的Start.jar

如果您需要带阴影的版本start.jar （例如对于Gradle），您可以通过Maven依赖关系来实现。

<dependency>
    <groupId>org.eclipse.jetty</groupId>
    <artifactId>jetty-start</artifactId>
    <version>9.4.24.v20191120</version>
    <classifier>shaded</classifier>
</dependency>
