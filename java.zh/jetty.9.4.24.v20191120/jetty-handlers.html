处理程序是处理收到的请求的Jetty组件。

许多Jetty用户从不需要编写Jetty处理程序，而是使用Servlet API。您可以将现有的Jetty处理程序重用于上下文，安全性，会话和servlet，而无需扩展。但是，某些用户可能有禁止使用完整Servlet API的特殊要求或占用空间的问题。对于他们来说，实现Jetty处理程序是一种以最少的麻烦提供动态Web内容的直接方法。

请参阅“ Jetty体系结构 ”部分，以了解有关处理程序与Servlet的更多信息。

处理程序API

Handler接口提供了Jetty内容生成或操作的核心。实现此接口的类用于协调请求，过滤请求并生成内容。

Handler接口的核心API是：

public void handle(String target, Request baseRequest, HttpServletRequest request, HttpServletResponse response)
    throws IOException, ServletException

此方法的实现可以处理请求并将请求传递到另一个处理程序（或servlet），也可以在传递请求之前修改和/或包装请求。这提供了三种处理程序样式：

  • 协调处理程序-将请求路由到其他处理程序的处理程序（ HandlerCollection ， ContextHandlerCollection ）
  • 过滤处理程序-扩充请求并将其传递给其他处理程序的处理程序（ HandlerWrapper ， ContextHandler ， SessionHandler ）
  • 生成处理程序-产生内容的处理程序（ ResourceHandler和ServletHandler ）

目标

处理程序的目标是应该处理传递的请求的资源的标识符。通常，这是从HTTP请求中解析的URI。但是，在两种关键情况下，目标可能与传递的请求的URI不同：

  • 如果请求已分派到命名资源（例如命名Servlet），则目标是该资源的名称。
  • 如果请求是通过以下方式发出的： RequestDispatcher ，目标是所包含资源的URI，并且与实际请求的URI不同。

请求与回应

在handle方法签名中使用的请求和响应对象是ServletRequest和ServletResponse 。这些是标准的API，在它们对请求和响应的操作上受到适度的限制。通常，需要访问以下类的Jetty实现： Request和Response 。但是，由于请求和响应可能由处理程序，过滤器和Servlet包裹，因此无法直接传递实现。以下方法从任何包装器下面检索核心实现对象：

Request base_request = request instanceof Request ? (Request)request : HttpConnection.getCurrentConnection().getHttpChannel().getRequest();
Response base_response = response instanceof Response ? (Response)response : HttpConnection.getCurrentConnection().getHttpChannel().getResponse();

请注意，如果处理程序将请求传递给另一个处理程序，则应使用传入的请求/响应对象，而不是基础对象。这是为了保留上游处理程序完成的所有包装。

派遣

dispatch参数指示呼叫处理的状态，可以是：

  • REQUEST == 1 -从连接器收到原始请求。
  • FORWARD == 2 -由RequestDispatcher转发的请求。
  • INCLUDE == 4 -RequestDispatcher包含的请求。
  • ERROR == 8 -容器将请求转发给错误处理程序。

这些对于servlet和相关处理程序来说最重要。例如，安全处理程序仅将身份验证和授权应用于REQUEST调度。

处理要求

处理程序可以通过以下方式处理请求：

  • 产生回应
  • 过滤请求和/或响应
  • 将请求和响应传递给另一个处理程序

产生回应

的OneHandler嵌入式示例显示了简单的处理程序如何生成响应。

您可以使用标准的Servlet响应API，该API通常会设置一些状态，内容标头，然后写出内容：

 response.setContentType("text/html");
 response.setStatus(HttpServletResponse.SC_OK);
 response.getWriter().println("<h1>Hello OneHandler</h1>");

处理程序指示它已完成对请求的处理，并且不应将请求传递给其他处理程序，这一点也很重要：

 Request base_request = (request instanceof Request) ? (Request)request:HttpConnection.getCurrentConnection().getHttpChannel().getRequest();
 base_request.setHandled(true);

过滤请求和/或响应

一旦获得基本请求或响应对象，就可以对其进行修改。通常，您将进行修改以实现：

  • 将URI分为contextPath，servletPath和pathInfo组件。
  • 将资源库与对静态内容的请求相关联。
  • 将会话与请求相关联。
  • 将安全主体与请求相关联。
  • 在请求期间更改URI和路径将转发到另一个资源。

您还可以更新请求的上下文：

  • 设置当前线程的上下文类加载器。
  • 设置线程本地以识别当前ServletContext 。

通常，Jetty将修改后的请求传递给另一个处理程序，然后在finally块中撤消修改：

 try
 {
    base_request.setSession(a_session);
    next_handler.handle(target,request,response,dispatch);
 }
 finally
 {
    base_request.setSession(old_session);
 }

实现HandlerWrapper类通常是这种样式的处理程序过滤器。

将请求和响应传递给另一个处理程序

处理程序可能只是检查请求并使用目标，请求URI或其他信息来选择另一个处理程序以将请求传递给该处理程序。这些处理程序通常会实现HandlerContainer接口。

示例包括：

  • 类HandlerCollection -处理程序的集合，其中，无论请求的状态如何，都会调用每个处理程序。通常用于将请求传递给ContextHandlerCollection ，然后RequestLogHandler 。
  • HandlerList -依次调用的处理程序列表，直到将请求状态设置为已处理。
  • ContextHandlerCollection -一组处理程序，根据上下文路径最佳匹配选择其中一个。
