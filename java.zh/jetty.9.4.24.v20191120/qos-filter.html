用法

Jetty支持Continuation（连续性），该连续性允许对HTTP请求进行非阻塞处理，因此可以以托管方式分配线程，以提供特定于应用程序的服务质量（QoS）。的QoSFilter是一个实现了某些QoS功能的实用程序servlet过滤器。

了解问题

等待资源

Web应用程序经常使用JDBC连接池来限制数据库上的同时负载。这样可以保护数据库免受峰值负载的影响，但会使Web应用程序容易受到线程不足的困扰。考虑一个具有20个连接的线程池，该线程池由Web应用程序使用，该Web应用程序通常每秒接收200个请求，每个请求保持JDBC连接50ms。这样的池平均每秒可以处理200 * 20 * 1000/50 = 400个请求。

但是，如果请求速率提高到每秒400以上，或者数据库变慢（由于查询量大）或暂时不可用，则线程池可以非常迅速地累积许多等待的请求。例如，如果网站“乱七八糟”或经历了一些其他暂时的流量突发，并且请求速率从每秒400个增加到500个请求，则每秒100个请求将加入那些等待JDBC连接的请求。通常，Web服务器的线程池仅包含数百个线程，因此突发或慢速DB仅需要持续几秒钟即可消耗整个Web服务器的线程池。这称为线程饥饿。线程匮乏的关键问题在于它会影响整个Web应用程序，甚至可能影响整个Web服务器。即使使用数据库的请求仅占Web服务器上总请求的一小部分，由于所有可用线程都在JDBC连接池上等待，因此所有请求都被阻止。这代表了负载下的非正常降级，并提供了非常差的服务质量。

优先资源

考虑承受极大负载的Web应用程序。此负载可能是由于受欢迎程度激增（slashdot），使用量激增（圣诞节或营业时间关闭），甚至是拒绝服务攻击造成的。在这种负载期间，通常希望不要将所有请求均等对待，并优先考虑高价值客户或管理用户。

Web服务器在极端负载下的典型行为是使用其所有线程来服务请求并建立未服务请求的积压。如果积压变得足够深，则请求将开始超时，并且用户会遇到失败以及延迟。

理想情况下，Web应用程序应该能够检查积压中的请求，并优先考虑高价值客户和管理用户。但是，使用标准的阻塞Servlet API，如果在处理期间不为该请求分配线程，就无法检查该请求。无法延迟低优先级请求的处理，因此，如果要重新分配资源，则低优先级请求必须全部失败。

应用QoSFilter

服务质量筛选器（QoSFilter）使用Continuations来避免线程匮乏，确定请求的优先级并在负载下进行适当的降级，以提供高质量的服务。当您将过滤器应用于Web应用程序中的特定URL时，它会限制针对这些URL处理的活动请求的数量。超出限制的任何请求都将被暂停。当请求完成对受限URL的处理时，一个等待的请求将恢复并可以处理。您可以为每个暂停的请求分配优先级，以便高优先级的请求在低优先级的请求之前恢复。

必需的JAR

要使用QoS过滤器，这些JAR文件必须在WEB-INF/lib ：

  • $ JETTY_HOME / lib / jetty-util.jar
  • $ JETTY_HOME / lib / jetty-servlets.jar –包含QoSFilter

样本配置

将配置放入Webapp的web.xml要么jetty-web.xml 。默认配置一次处理十个请求，首先处理更重要的请求，然后对其余的请求进行排队。此示例一次处理五十个请求：

<filter>
   <filter-name>QoSFilter</filter-name>
   <filter-class>org.eclipse.jetty.servlets.QoSFilter</filter-class>
   <init-param>
     <param-name>maxRequests</param-name>
     <param-value>50</param-value>
   </init-param>
 </filter>

配置QoS过滤器参数

信号量指示maxRequests限制。过滤器在尝试获取信号量时会等待一小段时间。的waitMs init参数控制等待，避免在信号量不久就可用的情况下挂起。如果无法获得信号量，Jetty会在容器的默认暂停期或设置为默认值的容器中暂停请求。 suspendMs初始化参数。

QoS过滤器使用以下初始化参数：

maxRequests
一次要处理的最大请求数。预设值为10。
maxPriority
可以分配给请求的最大有效优先级。高优先级值的请求比低优先级值的请求更重要。预设值为10。
等待女士
尝试接受新请求时要等待的时间（以毫秒为单位）。达到maxRequests限制时使用。默认值为50毫秒。
暂停
如果请求没有被立即接受，那么该请求将被挂起的时间长度（以毫秒为单位）。如果未设置，则应用容器的默认挂起时间。默认值为-1毫秒。
managedAttr
如果设置为true，则将此servlet设置为ServletContext过滤器名称作为属性名称的属性。这允许上下文外部机制（例如，通过ContextHandler.MANAGED_ATTRIBUTES ）来管理过滤器的配置。

映射到URL

您可以使用通过使用servlet名称或URL模式将QoSFilter映射到servlet的语法。在此示例中，URL模式将QoSFilter应用于Web应用程序上下文中的每个请求：

<filter-mapping>
   <filter-name>QoSFilter</filter-name>
   <url-pattern>/*</url-pattern>
 </filter-mapping>

设置请求优先级

值较高的请求具有较高的优先级。QoSFilter分配的默认请求优先级为：

  • 2-对于任何已验证的请求
  • 1-对于具有非新有效会话的任何请求
  • 0-对于所有其他请求

要自定义优先级，请子类化QoSFilter，然后覆盖getPriority(ServletRequest request)方法，以为请求返回适当的优先级。然后，您可以将此子类用作QoS过滤器。这是一个例子：

public class ParsePriorityQoSFilter extends QoSFilter
 {
     protected int getPriority(ServletRequest request)
     {
         String p = ((HttpServletRequest)request).getParameter("priority");
         if (p!=null)
             return Integer.parseInt(p);
         return 0;
     }
 }
