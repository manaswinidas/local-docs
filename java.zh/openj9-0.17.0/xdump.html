    -X转储

    OpenJ9会生成各种类型的诊断信息，以便在发生不同事件（例如常规保护故障）时进行分析。产生的转储由转储代理控制，转储代理在OpenJ9虚拟机（VM）启动时初始化。在大多数情况下，转储代理的默认设置就足够了。但是，您可以使用-Xdump命令行上的“选项”以微调转储代理设置。例如，您可以使用-Xdump选项以添加和删除各种VM事件的转储代理，更新默认转储设置，并限制产生的转储数量。

    大量选项和子选项可用于控制转储，这提供了很大的灵活性。

    Xdump选项生成器

    使用Xdump Option Builder工具可以帮助您指定正确的选项并避免不兼容。

    句法

        -Xdump:<parameter>

    下表列出了针对-Xdump ，其中提供了用法和配置信息：

    命令 结果
    -Xdump:help 显示常规转储帮助。
    -Xdump:events 列出可用的触发事件。
    -Xdump:request 列出其他VM请求。
    -Xdump:tokens 列出公认的标签标记。
    -Xdump:what 在启动时显示注册的代理。
    -Xdump::help 显示转储代理使用信息。

    以下选项可用于控制诊断数据的产生：

    参数 结果
    -Xdump:none 删除所有默认的转储代理和任何以前的转储选项。
    -Xdump:dynamic 启用对可插拔代理的支持
    -Xdump:nofailover 当默认或指定的转储位置已满时，放弃转储。
    -Xdump:directory= 指定所有要写入的转储类型的目录。此目录路径的前缀是所有非绝对转储文件名的路径，包括默认转储代理的文件名。
    -Xdump:suspendwith= 修改在写入转储文件时挂起VM线程的信号。使用更改默认信号号。（仅Linux®）
    -Xdump:: 为每个转储代理提供详细的子选项，以提供更精细的控制。

    通过指定子选项，可以在非常详细的级别上配置转储代理。的子选项是主要触发机制。下表列出了完整的子选项集：

    转储代理子选项 结果
    -Xdump::none 删除转储代理。
    -Xdump::defaults 打印转储代理的默认选项。
    -Xdump::events= 发生特定事件时触发转储代理。
    -Xdump::exec= 启动转储代理的外部应用程序。
    -Xdump::file= 指定将转储代理写入转储的位置。
    -Xdump::filter= 按通配符或事件过滤转储。
    -Xdump::msg_filter= 过滤异常消息中的文本字符串。
    -Xdump::opts= 由特定的转储代理用来选择要生成的转储文件的类型。
    -Xdump::priority=<0-999> 指定转储代理运行的优先级。
    -Xdump::range= 在特定的VM上启动和停止转储代理。
    -Xdump::request= 在启动转储代理之前，要求VM准备状态。

    你可以有多个-Xdump命令行上的选项。您还可以通过多个事件触发多种转储类型。例如，以下命令关闭堆转储文件的创建，并创建一个发生vmstart或vmstop事件时产生堆转储文件和Java™转储文件的转储代理：

    java -Xdump:heap:none -Xdump:heap+java:events=vmstart+vmstop -mp . -m <class> [args...]

    注意：后面有多个子选项Xdump子选项必须以逗号（，）分隔，例如：

    java -Xdump:java:events=vmstart,file=/STDERR/ -version

    有关这些参数和子选项的更多详细信息（包括示例），请参见Parameters 。

    倾销代理

    触发时，转储代理执行诊断任务。大多数转储代理都以某种形式的转储或跟踪文件保存有关VM状态的信息，以供以后分析。“工具”代理是一个例外，当发生特定事件时，可用于触发外部过程。

    倾销代理 描述
    堆 堆栈转储是非常基本的转储，其中线程的状态和Java堆栈被写入stderr。
    安慰 控制台转储是非常基本的转储，其中每个Java线程的状态都写入stderr。
    系统 系统转储捕获应用程序的原始进程映像或地址空间。
    工具 工具选项允许事件发生时启动外部进程。
    爪哇 Java转储是对VM的内部生成和格式化的分析，提供的信息包括存在的Java线程，已加载的类和堆统计信息。
    堆 堆转储捕获堆中的所有对象实例，包括每个对象地址，类型或类名称，大小以及对其他对象的引用。
    卡 快速浏览包含跟踪点数据的跟踪缓冲区。
    ceedump LE CEEDUMP转储是z /OS®格式的摘要系统转储，其中显示了VM进程中每个线程的堆栈跟踪，以及寄存器信息和每个寄存器的简短存储转储。
    准时 JIT编译器转储包含二进制格式的诊断数据。

    合并-Xdump代理

    如果您配置多个转储代理，则每个转储代理都会根据其配置对事件做出响应。但是，代表转储代理配置的内部结构可能与命令行不匹配，因为合并转储代理是为了提高效率。只要没有代理设置冲突，就可以合并两组选项。这意味着已安装的转储代理及其参数的列表由-Xdump:what可能无法以与原始分组相同的方式进行分组-Xdump配置它们的选项。

    例如，可以使用以下命令指定转储代理在类卸载时创建Java转储文件：

    java -Xdump:java:events=unload -Xdump:what

    该命令不会创建新的代理，从以下结果中可以看出-Xdump:what选项。

    ...
----------------------
-Xdump:java:
    events=gpf+user+abort+unload+traceassert+corruptcache,
    label=/home/user/javacore.%Y%m%d.%H%M%S.%pid.%seq.txt,
    range=1..0,
    priority=400,
    request=exclusive+preempt
----------------------

    配置是合并现有的Java转储剂事件GPF， 用户 中止 ，traceassert和corruptcache，因为没有与那些现有的代理新的卸载代理冲突指定的选项。

    在上一个示例中，如果更改了卸载代理程序的参数之一，使其与现有代理程序冲突，则无法合并它。例如，以下命令指定了不同的优先级，强制创建单独的代理：

    java -Xdump:java:events=unload,priority=100 -Xdump:what

    结果-Xdump:what命令中的选项如下。

    ...
----------------------
-Xdump:java:
    events=unload,
    label=/home/user/javacore.%Y%m%d.%H%M%S.%pid.%seq.txt,
    range=1..0,
    priority=100,
    request=exclusive+preempt
----------------------
-Xdump:java:
    events=gpf+user+abort+traceassert+corruptcache,
    label=/home/user/javacore.%Y%m%d.%H%M%S.%pid.%seq.txt,
    range=1..0,
    priority=400,
    request=exclusive+preempt
----------------------

    要合并转储代理， request ， filter ， opts ， label和range参数必须完全匹配。如果您指定了多个筛选相同字符串的代理，但保持所有其他参数相同，则将合并代理。例如：

    java -Xdump:none -Xdump:java:events=uncaught,filter=java/lang/NullPointerException -Xdump:java:events=unload,filter=java/lang/NullPointerException -Xdump:what

    该命令的结果如下：

    Registered dump agents
----------------------
-Xdump:java:
    events=unload+uncaught,
    filter=java/lang/NullPointerException,
    label=/home/user/javacore.%Y%m%d.%H%M%S.%pid.%seq.txt,
    range=1..0,
    priority=400,
    request=exclusive+preempt
----------------------

    转储事件

    转储代理由OpenJ9 VM操作期间发生的事件触发。可以过滤某些事件以改善输出的相关性。

    下表显示了可用作转储代理触发器的事件：

    事件 在...时触发。 筛选...。
    gpf 发生一般保护故障（GPF）。 不适用
    用户 VM从操作系统接收SIGQUIT（Linux，macOS®，AIX®，z / OS）或SIGBREAK（Windows™）信号。 不适用
    中止 VM从操作系统接收SIGABRT信号。 不适用
    vmstart 虚拟机已启动。 不适用
    vmstop 虚拟机停止。 退出代码；例如，filter=#129..#192#-42#255
    加载 加载了一个类。 班级名称;例如，filter=java/lang/String
    卸下 我们卸载了一个类。 不适用
    扔 引发异常。 异常类名；例如，filter=java/lang/OutOfMem*
    抓住 捕获到异常。 异常类名；例如，filter=*Memory*
    未捕获 应用程序未捕获Java异常。 异常类名；例如，filter=*MemoryError
    抛出 VM将引发Java异常。这与“ throw”事件不同，因为它仅针对VM内部检测到的错误情况才触发。 异常类名；例如， filter=java/lang/OutOfMem* 。
    thrstart 新线程启动。 不适用
    受阻 线程被阻塞。 不适用
    停止 线程停止。 不适用
    完整的 垃圾收集周期开始。 不适用

    要关闭所有堆转储（包括默认代理），但保持启用Java转储，请使用以下选项：

    -Xdump:system:none:events=systhrow,filter=java/lang/OutOfMemoryError

    • 如果将用户信号（kill -QUIT）发送到VM，则将Java线程的简短列表（包括它们的堆栈，状态和监视器信息）写入stderr。
    • 如果发生崩溃，则将有关崩溃位置，VM选项以及本机和Java堆栈跟踪的信息写入stderr。系统转储文件也被写入用户的主目录。

    注意：删除转储代理并指定新的转储配置可能需要很长的命令行选项。要重新使用命令行选项，请将新的转储配置保存在文件中，然后使用-Xoptionsfile选项。有关更多信息，请参见-Xoptionsfile 。

    defaults

    每种转储类型都有默认选项。要查看特定转储类型的默认选项，请使用-Xdump::defaults 。

    -Xdump:java:defaults:file=dumps/%pid/javacore-%seq.txt`

    或者，例如，在z / OS上，可以使用以下命令将作业名称添加到Java转储文件名称中：

    -Xdump:java:defaults:file=javacore.%job.%H%M%S.txt

    此选项不添加Java转储代理。它更新了Java转储代理的默认设置。除非被覆盖，否则其他Java转储代理将使用该文件名规范创建转储文件。

    注意：更改转储类型的默认值也会影响VM在初始化期间为该转储类型添加的默认代理。例如，如果您更改Java转储文件的默认文件名，则将更改默认Java转储代理使用的文件名。但是，更改默认范围选项不会更改默认Java转储代理使用的范围，因为这些代理会使用特定值覆盖范围选项。

    -Xdump:heap:events=vmstop

    exec=

    工具转储代理使用exec子选项指定要启动的外部应用程序。您可以使用以下命令来设置要针对特定转储代理运行的特定命令：

    -Xdump:<agent>:exec=<command>

    file=

    file子选项指定针对指定转储类型的诊断信息的写入位置。语法是-Xdump::file= 。

    java -Xdump:heap:events=vmstop,file=my.dmp

    在z / OS平台上生成系统转储文件时，请使用dsn选项而不是file选项。例如：

    java -Xdump:system:events=vmstop,dsn=%uid.MYDUMP

    写给STDOUT /STDERR

    添加以下选项之一，分别将Java转储文件写入STDOUT或STDERR：

-Xdump:java:file=/STDERR/

    filter=

    在应用程序的生存期内，某些VM事件会发生数千次。转储代理可以使用过滤器和范围来避免产生过多的转储文件。必须使用以下语法：

    -Xdump:<agent>:filter=<filter>

    通配符

    您可以在例外事件过滤器中使用通配符，方法是仅在过滤器的开头或结尾放置星号。由于第二个星号不在结尾，所以以下命令不起作用：

    -Xdump:java:events=throw,filter=*InvalidArgumentException#*.myVirtualMethod

    要解决此问题，请将此过滤器更改为以下字符串：

    -Xdump:java:events=throw,filter=*InvalidArgumentException#MyApplication.*

    类加载和异常事件

    您可以通过正在加载，抛出或捕获的类的名称来过滤类的加载（ load ）和异常（ throw ， catch ， uncaught ， systhrow ）事件。例如：

    -Xdump:java:events=load,filter=java/lang/String
-Xdump:java:events=throw,filter=java/lang/ArrayStoreException
-Xdump:java:events=catch,filter=java/lang/NullPointerException

    另外，您可以通过引发异常的方法的名称来过滤throw ， uncaught和systhrow异常事件。父类的名称必须包含完整的软件包名称，并使用正斜杠（/）作为分隔符。使用点（。）将方法名称与类名称分开。您可以使用星号（*）作为通配符，以包括所有方法（可选部分显示在方括号中）。例如：

    -Xdump:java:events=throw,filter=ExceptionClassName[#com/ibm/ThrowingClassName.throwingMethodName[#stackFrameOffset]]

    例如，当方法触发Java转储MyApplication.myMethod()抛出一个NullPointerException例外，请使用以下语法：

    -Xdump:java:events=throw,filter=java/lang/NullPointerException#com/ibm/MyApplication.myMethod

    堆栈帧偏移使您可以筛选调用throwing方法的方法的名称。如果从通用类或实用程序类引发异常，则此选项很有用。例如，当由调用的方法时触发Java转储MyApplication.main()抛出一个NullPointerException ，请使用以下语法：

    -Xdump:java:events=throw,filter=java/lang/NullPointerException#com/ibm/MyApplication.main#1

    堆栈帧偏移的默认值为零。

    您可以按Java方法名称过滤捕获异常事件（可选部分显示在方括号中）。例如：

    -Xdump:java:events=catch,filter=ExceptionClassName[#com/ibm/CatchingClassName.catchingMethodName]

    您可以按Java方法名称过滤throw ， uncaught和systhrowexception事件（括号中显示可选部分）：

    -Xdump:java:events=throw,filter=ExceptionClassName[#com/ibm/ThrowingClassName.throwingMethodName[#stackFrameOffset]]

    您可以按Java方法名称过滤catch异常事件（可选部分显示在方括号中）：

    -Xdump:java:events=catch,filter=ExceptionClassName[#com/ibm/CatchingClassName.catchingMethodName]

    注意：筛选器适用于stacktrace，并在每次重新抛出相同的异常时触发，这可能会导致多个Java核心文件。

    vmstop事件

    您可以使用一个或多个退出代码来过滤VM关闭事件（ vmstop ）：

    -Xdump:java:events=vmstop,filter=#129..192#-42#255

    慢事件

    您可以过滤慢速事件以将时间阈值从默认值50 ms更改为：

    -Xdump:java:events=slow,filter=#300ms

    分配事件

    您必须过滤分配事件以指定导致触发的对象的大小。您可以将过滤器大小设置为从零到在32位平台上的32位指针的最大值，或在64位平台上的64位指针的最大值。将较低的过滤器值设置为零会触发所有分配的转储。

    例如，要在大小大于5 Mb的分配上触发转储，请使用：

    -Xdump:stack:events=allocation,filter=#5m

    要触发大小在256Kb和512Kb之间的分配的转储，请使用：

    -Xdump:stack:events=allocation,filter=#256k..512k

    其他活动

    如果将过滤器应用于不支持过滤的事件，则过滤器将被忽略。

    msg_filter=

    您可以使用msg_filter子选项过滤异常消息中的文本字符串，从而减少生成的转储文件的数量。仅以下事件支持此选项： throw ， catch ， systhrow和uncaught 。

    使用以下语法在转储输出中包括消息过滤：

    -Xdump:<agent>:events=<event>,msg_filter=<filter>`

    哪里是要包含在转储文件中的异常中的文本字符串。此子选项支持将星号用作通配符。

    以下示例过滤器java/lang/VerifyError包含文本字符串类格式的异常：

    -Xdump:java:events=throw,filter=java/lang/VerifyError,msg_filter=*class format*

    opts=

    完整的语法是-Xdump::opts= 。

    堆转储代理使用此子选项来指定要生成的文件的类型。在z / OS上，系统转储代理使用此子选项来指定要生成的转储的类型。

    堆转储

    您可以指定PHD堆转储文件（PHD），经典文本堆转储文件（CLASSIC）或两者。默认为PHD文件。例如：

    -Xdump:heap:opts=PHD  
-Xdump:heap:opts=CLASSIC
-Xdump:heap:opts=PHD+CLASSIC

    z / OS系统转储

    您可以指定系统事务转储（IEATDUMP），LE转储（CEEDUMP）或两者。默认值为IEADUMP文件。例如：

    -Xdump:system:opts=IEATDUMP
-Xdump:system:opts=CEEDUMP
-Xdump:system:opts=IEATDUMP+CEEDUMP

    ceedump代理是指定LE转储的首选方式，例如：

    -Xdump:ceedump:events=gpf

    工具堆

    工具转储代理支持两个子选项，可以使用opts订阅。您可以使用opts = ASYNC异步运行外部进程。您还可以指定以毫秒为单位的延迟，该延迟会在启动命令后产生暂停。这两个选项可以单独使用，也可以一起使用。以下示例显示了用于启动运行新进程的不同选项myProgram ：

    -Xdump:tool:events=vmstop,exec=myProgram

    没有opts子选项，工具转储代理将启动该过程，并等待该过程结束再继续。

    -Xdump:tool:events=vmstop,exec=myProgram,opts=ASYNC

    什么时候opts=ASYNC指定后，工具转储代理将启动该过程，然后继续运行而无需等待新过程结束。

    -Xdump:tool:events=vmstop,exec=myProgram,opts=WAIT1000

    此选项启动进程，等待进程结束，然后再等待1秒钟（1000毫秒），然后再继续。

    -Xdump:tool:events=vmstop,exec=myProgram,opts=ASYNC+WAIT10000

    最后，最后一个示例将启动该过程，并等待10秒钟，然后继续，无论该过程是否仍在运行。如果您启动的进程没有结束，但是需要一些时间来正确初始化，则最后一种形式很有用。

    priority=<0-999>

    一个事件可以生成多个转储文件。产生每个转储文件的代理按顺序运行，其顺序由为每个代理设置的优先级关键字确定。该命令的完整语法为-Xdump::priority=<0-999> 。

    检查来自的输出-Xdump:what显示gpf事件会生成快照跟踪，Java转储文件和系统转储文件。在此示例中，系统转储首先运行，优先级为999。快照转储运行第二，优先级为500。Java转储最后运行，优先级为10：

    -Xdump:heap:events=vmstop,priority=123

    允许的最大值为999。首先启动优先级更高的转储代理。

    如果没有特别设置优先级，则根据转储类型采用默认值。可以使用以下命令显示特定类型的转储的默认优先级和其他默认值-Xdump::defaults 。例如：

    java -Xdump:heap:defaults -version

    默认-Xdump:heap设置：

      events=gpf+user
  filter=
  file=/home/user/heapdump.%Y%m%d.%H%M%S.%pid.phd
  range=1..0
  priority=500
  request=exclusive+compact+prepwalk
  opts=PHD

    range=

    您可以使用以下命令在特定的VM事件发生时启动和停止转储代理： range子选项：-Xdump::range=

    例如：

    -Xdump:java:events=fullgc,range=100..200

    注意：针对事件的范围= 1..0表示“每次发生”。

    对于systhrow以外的所有事件，VM默认转储代理的range子选项均设置为1..0。大多数systhrow事件与filter=java/lang/OutOfMemoryError将range子选项设置为1..4，这限制了在OutOfMemory条件最多为4。有关更多信息，请参阅默认转储代理 。

    如果添加新的转储代理并且未指定范围，则使用默认值1..0。

    request=

    在启动转储代理之前，请使用request子选项要求VM准备状态：-Xdump::request=

    下表列出了可用的子选项：

    子选项值 描述
    独家 请求对VM的独占访问。
    紧凑 运行垃圾回收。此选项会在生成转储文件之前从堆中删除所有无法访问的对象。
    前行 准备堆走。使用此选项时，还必须指定独占。
    连载 挂起其他转储，直到此转储完成。
    抢占 适用于Java转储代理，并控制是否强制抢占进程中的本机线程以收集堆栈跟踪。如果未指定此选项，则Java转储中仅收集Java堆栈跟踪。

    您可以使用+指定多个请求选项。例如：

    -Xdump:heap:request=exclusive+compact+prepwalk

    VM独占访问机制允许VM线程通过使用内部VM锁以受控方式停止其他VM线程的活动。当。。。的时候request=exclusive为转储代理程序指定了选项，正在生成转储的VM线程会在写入转储文件之前等待运行Java代码的线程停止运行，并等待垃圾回收操作完成。此过程有助于确保转储具有一致的数据。转储完成后，该机制将允许其他线程恢复。

    默认情况下，仅系统转储OutOfMemoryError例外请求独占访问。其他系统转储事件通常是由崩溃导致的。在这些情况下，由于在崩溃期间获取锁可能会出现问题，因此不需要独占访问。

    如果使用来请求系统转储com.ibm.jvm.Dump.SystemDump() API，使用默认的系统转储代理设置，并且不要求独占访问。但是，如果打算将系统转储文件用于Java堆内存分析，请在进行转储时使用以下选项请求独占访问：

    -Xdump:system:defaults:request=exclusive+compact+prepwalk

    这些设置可以避免在垃圾回收期间捕获包含运行中数据的转储文件。或者，您可以使用com.ibm.jvm.Dump.triggerDump() API并指定request=exclusive+compact+prepwalk在API调用上。

    有关的更多信息com.ibm.jvm.Dump API ，请参阅API参考信息。

    的默认设置request Java转储文件的子选项是request=exclusive+preempt 。要更改设置，以便生成Java转储文件而不会抢占线程来收集本机堆栈跟踪，请使用以下选项：

    -Xdump:java:request=exclusive

    通常，默认请求选项就足够了。

    Linux上的系统转储

    Linux没有提供用于从正在运行的进程生成系统转储的操作系统API。VM通过使用fork（）API启动与父VM进程相同的进程，从而在Linux上产生系统转储。然后，VM在子进程中生成SIGSEGV信号。SIGSEGV信号使Linux为子进程创建系统转储。父VM根据需要处理并重命名系统转储。 -Xdump选项，并且可能会将其他数据添加到转储文件中。

    子进程的系统转储文件包含父进程中使用的内存区域的精确副本。转储查看器可以从系统转储中获取有关Java线程，类和堆的信息。但是，转储查看器和其他系统转储调试器仅显示在子进程中运行的单个本机线程。

    您可以使用Linux kernel.core_pattern设置以指定系统转储的名称和路径。VM转储代理通过按以下名称重命名转储来覆盖Linux系统转储名称和路径。 -Xdump选项。如果kernel.core_pattern设置指定了与文件系统不同的文件系统-Xdump选项，VM转储代理可能无法更改文件路径。在这种情况下，VM重命名转储文件，但文件路径保持不变。您可以在JVMDUMP010I消息中找到转储文件的名称和位置。

    注意：如果您使用%t中的说明符kernel.core_pattern设置，VM不会重命名转储。VM无法确定Linux生成核心文件的确切时间，因此无法确定哪个Linux转储文件是正确的重命名文件。
