    -Xshareclass

    使用-Xshareclasses选项来启用，禁用或修改类共享行为。默认情况下，仅对引导程序类启用类数据共享（等效于指定-Xshareclasses:bootClassesOnly,nonFatal,silent ），除非您的应用程序在容器中运行。

    此选项可以采用许多参数，其中一些是高速缓存实用程序。缓存实用程序对指定的缓存执行所需的操作，而无需启动VM。您可以组合多个子选项，这些子选项用逗号分隔，但是缓存实用程序是互斥的。

    当您运行缓存实用程序时，该消息Could not create the Java virtual machine是期待。缓存实用程序不会创建虚拟机。

    某些缓存实用程序可以使用Java™以前版本中的缓存或由具有不同位宽的虚拟机（VM）创建的缓存。这些高速缓存称为“不兼容”高速缓存。

    另请参阅“ 类数据共享”主题，其中包括一些使用的最佳实践-Xshareclasses 。

    句法

        -Xshareclasses:<parameter>

    当您指定-Xshareclasses没有任何参数，也没有指定-Xscmx要么-XX:SharedCacheHardLimit选项，将使用默认大小创建共享类缓存，如下所示：

    • 对于64位平台，默认大小为300 MB，对于缓存的初始大小有一个“软”最大限制（ -Xscmx ）的大小为64MB，但以下情况除外：
      • 对于永久性缓存，如果可用磁盘空间小于6 GB，则默认大小设置为64 MB，并且-Xscmx大小未设置。
      • 对于Linux®或macOS®系统上的非持久高速缓存，高速缓存的大小受进程可以保留的最大内存量的限制（ SHMMAX ）。如果SHMMAX小于300MB，默认共享缓存大小设置为等于SHMMAX 。如果SHMMAX大于80 MB， -Xscmx设置为64 MB。如果SHMMAX小于80MB -Xscmx大小未设置。
    • 对于其他平台，默认大小为16MB。

    参量

    adjustmaxaot （缓存实用程序）

        -Xshareclasses:adjustmaxaot=<size>
    调整AOT数据允许的最大共享类缓存空间。当您使用-Xshareclasses:verbose选项，VM将控制台当前设置的未写入的字节数写入控制台。 -Xscmaxaot选项。

    adjustminaot （缓存实用程序）

        -Xshareclasses:adjustminaot=<size>
    调整为AOT数据保留的最小共享类缓存空间。使用-Xscminaot设置初始最小尺寸的选项。

    adjustmaxjitdata （缓存实用程序）

        -Xshareclasses:adjustmaxjitdata=<size>
    调整JIT数据允许的最大共享类高速缓存空间。当您使用-Xshareclasses:verbose选项，VM将控制台当前设置的未写入的字节数写入控制台。 -Xscmaxjitdata选项。

    adjustminjitdata （缓存实用程序）

        -Xshareclasses:adjustminjitdata=<size>
    调整为JIT数据保留的最小共享类缓存空间。使用-Xscminjitdata设置初始最小尺寸的选项。

    adjustsoftmx （缓存实用程序）

        -Xshareclasses:adjustsoftmx=<size>
    调整缓存的软最大大小。当您使用-Xshareclasses:verbose选项，VM将由于软最大大小的当前设置而将未存储的字节数写入控制台。有关最大软尺寸的更多信息，请参见-Xscmx 。

    allowClasspaths

        -Xshareclasses:allowClasspaths
    允许VM将类路径存储到使用以下命令创建的现有共享缓存中： restrictClasspaths选项。

    bootClassesOnly

        -Xshareclasses:bootClassesOnly
    禁用对除引导类加载器之外的其他类加载器加载的类进行缓存。如果使用此子选项，则nonfatal子选项也已设置，因此此子选项等效于指定-Xshareclasses:bootClassesOnly,nonfatal 。

    cacheDir

        -Xshareclasses:cacheDir=<directory>

    设置用于读取和写入高速缓存数据的目录。以下默认值适用：

    • 在Windows™系统上， 是用户的C:\Documents and Settings\\Local Settings\Application Data\javasharedresources目录。
    • 在其他操作系统上， 是用户的主目录，除非groupAccess指定参数，在这种情况下为/tmp/javasharedresources ，因为该组的某些成员可能无法访问用户的主目录。您必须具有足够的权限 。

    在AIX®，Linux，macOS和Windows系统上，VM将持久性高速缓存文件直接写入指定的目录中。持久性缓存文件可以安全地从文件系统中移动和删除。对于永久性高速缓存，该目录不得位于NFS挂载上。

    非永久性高速缓存存储在共享内存中，并具有描述内存位置的控制文件。控制文件存储在javasharedresources的子目录cacheDir指定。不要移动或删除此目录中的控制文件。的listAllCaches实用程序destroyAll实用程序和expire子选项仅在给定范围内起作用cacheDir 。

    在AIX，Linux和macOS系统上，如果您指定cacheDir=选项，将使用以下权限创建持久性缓存（ -rw-r--r-- ）：

    • 用户：读/写
    • 组：读取（如果还指定，则为读/写-Xshareclasses:groupAccess ）
    • 其他：只读

    否则，将使用与非持久性缓存相同的权限来创建持久性缓存。非永久缓存的权限为-rw-r----- ， 要么-rw-rw----如果您还指定-Xshareclasses:groupAccess 。

    注意：优良作法是设置特定于应用程序的缓存目录，以避免与默认缓存或其他未设置缓存目录的应用程序缓存共享默认缓存目录，因此，您的应用程序不会受到用户的影响跑步java -Xshareclasses:destroyAll 。请参阅课程数据共享：使用的最佳做法-Xshareclasses 。

    cacheDirPerm

    （不是Windows）
    -Xshareclasses:cacheDirPerm=<permission>

    在创建缓存目录时设置Unix样式的权限。 必须是0700-0777或1700-1777范围内的八进制数。如果无效，则VM会以适当的错误消息结尾。

    仅当您创建新的缓存目录时，才使用此子选项指定的权限。如果缓存目录已经存在，则此子选项将被忽略，并且缓存目录权限不会更改。

    如果将此子选项设置为0000，则使用默认目录权限。如果将此子选项设置为1000，则使用计算机的默认目录权限，但是启用了sticky位。

    如果缓存目录是平台默认目录， /tmp/javasharedresources ，此子选项将被忽略，并且缓存目录权限设置为0777。

    如果未设置此子选项，则根据以下条件使用默认权限：

    条件 权限
    缓存目录为/tmp/javasharedresources 。如果此目录已经存在且具有不同的权限，则在打开缓存时会更改权限。† 0777
    缓存目录是一个新目录，您还可以指定groupAcess子选项 0770
    缓存目录是一个新目录，您没有指定groupAccess子选项 0700
    缓存目录已存在，但不存在/tmp/javasharedresources 不变的
    †在z /OS®系统上，对现有缓存目录的权限保持不变，以避免生成RACF®错误，该错误会生成日志消息。
    注意：当默认值不合适时，最好为缓存目录明确设置权限。请参阅课程数据共享：使用的最佳做法-Xshareclasses 。

    cacheRetransformed

        -Xshareclasses:cacheRetransformed
    启用对使用JVMTI转换的类的缓存RetransformClasses功能。有关更多信息，请参阅JVMTI重定义和重新转换类 。

    选项enableBCI默认情况下启用。但是，如果您使用cacheRetransformed选项，此选项强制将缓存创建到-Xshareclasses:disableBCI模式。

    checkURLTimestamps

        -Xshareclasses:checkURLTimestamps
    造成时间戳记jar要么zip每次加载类时要检查的文件。如果时间戳已更改，则从jar要么zip文件，而不是来自共享缓存。默认情况下，此子选项未启用，反映了共享类缓存的旧行为。
    注意：引导程序的时间戳jar要么zip首次使用该文件加载类时，将对文件进行一次检查。

    createLayer

    （仅用于实验的64位）

        -Xshareclasses:createLayer
    创建分层的缓存。此子选项是实验性的；不要在生产环境中使用它。
    如果在创建分层缓存时有多个虚拟机处于竞争状态，则可以创建多个新的分层缓存。为避免这种情况，请使用-Xshareclasses:layer=子选项可创建具有特定层号的新分层缓存。看到layer有关分层缓存的更多信息。

    destroy （缓存实用程序）

        -Xshareclasses:destroy
    销毁由name ， cacheDir和nonpersistent子选项。仅当使用缓存的所有虚拟机均已终止并且用户具有足够的权限时，才能销毁该缓存。

    destroyAll （缓存实用程序）

        -Xshareclasses:destroyAll
    尝试销毁由指定的所有缓存cacheDir和nonpersistent子选项。
    在Windows和z / OS系统上，仅当使用缓存的所有VM已关闭并且用户具有足够的权限时，才可以销毁该缓存。
    注意：在z / OS上，当destroyAll从31位VM调用该选项时，不会破坏64位缓存。同样，当destroyAll从64位VM调用该选项时，不会破坏31位缓存。显示以下消息：
    JVMSHRC735I: Use a nn-bit VM to perform the requested operation on the
nn-bit shared cache "cachename" as the nn-bit VM
cannot verify that the shared memory was created by the VM.

    在AIX，Linux和macOS系统上：

    • 仅当所有正在使用的虚拟机都已关闭且用户具有足够的权限时，才可以销毁非持久性缓存。

    • 即使您使用此选项，仍在使用的持久性高速缓存仍然存在，但是它们与文件系统未链接，因此对于新的VM调用不可见。如果更新VM，然后重新启动已经存在持久共享缓存的应用程序，则VM将取消现有缓存的链接并创建一个新的缓存。由于未链接的缓存对新的虚拟机不可见，因此无法通过使用-Xshareclasses:listAllCaches选项，您将无法使用-Xshareclasses:printStats他们的选择。因此，您可以拥有多个未链接的高速缓存，这些高速缓存会占用文件系统空间，直到不再使用它们为止。

    destroyAllLayers

    （仅用于实验的64位）

        -Xshareclasses:destroyAllLayers
    销毁由指定的所有共享缓存层name子选项。例如， -Xshareclasses:name=Cache1,destroyAllLayers销毁所有称为Cache1 。如果您使用destroy例如，分层缓存上的子选项-Xshareclasses:name=Cache1,destroy ，只有缓存的顶层被破坏。
    有关分层缓存的更多信息，请参见layer子选项。

    destroyAllSnapshots （缓存实用程序）

    （不是Windows）
    -Xshareclasses:destroyAllSnapshots

    销毁由于指定的结果而可用的所有共享缓存快照cacheDir子选项。

    destroySnapshot （缓存实用程序）

    （不是Windows）
    -Xshareclasses:destroySnapshot

    销毁由name和cacheDir子选项。

    disableBCI

        -Xshareclasses:disableBCI

    listAllCaches （缓存实用程序）

    mprotect

    AIX，z / OS 31位：
    -Xshareclasses:mprotect=[default|all|none]

    Linux，Windows，macOS：

    -Xshareclasses:mprotect=[default|all|partialpagesonstartup|onfind|nopartialpages|none]

    哪里：

    • default ：默认情况下，除非正在更新特定的页面，否则包含缓存的内存页面始终受到保护。此保护有助于防止意外或故意损坏缓存。默认情况下，缓存头不受保护，因为这种保护会降低性能。在Linux，macOS和Windows系统上，在启动阶段之后，只要将新数据按以下顺序添加到共享类缓存中，Java虚拟机（VM）就会保护部分填充的页面：
      • VM会将所有部分填充的页面的内存保护更改为读/写。
      • VM将数据添加到缓存。
      • VM会将所有部分填充的页面的内存保护更改为只读。
    • all ：此值可确保所有缓存页面（包括标题）都受到保护。看注释。
    • partialpagesonstartup ：此值使VM在启动期间以及启动阶段之后保护部分填充的页面。此值仅在Linux，macOS和Windows系统上可用。
    • onfind ：指定此选项后，当虚拟机读取另一个虚拟机添加的缓存中的新数据时，将保护部分填充的页面。此选项仅在Linux，macOS和Windows系统上可用。
    • nopartialpages ：使用此值可以关闭对部分填充页面的保护。此值仅在Linux，macOS和Windows系统上可用。
    • none ：指定此值将禁用页面保护。

    注意：指定all对性能产生负面影响。您应该指定all仅用于问题诊断而不用于生产。指定值partialpagesonstartup要么onfind填充缓存时，也会对性能产生负面影响。当缓存已满或不再被修改时，不会有进一步的影响。

    restrictClasspaths

        -Xshareclasses:restrictClasspaths
    仅允许正在初始化共享缓存的第一个VM在缓存中存储类路径。除非随后的虚拟机允许将类路径存储在缓存中，否则allowClasspaths指定了选项。
    使用restrictClasspaths仅当应用程序旨在创建类型的类加载器时才选择此选项java.net.URLClassloader或其子类，以便它们的类路径对于应用程序实例是唯一的，但是从这些类路径加载的类是相同的。在这种情况下，一个虚拟机存储的应用程序类路径不能被另一个虚拟机使用。

    例如，考虑两个分别使用类路径CP1和CP2的VM，VM1和VM2，其中：

    • CP1：url1;url2;url3;tempurl1;url4;url5
    • CP2：url1;url2;url3;tempurl2;url4;url5

    这些类路径的区别仅在于一个条目，即tempurl 。的url1 ， url2 ， url3 ， url4和url5条目永远不会因运行而改变，而tempurl条目总是不同的。这种差异意味着从中加载的类url4要么url5并由VM1存储到共享缓存中，而VM2无法定位。因此，VM2尝试从中加载类url4要么url5会导致它存储自己的类路径CP2到共享缓存中，还为从中加载的类添加新的元数据url4要么url5 。将此类唯一类路径添加到共享缓存中没有用。此外，其他元数据可能会对连接到共享缓存的其他VM的性能产生不利影响。因为类从url4要么url5不会从共享缓存加载tempurl与原始版本不同，最好将tempurl作为类路径中的最后一个条目。

    在示例中所述的情况下， restrictClasspaths通过确保第一个VM初始化共享缓存，然后通过附加到共享缓存的后续VM阻止唯一类路径的添加，可以使用option选项来限制类路径的添加。请注意， restrictClasspaths在任何其他情况下，该选项可能会对VM使用现有缓存时的性能产生负面影响。

        -Xshareclasses:revalidateAotMethods=<method_specification>
    -Xshareclasses:revalidateAotMethods=help
    修改共享缓存以重新验证与方法规范匹配的AOT方法。使用此子选项可以重新验证通过使用invalidateAotMethods子选项。然后，经过重新验证的AOT方法可以加载到VM中，但不影响已处理该方法的VM。要了解有关用于的语法的更多信息 ，包括如何指定多个方法的方法，请参见方法规范语法 。

    silent

        -Xshareclasses:silent
    禁用所有共享的类消息，包括错误消息。显示无法恢复的错误消息，阻止VM初始化。

    snapshotCache （缓存实用程序）

    （不是Windows）
    -Xshareclasses:snapshotCache

    创建现有非持久共享缓存的快照文件。快照具有与共享缓存相同的名称和位置，如name和cacheDir子选项。拍摄快照时可以使用共享缓存，但是当VM尝试写入共享缓存时，VM将被阻止，同时将缓存数据复制到文件中。

    通常，在重新初始化系统后，快照文件用于通过以下方式将非持久性缓存的副本还原到共享内存中： restoreFromSnapshot子选项。由于此过程会从共享内存中删除所有非持久性缓存，因此从快照文件还原缓存可以提高VM启动性能，因为不必重新创建共享缓存的内容（包括类和AOT代码） 。

    仅当用户具有足够的权限来创建目标快照文件时，才能创建快照。如果已经存在相同名称的快照，则它将被覆盖。在支持持久性缓存的平台上， nonpersistent必须指定suboption才能创建快照。有关删除快照文件的信息，请参阅destroySnapshot和destroyAllSnapshots子选项。

    utilities

        -Xshareclasses:utilities
    可以添加到命令行末尾以禁用类数据共享。此子选项将覆盖先前在命令行中找到的类共享参数。这个子选项就像none ，但不禁用共享类实用程序API。有关共享类实用程序API的更多信息，请参阅获取有关共享缓存的信息 。

    verbose

        -Xshareclasses:verbose
    提供有关缓存I / O活动的详细输出，列出有关存储和找到的类的信息。每个类加载器都有一个唯一的ID（引导加载器始终为0），输出显示工作中的类加载器层次结构，其中类加载器向父级询问类是否自己找不到。通常会看到许多失败的请求。对于类加载器层次结构，此行为是预期的。标准选项-verbose:class如果启用了类共享，还将启用类共享详细输出。

    verboseAOT

        -Xshareclasses:verboseAOT
    当找到已编译的AOT代码或将其存储在缓存中时，启用详细输出。AOT代码是启发式生成的。您可能根本看不到为小型应用程序生成的任何AOT代码。您可以使用禁用AOT缓存noaot子选项。有关产生的消息列表，请参见消息指南 。

    verboseHelper

        -Xshareclasses:verboseHelper
    为Java Helper API启用详细输出。此输出显示您的类加载器如何使用Helper API。

    verboseIO

        -Xshareclasses:verboseIO
    提供有关缓存I / O活动的详细输出，列出有关存储和找到的类的信息。每个类加载器都有一个唯一的ID（引导加载器始终为0），输出显示工作中的类加载器层次结构，其中，类加载器必须先向其父提供类，然后才能自己加载。通常会看到许多失败的请求。对于类加载器层次结构，此行为是预期的。

    方法规范语法

    以下示例说明了在使用以下方法时如何指定多个方法规范findAotMethods ， invalidateAotMethods ， 要么revalidateAotMethods子选项。

    每个方法规范定义如下：

    <packagename>/<classname>[.<methodname>[(<parameters>)]]

    如果要在一个选项中包含多个方法规范，请用逗号分隔规范并将所有规范括在{大括号}中。例如：

    {<packagename/classname>}[.{<methodname>}[({<parameters>})]]
    • 您可以在大多数地方使用星号（*）作为通配符。
    • 您可以使用感叹号（！）在规范之前表示“ 除此以外的所有内容”。
    • 参数是可选的，但如果指定，则应将其括在括号中，并且必须使用以下本机签名格式：
      • B对于字节
      • C对于字符
      • D对于双
      • F浮动
      • I对于int
      • J很久了
      • S简而言之
      • Z对于布尔
      • L;对于对象
      • [在签名意味着数组之前

    如果要指定参数来区分方法，则可以使用-Xshareclasses:findAotMethods=* （使用通配符）以列出所有参数变体。从输出中复制所需方法的签名。例如，参数的签名

    (byte[] bytes, int offset, int length, Charset charset)

    是

    ([BIILjava/nio/charset/Charset;)

    这里有些例子：

    方法签名 火柴...
    * 所有AOT方法。
    java/lang/Object 中的所有AOT方法java.lang.Object类
    java/util/* 中的所有AOT类和方法java.util包
    java/util/HashMap.putVal 所有putVal中的方法java.util.HashMap类
    java/util/HashMap.hash(Ljava/lang/Object;) 私人的java.util.HashMap.hash(java.lang.Object)方法
    *.equals 所有equals所有类中的方法
    {java/lang/Object,!java/lang/Object.clone} 中的所有方法java.lang.Object除clone
    {java/util/*.*(),java/lang/Object.*(*)} 所有没有输入参数的类或方法java.util包以及其中的所有方法java.lang.Object
    {java/util/*.*(),!java/util/*.*()} 没有。
