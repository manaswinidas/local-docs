    -Xgcpolicy

    通过指定不同的垃圾收集策略来控制垃圾收集器的行为。

    句法

        -Xgcpolicy:<parameter>

    参量

    参数 默认
    balanced
    gencon 是
    metronome （仅适用于AIX®，Linux®x86）
    nogc
    optavgpause
    optthruput

    指定您希望OpenJ9 VM使用的垃圾收集策略：

    balanced

        -Xgcpolicy:balanced
    平衡的策略策略使用标记，清除，紧凑和分代样式的垃圾回收。并发标记阶段已禁用；使用并发垃圾回收技术，但不以对其他策略实现并发标记的方式使用。的balanced策略对Java™堆使用基于区域的布局。对这些区域进行单独管理，以减少大堆上的最大暂停时间，并提高垃圾收集的效率。该策略尝试通过匹配对象分配和生存率来避免全局收集。

    如果您由于全局垃圾回收（尤其是压缩）而导致应用程序暂停时间出现问题，则此策略可能会提高应用程序性能。如果您使用具有非统一内存体系结构（NUMA）特性的大型系统（仅x86和POWER™平台），则平衡策略可能会进一步提高应用程序吞吐量。

    注意：如果您在使用默认值的Docker容器中使用平衡GC策略seccomp Docker配置文件，您必须使用以下命令启动容器--security-opt seccomp=unconfined利用NUMA特性。如果您在Kubernetes中运行，则不需要这些选项，因为unconfined是默认设置（请参见Seccomp ）。

    有关此策略（包括何时使用）的更多信息，请参阅平衡垃圾收集策略 。

    默认值和选项

    初始堆大小为Xmx / 1024 ，四舍五入到最接近的2的幂，其中Xmx是可用的最大堆大小。您可以通过指定-Xms命令行上的选项。

    还可以在命令行上使用以下选项指定-Xgcpolicy:balanced ：

    • -Xclassgc

    • -Xdisableexcessivegc

    当指定以下选项时，其行为是不同的-Xgcpolicy:balanced ：

    -Xcompactgc
    当接收到System.gc（）调用时会发生压缩（默认）。压缩总是在所有其他收集类型上进行。
    -Xnocompactgc
    收到System.gc（）调用时，不会发生压缩。压缩总是在所有其他收集类型上进行。

    gencon

        -Xgcpolicy:gencon
    分代并发策略（默认）将并发标记阶段与分代垃圾回收结合使用，以帮助最大程度地减少在任何垃圾回收暂停中花费的时间。对于具有许多短期对象的应用程序（例如事务性应用程序），此策略特别有用。暂停时间可能比使用optthruput政策，同时仍然产生良好的吞吐量。堆碎片也减少了。

    metronome （仅AIX，仅Linux x86）

        -Xgcpolicy:metronome

    节拍器策略是具有较短暂停时间的增量式确定性垃圾收集器。依赖精确响应时间的应用程序可以避免垃圾收集活动可能导致的长时间延迟，从而利用此技术。节拍器策略在特定的硬件和操作系统配置上受支持。

    欲了解更多信息，请参阅使用Metronome垃圾收集 。

    默认值和选项

    -Xgc:synchronousGCOnOOM | -Xgc:nosynchronousGCOnOOM
    发生垃圾回收的一种情况是堆内存不足。如果堆中没有可用空间，请使用-Xgc:synchronousGCOnOOM在垃圾回收删除未使用的对象的同时停止您的应用程序。如果可用空间再次用完，请考虑降低目标利用率，以使垃圾收集有更多时间来完成。设置-Xgc:nosynchronousGCOnOOM表示当堆内存已满时，您的应用程序将停止并发出内存不足消息。默认是-Xgc:synchronousGCOnOOM 。
    -Xnoclassgc
    禁用类垃圾回收。此选项关闭与OpenJ9 VM不再使用的Java类关联的存储的垃圾回收。默认行为是-Xnoclassgc。
    -Xgc:targetPauseTime=N
    设置垃圾收集暂停时间，其中N是时间（以毫秒为单位）。指定此选项后，GC的暂停时间不得超过指定的值。如果未指定此选项，则默认暂停时间设置为3毫秒。例如，使用-Xgc:targetPauseTime=20导致GC在操作过程中暂停不超过20毫秒。
    -Xgc:targetUtilization=N

    将应用程序利用率设置为N% ;垃圾收集器会尝试使用每个时间间隔的最多（100-N）％。合理的值在50-80％的范围内。分配率低的应用程序可能能够以90％的速度运行。默认值为70％。

    本示例显示堆内存的最大大小为30 MB。垃圾收集器尝试使用每个时间间隔的25％，因为应用程序的目标利用率为75％。

    java -Xgcpolicy：metronome -Xmx30m -Xgc：targetUtilization = 75测试

    -Xgc:threads=N
    指定要运行的GC线程数。默认值为可用于该进程的处理器核心数。您可以指定的最大值是操作系统可用的处理器数量。
    -Xgc:verboseGCCycleTime=N

    N是摘要信息应转储的时间（以毫秒为单位）。

    注意：循环时间并不意味着摘要信息会在那时精确地转储，而是在满足该时间标准的最后一个垃圾收集事件过去时进行。

    -Xmx
    指定Java堆大小。与其他垃圾回收策略不同，实时Metronome GC不支持堆扩展。没有初始或最大堆大小选项。您只能指定最大堆大小。

    nogc

        -Xgcpolicy:nogc

    该策略仅处理内存分配和堆扩展，但不回收任何内存。如果可用的Java堆耗尽，则OutOfMemoryError异常触发，虚拟机停止。

    因为没有GC暂停，并且消除了分配的大部分开销，所以对运行时性能的影响已降至最低。因此，该策略为“无垃圾”应用程序提供了好处。有关某些可能的用例，请参见以下部分“何时使用nogc”。

    在将以下任何一种技术用于以下情况时，您应格外小心： nogc因为根据该策略永远不会释放内存：
    -完成
    -直接内存访问
    -弱，软和幻像引用

    也可以通过以下方式启用此策略-XX:+UseNoGC选项。

    有关更多详细信息，请参见JEP 318：Epsilon：无操作垃圾收集器 。

    何时使用nogc

    对于大多数Java应用程序，你不应该使用nogc 。但是，在某些特定情况下可能合适：

    开发过程中的测试

    • GC性能。使用nogc在测试其他GC策略的性能（包括提供低延迟基线）时作为基线。

    • 应用程序内存。使用nogc测试分配内存的设置。如果您使用-Xmx设置不应超过的堆大小，如果您的应用程序尝试超过内存限制，则会因堆转储而崩溃。

    运行具有最少GC需求或没有GC需求的应用程序

    • 你可能会用nogc当应用程序的生存期如此之短，以至于分配的内存永远不会耗尽时，因此运行完整的GC周期将浪费资源。

    • 同样，当对内存应用程序有充分的了解或很少需要回收内存时，您可能希望避免不必要的GC周期，并依靠故障转移机制偶尔根据需要重新启动VM。

    optavgpause

        -Xgcpolicy:optavgpause
    “优化暂停时间”策略使用并发标记和并发扫描阶段。暂停时间比optthruput ，但应用程序的吞吐量降低了，因为在应用程序运行时正在进行一些垃圾回收工作。如果堆大小较大（在64位平台上可用），请考虑使用此策略，因为此策略限制了增加堆大小对垃圾回收暂停时间的影响。但是，如果您的应用程序使用许多短期对象，则gencon政策可能会产生更好的效果。

    optthruput

        -Xgcpolicy:optthruput
    “优化吞吐量”策略禁用并发标记阶段。该应用程序在全局垃圾回收期间停止，因此可能会出现长时间的暂停。当主要的性能目标是高应用程序吞吐量而不是短暂的垃圾收集暂停时，此配置通常用于大型堆应用程序。如果您的应用程序不能忍受长时间的垃圾回收暂停，请考虑使用其他策略，例如gencon 。
