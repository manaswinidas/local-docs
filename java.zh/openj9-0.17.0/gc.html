    垃圾收集

    VM中的内存管理过程由分配器和垃圾收集器（GC）处理。这些组件在保留用于VM处理的内存区域（称为Java堆）上运行。分配器为Java对象分配Java堆的区域。当对象具有从根引用（例如在线程堆栈中找到的根引用）开始的对它们的引用链时，它们被视为活动对象。当该引用或指针不再存在时，这些对象被视为垃圾 。垃圾收集器的作用是管理Java堆中的存储并通过删除垃圾来回收内存。

    分配者

    分配器是内存管理的一个组成部分，通常被垃圾收集任务所掩盖。但是，这是整个过程的关键，即使不是很小的一部分。分配器管理可用内存池以及如何使用可用内存。它还负责根据应用程序，类库或VM的请求为对象分配Java堆中的存储区域。每个分配都需要一个堆锁来停止尝试同时访问同一内存区域的并发线程。分配对象后，将释放堆锁。如果没有足够的空间来分配对象，则分配失败，将释放堆锁，并调用GC。如果GC设法恢复堆上的某些空间，则分配器可以恢复操作。如果GC无法恢复足够的空间，它将返回一个OutOfMemoryError例外。

    为每个分配获取堆锁将是一项繁重的操作，并且会影响性能。若要解决此问题，将小对象分配给线程本地堆（TLH）。

    线程本地堆（TLH）

    为了提高性能，在堆中为不同的线程保留了分配缓存。这些分配缓存被称为线程本地堆（TLH），并允许每个线程从其缓存中分配内存而无需获取堆锁。TLH通常用于小于512字节的小型对象（在64位VM上为768字节），但是如果有足够的空间，则可以从缓存中分配较大的对象。

    如果线程分配了很多对象，则分配器会为该线程分配更大的TLH，以减少堆锁上的争用。

    TLH是预定义的，初始默认大小为2 KB。每次TLH刷新时，该线程的请求大小都会增加（默认4 KB）。请求的大小可以增长到预定义的最大值（默认为128 KB）。

    在每个GC周期之后，每个线程的TLH请求大小都会减少，有时会减少多达50％，以考虑那些降低了线程分配率并且不再需要大型TLH的线程。

    对于非常不活跃的线程，请求的大小甚至可以降至初始值以下，降至预定义的最小值（512/768字节）。对于非常活跃的线程，在下一次GC发生之前，可能已达到TLH请求的最大大小。

    较大的TLH可以帮助减少堆锁争用，但也可能降低堆利用率并增加堆碎片。

    以下选项控制请求的TLH大小：

    • -Xgc:tlhMaximumSize=
    • -Xgc:tlhInitialSize=
    • -Xgc:tlhIncrementSize=

    通常，当最大TLH大小增加时，您还应按比例增加增量，以便活动线程可以更快地达到最大请求的TLH大小。

    垃圾收集

    为了防止应用程序内存不足，必须回收Java堆中不再需要的对象。此过程称为垃圾收集（GC）。收集垃圾时，垃圾收集器必须获得对堆的独占访问权，这将导致应用程序在清理完成时暂停。此暂停通常称为“世界停止”暂停，因为应用程序必须暂停直到该过程完成。通常，GC过程的第一步是标记可访问的对象，这意味着它们仍在使用中。下一步是清除未标记的对象以回收内存。除非堆变得非常零碎，否则并不总是需要执行最后一步，即压缩堆。

    Eclipse OpenJ9具有许多针对不同类型的应用程序和工作负载设计的GC策略。选择正确的策略很大程度上取决于您的使用情况和性能目标。

    代际并发政策

    如果您有一个事务处理应用程序，其中包含许多短期对象，则“世代并发（ -Xgcpolicy:gencon ）GC策略可能是最合适的，其目的是在不影响吞吐量的情况下最大程度地减少GC暂停时间。这是VM使用的默认策略，因此，如果要使用它，则在启动应用程序时无需在命令行上指定它。

    随着gencon根据政策，Java堆分为两个主要区域： 托儿所区域（创建新对象的区域）和保有权区域（ 保管区域），如果对象达到保有权年龄，则将其移动。

    保育区又分为两个区域， 分配空间和幸存者空间。下图说明了GC的过程，该过程显示了4个主要事件的序列：

    1. 在分配空间中创建对象。
    2. 分配空间已满。
    3. 将运行本地GC清理过程，将可到达的对象复制到幸存者空间中，或者如果它们已经达到使用期限，则复制到使用区域中。任何无法到达的物体均保持不变，然后清除。
    4. 分配空间和幸存者空间交换角色。原始幸存者空间将成为在其中创建新对象的分配空间，而原始分配空间将成为为下一个本地GC清理过程准备的幸存者空间。

    分配空间和幸存空间的相对大小通过称为倾斜的技术动态调整。首次创建苗圃区域时，它会在分配空间和幸存者空间之间平均分配 。如果在执行GC清除过程后， 幸存者区域所需的空间量较小，则可以通过倾斜调整两个空间之间的边界。例如，如果幸存者空间仅需要苗圃面积的10％，则调整倾斜比率以将90％的苗圃面积分配给分配空间。如果有更多空间用于新对象，则可以延迟垃圾回收。

    对象的使用期限由VM确定，反映了对象在分配空间和幸存者空间之间被复制的次数。年龄在1到14之间，并且VM会根据苗圃区域中使用的总空间量对它进行动态调整。例如，如果对象的保有权年龄为5，则已在分配空间和幸存者空间之间来回复制了5次。如果VM根据苗圃区域中剩余空间的百分比将使用期限设置为5，则下一次清理将对象从苗圃移动到保管区域。您可以设置的初始年龄任期 -Xgc:scvTenureAge选项。您还可以通过设置以下选项来防止VM动态调整使用期限： Xgc:scvNoAdaptiveTenure选项，以便在VM的整个运行期间保持初始使用期限。

    在使用权区域内，新对象被分配到小对象区域（SOA）中，如先前的图所示（请参见3.）。为无法分配到SOA中以最大程度减少碎片的大于64 KB的对象预留了一个较大的对象区域（LOA）。LOA是默认分配的，但如果没有填充，则会在几个GC周期后减少和删除。为了防止创建LOA，您可以指定-Xnoloa启动应用程序时在命令行上使用选项。当使用权区接近满时，将触发全局GC。

    本地GC清除通过在幼儿园区域中频繁地回收内存来减少停顿时间，对于具有许多短期对象的事务性应用程序，该区域具有可回收的最大空间。但是，随着时间的流逝， 保有权区域可能会变满。因此，虽然当地的GC清理过程中对幼儿园区域经营，并发全球GC过程中也一起运行正常的程序执行，以纪念和任期区域中删除不可达的对象。这两种GC方法相结合，可以在较短的暂停时间和稳定的吞吐量之间取得良好的平衡。

    一种特殊的模式gencon该策略被称为并发清除 （ -Xgc:concurrentScavenge ），旨在通过与运行的应用程序线程并行收集托儿所垃圾来最大程度地减少停顿世界的时间。可以通过基于硬件的支持和基于软件的支持来启用此模式。

    • 基于硬件的支持：（IBMZ®和z /OS®上的Linux）此模式在具有保护存储（GS）功能的IBM z14™和更高版本的大型机系统上有效。GS设备提供基于硬件的支持，以检测应用程序何时访问对对象的潜在过时引用。这意味着垃圾收集器可以在不停止应用程序的情况下开始处理堆的各个部分中的对象，因为GS设备可以立即发现对对象的访问并发送通知。准备好要清除的对象可以移动，并且可以重置对其的引用。您可以在以下博客文章中了解有关此模式的更多信息：

      • 通过并发清除和受保护的存储设施减少垃圾收集的暂停时间
      • 如何使用受保护的存储设施进行并发清理

    • 基于软件的支持：（64位：Linux on（x86-64，POWER，IBMZ®），AIX®和z /OS®）借助基于软件的支持，可以启用Concurrent Scavenge，而无需任何必备硬件尽管性能吞吐量不如基于硬件的支持。

    有关启用并发清除的更多信息，请参见-Xgc：concurrentScavenge选项。

    其他政策

    OpenJ9具有以下替代GC策略：

    • -Xgcpolicy:balanced将Java堆划分为多个区域，对这些区域进行单独管理，以减少大堆上的最大暂停时间，并提高垃圾回收的效率。该政策的目的是通过匹配对象分配和生存率来避免全局收集。如果由于全局垃圾回收（尤其是压缩）而导致应用程序暂停时间出现问题，则此策略可能会提高应用程序性能，尤其是在具有非统一内存体系结构（NUMA）特征的大型系统（x86和POWER平台）上。
    • -Xgcpolicy:metronome设计用于需要精确响应时间的应用。垃圾收集会以小的可中断步骤进行，以避免世界停顿。该策略仅在x86 Linux和AIX平台上可用。
    • -Xgcpolicy:nogc仅处理内存分配和堆扩展，但不回收任何内存。因此，GC对运行时性能的影响已降至最低，但是如果可用的Java堆耗尽了， OutOfMemoryError异常触发，虚拟机停止。
    • -Xgcpolicy:optavgpause使用并发的标记和扫描阶段，这意味着与optthruput相比，减少了暂停时间，但以牺牲一些性能吞吐量为代价。
    • -Xgcpolicy:optthruput通过禁用并发标记阶段针对吞吐量进行了优化，这意味着在进行垃圾收集时，应用程序将停止长时间的停顿。当主要的性能目标是高应用程序吞吐量而不是短暂的垃圾收集暂停时，您可以考虑使用此策略。

    有关这些垃圾收集策略和选项的更多信息，请参见-Xgcpolicy 。

    故障排除

    您可以通过打开详细的垃圾收集日志来诊断垃圾收集操作的问题。默认情况下，信息被打印到STDERR，但是可以通过指定-Xverbosegclog选项。日志文件包含有关所有操作的详细信息，包括初始化，停产处理，完成，引用处理和分配失败。有关更多信息，请参见详细垃圾回收。

    如果详细日志没有提供足够的信息来帮助您诊断GC问题，则可以使用GC跟踪来更精细地分析操作。有关更多信息，请参见-Xtgc 。
