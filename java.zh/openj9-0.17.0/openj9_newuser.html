    OpenJ9的新手？

    Eclipse OpenJ9虚拟机（VM）实现了Java虚拟机规范 。大多数Java应用程序都应在包含OpenJ9 VM的OpenJDK上运行，而无需进行任何更改。但是，由于它是一个独立的实现，因此与HotSpot VM（它是默认的OpenJDK VM，也包含在Oracle JDK中）相比有一些差异。

    命令行选项

    尽管OpenJ9实现了自己的命令行界面，但VM出于兼容性考虑识别并接受了许多Hotspot选项。任何-XX:默认情况下，VM无法识别的选项将被忽略，以防止应用程序启动失败。您可以使用-XX：-IgnoreUnrecognizedXXColonOptions选项关闭此行为。

    有关兼容选项的列表，请参见命令行选项部分中的切换到OpenJ9 。

    垃圾收集政策

    Eclipse OpenJ9具有许多针对不同类型的应用程序和工作负载设计的GC策略。默认情况下，OpenJ9使用分代并发（ gencon ）GC策略，最适合具有许多短期对象的事务性应用程序。该策略旨在在不影响吞吐量的情况下最小化GC暂停时间。如果您使用的是Java 8， gencon政策类似于ParallelGC策略，这是默认的HotSpot策略。如果您使用的是Java 11，则OpenJ9平衡（ balanced ）策略与默认的HotSpot策略最相似。

    如果您有不同类型的工作负载，则可能要选择其他GC策略。有关可用策略的摘要，请参见垃圾回收 。有关OpenJ9 GC策略之间的差异以及如何为您的应用程序选择正确的策略的更多信息，请参阅Eclipse OpenJ9：垃圾回收的权衡和使用OpenJ9进行调整 。要使用其他策略启动应用程序，请使用-Xgcpolicy命令行上的选项。

    操作工具

    如果您是Java应用程序开发人员，或者负责管理Java运行时环境的大型服务器或桌面部署，则可能会使用许多工具进行监视，管理和故障排除。由于OpenJ9是独立的实现，因此它针对这些领域采用了自己的方法，并且在某些情况下还提供了自己的独特工具。

    在其他情况下，已添加工具以与参考实现兼容，但是这些工具的行为可能与HotSpot中的等效工具不同。有关这些工具的列表，请参见“工具”部分中的“ 切换到OpenJ9 ”。

    转储，日志和跟踪文件

    OpenJ9包含广泛的跟踪和调试功能，以帮助识别，隔离和解决运行时问题。

    • 转储文件：默认情况下，会针对某些事件生成各种类型的转储，也可以使用以下命令针对整个事件范围触发com.ibm.jvm.Dump API或通过指定-Xdump命令行上的选项。转储包括Java转储 ， 堆转储 ， 系统转储 ，JIT转储，堆栈转储和快照转储（跟踪点数据）。有关更多信息，请参见-Xdump选项。
    • 详细的日志文件： OpenJ9的某些组件还可以产生详细的输出或日志文件以帮助确定问题，包括类数据共享 ， 垃圾回收和JIT编译器 。
    • 跟踪文件： OpenJ9实现包含用于记录信息和特殊情况的广泛跟踪点，对性能的影响最小。默认情况下，某些跟踪点处于启用状态。其他可以按需启用。有关更多信息，请参见-Xtrace跟踪Java应用程序和VM的选项，以及-Xtgc跟踪垃圾收集的选项。

    如果您熟悉如何将HotSpot用作Oracle JDK或OpenJDK的一部分，则您可能会使用VM随附的监视和诊断工具。 OpenJ9采用了另一种方法来提供相似的数据。 Java转储文件无需运行许多不同的工具来获取不同的信息，而是在一个地方提供了一组全面的信息。您可以在OpenJ9 Java转储中找到以下信息：

    • VM在其上运行的系统和可用资源。
    • Java执行环境，包括从命令行设置的选项。
    • VM使用的本机内存，按VM组件细分。
    • VM中对象堆和内部VM结构（例如JIT代码缓存）的内存使用情况。
    • 在运行时锁定保护共享资源的操作。
    • Java线程，本机线程和堆栈跟踪。
    • 挂钩接口，用于性能分析。
    • 有关共享类缓存的详细信息（如果使用）。
    • 有关类加载器的详细信息，以及已加载的库和类的列表。

    有关更多信息，请参见Java dump 。

    工具类

    OpenJ9支持在Eclipse市场中找到的许多监视和诊断工具。每个工具都提供一个图形用户界面来帮助您可视化数据，并且在某些情况下，可以提供调整或调试建议。

    • Health Center：以最小的网络开销提供对正在运行的应用程序的实时监视。您可以监视整个操作范围，包括类加载，CPU使用率，GC堆和暂停时间，I / O活动，锁争用，方法跟踪，本机内存使用率，性能分析和活动线程。有关更多信息，请阅读Health Center文档 。
    • 垃圾收集内存Vizualizer（GCMV）：绘制一段时间内的GC和本机内存数据。您可以查看数据并将其保存为报告，原始日志，列表数据或图形格式。该工具可使用各种可视格式显示的数据来帮助诊断问题，例如内存泄漏。还提供了调整建议。有关更多信息，请阅读GCMV文档 。
    • 内存分析器：检查Java对象堆以帮助查找内存泄漏或减少内存消耗。可通过DTFJ界面获得对OpenJ9的支持（从Eclipse帮助菜单进行安装；安装新软件>使用“ IBM Diagnostic Tool Framework for Java”> IBM Monitoring and Diagnostic Tools> Diagnostic Tool Framework for Java）。可以在项目网站页面上找到有关Eclipse MAT的更多信息。
    • 交互式诊断数据资源管理器（IDDE）：可以替代OpenJ9 转储查看器的 GUI，它可以检查OpenJ9系统转储的内容。有关更多信息，请阅读IDDE文档 。

    如果您熟悉如何将HotSpot用作Oracle JDK或OpenJDK的一部分，则Java VisualVM实用程序在功能上类似于Health Center。正式不支持HotSpot附带的大多数其他工具，但是OpenJ9中通过命令行选项，转储代理和AttachAPI提供了等效功能。

    介面

    OpenJ9提供以下接口，可用于监视和诊断目的：

    • JVMTI ： OpenJ9支持Java虚拟机工具接口（JVMTI），并提供扩展，这些扩展允许JVMTI工具获取诊断信息或触发VM中的诊断操作。有关此接口的更多信息，请参见Java虚拟机工具接口 。
    • DTFJ接口 ： Java诊断工具框架（DTFJ）API允许编写自定义应用程序，这些应用程序可以访问系统转储或Java转储中的各种信息。DTFJ可以与Eclipse Memory Analyzer Toolkit（MAT）一起使用，以检查Java对象堆中是否存在内存泄漏并减少内存消耗。有关DTFJ的更多信息，请参见Java诊断工具框架 。
    • java.lang.management API： OpenJ9为该标准API提供了MXBean附加和扩展，使您能够使用JConsole之类的工具来监视和管理Java应用程序。有关更多信息，请参阅IBM®Knowledge Center中的MBean和MXBean 。

    如果您熟悉如何将HotSpot用作Oracle JDK或OpenJDK的一部分，则可以利用com.sun.management接口。尽管OpenJ9实现了其中一些接口，但其中一些是特定于HotSpot VM的。下表指示了可用于OpenJ9中等效功能的替代类或机制：

    热点特定的课程 OpenJ9的替代品
    HotSpotDiagnosticMXBean OpenJ9DiagnosticsMXBean （用于堆转储）
    MissionControl 使用健康中心
    MissionControlMXBean 使用健康中心
    ThreadMXBean JvmCpuMonitorMXBean （用于线程时间）
    VMOption OpenJ9 Java转储（选项-Xdump:java ）
    DiagnosticCommandMBean 没有

    注意： OpenJ9实现以下功能com.sun.management接口： GarbageCollectorMXBean ， GarbageCollectionNotificationInfo ， GcInfo ， OperatingSystemMXBean ， UnixOperatingSystemMXBean 。

    其他差异

    本主题描述了HotSpot VM和Eclipse OpenJ9 VM之间的区别。因此，如果您当前正在将OpenJDK与默认的Hotspot VM一起使用，并且想要切换为将OpenJDK与OpenJ9 VM一起使用，则这些可能是您唯一需要关注的区别。但是，如果您使用的是Oracle JDK，则可能需要从AdoptOpenJDK社区了解组成Oracle JDK或OpenJDK的其他组件之间的区别。有关更多信息，请阅读《 迁移指南》 。
