    -X检查

    您可以使用-Xcheck选项以在OpenJ9虚拟机（VM）启动期间运行检查，例如内存检查或对JNI功能的检查。

    句法

        -Xcheck:<parameter>

    参量

    参数 影响
    classpath 检查类路径并报告错误，例如缺少目录或JAR文件。
    dump 检查操作系统中可能会截断系统转储的设置。（仅AIX®和Linux®）
    gc 对垃圾收集运行其他检查。
    jni 对JNI函数运行其他检查。
    memory 使用严格的检查来确定VM内部的内存泄漏，这些检查会导致VM在故障时退出。
    vm 在VM上执行其他检查。

    classpath

        -Xcheck:classpath
    检查类路径并报告错误，例如缺少目录或JAR文件。

    dump

    仅AIX和Linux

        -Xcheck:dump

    在VM启动期间检查操作系统设置。如果操作系统具有可能会截断系统转储的设置，则会发出消息。

    在AIX系统上，以下消息是可能的：

    • JVMJ9VM133W The system core size hard ulimit is set to , system dumps may be truncated
    此消息表明已设置AIX操作系统用户限制，以将系统转储的大小限制为指示的值。如果虚拟机产生了系统转储，则它可能会被截断，因此在调查崩溃原因和其他问题方面的价值将大大降低。有关如何在AIX上设置用户限制的更多信息，请参阅在AIX和Linux系统上设置系统资源限制 。
    • JVMJ9VM134W The system fullcore option is set to FALSE, system dumps may be truncated
    此消息表明AIX操作系统Enable full CORE dump选项设置为FALSE 。此设置可能会导致系统转储被截断。有关如何在AIX上正确设置此选项的更多信息，请参阅在AIX和Linux系统上设置系统资源限制 。

    在Linux系统上，可能显示以下消息：

    • JVMJ9VM133W The system core size hard ulimit is set to , system dumps may be truncated.
    此消息表明Linux操作系统用户限制已设置为将系统转储的大小限制为指示的值。如果虚拟机产生了系统转储，则它可能会被截断，因此在调查崩溃原因和其他问题时的价值将大大降低。查看为您的操作系统提供的文档以正确配置ulimits 。有关更多信息，请参阅在AIX和Linux系统上设置系统资源限制 。
    • JVMJ9VM135W /proc/sys/kernel/core_pattern setting "|/usr/libexec/abrt-hook-ccpp %s %c %p %u %g %t e" specifies that core dumps are to be piped to an external program. The JVM may be unable to locate core dumps and rename them.
    此消息表示在操作系统中将外部程序abrt-hook-ccpp配置为拦截生成的所有系统转储文件。该程序是自动错误报告工具（ABRT）的一部分。
    有关更多信息，请参见自动错误报告工具 。该工具可能会通过重命名或截断系统转储来干扰VM的系统转储文件处理。查看ABRT工具的配置以及该工具编写的消息。 /var/log/messages 。如果从VM生成系统转储时出现问题，请考虑禁用ABRT。
    • JVMJ9VM135W /proc/sys/kernel/core_pattern setting "|/usr/share/apport/apport %p %s %c" specifies that core dumps are to be piped to an external program. The JVM may be unable to locate core dumps and rename them.
    此消息意味着在操作系统中将外部程序apport配置为拦截生成的任何系统转储文件。有关此工具的更多信息，请参阅： 分配该工具可能会通过重命名或截断系统转储来干扰VM的系统转储文件处理。查看Apport工具的配置以及该工具在其中编写的消息/var/log/apport.log 。如果从VM生成系统转储时出现问题，请考虑禁用Apport工具。
    • JVMJ9VM136W "/proc/sys/kernel/core_pattern setting "/tmp/cores/core.%e.%p.%h.%t " specifies a format string for renaming core dumps. The JVM may be unable to locate core dumps and rename them.
    此消息表明Linux /proc/sys/kernel/core_pattern选项设置为重命名系统转储。操作系统转储名称中使用的令牌可能会干扰VM的系统转储文件处理，尤其是与VM中指定的文件名冲突-Xdump选项。如果从VM生成系统转储时出现问题，请考虑更改/proc/sys/kernel/core_pattern设置为默认值core 。

    gc

        -Xcheck:gc[:help][:<scan options>][:<verify options>][:<misc options>]
    对垃圾收集运行其他检查。默认情况下，不进行任何检查。有许多可用的扫描，验证和其他子选项。如果未指定任何选项，则将运行所有可能的扫描和验证子选项，以及其他详细和检查子选项。有关更多信息，请参见输出-Xcheck:gc:help 。

    jni

        -Xcheck:jni[:help][:<option>]
    对JNI函数运行其他检查。默认情况下，不进行任何检查。有关更多信息，请参见输出-Xcheck:jni:help 。

    memory

        -Xcheck:memory[:<option>]
    通过使用严格检查来确定VM内部的内存泄漏，这些检查会导致VM在故障时退出。
    限制：您不能包括-Xcheck:memory在选项文件中（请参阅-Xoptionsfile ）。

    可用参数如下：

    :all
    （如果未指定任何选项，则为默认值）启用检查每个释放和分配调用中所有分配和释放的块的功能。堆的检查是最彻底的。它通常会导致VM在引起几乎所有与内存相关的问题后立即退出。此选项对性能的影响最大。
    :callsite=

    每次显示呼叫站点信息 。取消分配不计算在内。呼叫地点信息显示在表格中，其中包含每个呼叫地点的单独信息。统计数据包括：

    • 自上次报告以来分配和免费请求的数量和大小。
    • 每个站点中负责最大分配的分配请求数。

    呼叫站点显示为sourcefile:linenumber用于C代码，用于汇编代码的汇编函数名称。

    不提供呼叫站点信息的呼叫站点将累积到“未知”条目中。

    :failat=

    导致内存分配失败（返回NULL） 。例如，设置到13会使第14个分配返回NULL。取消分配不计算在内。使用此选项可确保VM代码可靠地处理分配失败。此选项对于检查分配站点的行为很有用，而不是设置特定的分配限制。

    :ignoreUnknownBlocks
    忽略释放未使用内存分配的内存的尝试-Xcheck:memory工具。而是在运行结束时输出的-Xcheck：memory统计信息指示已释放的“未知”块的数量。
    :mprotect=[top|bottom]
    锁定受支持平台上的内存页面，如果在访问分配的块之前或之后进行填充以进行读取或写入，则会导致程序停止。在返回给用户的块的每一侧都锁定了一个额外的页面。
    如果您不要求一页内存的精确倍数，则不会锁定内存一侧的区域。的top和bottom选项控制存储区域的哪一侧被锁定。 top将您的内存块对齐到页面顶部（较低的地址），因此缓冲区不足会导致应用程序失败。 bottom将您的内存块对齐到页面底部（地址较高），以便缓冲区溢出会导致应用程序失败。
    标准填充扫描可在使用时检测缓冲区不足top使用时缓冲区溢出bottom 。
    :nofree
    保留已使用的块列表，而不是释放内存。检查此列表以及当前已分配块的列表，以检查每次分配和释放时的内存损坏。使用此选项可检测到悬空指针（释放目标内存后被“取消引用”的指针）。该选项不能与长时间运行的应用程序（例如WebSphere®Application Server）可靠地配合使用，因为VM永远不会重用或释放“释放”的内存。
    :noscan
    检查未释放的块。此选项对性能几乎没有影响，但是未检测到内存损坏。此选项仅与subAllocator ， callsite和callsitesmall 。
    :quick
    仅启用块填充，并用于检测基本堆损坏。每个分配的块都填充有前哨字节，这些字节在每次分配和释放时都会进行验证。块填充比检查每个块的默认设置要快，但效果不如预期。
    :skipto=
    使程序仅检查在之后发生的分配 。取消分配不计算在内。当早期分配不会引起内存问题时，使用此选项可以加快VM启动。VM在启动期间执行大约250多个分配。
    :subAllocator[=]
    为所有VM分配一个专用且连续的内存区域。此选项有助于确定用户JNI代码还是VM造成内存损坏。虚拟机损坏subAllocator堆表明VM引起了问题；用户分配的内存损坏表明用户代码正在损坏内存。通常，用户和VM分配的内存是交错的。
    :zero
    新分配的块设置为0，而不是用0xE7E7xxxxxxxxE7E7图案。将这些块设置为0可帮助您确定呼叫站点是否期望内存归零，在这种情况下，分配请求后跟memset(pointer, 0, size) 。

    vm

        -Xcheck:vm[:<option>]
    在VM上执行其他检查。默认情况下，不执行检查。有关更多信息，请运行-Xcheck:vm:help 。
