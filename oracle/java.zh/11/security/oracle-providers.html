JDK提供程序简介

Java平台定义了一组跨越主要安全领域的API，包括加密，公共密钥基础结构，身份验证，安全通信和访问控制。这些API使开发人员可以轻松地将安全机制集成到其应用程序代码中。

Java密码体系结构（JCA）及其提供程序体系结构是Java开发工具包（JDK）的核心概念。假定读者对这种体系结构有扎实的理解。

提醒：出于历史原因和所提供服务的类型，JDK中的加密实现是通过几个不同的提供程序（“ SUN”，“ SunJSSE”，“ SunJCE”，“ SunRsaSign”）分发的。通用应用程序不应从特定提供商那里请求加密服务。那是：

getInstance("...", "SunJCE");  // not recommended

与

getInstance("...");            // recommended

否则，应用程序将绑定到其他Java实现上可能不可用的特定提供程序。他们也可能无法利用优先级高于特定请求提供者的可用优化提供者（例如，通过PKCS11的硬件加速器或Microsoft MSCAPI之类的本机OS实现）。

密码算法的导入限制

默认情况下，应用程序可以使用任何强度的密码算法。但是，由于某些地区的进口法规，您可能必须限制这些算法的强度。JDK在目录中提供了两组不同的权限策略文件/conf/security/policy决定了加密算法的强度。密码强度配置中提供了有关管辖区策略文件以及如何激活它们的信息。

请咨询您的进出口控制顾问或律师，以确定您所在位置的确切要求。

密码变换

的javax.crypto.Cipher.getInstance(String transformation)工厂方法生成Cipher对象使用形式algorithm / mode / padding的转换 。如果省略了模式/填充，则对于许多对称密码，SunJCE和SunPKCS11提供程序都将ECB用作默认模式，将PKCS5Padding用作默认填充。

建议使用完全指定算法，模式和填充的转换，而不要依赖默认值。默认值是特定于提供程序的，并且在提供程序之间可能有所不同。

注意：

ECB适用于单个数据块，并且可以并行化，但绝对不应用于多个数据块。

SunPKCS11提供程序

加密令牌接口标准（ PKCS＃11 ）为加密机制（例如硬件加密加速器和智能卡）提供了本机编程接口。正确配置后， SunPKCS11提供程序使应用程序可以使用标准JCA / JCE API来访问本机PKCS＃11库。的SunPKCS11提供程序本身不包含加密功能 ，它只是Java环境和本机PKCS11提供程序之间的通道。PKCS＃11参考指南对该提供程序进行了更详细的介绍。

SUN提供商

JDK 1.1引入了Provider建筑。第一个JDK提供程序被命名为SUN ，其中包含两种类型的密码服务（ MessageDigest和Signature ）。在以后的版本中，添加了其他机制（ SecureRandom ， KeyPairGenerator ， KeyFactory ， 等等）。

当时有效的美国出口法规对JDK可以在国际上提供的密码功能类型进行了重大限制。因此， SUN提供者历来都包含未直接加密或解密数据的加密引擎。

SunJSSE提供程序

演算法

Java安全套接字扩展（JSSE）最初是作为单独的“可选软件包”（也简称为“标准扩展”）发布的，可用于JDK 1.2。 n和1.3。 n 。的SunJSSE提供程序是作为此版本的一部分引入的。

在早期的JDK版本中，JDK中没有可用的RSA签名提供程序，因此SunJSSE为了使用常用的基于RSA的证书，必须提供自己的RSA实现。JDK 5引入了SunRsaSign提供程序，它提供了以下所有功能（以及更多功能） SunJSSE提供者。针对JDK 5.0及更高版本的应用程序应请求SunRsaSign提供商。为了向后兼容，RSA算法仍可通过该提供程序使用，但实际上是在SunRsaSign提供者。

使用椭圆曲线密码术的密码套件（ECDSA，ECDH，ECDHE和ECDH_anon）需要满足以下要求的JCE密码提供程序：

  • 提供者必须实现包中的类和接口所定义的ECC java.security.spec和java.security.interfaces 。的getAlgorithm()椭圆曲线关键对象的方法必须返回字符串“ EC”。

  • 提供者必须支持Signature算法SHA1withECDSA和NONEwithECDSA， KeyAgreement算法ECDH，以及KeyPairGenerator和一个KeyFactory用于算法EC。如果缺少这些算法之一，则SunJSSE不允许使用EC密码套件。

  • 提供者必须支持RFC 4492规范第5.1.1节中引用的所有SECG曲线（另请参见附录A，“等效曲线（参考）”）。在证书中，点应使用未压缩形式编码，曲线应使用未压缩形式编码。 namedCurve选择，即使用对象标识符。

严格检查加密的PreMasterSecret版本号

在JDK 7发行版之前，SSL / TLS实现未在PreMasterSecret中检查版本号，并且默认情况下SSL / TLS客户端未发送正确的版本号。除非系统属性com.sun.net.ssl.rsaPreMasterSecretFix被设定为true ，则TLS客户端会发送活动的协商版本，但不会发送客户端支持的预期最高版本。

为了兼容，此行为在SSL 3.0版和TLS 1.0版中保留。但是，对于TLS 1.1或更高版本，该实现会严格按照RFC 5246的要求检查PreMasterSecret版本号。客户端始终发送正确的版本号，服务器严格检查版本号。系统属性， com.sun.net.ssl.rsaPreMasterSecretFix ，在TLS 1.1或更高版本中不使用。

SunPCSC提供商

SunPCSC提供程序使应用程序可以使用Java智能卡I / O API与基础操作系统的PC / SC智能卡堆栈进行交互。有关详细信息，请查阅您的操作系统文档。

在Solaris和Linux平台上，SunPCSC通过以下方式访问PC / SC堆栈： libpcsclite.so图书馆。它在目录中查找该库/usr/$LIBISA和/usr/local/$LIBISA ，在哪里$LIBISA扩展到lib/64在64位Solaris平台上lib64在64位Linux平台上。系统属性sun.security.smartcardio.library也可以设置为备用文件的完整文件名libpcsclite.so实施。在Windows平台上，SunPCSC始终调用winscard.dll而且不需要或不可能进行Java级配置。

如果主机平台上有PC / SC，则可以通过以下方式获取SunPCSC实施： TerminalFactory.getDefault()和TerminalFactory.getInstance("PC/SC") 。如果PC / SC不可用或配置不正确，请getInstance()呼叫将失败，并出现NoSuchAlgorithmException和getDefault()将返回不支持任何终端的JRE内置实现。

SunEC提供商

SunEC提供者实现了椭圆曲线密码术（ECC）。与RSA等传统密码系统相比，ECC具有较小的密钥大小，可提供同等的安全性，从而可加快计算速度，降低功耗并节省内存和带宽。

应用程序现在可以使用标准的JCA / JCE API来访问ECC功能，而无需依赖外部ECC库（通过SunPKCS11），就像JDK 6发行版一样。

OracleUcrypto提供程序配置文件

的OracleUcrypto提供者有一个名为ucrypto-solaris.cfg驻留在$JAVA_HOME/conf/security目录。修改此配置文件以指定默认情况下禁用的算法。例如，以下配置文件默认情况下以CFB128模式禁用AES：

#
# Configuration file for the OracleUcrypto provider
#
disabledServices = {
  Cipher.AES/CFB128/PKCS5Padding
  Cipher.AES/CFB128/NoPadding
}
