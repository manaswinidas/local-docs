Java安全性概述

Java安全性包括大量的API，工具和常用安全性算法，机制和协议的实现。Java安全API涵盖了广泛的领域，包括加密，公共密钥基础结构，安全通信，身份验证和访问控制。Java安全技术为开发人员提供了用于编写应用程序的全面安全框架，还为用户或管理员提供了一组工具来安全地管理应用程序。

Java安全性简介

JDK在设计时特别强调安全性。Java语言本身是类型安全的，并且提供自动垃圾回收，从而增强了应用程序代码的健壮性。安全的类加载和验证机制可确保仅执行合法的Java代码。Java安全体系结构包括大量的应用程序编程接口（API），工具以及常用安全算法，机制和协议的实现。

Java安全性API涵盖了广泛的领域。加密和公共密钥基础结构（PKI）接口为开发安全应用程序提供了基础。执行身份验证和访问控制的接口使应用程序可以防止对受保护资源的未授权访问。

这些API允许算法和其他安全服务的多种互操作实现。服务是在providers中实现的，这些providers通过标准接口插入到JDK中，这使得应用程序可以轻松获得安全服务，而不必了解其实现情况。这使开发人员可以专注于如何将安全性集成到其应用程序中，而不是如何实际实现复杂的安全性机制。

JDK包括许多实现一组核心安全服务的提供程序。它还允许安装其他自定义提供程序。这使开发人员可以使用新的安全机制扩展平台。

Java语言安全性和字节码验证

Java语言被设计为类型安全且易于使用。它提供自动内存管理，垃圾回收和阵列范围检查。这减轻了开发人员的总体编程负担，从而减少了细微的编程错误，并使代码更安全，更健壮。

编译器将Java程序转换为与机器无关的字节码表示形式。调用字节码验证程序以确保在Java运行时中仅执行合法的字节码。它检查字节码是否符合Java语言规范，并且没有违反Java语言规则或名称空间限制。验证程序还会检查内存管理违规，堆栈下溢或上溢以及非法数据类型转换。字节码经过验证后，Java运行时会为执行做好准备。

此外，Java语言定义了可以分配给Java类，方法和字段的不同访问修饰符，从而使开发人员可以适当地限制对其类实现的访问。该语言定义了四个不同的访问级别：

  • private ：最严格的修饰符；不允许在定义了私有成员（例如方法）的特定类之外进行访问。

  • protected ：允许访问同一包中的任何子类或其他类。

  • Package-private：如果未指定，则这是默认访问级别；允许访问同一包中的类。

  • public ：不再保证该元素可在任何地方访问；可访问性取决于包含该元素的包是否由其定义模块导出，以及包含试图访问它的代码的模块是否可读取该模块。

基本安全架构

JDK定义了一组跨越主要安全领域的API，包括加密，公共密钥基础结构，身份验证，安全通信和访问控制。这些API使开发人员可以轻松地将安全性集成到他们的应用程序代码中。

API的设计遵循以下原则：

Implementation independence
应用程序不需要自己实现安全性。相反，他们可以从JDK请求安全服务。安全服务是在提供程序中实现的（请参阅“ 安全提供程序 ”部分），这些提供程序通过标准接口插入到JDK中。一个应用程序可能依赖多个独立的提供程序来提供安全功能。
Implementation interoperability

提供程序可跨应用程序互操作。具体来说，如果应用程序不依赖特定提供者的默认值，则该应用程序不会绑定到该特定提供者。

Algorithm extensibility
JDK包括许多内置提供程序，这些提供程序实现了当今广泛使用的一组基本安全服务。但是，某些应用程序可能依赖于尚未实施的新兴标准或专有服务。JDK支持安装实现此类服务的自定义提供程序。

安全提供者

的java.security.Provider类在Java平台中封装了安全提供程序的概念。它指定了提供者的名称并列出了它实现的安全服务。可以同时配置多个提供程序，并按优先顺序列出。当请求安全服务时，将选择实现该服务的最高优先级提供程序。

Java密码学

Java密码体系结构是用于访问和开发Java平台的密码功能的框架。

它包括用于多种加密服务的API，其中包括：

  • 消息摘要算法
  • 数字签名算法
  • 对称的批量和流加密
  • 非对称加密
  • 基于密码的加密（PBE）
  • 椭圆曲线密码学（ECC）
  • 关键协议算法
  • 密钥生成器
  • 消息验证码（MAC）
  • 安全随机数生成器
出于历史（出口控制）的原因，加密API分为两个不同的程序包：

  • 的java.security和java.security.*套餐包含的类not受到出口管制（如Signature和MessageDigest ）

  • 的javax.crypto程序包包含受导出控制的类（例如Cipher和KeyAgreement ）

密码接口是基于提供程序的，从而允许多种且可互操作的密码实现。某些提供程序可能会在软件中执行加密操作；其他人可以对硬件令牌（例如，在智能卡设备或硬件密码加速器上）执行操作。实施出口管制服务的提供商必须由Oracle JCE证书颁发机构颁发的证书进行数字签名。

Java平台包括许多最常用的加密算法的内置提供程序，包括RSA，DSA和ECDSA签名算法，AES加密算法，SHA-2消息摘要算法和Diffie-Hellman（DH）。和椭圆曲线Diffie-Hellman（ECDH）密钥协商算法。大多数内置提供程序都使用Java代码实现加密算法。

Java平台还包括一个内置提供程序，该提供程序充当与本机PKCS＃11（v2.x）令牌的桥梁。此提供者，名为SunPKCS11允许Java应用程序无缝访问位于PKCS＃11兼容令牌上的加密服务。

在Windows上，Java平台包括一个内置提供程序，该提供程序充当与本机Microsoft CryptoAPI的桥梁。此提供者，名为SunMSCAPI允许Java应用程序通过CryptoAPI无缝访问Windows上的加密服务。

公钥基础设施

公钥基础结构（PKI）是用于基于公钥加密技术实现安全信息交换的框架的术语。它允许将（个人，组织等的）身份绑定到数字证书，并提供一种验证证书真实性的方法。PKI包含密钥，证书，公共密钥加密以及生成和数字签名证书的受信任证书颁发机构（CA）。

Java平台包括针对X.509数字证书和证书吊销列表（CRL）的API和提供程序支持，以及与PKIX兼容的证书路径构建和验证。与PKI相关的类位于java.security和java.security.cert包。

密钥和证书存储

Java平台通过密钥和证书存储提供了加密密钥和证书的长期持久存储。具体来说， java.security.KeyStore类表示密钥存储区 ，加密密钥和/或受信任证书的安全存储库（例如在认证路径验证期间使用），以及java.security.cert.CertStore类代表证书存储 ，这是不相关且通常不可信的证书的公共存储库，并且可能是大型存储库。一种CertStore也可以存储CRL。

KeyStore和CertStore实现按类型区分。Java平台包括标准的PKCS11和PKCS12密钥存储类型（其实现符合RSA Security的相应PKCS规范）。它还包含一个称为JKS（代表Java密钥库）的基于文件的专有密钥存储类型，以及一个称为DKS（域密钥存储）的类型，DKS是作为单个逻辑密钥存储呈现的密钥存储的集合。

Java平台包括一个特殊的内置密钥库， cacerts ，其中包含许多知名的受信任CA的证书。keytool实用程序能够列出包含在其中的证书cacerts 。请参阅《 Java平台标准版工具参考》中的 keytool 。

Java加密一节中提到的SunPKCS11提供程序包括一个PKCS11 KeyStore实施。这意味着Java应用程序可以通过以下方式访问和使用安全硬件（例如智能卡）中的密钥和证书： KeyStore API。请注意，可能不允许智能卡钥匙离开设备。在这种情况下， java.security.Key由返回的对象KeyStore API可能只是对密钥的引用（也就是说，它不包含实际的密钥材料）。这样的Key对象只能用于在实际密钥所在的设备上执行加密操作。

Java平台还包括LDAP证书存储类型（用于访问LDAP目录中存储的证书），以及内存中收集证书存储类型（用于访问存储在LDAP目录中的证书）。 java.util.Collection宾语）。

认证方式

身份验证是确定用户身份的过程。在Java运行时环境中，这是识别正在执行的Java程序的用户的过程。在某些情况下，此过程可能依赖于Java密码学小节中描述的服务。

Java平台提供的API使应用程序能够通过可插入的登录模块执行用户身份验证。应用程序调用LoginContext类（在javax.security.auth.login软件包），而后者又引用了配置。该配置指定哪个登录模块（ javax.security.auth.spi.LoginModule接口）用于执行实际的身份验证。

安全通讯

非预期接收者可以访问通过网络传输的数据。当数据包括密码和信用卡号之类的私人信息时，必须采取措施使未经授权的人员无法理解该数据。同样重要的是要确保您正在将数据发送到适当的一方，并且在传输过程中没有有意或无意地修改了数据。

密码学构成安全通信所需的基础；请参阅“ Java密码学 ”部分。Java平台还为许多标准的安全通信协议提供API支持和提供程序实现。

TLS和DTLS协议

传输层安全性（TLS）及其前身安全套接字层（SSL）是加密协议，可在两个通信对等之间提供安全通道。TLS通过提供身份验证，机密性和完整性属性来通过不可信或潜在的敌对网络进行通信，从而结合了加密过程。TLS在可靠的，面向流的传输通道（通常是传输控制协议（TCP））上运行。TLS与应用程序协议无关。更高级别的协议（例如超文本传输协议（HTTP））可以透明地位于TLS之上。

数据报传输层安全性（DTLS）协议基于面向流的TLS协议，旨在为数据报传输提供类似的安全属性，例如用户数据报协议（UDP），该协议不提供可靠或按顺序的数据传递。

JDK提供API和SSL，TLS和DTLS协议的实现，其中包括数据加密，消息完整性以及服务器和客户端身份验证的功能。应用程序可以使用（D）TLS通过任何应用程序协议（例如TCP / IP之上的HTTP）在两个对等点之间提供安全的数据传递。

的javax.net.ssl.SSLSocket类表示一个网络套接字，该套接字将TLS支持封装在普通流套接字（ java.net.Socket ）。某些应用程序可能希望使用备用数据传输抽象（例如，New-I / O）；的javax.net.ssl.SSLEngine类可用于产生和使用TLS / DTLS数据包。

JDK还包括支持可插拔（基于提供商的）密钥管理器和信任管理器概念的API。A key manager由javax.net.ssl.KeyManager类，并管理用于执行身份验证的密钥。trust manager由TrustManager类（在同一包中），并根据其管理的密钥存储区中的证书来决定信任谁。

JDK包括一个内置的提供程序，该提供程序实现SSL / TLS / DTLS协议：

  • SSL 3.0
  • TLS 1.0
  • TLS 1.1
  • TLS 1.2
  • TLS 1.3
  • DTLS 1.0
  • DTLS 1.2

简单身份验证和安全层（SASL）

简单身份验证和安全层（SASL）是Internet标准，它指定用于身份验证的协议以及客户端和服务器应用程序之间安全层的可选建立。SASL定义了如何交换身份验证数据，但它本身并未指定该数据的内容。它是一个框架，特定的身份验证机制可用于指定身份验证数据的内容和语义。Internet社区为各种安全级别和部署方案定义了许多标准SASL机制。

java.security.sasl模块中的Java SASL API为使用SASL机制的应用程序定义类和接口。它被定义为与机制无关；使用API的应用程序不需要硬连接到使用任何特定的SASL机制。应用程序可以根据所需的安全功能选择要使用的机制。该API支持客户端和服务器应用程序。的javax.security.sasl.Sasl类用于创建SaslClient和SaslServer对象。

SASL机制的实现在提供程序包中提供。每个提供程序可以支持一种或多种SASL机制，并通过标准提供程序体系结构进行注册和调用。

Java平台包括一个内置提供程序，该提供程序实现了以下SASL机制：

  • CRAM-MD5，DIGEST-MD5，EXTERNAL，GSSAPI，NTLM和PLAIN客户端机制
  • CRAM-MD5，DIGEST-MD5，GSSAPI和NTLM服务器机制

通用安全服务API和Kerberos

Java平台包含一个API，该API具有用于通用安全服务应用程序编程接口（GSS-API）的Java语言绑定，该API位于java.security.jgss模块。GSS-API为应用程序员提供了对各种基础安全机制之上的安全服务的统一访问。Java GSS-API当前需要使用Kerberos v5机制，并且Java平台包括此机制的内置实现。目前，无法插入其他机制。

Java平台还包括简单和受保护的GSS-API协商机制（SPNEGO）GSS-API机制的内置实现。

在两个应用程序可以使用GSS-API安全地交换消息之前，它们必须建立联合的安全上下文。上下文封装了共享状态信息，其中可能包括例如加密密钥。这两个应用程序都创建并使用org.ietf.jgss.GSSContext对象来建立和维护组成安全上下文的共享信息。一旦建立了安全上下文，就可以使用它来准备要交换的安全消息。

Java GSS API位于org.ietf.jgss包。Java平台还定义了基本的Kerberos类，例如KerberosPrincipal ， KerberosTicket ， KerberosKey和KeyTab ，它们位于javax.security.auth.kerberos包。

访问控制

Java平台中的访问控制体系结构可保护对敏感资源（例如，本地文件）或敏感应用程序代码（例如，类中的方法）的访问。所有访问控制决策均由安全管理器调解，由java.lang.SecurityManager类。一种SecurityManager必须将其安装到Java运行时中才能激活访问控制检查。

通过java默认情况下，命令不与SecurityManager已安装。为了使用SecurityManager ，要么应用程序本身必须通过setSecurityManager方法（在java.lang.System类），或java必须使用-Djava.security.manager命令行上的参数。

权限

权限表示对系统资源的访问。为了允许对applet（或使用安全管理器运行的应用程序）进行资源访问，必须将相应的权限显式授予尝试访问的代码。

当类加载器将Java代码加载到Java运行时时，类加载器会自动将以下信息与该代码关联：

  • 代码从何处加载
  • 谁签署了代码（如果有的话）
  • 授予代码的默认权限

不管代码是通过不受信任的网络（例如，小应用程序）下载还是从文件系统（例如，本地应用程序）下载，该信息都与代码相关联。URL表示从中加载代码的位置，签名者的证书链表示代码签署者，而默认权限由表示。 java.security.Permission对象。

自动授予下载代码的默认权限包括将网络连接返回到其原始主机的能力。自动授予从本地文件系统加载的代码的默认权限包括从其所在目录以及该目录的子目录读取文件的能力。

注意，执行代码的用户的身份在类加载时不可用。必要时，由应用程序代码负责对最终用户进行身份验证（请参阅“ 身份验证 ”部分）。用户通过身份验证后，应用程序可以通过调用doAs中的方法javax.security.auth.Subject类。

安全政策

类加载器将有限的默认权限集授予代码。管理员可以通过安全策略灵活地管理其他代码权限。

Java SE将安全策略的概念封装在java.security.Policy类。只有一个Policy在任何给定时间将对象安装到Java运行时中。的基本责任Policy目的是确定是否允许对受保护资源的代码访问（由加载的位置，签名的人和执行的人来表征）。怎么Policy对象使得此确定取决于实现。例如，它可以查询包含授权数据的数据库，或者可以联系其他服务。

Java SE包含一个默认值Policy从安全属性文件中配置的一个或多个ASCII（UTF-8）文件中读取其授权数据的实现。这些策略文件包含授予代码的确切权限集：特别是授予从特定位置加载，由特定实体签名并以特定用户身份执行的代码的确切权限集。每个文件中的策略条目必须符合文档化的专有语法，并且可以通过简单的文本编辑器或图形文件来组成policytool效用。

访问控制执行

Java运行时跟踪程序执行时进行的Java调用的顺序。当请求访问受保护资源时，默认情况下，将评估整个调用堆栈以确定是否允许请求的访问。

如前所述，资源受SecurityManager 。JDK和应用程序中的安全敏感代码通过如下代码保护对资源的访问：

SecurityManager sm = System.getSecurityManager();
if (sm != null) {
   sm.checkPermission(perm);
}

权限对象perm对应于请求的访问权限。例如，如果尝试读取文件/tmp/abc ，则权限可以构造如下：

Permission perm = new java.io.FilePermission("/tmp/abc", "read");

默认实现SecurityManager将其决定委托给java.security.AccessController实施。的AccessController遍历调用堆栈，传递给已安装的安全性Policy堆栈中的每个代码元素，以及请求的权限（例如， FilePermission在上一个示例中）。的Policy根据管理员配置的权限确定是否授予请求的访问权限。如果未授予访问权限，则AccessController抛出一个java.lang.SecurityException.

XML签名

Java XML数字签名API是用于生成和验证XML签名的标准Java API。

XML签名可以应用于任何类型的数据（XML或二进制）（请参阅XML签名语法和处理 ）。结果签名以XML表示。 XML签名可用于保护您的数据并提供数据完整性，消息身份验证和签名者身份验证。

该API旨在支持W3C XML签名语法和处理建议书的所有必需或推荐功能。该API是可扩展和可插入的，并且基于Java密码服务提供程序体系结构。

java.xml.crypto模块中的Java XML数字签名API由六个软件包组成：

  • javax.xml.crypto
  • javax.xml.crypto.dsig
  • javax.xml.crypto.dsig.keyinfo
  • javax.xml.crypto.dsig.spec
  • javax.xml.crypto.dom
  • javax.xml.crypto.dsig.dom

从历史上看，随着Java SE中添加了新类型的安全服务（有时最初是扩展名），各种缩写都被用来指代它们。由于这些首字母缩写词仍在Java安全文档中使用，因此在此解释它们的含义：

  • JSSE（Java安全套接字扩展）指的是与SSL相关的服务，如TLS和DTLS协议一节中所述

  • JCE（Java密码学扩展）指的是Java密码学小节中描述的加密服务。

  • JAAS（Java身份验证和授权服务）指的是分别在Authentication和Access Control部分中所述的身份验证和基于用户的访问控制服务。
