何时使用Java GSS-API和JSSE

Java GSS-API和JSSE为您提供了相同的基本安全功能集：

  1. 客户端服务器认证
  2. 传输数据的加密和完整性保护

但是，两者之间存在一些关键差异。本文档列出了其中的一些内容，以帮助您确定哪种方法更适合您的环境：

  1. Kerberos单一登录支持

    GSS-API包含对Kerberos的支持，作为一种强制性的安全机制。这意味着，如果您的桌面具有Kerberos支持，则可以编写基于Java GSS-API的应用程序，这些应用程序永远不会提示用户输入密码。

    RFC 2712为TLS定义了Kerberos密码套件，但是该文档已过时，并且不支持AES等现代加密类型。

  2. 通讯API

    JSSE支持基于套接字的API。 JSSE套接字扩展了在java.net和JSSE套接字工厂扩展了在javax.net 。因此，如果您编写的应用程序需要通过套接字工厂配置其安全性，那么JSSE可能更适合您。JSSE套接字需要使用一些可靠的传输方式。通常，实现使用TCP。

    另一方面，Java GSS-API是基于令牌的API，它依赖于应用程序进行通信。这意味着应用程序可以使用TCP套接字，UDP数据报或任何其他允许其传输Java GSS-API生成的令牌的通道。如果您的应用程序具有不同的通信协议需求，那么Java GSS-API可能更适合您。Java GSS-API可以使用输入和输出流读取和写入其令牌。但是，您将需要自己设置流。

  3. 凭证委派

    Java GSS-API允许客户端在使用Kerberos时将其凭据委派给服务器。如果您的应用程序将部署在多层环境中，而中介层在与后端层对话时需要模拟客户端，那么Java GSS-API可能更适合您。

  4. 选择性加密

    因为Java GSS-API是基于令牌的，所以您可以选择选择性地加密某些消息，但不是全部。如果您的应用程序需要插入纯文本和密文消息，则Java GSS-API可能更适合您。

  5. 协议要求

    JSSE提供了RFC 5246中定义的TLS协议的实现。Java GSS-API提供了RFC 5653中定义的GSS-API框架的实现，以及RFC 1964中定义的Kerberos Version 5机制的实现。（在Microsoft Windows平台上，这可以称为带有Kerberos的SSPI。）某些服务器（例如HTTPS服务器）将要求您使用TLS，在这种情况下，JSSE将非常适合您。其他服务器，例如使用SASL的LDAP服务器，可能需要带有Kerberos的GSS-API，在这种情况下，Java GSS-API将适合您。
