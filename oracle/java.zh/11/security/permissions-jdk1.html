JDK中的权限

权限表示对系统资源的访问。为了允许对applet（或使用安全管理器运行的应用程序）进行资源访问，必须将相应的权限显式授予尝试访问的代码。

权限通常具有名称（通常称为“目标名称”），在某些情况下还具有一个或多个动作的逗号分隔列表。例如，以下代码创建了一个FilePermission表示对名为的文件的读取访问权的对象abc在里面/tmp目录：

perm = new java.io.FilePermission("/tmp/abc", "read");

在这里，目标名称是“ /tmp/abc “，并且操作字符串为” read ”。

重要：

上面的语句创建一个权限对象。权限对象代表但不授予对系统资源的访问权限。根据有效的策略，将权限对象构建并分配（“授予”）代码。将权限对象分配给某些代码后，将以指定的方式为该代码授予访问权限对象中指定的系统资源的权限。当前的安全管理器在做出访问决策时也可以构造一个权限对象。在这种情况下，（目标）权限对象是基于所请求的访问创建的，并与发出请求的代码所授予和持有的权限对象进行对照。

Java应用程序环境的策略由Policy对象表示。在里面"JavaPolicy"在策略实施中，可以在一个或多个策略配置文件中指定该策略。策略文件指定允许来自指定代码源的代码具有哪些权限。以下是一个示例策略文件条目，该条目从/home/sysadmin目录对该文件的读取权限/tmp/abc ：

grant codeBase "file:/home/sysadmin/" {
    permission java.io.FilePermission "/tmp/abc", "read";
};

要了解有关策略文件位置和策略文件中的授予权限的更多信息，请参阅默认策略实现和策略文件语法 。

从技术上讲，每当尝试进行资源访问时，执行线程直到该点遍历的all代码all必须具有对该资源访问的许可，除非该线程上的某些代码已标记为“特权”。请参阅附录A：特权块的API 。

如果FileInputStream方法（在这种情况下为构造函数）的调用方式为"/test/MyTestFile"作为name论点，如

FileInputStream("/test/MyTestFile");

然后，为了使调用成功，必须在当前策略中设置以下权限，以允许对该文件进行读取访问"/test/MyTestFile" ：

java.io.FilePermission "/test/MyTestFile", "read";

更具体地说，该权限必须如上所述明确设置，或由其他权限隐含，例如：

java.io.FilePermission "/test/*", "read";

允许对文件中的任何文件进行读取访问"/test"目录。

在某些情况下，花括号中的术语与特定方法参数的名称并不完全相同，而是用来表示相关值。这是一个例子：

java.net.DatagramSocket public synchronized void receive(DatagramPacket p); checkAccept（{host}，{port}） java.net.SocketPermission "{host}:{port}", "accept";

在此，适当的主机和端口值由receive方法并传递给checkAccept 。

默认策略实施

在策略参考实现中，可以在一个或多个策略配置文件中指定策略。配置文件指定允许来自指定代码源的代码具有哪些权限，并由指定的主体执行。每个配置文件都必须使用UTF-8编码。

默认情况下，存在单个系统范围的策略文件和单个（可选）用户策略文件。默认情况下，总是授予平台类加载器或其祖先加载的JDK模块所需的权限。

首次调用其getPermissions方法或每次调用其refresh方法时，都会初始化Policy参考实现。初始化涉及解析策略配置文件（请参阅策略文件语法 ），然后填充Policy对象。

默认策略文件位置

默认情况下，有一个系统范围的策略文件和一个（可选）用户策略文件。初始化策略后，将首先加载系统策略，然后将用户策略添加到其中。如果两个策略都不存在，则使用内置策略。此内置策略与java.policy与JRE一起安装的文件。

系统策略文件位置

默认情况下，系统策略文件为/conf/security/java.policy 。

系统策略文件旨在授予系统范围的代码权限。的java.policy与JDK一起安装的文件允许任何人侦听动态端口，并允许任何代码读取对安全性不敏感的某些“标准”属性，例如os.name和file.separator属性。

用户策略文件位置

默认情况下，用户策略文件为/.java.policy 。

策略文件的位置和格式

策略文件位置在安全属性文件中指定/conf/security/java.security 。

策略文件位置被指定为名称具有以下形式的属性的值：

policy.url.n

这里， n是一个数字。您可以按以下格式在一行中指定每个此类属性值：

policy.url.n=URL

这里， URL是URL规范。例如，默认系统和用户策略文件在安全属性文件中定义为：

policy.url.1=file:${java.home}/conf/security/java.policy
policy.url.2=file:${user.home}/.java.policy

（有关通过特殊语法指定属性值的信息，请参见策略文件中的属性扩展 ，例如指定java.home属性值通过${java.home} ）

实际上，您可以指定许多URL（包括“ http:// ”），所有指定的策略文件将被加载。您也可以注释掉或更改第二个注释以禁用读取默认用户策略文件。

该算法开始于policy.url.1 ，并一直递增，直到找不到URL。因此，如果你有policy.url.1和policy.url.3和policy.url.3将永远不会被读取。

在运行时指定其他策略文件

调用应用程序执行时，还可以指定其他或不同的策略文件。这可以通过-Djava.security.policy命令行参数，用于设置java.security.policy属性。例如，如果使用以下命令，则其中someURL是一个URL，它指定策略文件的位置，然后除安全属性文件中指定的所有策略文件外，还将加载指定的策略文件。 

java -Djava.security.manager -Djava.security.policy=someURL SomeApp

该URL可以是任何常规URL，也可以只是当前目录中策略文件的名称，例如：

java -Djava.security.manager -Djava.security.policy=mypolicy SomeApp

的-Djava.security.manager选项可确保安装了默认的安全管理器，从而对应用程序进行策略检查。如果应用程序SomeApp安装了安全管理器，则不是必需的。

如果使用以下命令（请注意双精度等于），则仅使用指定的策略文件；安全属性文件中指示的所有内容都将被忽略。 

java -Djava.security.manager -Djava.security.policy==someURL SomeApp

注意：

的策略文件值-Djava.security.policy如果该选项被忽略policy.allowSystemProperty安全属性文件中的property属性设置为false。默认值为true。

修改政策实施

可以通过编辑安全属性文件来修改策略参考实现，该文件是java.security文件在conf/security JDK的目录。

只要前者是抽象Policy类的子类并实现getPermissions方法（和其他必要的方法），就可以提供替代策略类来代替Policy参考实现类。

您可以设置的属性类型之一java.security具有以下形式：


    policy.provider=PolicyClassName

PolicyClassName必须指定所需的标准名称Policy实现类。

此属性的默认安全属性文件条目如下：


    policy.provider=sun.security.provider.PolicyFile

要进行自定义，您可以更改属性值以指定另一个类，如


    policy.provider=com.mycom.MyPolicy

策略文件语法

JDK安装的策略配置文件指定对来自指定代码源的代码授予什么权限（哪些类型的系统资源访问权限），并作为指定的主体执行。

为了允许小程序（或在安全管理器下运行的应用程序）执行安全操作（例如，读取或写入文件），必须授予小程序（或应用程序）对该特定操作的许可。在策略参考实现中，必须通过策略配置文件中的授予条目来授予该权限。有关更多信息，请参见下文和Java安全体系结构规范 。（唯一的例外是代码始终自动具有从其相同（URL）位置以及该位置的子目录读取文件的权限；这样做不需要显式权限。）

策略配置文件实质上包含条目列表。它可能包含一个“密钥库”条目，并且包含零个或多个“授予”条目。

密钥库条目

keystore是私钥及其相关的数字证书（例如X.509证书链）的数据库，这些证书对相应的公钥进行身份验证。keytool实用程序用于创建和管理密钥库。在策略配置文件中指定的密钥库用于查找在文件的授权条目中指定的签名者的公钥。如果任何授权条目指定了签名者别名，或者任何授权条目指定了主体别名，则密钥库条目必须出现在策略配置文件中。

目前只能有一个keystore / keystorePasswordURL策略文件中的条目（第一个条目之后的其他条目将被忽略）。该条目可以出现在文件的授予条目之外的任何位置。它具有以下语法：


keystore "some_keystore_url", "keystore_type", "keystore_provider";
keystorePasswordURL "some_password_url";

这里，

some_keystore_url
指定密钥库的URL位置。
some_password_url
指定密钥库密码的URL位置。
keystore_type
指定密钥库类型。
keystore_provider
指定密钥库提供程序。

注意：

  • 来自的输入流some_keystore_url传递给KeyStore.load方法。

  • 如果将NONE指定为URL，则将空流传递给KeyStore.load方法。如果KeyStore不是基于文件的，则不应在URL中指定NONE。例如，如果它位于硬件令牌设备上。

  • URL是相对于策略文件位置的。如果在安全属性文件中将策略文件指定为：

        policy.url.1=http://foo.example.com/fum/some.policy

    并且该策略文件具有一个条目：

        keystore ".keystore";

    那么将从以下位置加载密钥库：

        http://foo.example.com/fum/.keystore

  • 该URL也可以是绝对的。

密钥库类型定义了密钥库信息的存储和数据格式，以及用于保护密钥库中的私钥和密钥库本身完整性的算法。默认类型为“ PKCS12”。因此，如果密钥库类型是“ PKCS12”，则不需要在密钥库条目中指定它。

赠款

始终认为执行的代码来自特定的“代码源”（由类型为object的对象表示） CodeSource ）。代码源不仅包括代码所源自的位置（URL），而且还包括对包含对应于用于对代码进行签名的私有密钥的公共密钥的证书的引用。代码源中的证书由用户密钥库中的符号别名引用。代码也被认为是作为特定主体执行的（由类型为object的对象表示） Principal ）或校长小组。

每个授予条目包括一个或多个“权限条目”，其后是可选的codeBase ， signedBy ，以及用于指定要授予权限的代码的主体名称/值对。授予条目的基本格式如下：


  grant signedBy "signer_names", codeBase "URL",
        principal principal_class_name "principal_name",
        principal principal_class_name "principal_name",
        ... {

      permission permission_class_name "target_name", "action", 
          signedBy "signer_names";
      permission permission_class_name "target_name", "action", 
          signedBy "signer_names";
      ...
  };

上面的所有非斜体项必须按原样显示（尽管大小写无关紧要，但有些是可选的，如下所述）。斜体项目表示变量值。

授权条目必须以单词开头grant 。

SignedBy，Principal和CodeBase字段

的signedBy ， codeBase和principal值是可选的，并且这些字段的顺序无关紧要。

一种signedBy value表示密钥库中存储的证书的别名。该证书中的公钥用于验证代码上的数字签名；您授予对由别名指定的密钥库条目中的与公钥相对应的私钥签名的代码的权限。

的signedBy value可以是多个别名的逗号分隔列表。例如“亚当，夏娃，查尔斯”，意思是“由亚当，夏娃和查尔斯签名”；关系是AND，而不是OR。更准确地说，类似“由亚当签名的代码”这样的语句意味着“包含在JAR中的类文件中的代码，该文件使用与密钥存储库中的公共密钥证书相对应的私钥进行了签名，其条目由亚当别名”。

的signedBy字段是可选的，如果省略，则表示“任何签名者”。代码是否签名或由谁签名都没有关系。

主体值指定一个class_name / principal_name对，必须存在于执行线程的主体集中。主体集通过主题与执行代码相关联。

的principal_class_name可以设置为通配符值*，以使其与任何Principal类。除此之外principal_name也可以设置为通配符值*，以使其与任何Principal名称。设定principal_class_name要么principal_name *，不要用引号将*引起来。另外，如果指定通配符主体类，则还必须指定通配符主体名称。

主体字段是可选的，因为如果省略它，则表示“任何主体”。

密钥库别名替换

如果校长class_name / principal_name对被指定为单引号字符串，然后将其视为密钥库别名。向密钥库查询（通过别名）并查询X509证书。如果找到一个，则主体class_name将自动视为javax.security.auth.x500.X500Principal和principal_name自动被视为证书中主题的专有名称。如果找不到X509证书映射，则忽略整个授予条目。

一种codeBase值指示代码源位置；您授予从该位置进行编码的权限。一个空的codeBase条目表示“任何代码”；代码来自哪里都没有关系。

Windows系统上的文件路径规范

当您指定一个java.io.FilePermission ，“ target_name ”是文件路径。在Windows系统上，每当您直接在字符串中指定文件路径（而不是在代码库URL中）时，都需要为路径中的每个实际单个反斜杠包括两个反斜杠，如下所示：

    grant {
        permission java.io.FilePermission "C:\\users\\cathy\\foo.bat", "read";
    };

之所以需要这样做，是因为字符串是由令牌处理程序处理的（ java.io.StreamTokenizer ），它允许将“ \”用作转义字符串（例如，“ \ n”表示换行），因此需要两个反斜杠来表示单个反斜杠。令牌化程序处理完上述文件路径字符串后，将双反斜杠转换为单反斜杠，最终结果是

    "C:\users\cathy\foo.bat"

策略文件示例

以下策略配置文件包含两个条目：

  // If the code is signed by "Duke", grant it read/write access to all 
  // files in /tmp:
  grant signedBy "Duke" {
      permission java.io.FilePermission "/tmp/*", "read,write";
  };

  // Grant everyone the following permission:
  grant { 
      permission java.util.PropertyPermission "java.vendor", "read";
  };

以下策略配置文件指定只有满足以下条件的代码才能调用Security类中的方法来添加或删除提供程序或设置安全属性：

  • 该代码是从位于“ /home/sysadmin/本地文件系统上的目录。
  • 可以使用别名“”引用的公钥来验证签名。 sysadmin ”。 
  grant signedBy "sysadmin", codeBase "file:/home/sysadmin/*" {
      permission java.security.SecurityPermission "Security.insertProvider.*";
      permission java.security.SecurityPermission "Security.removeProvider.*";

策略条目的任何一个组成部分（或两者）都可能丢失。

以下是一个策略配置文件，其中codeBase不见了：

  grant signedBy "sysadmin" {
      permission java.security.SecurityPermission "Security.insertProvider.*";
      permission java.security.SecurityPermission "Security.removeProvider.*";
  };

如果该政策有效，那么JAR文件中的代码将由“ sysadmin ”可以添加/删除提供程序，无论JAR文件来自何处。

以下是没有签名者的策略配置文件：

  grant codeBase "file:/home/sysadmin/-" {
      permission java.security.SecurityPermission "Security.insertProvider.*";
      permission java.security.SecurityPermission "Security.removeProvider.*";
  };

在这种情况下，来自"home/sysadmin/"本地文件系统上的目录可以添加/删除提供程序。该代码不需要签名。

以下是一个策略配置文件，其中codeBase也不signedBy已经包括了：

  grant {
      permission java.security.SecurityPermission "Security.insertProvider.*";
      permission java.security.SecurityPermission "Security.removeProvider.*";
  };

在这里，缺少两个代码源组件，任何代码（无论它来自何处，是否被签名或由谁签名）都可以添加/删除提供程序。

以下是基于主体的条目：

  grant principal javax.security.auth.x500.X500Principal "cn=Alice" {
      permission java.io.FilePermission "/home/Alice", "read, write";
  };

这允许任何以X500Principal身份执行的代码， cn=Alice “，允许读写” /home/Alice ”。

以下代表具有通配符值的基于主体的条目：

  grant principal javax.security.auth.x500.X500Principal * {
      permission java.io.FilePermission "/tmp", "read, write";
  };

这允许以X500Principal（无论专有名称）执行的任何代码，均具有读取和写入“ /tmp ”。

以下示例显示了具有代码源和主体信息的Grant语句：

        signedBy "Duke",

这允许从“ www.games.example.com “， 被...签名 ” Duke ”，并由“ cn=Alice ”，允许对“ /tmp/games “ 目录。

以下示例显示了具有KeyStore别名替换的Grant语句：

  grant principal "alice" {

策略文件中的属性扩展

在策略文件和安全属性文件中可以进行属性扩展。

属性扩展类似于在shell中扩展变量。也就是说，当一个字符串像

    ${some.property}

如果出现在策略文件或安全属性文件中，它将被展开为系统属性的值。例如，

    permission java.io.FilePermission "${user.home}", "read";

将会扩展${user.home}以使用“ user.home”系统属性的值。如果该属性的值为“ /home/cathy ”，那么以上等同于

    permission java.io.FilePermission "/home/cathy", "read";

为了协助独立于平台的政策文件，您还可以使用特殊符号“ ${/} ”，这是${file.separator} ”。这允许像

    permission java.io.FilePermission "${user.home}${/}*", "read";

如果“ user.home “财产是/home/cathy ，并且您使用的是Solaris，Linux或macOS，则以上代码会转换为：

    permission java.io.FilePermission "/home/cathy/*", "read";

另一方面，如果“ user.home “值是C:\users\cathy并且您在Windows系统上，以上代码将转换为：

    permission java.io.FilePermission "C:\users\cathy\*", "read";

另外，在特殊情况下，如果您在代码库中扩展属性，例如

    grant codeBase "file:${my.libraries}/api/"

那么任何文件分隔符都将自动转换为/字符。例如，假设my.libraries是C:\Users\me\lib 。因此，在Windows系统上，以上代码将转换为

    grant codeBase "file:C:/Users/me/lib/api/"

因此，您不需要使用${/}在代码库字符串中（您不应这样做）。属性扩展发生在策略文件中允许双引号字符串的任何地方。这包括"signer_names" ， "URL" ， "target_name"和"action"字段。是否允许属性扩展由“ policy.expandProperties安全属性文件中的“”属性。如果此属性的值为true（默认值），则允许扩展。

注意：

您不能使用嵌套属性；他们将无法正常工作。例如，

    "${user.${foo}}"

即使“ foo “属性设置为” home ”。原因是属性解析器无法识别嵌套属性；它只是寻找第一个“ ${ ”，然后继续查找，直到找到第一个“ }并尝试解释结果（在这种情况下， ${user.$foo} “）作为属性，但如果没有这样的属性，则失败。

注意：

如果无法在授予条目，权限条目或密钥库条目中扩展属性，则将忽略该条目。例如，如果系统属性为“ foo “”未定义，您具有：

    grant codeBase "${foo}" {
        permission ...;
        permission ...;
    };

那么该授权条目中的所有权限都将被忽略。如果你有

    grant {
        permission Foo "${foo}";
        permission Bar "barTarget";
    };

然后只有“ permission Foo... ”条目将被忽略。最后，如果你有

    keystore "${foo}";

那么密钥库条目将被忽略。

策略文件中的常规扩展

策略文件中也支持通用形式的扩展。例如，权限名称可能包含以下格式的字符串：

${{protocol:protocol_data}}

如果这样的字符串出现在权限名称中，则protocol中的值将确定应该发生的扩展的确切类型，并且protocol_data用于帮助执行扩展。 protocol_data可能为空，在这种情况下，上面的字符串应采用以下形式：

${{protocol}}

默认策略文件实现中支持两种协议：

  1. ${{self}}

    协议self ，表示整个字符串的替换， ${{self}} ，具有一个或多个主体类/名称对。确切执行的替换取决于权限所属的Grant子句的内容。

    如果Grant子句不包含任何主体信息，则该权限将被忽略（权限包含${{self}}目标名称中的“仅”在基于委托人的授予子句的上下文中有效）。例如， BarPermission在以下Grant子句中将始终被忽略：

    grant codebase "www.example.com", signedby "duke" {
    permission BarPermission "... ${{self}} ...";
};

    如果Grant子句包含主体信息， ${{self}}将被替换为相同的主要信息。例如， ${{self}}在BarPermission将被替换为javax.security.auth.x500.X500Principal "cn=Duke"在以下Grant子句中：

    grant principal javax.security.auth.x500.X500Principal "cn=Duke" {
    permission BarPermission "... ${{self}} ...";
};

    如果Grant子句中有逗号分隔的主体列表，则${{self}}将被相同的逗号分隔列表或主体替换。如果在Grant子句中通配了主体类和名称， ${{self}}被替换为与Subject在当前AccessControlContext 。

    下面的示例描述了一个场景self和KeyStore别名替换一起：

    keystore "http://foo.example.com/blah/.keystore";

grant principal "duke" {
    permission BarPermission "... ${{self}} ...";
};

    在上面的示例中， duke ”将首先扩展为javax.security.auth.x500.X500Principal "cn=Duke"假设与KeyStore别名” duke “，具有主题专有名称“ cn=Duke ”。下一个， ${{self}}将被授予子句中刚刚扩展的相同主要信息替换： javax.security.auth.x500.X500Principal "cn=Duke" 。

  2. ${{alias:alias_name}}

    协议alias ，表示java.security。KeyStore别名替换。的KeyStore使用的是密钥库条目中指定的密钥 。 alias_name代表别名KeyStore 。 ${{alias:alias_name}}被替换为javax.security.auth.x500.X500Principal "DN" ，其中DN表示属于alias_name的证书的主题可分辨名称。例如：

    keystore "http://foo.example.com/blah/.keystore";

grant codebase "www.example.com" {
    permission BarPermission "... ${{alias:duke}} ...";
};

    在上面的示例中，与别名duke关联的X.509证书是从KeyStore ， foo.example.com/blah/.keystore 。假设duke的证书指定“ o=dukeOrg, cn=duke “”作为主题专有名称，然后${{alias:duke}}被替换为javax.security.auth.x500.X500Principal "o=dukeOrg, cn=duke" 。

    在以下错误情况下，权限条目将被忽略：

    • 密钥库条目未指定
    • 没有提供alias_name
    • 无法检索alias_name的证书
    • 检索到的证书不是X.509证书

附录A：FilePermission路径名规范化默认情况下处于禁用状态

规范路径是不包含任何链接或快捷方式的路径。执行路径名称规范化FilePermission对象可能会对性能产生负面影响。

在JDK 9之前，当两个FilePermission对象进行了比较。这允许程序使用与授予代理程序名称不同的名称访问文件。 FilePermission对象在策略文件中，只要该对象指向同一文件即可。因为规范化必须访问底层文件系统，所以它可能会很慢。

在JDK 9中，默认情况下禁用路径名规范化。这意味着两个FilePermission如果一个对象使用绝对路径，另一个使用相对路径，或者一个使用符号链接，另一个使用目标，或者一个使用Windows长名称，另一个使用DOS样式，则对象彼此不相等8.3名称。即使它们都指向文件系统中的同一文件，也是如此。

因此，如果将路径名授予了FilePermission对象添加到策略文件中，则程序也应该使用相同的路径名样式访问该文件。例如，如果策略文件中的路径名使用符号链接，则程序也应使用该符号链接。使用目标路径名访问文件将导致权限检查失败。

相容层

添加了兼容性层，以确保授予FilePermission相对路径的对象将允许应用程序使用绝对路径（反之）访问文件。这适用于默认策略提供程序和Limited doPrivileged调用。

例如，一个FilePermission相对路径名称为的文件上的对象"a"不再意味着FilePermission对象在同一文件上，其绝对路径名为"/pwd/a" （ "pwd"是当前的工作目录）。授予代码FilePermission阅读对象"a"允许该代码也读取"/pwd/a"启用安全管理器时。

兼容性层不包括符号链接和目标之间的转换，也不包括Windows长名称和DOS样式8.3名称之间的转换，或可以规范化为相同名称的任何其他不同名称形式。
