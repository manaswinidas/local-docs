故障排除

下面列出了尝试登录时可能会出现的一些问题以及解决这些问题的建议。

  • 可配置的Kerberos设置： Kerberos密钥分发中心（KDC）名称和领域设置在Kerberos配置文件中或通过系统属性提供java.security.krb5.kdc和java.security.krb5.realm.布尔选项refreshKrb5Config可以在条目中指定Krb5LoginModule在JAAS配置文件中。如果此选项设置为true ，则配置值将在刷新登录方法之前刷新Krb5LoginModule叫做。

    注意：

    切换Kerberos配置时，要求refreshKrb5Config应该设置为true.未能设置此值可能导致意外结果。

  • java.lang。javax.security.auth.login上的SecurityException。Configuration.getConfiguration

    Cause ：处理JAAS登录配置文件时出现问题，可能是由于该文件中的语法错误所致。

    Solution ：仔细检查配置文件中是否有错误。有关登录配置文件中所需语法的信息，请参见附录B：JAAS登录配置文件。

  • javax.security.auth.login。LoginException：KrbException：预身份验证信息无效（24）-预身份验证失败

    Cause 1:输入的密码错误。

    Solution 1 ：验证密码。

    Cause 2:如果您使用密钥标签来获取密钥（例如，通过设置useKeyTab选择true （在JAAS登录配置文件中的Krb5LoginModule条目中），则自更新密钥表以来，密钥可能已更改。

    Solution 2 ：请查阅Kerberos文档以生成新的密钥表并使用该密钥表。

    Cause 3:时钟偏斜-如果KDC和客户端上的时间相差很大（通常为5分钟），则可以返回此错误。

    Solution 3 ：同步时钟（或由系统管理员同步）。Cause 4: Kerberos领域名称并非全为大写。

    解决方案4 ：将Kerberos领域名称全部大写。注意 ：建议使用所有大写的领域名称。请参见领域名称和主机名的命名约定 。

  • GSSException：没有提供有效的凭据（机制级别：尝试获取新的INITIATE凭据失败！（空值）） 。 。 。由：javax.security.auth.login引起。LoginException：时钟偏斜太大

    Cause ：Kerberos需要在KDC和客户端上的时间进行松散同步。（默认值为5分钟内。）如果不是这种情况，您将收到此错误。

    Solution ：同步时钟（或由系统管理员同步）。

  • javax.security.auth.login。LoginException：KrbException：空领域名称（601）-未指定默认领域

    Cause ：Kerberos配置文件中未指定默认领域krb5.conf （如果使用），作为用户名的一部分提供，或通过java.security.krb5.realm系统属性。

    Solution ：验证您的Kerberos配置文件（如果使用）是否包含指定默认领域的条目，或通过设置java.security.krb5.realm使用Kerberos进行身份验证时，系统属性和/或将其包含在用户名中。

  • javax.security.auth.login。LoginException：java.net。SocketTimeoutException：接收超时

    Solution ：验证Kerberos KDC是否已启动并正在运行。

  • GSSException：没有提供有效的凭据（机制级别：找不到任何Kerberos票证）

    Cause ：如果没有获得有效的Kerberos凭据，则可能会发生这种情况。特别是，如果您希望基础机制获取凭据，却忘记了通过设置javax.security.auth.useSubjectCredsOnly系统属性值false （例如，通过-Djavax.security.auth.useSubjectCredsOnly=false在您的执行命令中）。

    Solution ：请务必设置javax.security.auth.useSubjectCredsOnly系统属性值false如果您希望底层机制获取凭据，而不是您的应用程序或包装程序（例如某些教程使用的Login实用程序）获取证书，则使用JAAS进行身份验证。

  • javax.security.auth.login。LoginException：无法加载配置文件（无此文件或目录）

    Cause ：教程的示例执行命令通过为以下示例设置值来指定默认的Kerberos领域和KDC。 java.security.krb5.realm和java.security.krb5.kdc系统属性。如果愿意，您可以改为krb5.conf使用的Kerberos配置文件。这样的文件包含有关默认领域和KDC是什么的信息。要使用krb5.conf文件，您可以设置系统属性java.security.krb5.conf （而不是realm和kdc属性）以指定文件的位置，或者您未设置任何这些属性，因此尝试查找krb5.conf文件放在默认位置。如果找不到该文件，则会出现错误“无法加载配置文件（无此文件或目录）”。

    Solution ：验证Kerberos配置文件krb5.conf可用且可读。查看Kerberos要求，以获取有关如何指定安装位置的信息。 krb5.conf文件，如果您未明确指出位置，则默认情况下将搜索该文件。

  • javax.security.auth.login。LoginException：KrbException：KDC不支持加密类型（14）-KDC不支持加密类型

    Cause 1 ：您的KDC不支持请求的加密类型。

    Solution 1 ：Oracle的Kerberos实现支持以下加密类型： aes256-cts-hmac-sha1-96 ， aes128-cts-hmac-sha1-96 ， des3-cbc-sha1 ， arcfour-hmac-md5 ， des-cbc-crc和des-cbc-md5 。

    应用程序可以通过在Kerberos配置文件中指定以下标记来选择所需的加密类型krb5.conf ：

    [libdefaults]
default_tkt_enctypes = des-cbc-md5 des-cbc-crc des3-cbc-sha1
default_tgs_enctypes = des-cbc-md5 des-cbc-crc des3-cbc-sha1
permitted_enctypes = des-cbc-md5 des-cbc-crc des3-cbc-sha1

    如果未指定，则默认值为：

    aes256-cts-hmac-sha1-96 aes128-cts-hmac-sha1-96 des3-cbc-sha1 arcfour-hmac-md5

    如果allow_weak_crypto在krb5.conf设置为true，则des-cbc-crc和des-cbc-md5也受支持。

    Cause 2 ：在某些Windows平台上使用本机票证缓存时，抛出此异常。Microsoft已添加了一项新功能，即他们不再导出票证授予票证（TGT）的会话密钥。结果，在Windows上获得的本机TGT具有“空”会话密钥和空EType。

    Solution 2:您需要更新Windows注册表以禁用此新功能。注册表项allowtgtsessionkey应该添加并正确设置，以允许在Kerberos票证授予票证中发送会话密钥。

    有关更多信息，请参见Microsoft支持中心的Kerberos Ticket-Granting-Ticket中允许发送会话密钥的注册表项 。通常，以下是必需的注册表设置：

    HKEY_LOCAL_MACHINE\System\CurrentControlSet\Control\Lsa\Kerberos\Parameters
Value Name: allowtgtsessionkey
Value Type: REG_DWORD
Value: 0x01  ( default is 0 )

    默认情况下，该值为0；否则为0。将其设置为“ 0x01”可以将会话密钥包含在TGT中。

  • KDC的回复与预期不符

    Cause ：KDC发送了客户端无法理解的响应。

    Solution ：确认您已正确设置所有krb5.conf文件配置参数，并查阅您的KDC供应商指南。

    注意：

    可以通过设置系统属性来启用调试模式sun.security.krb5.debug为“ true”。此设置使您可以跟踪程序对Kerberos V5协议的执行。
