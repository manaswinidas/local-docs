一个致命的错误日志

描述致命错误日志，其位置和内容。

发生致命错误时，将创建致命错误日志。它包含信息以及发生致命错误时获得的状态。

注意：

此文件的格式在更新版本中可能会稍有更改。

本附录包含以下部分：

  • 致命错误日志的位置

  • 致命错误日志的描述

  • 标题格式

  • 线程节格式

  • 处理部分格式

  • 系统部分格式

致命错误日志的位置

要指定将在何处创建日志文件，请使用product标志-XX:ErrorFile=file ，在哪里file表示日志文件位置的完整路径。

子串%%在file变量中转换为%和子字符串%p转换为过程的PID。

在下面的示例中，错误日志文件将被写入目录/var/log/java并将被命名java_errorpid.log ：

java -XX:ErrorFile=/var/log/java/java_error%p.log

如果-XX:ErrorFile=file未指定标志，则默认日志文件名为hs_err_pid.log ，在哪里pid是过程的PID。

另外，如果-XX:ErrorFile=file如果未指定flag，则系统尝试在进程的工作目录中创建文件。如果无法在工作目录中创建文件（空间不足，权限问题或其他问题），则会在操作系统的临时目录中创建文件。在Oracle Solaris和Linux操作系统上，临时目录为/tmp 。在Windows上，临时目录由TMP环境变量。如果未定义该环境变量，则TEMP使用环境变量。

致命错误日志的描述

致命错误日志文件的描述以及包含在致命错误时获得的信息的部分。

错误日志包含在发生致命错误时获得的信息，并在可能的情况下包括以下信息：

  • 引发致命错误的操作异常或信号

  • 版本和配置信息

  • 有关引发致命错误的线程的详细信息以及线程的堆栈跟踪

  • 正在运行的线程及其状态列表

  • 有关堆的摘要信息

  • 加载的本机列表

  • 命令行参数

  • 环境变量

  • 有关操作系统和CPU的详细信息

注意：

在某些情况下，此信息的子集仅输出到错误日志。当致命错误的严重性严重到使错误处理程序无法恢复并报告所有详细信息时，就会发生这种情况。

错误日志是一个文本文件，由以下部分组成：

  • 标头，提供崩溃的简短描述。请参阅标题格式 。

  • 具有线程信息的部分。请参阅线程节格式 。

  • 包含过程信息的部分。请参阅过程节格式 。

  • 包含系统信息的部分。请参阅系统节格式 。

注意：

此处描述的致命错误日志的格式基于Java SE 6。格式可能与其他版本不同。

标题格式

每个致命错误日志文件开头的标头部分均包含对该问题的简短描述。

标头也被打印到标准输出，并可能显示在应用程序的输出日志中。

标头包含指向HotSpot虚拟机错误报告页面的链接，用户可以在其中提交错误报告。

#
# A fatal error has been detected by the Java Runtime Environment:
#
#  SIGSEGV (0xb) at pc=0x00007f0f159f857d, pid=18240, tid=18245
#
# JRE version: Java(TM) SE Runtime Environment (9.0+167) (build 9-ea+167)
# Java VM: Java HotSpot(TM) 64-Bit Server VM (9-ea+167, mixed mode, tiered, compressed oops, g1 gc, linux-amd64)
# Problematic frame:
# C  [libMyApp.so+0x57d]  Java_MyApp_readData+0x11
#
# Core dump will be written. Default location: /cores/core.18240)
#
# If you would like to submit a bug report, please visit:
#   http://bugreport.java.com/bugreport/crash.jsp
# The crash happened outside the Java Virtual Machine in native code.
# See problematic frame for where to report the bug.
#

该示例显示VM因意外信号而崩溃。

下一行描述了信号类型，引起信号的程序计数器（pc），进程ID和线程ID，如以下示例所示。

#  SIGSEGV (0xb) at pc=0x00007f0f159f857d, pid=18240, tid=18245
      |      |           |                    |         +--- thread id
      |      |           |                    +------------- process id
      |      |           +--------------------------- program counter
      |      |                                        (instruction pointer)
      |      +--------------------------------------- signal number
      +---------------------------------------------- signal name

下一行包含VM版本（客户端VM或服务器VM），指示应用程序是在混合模式还是解释模式下运行以及指示是否启用了类文件共享，如以下行所示。

# Java VM: Java HotSpot(TM) 64-Bit Server VM (9-ea+167, mixed mode, tiered, compressed oops, g1 gc, linux-amd64)

下一个信息是导致崩溃的功能框，如以下示例所示。

# Problematic frame:
# C  [libMyApp.so+0x57d]  Java_MyApp_readData+0x11
  |              +-- Same as pc, but represented as library name and offset.
  |                  For position-independent libraries (JVM and most shared
  |                  libraries), it is possible to inspect the instructions
  |                  that caused the crash without a debugger or core file
  |                  by using a disassembler to dump instructions near the
  |                  offset.
  +----------------- Frame type

在该示例中，“ C”帧类型指示本地C帧。表A-1显示了可能的帧类型。

内部错误将导致VM错误处理程序生成类似的错误转储。但是，头格式不同。内部错误的示例是guarantee()失败，断言失败， ShouldNotReachHere()等等。下面的示例显示内部错误的标头格式。 

#
# An unexpected error has been detected by HotSpot Virtual Machine:
#
# Internal Error (4F533F4C494E55583F491418160E43505000F5), pid=10226, tid=16384
#
# Java VM: Java HotSpot(TM) Client VM (1.6.0-rc-b63 mixed mode)

在上面的标题中，没有信号名称或信号编号。相反，第二行现在包含Internal Error和一个十六进制长字符串。此十六进制字符串对检测到错误的源模块和行号进行编码。通常，此“错误字符串”仅对在HotSpot虚拟机上工作的工程师有用。

错误字符串对行号进行编码，因此它会随着每次代码的更改和发布而更改。在一个版本（例如1.6.0）中具有给定错误字符串的崩溃可能与更新版本（例如1.6.0_01）中的相同崩溃不符，即使字符串匹配。

注意：

不要假定在与给定错误字符串关联的一种情况下有效的解决方法或解决方案在与该错误字符串关联的另一种情况下有效。请注意以下事实：

  • 具有相同根本原因的错误可能具有不同的错误字符串。

  • 具有相同错误字符串的错误可能具有完全不同的根本原因。

因此，在对错误进行故障排除时，不应将错误字符串用作唯一标准。

线程节格式

有关崩溃的线程的信息。

如果多个线程同时崩溃，则仅打印一个线程。

线程信息

线程部分的第一部分显示了导致致命错误的线程，如以下示例所示。

Current thread (0x00007f102c013000):  JavaThread "main" [_thread_in_native, id=18245, stack(0x00007f10345c0000,0x00007f10346c0000)]
                    |                      |         |            |             |             + stack     
                    |                      |         |            |             +------------------ ID
                    |                      |         |            +------------------------------- state
                    |                      |         +-------------------------------------------- name
                    |                      +------------------------------------------------------ type
                    +----------------------------------------------------------------------------- pointer

线程指针是指向Java VM内部线程结构的指针。除非您正在调试实时Java VM或核心文件，否则通常没有兴趣。

以下列表显示了可能的线程类型。

  • JavaThread

  • VMThread

  • CompilerThread

  • GCTaskThread

  • WatcherThread

  • ConcurrentMarkSweepThread

表A-2显示了重要的线程状态。

表A-2线程状态

线程状态 描述

_thread_uninitialized

没有创建线程。仅在内存损坏的情况下才会发生这种情况。

_thread_new

线程已创建，但尚未启动。

_thread_in_native

线程正在运行本机代码。该错误可能是本机代码中的错误。

_thread_in_vm

线程正在运行VM代码。

_thread_in_Java

线程正在运行解释的或编译的Java代码。

_thread_blocked

线程被阻塞。

..._trans

如果前面的任何状态后跟字符串_trans ，则意味着线程正在更改为其他状态。

输出中的线程ID是本机线程标识符。

如果Java线程是守护程序线程，则在线程状态之前打印字符串守护程序。

信号信息

错误日志中的以下信息描述了导致VM终止的意外信号。在Windows系统上，输出如下例所示。

siginfo: ExceptionCode=0xc0000005, reading address 0xd8ffecf1

在上面的示例中，异常代码为0xc0000005 （ ACCESS_VIOLATION ），并且当线程尝试读取地址时发生异常0xd8ffecf1 。

在Oracle Solaris和Linux操作系统上，信号编号（ si_signo ）和信号代码（ si_code ）用于识别异常，如下所示：

siginfo: si_signo: 11 (SIGSEGV), si_code: 1 (SEGV_MAPERR), si_addr: 0x0000000000000000

该日志包含两个线程堆栈。

  • 第一个线程堆栈是Native frames ，它显示显示所有函数调用的本机线程。但是，此线程堆栈未考虑运行时编译器内联的Java方法；它不考虑运行时编译器的内联方法。如果内联方法，则它们似乎是父级堆栈框架的一部分。

    本机框架的线程堆栈中的信息提供了有关崩溃原因的重要信息。通过从上至下分析列表中的库，通常可以确定哪个库可能引起了问题，并将其报告给负责该库的适当组织。

  • 第二个线程堆栈是Java frames ，它会打印包含内联方法的Java框架，并跳过本机框架。根据崩溃的情况，可能无法打印本机线程堆栈，但是可能会打印Java框架。

处理部分在线程部分之后打印。

它包含有关整个过程的信息，包括该过程的线程列表和内存使用情况。

一些系统要求交换空间至少为实际物理内存大小的两倍，而其他系统则没有任何要求。通常，如果物理内存和交换空间都快满了，那么就有充分的理由怀疑崩溃是由于内存不足引起的。

在Linux系统上，内核可能会将大多数未使用的物理内存转换为文件缓存。当需要更多内存时，Linux内核会将高速缓存内存返回给应用程序。这由内核透明地处理，但是这意味着，当仍然有足够的物理内存可用时，致命错误处理程序报告的未使用物理内存量可能接近于零。

错误日志的SYSTEM部分中的最终信息是vm_info ，这是嵌入在其中的版本字符串libjvm.so/jvm.dll 。每个Java VM都有自己的独特之处vm_info串。如果您不确定致命错误日志是否由特定的Java VM生成，请检查版本字符串。
