对象排序

一种Listl可以分类如下。

Collections.sort(l);

如果List由组成String元素，它将按字母顺序排序。如果包含Date元素，它将按时间顺序排序。这是怎么发生的？ String 和Date都实现了Comparable接口。 Comparable实现为类提供自然排序 ，这允许该类的对象自动排序。下表总结了实现的一些更重要的Java平台类Comparable 。

如果您尝试对列表进行排序，则其元素不会实现Comparable ， Collections.sort(list)会抛出一个ClassCastException 。同样， Collections.sort(list, comparator)会抛出一个ClassCastException如果您尝试对列表进行排序，则其列表中的元素无法使用comparator 。可以相互比较的元素称为相互可比较 。尽管不同类型的元素可以相互比较，但此处列出的所有类都不允许类间比较。

这是您真正需要了解的所有信息Comparable接口，如果您只是想对可比较元素的列表进行排序或创建它们的排序集合。如果您想实现自己的下一部分，那么下一部分将是您感兴趣的Comparable类型。

编写自己的可比较类型

的Comparable接口由以下方法组成。

public interface Comparable<T> {
    public int compareTo(T o);
}

的compareTo方法将接收对象与指定对象进行比较，并根据接收对象是否小于，等于或大于指定对象，返回负整数，0或正整数。如果无法将指定对象与接收对象进行比较，则该方法将抛出一个ClassCastException 。

的following class representing a person's name贯彻Comparable 。

import java.util.*;

public class Name implements Comparable<Name> {
    private final String firstName, lastName;

    public Name(String firstName, String lastName) {
        if (firstName == null || lastName == null)
            throw new NullPointerException();
        this.firstName = firstName;
        this.lastName = lastName;
    }

    public String firstName() { return firstName; }
    public String lastName()  { return lastName;  }

    public boolean equals(Object o) {
        if (!(o instanceof Name))
            return false;
        Name n = (Name) o;
        return n.firstName.equals(firstName) && n.lastName.equals(lastName);
    }

    public int hashCode() {
        return 31*firstName.hashCode() + lastName.hashCode();
    }

    public String toString() {
	return firstName + " " + lastName;
    }

    public int compareTo(Name n) {
        int lastCmp = lastName.compareTo(n.lastName);
        return (lastCmp != 0 ? lastCmp : firstName.compareTo(n.firstName));
    }
}

为了使前面的示例简短，该类在某种程度上受到限制：它不支持中间名，它需要名字和姓氏，并且不以任何方式进行国际化。但是，它说明了以下重要点：

  • Name对象是不可变的 。在所有其他条件相同的情况下，不可变类型是必经之路，特别是对于将用作元素的对象Set s或作为键Map s。如果您在集合中修改其元素或键，则这些集合将中断。
  • 构造函数检查其参数是否为null 。这样可以确保所有Name对象的格式正确，因此其他方法都不会抛出NullPointerException 。
  • 的hashCode方法已重新定义。这对于重新定义equals方法。（相等的对象必须具有相等的哈希码。）
  • 的equals方法返回false如果指定的对象是null或类型不正确。的compareTo在这种情况下，方法将抛出运行时异常。这两种行为都是相应方法的一般合同所要求的。
  • 的toString方法已重新定义，因此它会打印Name以人类可读的形式。这总是一个好主意，特别是对于将要放入集合中的对象。各种收藏类型” toString方法取决于toString元素，键和值的方法。

由于本节是关于元素顺序的，所以让我们再谈一些Name的compareTo方法。它实现了标准的名称排序算法，其中姓氏优先于名字。这正是您想要的自然顺序。如果自然排序不自然，那确实会造成混乱！

看看如何compareTo实现，因为它很典型。首先，您比较对象的最高有效部分（在本例中为姓）。通常，您可以只使用零件类型的自然顺序。在这种情况下，该部分是String自然的（字典顺序）正是我们所需要的。如果比较得出的结果等于零，则表示等于，那么您就可以完成：您只需返回结果即可。如果最重要的部分相等，则继续比较下一个最重要的部分。在这种情况下，只有两个部分-名和姓。如果有更多部分，则以明显的方式进行操作，比较各部分，直到找到两个不相等的部分，或者您正在比较最低有效部分，此时将返回比较结果。

只是为了证明一切正常，这里是a program that builds a list of names and sorts them 。

import java.util.*;

public class NameSort {
    public static void main(String[] args) {
        Name nameArray[] = {
            new Name("John", "Smith"),
            new Name("Karl", "Ng"),
            new Name("Jeff", "Smith"),
            new Name("Tom", "Rich")
        };

        List<Name> names = Arrays.asList(nameArray);
        Collections.sort(names);
        System.out.println(names);
    }
}

如果您运行此程序，则显示的内容如下。

[Karl Ng, Tom Rich, Jeff Smith, John Smith]

行为的限制有四个compareTo方法，我们将不再讨论，因为它们是相当技术性和无聊的，最好留在API文档中。所有实现的类都非常重要Comparable遵守这些限制，因此请阅读有关文档Comparable如果您正在编写实现它的类。尝试对违反限制的对象列表进行排序具有未定义的行为。从技术上讲，这些限制可确保自然顺序是实现该顺序的类的对象的总顺序 ；这是确保正确定义排序所必需的。

比较器

如果要按自然顺序以外的顺序对某些对象进行排序怎么办？或者，如果您想对一些未实现的对象进行排序，该怎么办Comparable ？要执行上述任一操作，您需要提供一个Comparator —封装顺序的对象。如Comparable界面Comparator接口由单个方法组成。

public interface Comparator<T> {
    int compare(T o1, T o2);
}

的compare方法比较其两个参数，根据第一个参数是否小于，等于或大于第二个参数，返回负整数0或正整数。如果任何一个参数的类型都不适合Comparator ， compare方法抛出一个ClassCastException 。

大部分关于Comparable适用于Comparator也一样写一个compare方法几乎与编写compareTo方法，除了前者获取两个作为参数传递的对象。的compare方法必须遵守与以下四个相同的技术限制Comparable的compareTo出于相同原因的方法— Comparator必须在比较对象上得出总顺序。

假设您有一个名为Employee ， 如下。

public class Employee implements Comparable<Employee> {
    public Name name()     { ... }
    public int number()    { ... }
    public Date hireDate() { ... }
       ...
}

假设的自然排序Employee实例是Name按员工姓名排序（如上例中所定义）。不幸的是，老板要求按年长顺序列出员工名单。这意味着我们必须做一些工作，但不多做。以下程序将生成所需的列表。

import java.util.*;
public class EmpSort {
    static final Comparator<Employee> SENIORITY_ORDER = 
                                        new Comparator<Employee>() {
            public int compare(Employee e1, Employee e2) {
                return e2.hireDate().compareTo(e1.hireDate());
            }
    };

    // Employee database
    static final Collection<Employee> employees = ... ;

    public static void main(String[] args) {
        List<Employee> e = new ArrayList<Employee>(employees);
        Collections.sort(e, SENIORITY_ORDER);
        System.out.println(e);
    }
}

的Comparator在程序中相当简单。它依赖于Date应用于由hireDate访问器方法。请注意Comparator将第二个参数的聘用日期传递给第一个参数，而不是相反。原因是最近雇用的员工的最低年龄；按照聘用日期的顺序进行排序会使列表按相反的年资顺序排列。人们有时用来实现此效果的另一种技术是保持参数顺序，但否定比较结果。

// Don't do this!!
return -r1.hireDate().compareTo(r2.hireDate());

您应该始终使用前一种技术来支持后者，因为不能保证后者可以工作。原因是compareTo方法可以返回任何负数int如果其参数小于调用它的对象。有一个负面的int否定后仍然是负面的，看起来很奇怪。

-Integer.MIN_VALUE == Integer.MIN_VALUE

的Comparator在前面的程序中可以很好地进行排序List ，但确实有一个不足：它不能用于订购已排序的集合，例如TreeSet ，因为它会生成与 equals 不兼容的排序。这意味着Comparator等同于equals方法没有。特别是，在同一日期雇用的任何两名雇员将进行比较。当您排序List ，这无关紧要；但是当您使用Comparator订购分类的收藏品是致命的。如果你用这个Comparator将同一天雇用的多名员工插入TreeSet ，只会将第一个添加到集合中；第二个将被视为重复元素，将被忽略。

要解决此问题，只需调整Comparator这样它就产生了与 equals 。换句话说，对其进行调整，以便在使用时唯一的元素被视为相等compare是那些当使用equals 。这样做的方法是执行两部分比较（至于Name ），第一部分是我们感兴趣的部分（在本例中为雇用日期），第二部分是唯一标识对象的属性。在这里，员工人数是显而易见的属性。这是Comparator结果。

static final Comparator<Employee> SENIORITY_ORDER = 
                                        new Comparator<Employee>() {
    public int compare(Employee e1, Employee e2) {
        int dateCmp = e2.hireDate().compareTo(e1.hireDate());
        if (dateCmp != 0)
            return dateCmp;

        return (e1.number() < e2.number() ? -1 :
               (e1.number() == e2.number() ? 0 : 1));
    }
};

最后一点：您可能会想替换最后一个return中的陈述Comparator与更简单：

return e1.number() - e2.number();

除非您完全确定没有人会有负号，否则请不要这样做！该技巧通常不起作用，因为有符号整数类型不足以表示两个任意有符号整数的差。如果i是一个大的正整数， j是一个大的负整数， i - j将溢出，并返回一个负整数。所结果的comparator违反了我们一直在谈论的四个技术限制（传递性）之一，并产生了可怕的，微妙的错误。这不是纯粹的理论问题；人们被它烧死了。
