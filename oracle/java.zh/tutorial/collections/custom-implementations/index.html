课程：自定义集合实现

许多程序员永远不需要实现自己的Collection的班级。您可以使用本章前面各节中描述的实现进行深入研究。但是，有一天您可能想编写自己的实现。借助于Java平台提供的抽象实现，这样做非常容易。在讨论如何编写实现之前，让我们讨论为什么要编写实现。

编写实现的理由

以下列表说明了自定义的种类Collection您可能想要实现。它并非旨在详尽无遗：

  • 持久性 ：所有内置Collection实现驻留在主内存中，并且在程序退出时消失。如果希望在下次程序启动时仍然存在一个集合，则可以通过在外部数据库上构建单板来实现它。这样的集合可以被多个程序同时访问。
  • 特定于应用程序 ：这是一个非常广泛的类别。一个例子是不可修改的Map包含实时遥测数据。这些键可以表示位置，并且可以响应这些位置从这些位置的传感器读取值get操作。
  • 高性能，专用 ：与通用实现相比，许多数据结构利用受限的使用来提供更好的性能。例如，考虑一个List包含长期运行的相同元素值。这样的列表（在文本处理中经常出现）可以进行游程长度编码 - 游程可以表示为包含重复元素和连续重复次数的单个对象。此示例很有趣，因为它需要权衡两个方面的性能：与ArrayList 。
  • 高性能，通用 ：Java Collections Framework的设计人员试图为每个接口提供最佳的通用实现，但是本可以使用许多很多数据结构，并且每天都在发明新的数据结构。也许您可以更快地提出一些建议！
  • 增强的功能 ：假设您需要高效的bag实现（也称为多集 ）： Collection提供恒定时间的遏制检查，同时允许重复的元素。在顶上实现这样的集合是相当简单的HashMap 。
  • 便利性 ：您可能需要其他实现，这些实现除了Java平台提供的便利之外，还提供其他便利。例如，您可能经常需要List代表连续范围的实例Integer s。
  • 适配器 ：假设您使用的是具有自己专用集合API的旧版API。您可以编写适配器实现，以允许这些集合在Java Collections Framework中运行。适配器实现是一种薄木皮，它通过将后一种类型的操作转换为前一种类型的操作来包装一种类型的对象，并使它们的行为类似于另一种类型的对象。

如何编写自定义实现

编写自定义实现非常容易。Java Collections Framework提供了专门设计用于促进自定义实现的抽象实现。我们将从以下实现示例开始Arrays.asList 。

public static <T> List<T> asList(T[] a) {
    return new MyArrayList<T>(a);
}

private static class MyArrayList<T> extends AbstractList<T> {

    private final T[] a;

    MyArrayList(T[] array) {
        a = array;
    }

    public T get(int index) {
        return a[index];
    }

    public T set(int index, T element) {
        T oldValue = a[index];
        a[index] = element;
        return oldValue;
    }

    public int size() {
        return a.length;
    }
}

信不信由你，这与包含在其中的实现非常接近java.util.Arrays 。就这么简单！您提供了一个构造函数， get ， set和size方法，以及AbstractList剩下的全部。你得到ListIterator ，批量操作，搜索操作，哈希码计算，比较和字符串表示是免费的。

假设您想使实现更快一些。抽象实现的API文档精确描述了每种方法的实现方式，因此您将知道要覆盖哪些方法才能获得所需的性能。前面的实现的性能很好，但是可以改进一点。特别是toArray方法遍历List ，一次复制一个元素。有了内部表示形式，仅克隆阵列就变得更快，更明智。

public Object[] toArray() {
    return (Object[]) a.clone();
}

加上此覆盖和更多类似的覆盖后，此实现正是在java.util.Arrays 。为了完全公开，使用其他抽象实现会有些困难，因为您将不得不编写自己的迭代器，但是仍然没有那么困难。

以下列表总结了抽象的实现：

  • AbstractCollection - 一种Collection那既不是Set也不是List 。至少，您必须提供iterator和size方法。
  • AbstractSet - 一种Set ;用途与AbstractCollection 。
  • AbstractList - 一种List由随机访问数据存储（例如数组）备份。至少，您必须提供positional access方法 （ get以及（可选） set ， remove和add ）和size方法。抽象类负责listIterator （和iterator ）。
  • AbstractSequentialList - 一种List由顺序访问数据存储（例如链表）备份。至少，您必须提供listIterator和size方法。抽象类负责位置访问方法。（这与AbstractList ）
  • AbstractQueue —至少，您必须提供offer ， peek ， poll和size方法和iterator支持remove 。
  • AbstractMap - 一种Map 。至少您必须提供entrySet视图。通常使用AbstractSet类。如果Map是可修改的，您还必须提供put方法。

编写自定义实现的过程如下：

  1. 从前面的列表中选择适当的抽象实现类。
  2. 提供该类的所有抽象方法的实现。如果您的自定义集合是可修改的，那么您还必须覆盖一种或多种具体方法。抽象实现类的API文档将告诉您要覆盖哪些方法。
  3. 测试并在必要时调试实现。现在，您有了一个有效的自定义集合实现。
  4. 如果您担心性能，请阅读抽象实现类的API文档，以了解要继承其实现的所有方法。如果看似太慢，请覆盖它们。如果覆盖任何方法，请确保在覆盖前后测量该方法的性能。您需要花多少精力来调整性能，这取决于实现将获得多少使用以及其使用对性能的重要性。（通常最好省略此步骤。）
