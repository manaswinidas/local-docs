音序器简介

在MIDI的世界，一个序是任何硬件或软件设备，可以精确地播放或录制时间标记的MIDI消息的序列 。同样，在Java Sound API中， Sequencer抽象接口定义可以播放和记录序列的对象的属性MidiEvent对象。一种Sequencer通常加载这些MidiEvent从标准MIDI文件中进行音序或将它们保存到这样的文件中。序列也可以编辑。以下页面说明了如何使用Sequencer对象以及相关的类和接口来完成此类任务。

对什么Sequencer可以想象它与录音机类似，在很多方面它都类似于音序器。磁带录音机播放音频，而音序器播放MIDI数据。音序是MIDI音乐数据的多轨，线性，有时间顺序的录音，音序器可以各种速度播放，倒带，穿梭到特定点，记录或复制到文件中进行存储。

传输和接收MIDI消息说明设备通常具有Receiver对象Transmitter对象，或两者兼而有之。要播放音乐，设备通常会接收MidiMessages通过一个Receiver ，而后者通常是从Transmitter属于Sequencer 。拥有此设备的设备Receiver可能是Synthesizer ，它将直接生成音频，或者可能是MIDI输出端口，该端口通过物理电缆将MIDI数据传输到某些外部设备。同样，要录制音乐，需要设置一系列时间戳MidiMessages通常发送到Receiver拥有Sequencer ，将它们放在Sequence宾语。通常，发送消息的对象是Transmitter与硬件输入端口关联的端口，该端口中继从外部乐器获取的MIDI数据。但是，负责发送消息的设备可能是其他设备Sequencer ，或任何其他具有Transmitter 。此外，如前所述，程序无需使用任何程序即可发送消息Transmitter完全没有

一种Sequencer本身既有Receivers和Transmitters 。在录制时，它实际上获得了MidiMessages通过其Receivers 。在播放期间，它使用Transmitters发送MidiMessages存储在Sequence它已记录（或从文件加载）的内容。

一种思考角色的方法Sequencer Java Sound API中的代码是作为聚合器和“反聚合器”的MidiMessages 。一系列分开的MidiMessages ，每个都是独立的，发送到Sequencer加上自己的时间戳，以标记音乐活动的时间。这些MidiMessages封装在MidiEvent对象并收集在Sequence通过对象的动作Sequencer.record方法。一种Sequence是包含以下内容的聚合的数据结构MidiEvents ，通常代表一系列音符，通常是整首歌曲或一首乐曲。在播放时， Sequencer再次提取MidiMessages来自MidiEvent中的对象Sequence然后将它们传输到一个或多个设备，这些设备要么将它们呈现为声音，保存它们，对其进行修改，要么将它们传递给其他设备。

一些定序器可能既没有发送器也没有接收器。例如，他们可能会创建MidiEvents由于键盘或鼠标事件而从头开始，而不是接收MidiMessages通过Receivers 。同样，他们可以通过直接与内部合成器（实际上可能与音序器是同一对象）通信来播放音乐，而不是发送MidiMessages到一个Receiver与单独的对象关联。但是，本讨论的其余部分假定使用Receivers和Transmitters 。

何时使用音序器

如发送和接收MIDI消息中所述，应用程序可以不使用音序器直接将MIDI消息发送到设备。该程序仅调用Receiver.send每次要发送消息时的方法。这是一种直接的方法，当程序本身实时创建消息时很有用。例如，考虑一个允许用户通过单击屏幕上的钢琴键盘来弹奏音符的程序。当程序获得鼠标按下事件时，它会立即将适当的Note On消息发送到合成器。

如前所述，程序可以在每个发送到设备接收器的MIDI消息中包含一个时间戳。但是，此类时间戳仅用于微调时序，以校正处理延迟。呼叫者通常无法设置任意时间戳；传递给的时间值Receiver.send必须接近当前时间，否则接收设备可能无法正确安排邮件。这意味着，如果应用程序想要为整个音乐提前创建MIDI消息队列（而不是为响应实时事件而创建每个消息），则必须非常谨慎地安排每个音乐调用Receiver.send在几乎正确的时间。

幸运的是，大多数应用程序不必关心这种调度。而不是调用Receiver.send本身，程序可以使用Sequencer对象来管理它的MIDI消息队列。音序器负责安排和发送消息，换句话说，以正确的时间播放音乐。通常，每当需要将MIDI消息的非实时序列转换为实时序列（如在播放中），反之亦然（如记录中）时，使用音序器是有利的。音序器最常用于播放MIDI文件中的数据和录制来自MIDI输入端口的数据。

了解序列数据

在检查之前Sequencer API，它有助于了解序列中存储的数据类型。

音序和音轨

在Java Sound API中，音序器以组织录制的MIDI数据的方式严格遵循“标准MIDI文件”规范。如上所述， Sequence是...的集合MidiEvents ，及时组织。但是还有更多的结构Sequence不仅仅是线性系列MidiEvents ： 一种Sequence实际上包含全局时序信息以及Tracks ，这是Tracks自己持有MidiEvent数据。因此，音序器播放的数据由三层对象组成： Sequencer ， Track和MidiEvent 。

在这些对象的常规使用中， Sequence代表完整的音乐作品或作品的一部分，每个Track对应于合奏中的声音或播放器。在此模型中，所有数据都在特定的Track因此也将被编码到为该声音或播放器保留的特定MIDI通道中。

这种组织数据的方式对于编辑序列很方便，但是请注意，这只是常规的使用方式Tracks 。定义中没有任何内容Track使其无法包含以下各项的类MidiEvents在不同的MIDI通道上例如，可以混合整个多通道MIDI合成并将其录制到一个Track 。同样，根据定义，类型0（与类型1和类型2相对）的标准MIDI文件仅包含一个轨道。所以Sequence从这样的文件中读取的文件必然会有一个Track宾语。

MidiEvents和壁虱

如MIDI包概述中所述 ，Java Sound API包括MidiMessage与构成大多数标准MIDI消息的原始两字节或三字节序列相对应的对象。一种MidiEvent只是一个包装MidiMessage以及指定事件发生时间的随附计时值。（然后，我们可以说一个序列实际上是由四或五级数据层次结构组成，而不是三级数据，因为表面上的最低级别， MidiEvent ，实际上包含一个较低级别的MidiMessage ，同样MidiMessage对象包含包含标准MIDI消息的字节数组。）

在Java Sound API中，有两种不同的方法MidiMessages可以与时序值关联。一种是上面“何时使用音序器”中提到的方法。在不使用发送器的情况下向接收者发送消息和理解时间戳的过程中详细描述了此技术。在那里，我们看到了send的方法Receiver需要一个MidiMessage参数和时间戳参数。这种时间戳只能用微秒表示。

另一种方式MidiMessage可以通过将其封装在MidiEvent 。在这种情况下，时间用称为ticks的抽象单位表示。

tick的持续时间是多长？它可以在序列之间变化（但不能在序列中变化），并且其值存储在标准MIDI文件的标头中。刻度的大小以两种类型的单位之一给出：

  • 每四分音符的脉冲数（滴答声），缩写为PPQ
  • 每帧刻度，也称为SMPTE时间码（电影电视工程师协会采用的标准）

如果单位是PPQ，则刻度的大小表示为四分音符的分数，它是一个相对而非绝对的时间值。四分音符是一个音乐持续时间值，通常对应于音乐的一个节拍（4/4次中小节的四分之一）。四分音符的持续时间取决于速度，如果序列中包含速度变化事件，则该速度会在音乐过程中发生变化。因此，如果序列的定时增量（滴答声）出现了，例如每四分音符出现96次，则每个事件的定时值将以音乐来衡量该事件的位置，而不是绝对时间值。

另一方面，在SMPTE的情况下，单位测量的是绝对时间，因此节奏概念不适用。实际上有四种不同的SMPTE约定可用，它们是指每秒的运动图像帧数。每秒的帧数可以是24、25、29.97或30。使用SMPTE时间码时，刻度的大小表示为一帧的一部分。

在Java Sound API中，您可以调用Sequence.getDivisionType了解特定顺序使用哪种类型的单位（即PPQ或SMPTE单位之一）。然后，您可以在调用后计算刻度的大小Sequence.getResolution 。如果划分类型为PPQ，则后一种方法返回每四分音符的刻度数；如果划分类型为SMPTE约定之一，则后一种方法返回每SMPTE帧的刻度数。对于PPQ，您可以使用以下公式获得刻度的大小：

ticksPerSecond =  
    resolution * (currentTempoInBeatsPerMinute / 60.0);
tickSize = 1.0 / ticksPerSecond;

对于SMPTE，此公式为：

framesPerSecond = 
  (divisionType == Sequence.SMPTE_24 ? 24
    : (divisionType == Sequence.SMPTE_25 ? 25
      : (divisionType == Sequence.SMPTE_30 ? 30
        : (divisionType == Sequence.SMPTE_30DROP ?

            29.97))));
ticksPerSecond = resolution * framesPerSecond;
tickSize = 1.0 / ticksPerSecond;

Java Sound API在序列中对时序的定义与标准MIDI文件规范的镜像相同。但是，有一个重要的区别。包含在中的报价值MidiEvents测量累计时间，而不是增量时间。在标准MIDI文件中，每个事件的定时信息都测量自序列中上一个事件开始以来经过的时间。这称为增量时间。但是在Java Sound API中，刻度不是增量值；它们是前一个事件的时间值加上差值。换句话说，在Java Sound API中，每个事件的时序值始终大于序列中前一个事件的时序值（如果假定这些事件是同时发生的，则该值等于）。每个事件的计时值衡量自序列开始以来经过的时间。

总而言之，Java Sound API以MIDI滴答声或微秒表示计时信息。 MidiEvents根据MIDI滴答存储时序信息。tick的持续时间可以从Sequence's全局定时信息，如果序列使用基于速度的定时，则为当前音乐速度。与MidiMessage发送到Receiver另一方面，始终以微秒表示。

这种设计的一个目标是避免时间观念的冲突。这是一个工作Sequencer解释其时间单位MidiEvents ，它可能具有PPQ单位，并考虑到当前速度将其转换为绝对时间（以微秒为单位）。定序器还必须表示相对于打开接收消息的设备的时间的微秒。请注意，定序器可以具有多个发送器，每个发送器将消息传递到可能与完全不同的设备关联的不同接收器。然后，您会看到，定序器必须能够同时执行多个转换，并确保每个设备都收到适合其时间概念的时间戳。

使事情变得更加复杂的是，不同的设备可能会基于不同的来源（例如操作系统的时钟或声卡维护的时钟）来更新其时间概念。这意味着它们的时序可能会相对于音序器漂移。为了与定序器保持同步，某些设备允许自己成为定序器的时间概念的“奴隶”。设置主机和从机将在下面讨论MidiEvent 。
