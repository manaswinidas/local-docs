使用高级音序器功能

到目前为止，我们专注于MIDI数据的简单播放和记录。本节将简要介绍通过以下方法可以使用的一些更高级的功能Sequencer界面和Sequence类。

移至序列中的任意位置

那里有两个Sequencer获得定序器当前序列位置的方法。第一个：

long getTickPosition()

从序列的开头返回以MIDI节拍为单位的位置。第二种方法：

long getMicrosecondPosition()

返回当前位置（以微秒为单位）。此方法假定序列以存储在MIDI文件或MIDI文件中的默认速率播放。 Sequence 。如果您按照以下说明更改了播放速度，则不会返回其他值。

您可以根据一个或另一个单位类似地设置音序器的当前位置：

void setTickPosition(long tick)

要么

void setMicrosecondPosition(long microsecond)

改变播放速度

如前所述，序列的速度由其速度表示，其速度会随着序列的变化而变化。序列可以包含封装标准MIDI速度改变消息的事件。当音序器处理此类事件时，它会更改播放速度以反映所指示的速度。此外，您可以通过调用以下任意一种方式以编程方式更改速度Sequencer方法：

    public void setTempoInBPM(float bpm)
    public void setTempoInMPQ(float mpq)
    public void setTempoFactor(float factor)

这些方法中的前两种分别以每分钟节拍或每四分音符微秒为单位设置节奏。速度将保持在指定的值，直到再次调用这些方法之一，或者直到序列中遇到速度变化事件为止，此时新的指定速度将覆盖当前速度。

第三种方法setTempoFactor ，本质上是不同的。它可缩放为定序器设置的任何速度（无论是通过速度变化事件还是通过上述前两种方法之一）。默认标量为1.0（不变）。尽管此方法使播放或录制的速度快于或慢于标称速度（除非系数为1.0），但它不会改变标称速度。换句话说，由getTempoInBPM和getTempoInMPQ即使速度因素确实影响播放或录制的实际速率，也不受速度因素的影响。同样，如果通过速度变化事件或前两种方法之一来改变速度，它仍然会根据上次设置的速度因子进行缩放。但是，如果加载新序列，则速度因子会重置为1.0。

请注意，当序列的分割类型是SMPTE类型之一而不是PPQ时，所有这些速度变化指令均无效。

静音或独奏序列中的单个音轨

音序器用户通常可以方便地关闭某些音轨，以更清楚地听到音乐中正在发生的事情。功能齐全的音序器程序使用户可以选择在播放过程中应该播放哪些音轨。（更准确地说，由于音序器实际上不会自己创建声音，因此用户可以选择哪些音轨会影响音序器产生的MIDI消息流。）通常，每个轨道上都有两种图形控件： 静音按钮和独奏按钮。如果启用了静音按钮，则在禁用静音按钮之前，该音轨在任何情况下都不会发出声音。独奏是鲜为人知的功能。大致与静音相反。如果激活任何轨道上的独奏按钮，则仅会激活其独奏按钮被激活的轨道。此功能使用户可以快速试听少量轨道，而不必使所有其他轨道静音。静音按钮通常比独奏按钮优先：如果两个按钮均被激活，则不会发出声音。

使用Sequencer方法，静音或独奏曲目（以及查询曲目的当前静音或独奏状态）很容易实现。假设我们已经获得了默认值Sequencer并且我们已经将序列数据加载到其中。将序列中的第五条音轨静音可以如下完成：

    sequencer.setTrackMute(4, true);
    boolean muted = sequencer.getTrackMute(4);
    if (!muted) { 
        return;         // muting failed
    }

关于上述代码段，需要注意几件事。首先，序列的轨道从0开始编号，以轨道总数减去1结束。另外，第二个论点setTrackMute是一个布尔值。如果为真，则请求将轨道静音；否则，请求是取消静音指定的曲目。最后，为了测试静音是否生效，我们调用了Sequencer getTrackMute方法，向其传递我们正在查询的曲目号。如果返回true ，就像我们在这种情况下所期望的那样，则静音请求起作用了。如果返回false ，然后失败了。

静音请求可能由于各种原因而失败。例如，在setTrackMute呼叫可能会超出曲目总数，或者音序器可能不支持静音。通过致电getTrackMute ，我们可以确定请求是成功还是失败。

顺便说一句，返回的布尔值getTrackMute确实可以告诉我们是否发生了故障，但无法告诉我们为什么发生。我们可以测试一下是否由于将无效的曲目号传递给setTrackMute方法。为此，我们将getTracks的方法Sequence ，它返回一个包含序列中所有轨道的数组。如果在setTrackMute调用超出了此数组的长度，那么我们知道我们指定了无效的轨道号。

如果静音请求成功，则在我们的示例中，在播放序列时，第五音轨将不会发出声音，而当前已静音的任何其他音轨也不会发出声音。

独奏音轨的方法和技术与静音的方法和技术非常相似。要独奏曲目，请调用setTrackSolo的方法Sequence:

void setTrackSolo(int track, boolean bSolo)

如setTrackMute ，第一个参数指定从零开始的音轨编号，第二个参数指定true ，指定轨道应处于独奏模式；否则，曲目不应独奏。

默认情况下，曲目既不会静音也不会单独播放。

与其他MIDI设备同步

Sequencer有一个叫做Sequencer.SyncMode 。一种SyncMode object表示MIDI音序器的时间概念可以与主设备或从设备同步的方式之一。如果音序器正在与主机同步，则音序器会响应于来自主机的某些MIDI消息修改其当前时间。如果音序器具有从机，则音序器会类似地发送MIDI消息以控制从机的定时。

有三种预定义的模式，它们为音序器指定可能的母版： INTERNAL_CLOCK ， MIDI_SYNC和MIDI_TIME_CODE 。如果音序器从另一台设备接收MIDI消息，则后两者可以工作。在这两种模式下，分别基于系统实时定时时钟消息或MIDI时间代码（MTC）消息重置音序器的时间。（有关这些消息类型的更多信息，请参见MIDI规范。）这两种模式也可以用作从机模式，在这种情况下，音序器会将相应类型的MIDI消息发送到其接收器。第四种模式NO_SYNC用于指示定序器不应向其接收器发送定时信息。

通过调用setMasterSyncMode具有支持的方法SyncMode对象作为参数，您可以指定如何控制音序器的时序。同样， setSlaveSyncMode该方法确定定序器将向其接收器发送什么定时信息。此信息控制使用定序器作为主定时源的设备的定时。

指定特殊事件侦听器

序列的每个轨道可以包含许多不同种类的MidiEvents 。这些事件包括Note On和Note Off消息，程序更改，控件更改和元事件。Java Sound API为这些事件类型的后两种（控件更改事件和元事件）指定“侦听器”接口。当序列播放期间发生此类事件时，您可以使用这些接口来接收通知。

支持ControllerEventListener界面可以在以下情况下接收通知： Sequencer处理特定的控制更改消息。控制更改消息是MIDI消息的标准类型，它表示MIDI控制器（例如弯音轮或数据滑块）的值发生变化。（有关控制更改消息的完整列表，请参见MIDI规范。）在序列播放期间处理此类消息时，该消息会指示从定序器接收数据的任何设备（可能是合成器）更新某些参数的值。该参数通常控制声音合成的某些方面，例如，如果控制器是弯音轮，则当前听起来的音符的音高。当记录序列时，控制更改消息意味着创建该消息的外部物理设备上的控制器已经移动，或者已经用软件模拟了这种移动。

这是ControllerEventListener使用接口。假设您已经开发了一个实现ControllerEventListener接口，这意味着您的类包含以下方法：

    void controlChange(ShortMessage msg)

我们还假设您已经创建了一个类的实例，并将其分配给一个名为myListener 。如果在程序中的某些位置包含以下语句：

    int[] controllersOfInterest = { 1, 2, 4 };
    sequencer.addControllerEventListener(myListener,
        controllersOfInterest);

然后你班的controlChange每当音序器处理MIDI控制器编号1、2或4的控制更改消息时，都会调用此方法。换句话说，当Sequencer处理一个设置任何已注册控制器的值的请求， Sequencer将调用您班级的controlChange方法。（请注意，在MIDI 1.0规范中详细说明了将MIDI控制器编号分配给特定控制设备的情况。）

的controlChange方法通过了ShortMessage包含受影响的控制器编号，以及设置了控制器的新值。您可以使用ShortMessage.getData1方法，以及使用ShortMessage.getData2方法。

另一种特殊事件侦听器由MetaEventListener接口。根据标准MIDI文件1.0规范，元消息是MIDI有线协议中不存在但可以嵌入MIDI文件中的消息。它们对合成器没有意义，但可以由音序器解释。元消息包括指令（例如速度改变命令），歌词或其他文本以及其他指示符（例如音轨结尾）。

的MetaEventListener机制类似于ControllerEventListener 。实施MetaEventListener接口的任何类中的实例，当实例MetaMessage由定序器处理。这涉及向类添加以下方法：

void meta(MetaMessage msg)

通过将此类的实例作为参数传递给Sequencer addMetaEventListener方法：

boolean b = sequencer.addMetaEventListener
        (myMetaListener);

这与采取的方法略有不同。 ControllerEventListener界面，因为您必须注册才能接收全部MetaMessages,不只是感兴趣的人。如果音序器遇到MetaMessage按顺序，它将调用myMetaListener.meta传递给MetaMessage遇到。的meta方法可以调用getType在它的MetaMessage参数获取0到127之间的整数，该整数表示消息类型，这是标准MIDI文件1.0规范定义的。
