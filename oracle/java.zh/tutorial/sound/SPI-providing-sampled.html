提供音频采样服务

如您所知，Java Sound API包含两个软件包， javax.sound.sampled.spi和javax.sound.midi.spi定义了声音服务开发人员要使用的抽象类。通过实现和安装这些抽象类之一的子类，服务提供者可以注册新服务，从而扩展了运行时系统的功能。此页面告诉您如何使用javax.sound.sampled.spi包，以提供用于处理采样音频的新服务。

共有四个抽象类javax.sound.sampled.spi软件包，代表可以为采样音频系统提供的四种不同类型的服务：

  • AudioFileWriter提供完善的文件编写服务。这些服务使应用程序可以将音频数据流写入特定类型的文件。
  • AudioFileReader提供文件读取服务。这些服务使应用程序能够确定声音文件的特性，并获得可以从中读取文件的音频数据的流。
  • FormatConversionProvider提供用于转换音频数据格式的服务。这些服务允许应用程序将音频流从一种数据格式转换为另一种数据格式。
  • MixerProvider提供对特定类型混合器的管理。该机制允许应用程序获取有关给定类型的混合器的信息并访问其实例。

为了概括前面的讨论，服务提供商可以扩展运行时系统的功能。典型的SPI类有两种类型的方法：一种响应特定提供商提供的服务类型的查询，另一种直接执行新服务或返回实际提供服务的对象实例。运行时环境的服务提供者机制可向音频系统注册已安装服务，并管理新服务提供者类。

本质上，服务实例与应用程序开发人员存在双重隔离。应用程序永远不会直接创建其音频处理任务所需的服务对象实例，例如混合器或格式转换器。该程序甚至也不直接从管理它们的SPI类中直接请求这些对象。应用程序向AudioSystem中的对象javax.sound.sampled包装，以及AudioSystem依次使用SPI对象处理这些查询和服务请求。

新音频服务的存在可能对用户和应用程序程序员都是完全透明的。所有应用程序引用都是通过javax.sound.sampled包装，主要AudioSystem ，而新服务可能提供的特殊处理通常被完全隐藏。

在此讨论中，我们将继续以前的约定，即使用类似名称来引用新的SPI子类。 AcmeMixer和AcmeMixerProvider 。

提供音频文件编写服务

让我们开始AudioFileWriter ，是较简单的SPI类之一。

实现以下方法的子类AudioFileWriter必须提供一组方法的实现，以处理有关该类支持的文件格式和文件类型的查询，还必须提供将提供的音频数据流实际写出到File要么OutputStream 。

AudioFileWriter包括两个在基类中具有具体实现的方法：

boolean isFileTypeSupported(AudioFileFormat.Type fileType) 
boolean isFileTypeSupported(AudioFileFormat.Type fileType, AudioInputStream stream)

这些方法中的第一种方法通知调用者此文件编写器是否可以写入指定类型的声音文件。此方法是一般查询，它将返回true如果文件写入器可以写入该类型的文件，则假定该文件写入器已获得适当的音频数据。但是，写入文件的能力可能取决于传递给文件编写器的特定音频数据的格式。文件编写器可能不支持每种音频数据格式，或者文件格式本身可能会施加约束。（并非所有类型的音频数据都可以写入各种声音文件。）第二种方法更具体，然后询问是否特定AudioInputStream可以写入特定类型的文件。

通常，您不需要重写这两个具体方法。每个只是一个包装器，它调用其他两个查询方法之一并遍历返回的结果。这另外两个查询方法是抽象的，因此需要在子类中实现：

abstract AudioFileFormat.Type[] getAudioFileTypes() 
abstract AudioFileFormat.Type[] getAudioFileTypes(AudioInputStream stream)

这些方法直接对应于前两种方法。每个方法都返回一个数组，其中包含所有受支持的文件类型-第一种方法通常支持所有文件类型，而第二种方法则支持特定音频流支持的所有文件类型。第一种方法的典型实现可能只是返回文件编写器的构造函数初始化的数组。第二种方法的实现可能会测试流的AudioFormat对象以查看其是否为请求的文件类型支持的数据格式。

最后两种方法AudioFileWriter做实际的文件写入工作：

abstract int write(AudioInputStream stream, 
     AudioFileFormat.Type fileType, java.io.File out) 
abstract int write(AudioInputStream stream, 
     AudioFileFormat.Type fileType, java.io.OutputStream out)

这些方法将代表音频数据的字节流写入第三个参数指定的流或文件。具体操作方式取决于指定文件类型的结构。的write方法必须以这种格式的声音文件规定的方式写入文件的标题和音频数据（无论是标准类型的声音文件还是新的可能专有的声音文件）。

提供音频文件阅读服务

的AudioFileReader类由子类实际需要实现的六个抽象方法组成，两个不同的重载方法，每个方法都可以采用File ， URL ， 要么InputStream论点。这些重载方法中的第一个接受有关指定文件的文件格式的查询：

abstract AudioFileFormat getAudioFileFormat(java.io.File file) 
abstract AudioFileFormat getAudioFileFormat(java.io.InputStream stream) 
abstract AudioFileFormat getAudioFileFormat(java.net.URL url)

典型的实现getAudioFileFormat方法读取并解析声音文件的标头，以确定其文件格式。请参阅AudioFileFormat类的描述，以了解需要从标头读取哪些字段，并参考特定文件类型的规范以了解如何解析标头。

因为提供流作为此方法的参数的调用方希望该方法不更改该流，所以文件读取器通常应从标记该流开始。读取到标头的末尾后，应将流重置为其原始位置。

另一个超载AudioFileReader方法通过返回一个AudioInputStream来提供文件读取服务，从该文件中可以读取文件的音频数据：

abstract AudioInputStream getAudioInputStream(java.io.File file) 
abstract AudioInputStream getAudioInputStream(java.io.InputStream stream) 
abstract AudioInputStream getAudioInputStream(java.net.URL url)

通常， getAudioInputStream返回一个AudioInputStream缠绕到文件数据块的开头（在标头之后），以备读取。但是，可以想象，文件读取器返回一个AudioInputStream其音频格式表示从文件中包含的内容以某种方式解码的数据流。重要的是该方法返回一个格式化的流，可以从中读取文件中包含的音频数据。的AudioFormat封装在返回中AudioInputStream物件会告知呼叫者串流的资料格式，通常但不一定与档案本身的资料格式相同。

通常，返回的流是的实例AudioInputStream ;您不太可能需要继承AudioInputStream 。

提供格式转换服务

一种FormatConversionProvider子类转换AudioInputStream将一种音频数据格式转换为另一种格式。前一个（输入）流称为源流 ，后一个（输出）流称为目标流。回想一下AudioInputStream包含一个AudioFormat和AudioFormat依次包含一种特定类型的数据编码，由AudioFormat.Encoding宾语。源流中的格式和编码称为源格式和源编码，目标流中的格式和编码也称为目标格式和目标编码。

转换工作以重载的抽象方法执行FormatConversionProvider叫getAudioInputStream 。该类还具有抽象查询方法，用于了解所有受支持的目标和源格式以及编码。有一些用于查询特定转换的具体包装方法。

的两个变体getAudioInputStream是：

abstract AudioInputStream getAudioInputStream(AudioFormat.Encoding targetEncoding, 
     AudioInputStream sourceStream)

和

abstract AudioInputStream getAudioInputStream(AudioFormat targetFormat, 
     AudioInputStream sourceStream)

根据调用方是指定完整目标格式还是仅格式的编码，这些参数在第一个参数中有所不同。

典型的实现getAudioInputStream通过返回一个新的子类来工作AudioInputStream环绕原始文件（来源） AudioInputStream并在以下情况下对其数据进行数据格式转换： read方法被调用。例如，考虑新的情况FormatConversionProvider子类称为AcmeCodec ，它可以与新AudioInputStream子类称为AcmeCodecStream 。

实施AcmeCodec's第二getAudioInputStream方法可能是：

public AudioInputStream getAudioInputStream
      (AudioFormat outputFormat, AudioInputStream stream) {
        AudioInputStream cs = null;
        AudioFormat inputFormat = stream.getFormat();
        if (inputFormat.matches(outputFormat) ) {
            cs = stream;
        } else {
            cs = (AudioInputStream)
                (new AcmeCodecStream(stream, outputFormat));
            tempBuffer = new byte[tempBufferSize];
        }
        return cs;
    }

实际的格式转换发生在新的read返回的方法AcmeCodecStream ，是的子类AudioInputStream 。再次，访问此内容的应用程序返回AcmeCodecStream只需将其作为AudioInputStream ，无需了解其实施的详细信息。

其他方法FormatConversionProvider所有这些都允许查询有关对象支持的输入和输出编码以及格式。需要实现以下四种抽象方法：

abstract AudioFormat.Encoding[] getSourceEncodings() 
abstract AudioFormat.Encoding[] getTargetEncodings() 
abstract AudioFormat.Encoding[] getTargetEncodings(
    AudioFormat sourceFormat) 
abstract  AudioFormat[] getTargetFormats(
    AudioFormat.Encoding targetEncoding, 
    AudioFormat sourceFormat)

如在查询方法中AudioFileReader在上面讨论的类中，这些查询通常通过检查对象的私有数据并针对后两种方法将它们与参数进行比较来处理。

其余四个FormatConversionProvider方法是具体的，通常不需要重写：

boolean isConversionSupported(
    AudioFormat.Encoding targetEncoding,
    AudioFormat sourceFormat) 
boolean isConversionSupported(AudioFormat targetFormat, 
    AudioFormat sourceFormat) 
boolean isSourceEncodingSupported(
    AudioFormat.Encoding sourceEncoding) 
boolean isTargetEncodingSupported(
    AudioFormat.Encoding targetEncoding)

与AudioFileWriter.isFileTypeSupported() ，这些方法中每个方法的默认实现实质上是一个包装器，该包装器调用其他查询方法之一并遍历返回的结果。

提供新型搅拌机

顾名思义， MixerProvider提供混合器实例。每个混凝土MixerProvider子类充当Mixer应用程序使用的对象。当然，定义一个新的MixerProvider仅在以下情况下有意义： Mixer接口也被定义。如在FormatConversionProvider上面的例子getAudioInputStream方法返回的子类AudioInputStream进行了转换的新班级AcmeMixerProvider有一种方法getMixer返回另一个新类的实例，该实例实现Mixer接口。我们称之为后一类AcmeMixer 。特别是如果混合器是通过硬件实现的，则提供程序可能仅支持所请求设备的一个静态实例。如果是这样，它应返回此静态实例以响应对的每次调用getMixer 。

以来AcmeMixer支持Mixer接口，应用程序不需要任何其他信息即可访问其基本功能。但是，如果AcmeMixer支持未在Mixer接口，并且供应商希望使该扩展功能可被应用程序访问，当然应该将混合器定义为带有附加的，经过良好记录的公共方法的公共类，以便希望使用此扩展功能的程序可以进口AcmeMixer并投射由返回的对象getMixer这种类型。

其他两种方法MixerProvider是：

abstract Mixer.Info[] getMixerInfo()

和

boolean isMixerSupported(Mixer.Info info)

这些方法允许音频系统确定此特定提供程序类是否可以产生应用程序需要的设备。换句话说， AudioSystem对象可以遍历所有已安装的对象MixerProviders看看有哪些可以提供应用程序请求的设备AudioSystem 。的getMixerInfo方法返回一个对象数组，其中包含有关此提供程序对象可用的混合器类型的信息。系统可以将这些信息对象以及其他提供程序中的信息对象传递给应用程序。

一个MixerProvider可以提供多种混合器。当系统调用MixerProvider's getMixerInfo方法，它获取信息对象的列表，这些信息对象标识此提供程序支持的各种混合器。然后系统可以调用MixerProvider.getMixer(Mixer.Info)获得感兴趣的每个混频器。

您的子类需要实现getMixerInfo ，因为它很抽象。的isMixerSupported方法是具体的，通常不需要重写。默认实现只是比较提供的Mixer.Info到返回的数组中的每个getMixerInfo 。
