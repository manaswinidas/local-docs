提供MIDI服务

服务提供商接口简介介绍了javax.sound.sampled.spi和javax.sound.midi.spi包定义了声音服务开发人员要使用的抽象类。通过实现这些抽象类之一的子类，服务提供者可以创建扩展了运行时系统功能的新服务。上一节介绍了javax.sound.sampled.spi包。本节讨论如何使用javax.sound.midi.spi包，以提供用于处理MIDI设备和文件的新服务。

共有四个抽象类javax.sound.midi.spi软件包，代表可以为MIDI系统提供的四种不同类型的服务：

  • MidiFileWriter提供MIDI文件写入服务。这些服务使应用程序可以将MIDI保存到MIDI文件中。 Sequence它已经生成或处理的。
  • MidiFileReader提供返回MIDI的文件读取服务Sequence从MIDI文件中供应用程序使用。
  • MidiDeviceProvider提供一种或多种特定类型的MIDI设备（可能包括硬件设备）的实例。
  • SoundbankReader提供声音库文件读取服务。的具体子类SoundbankReader解析给定的音库文件，产生一个Soundbank可以加载到Synthesizer 。

应用程序将不会直接创建服务对象的实例-无论是提供者对象，例如MidiDeviceProvider ，或对象，例如Synthesizer ，由提供者对象提供。该程序也不会直接引用SPI类。而是，应用程序向MidiSystem中的对象javax.sound.midi包装，以及MidiSystem依次使用的具体子类javax.sound.midi.spi类来处理这些请求。

提供MIDI文件写入服务

共有三种标准MIDI文件格式，Java Sound API的实现可以支持所有三种格式：Type 0，Type 1和Type 2。这些文件格式在文件中的MIDI序列数据的内部表示形式方面有所不同，并且适用于不同种类的序列。如果实现本身不支持所有三种类型，则服务提供商可以为未实现的类型提供支持。标准MIDI文件格式也有一些变体，其中一些是专有的，第三方供应商也可以类似地支持它们。

MIDI文件的写入能力由以下的具体子类提供： MidiFileWriter 。这个抽象类直接类似于javax.sampled.spi.AudioFileWriter 。同样，这些方法分为查询方法和用于实际写入文件的方法，查询方法用于学习可以写入哪些类型的文件。与AudioFileWriter ，其中两种查询方法是具体的：

boolean isFileTypeSupported(int fileType)
boolean isFileTypeSupported(int fileType, Sequence sequence)

其中的第一个提供有关文件编写器是否可以写入指定类型的MIDI文件类型的常规信息。第二种方法更具体：询问是否可以将特定音序写入指定类型的MIDI文件。通常，您不需要覆盖这两个具体方法中的任何一个。在默认实现中，每个调用另一个两个相应查询方法之一，并迭代返回的结果。作为抽象，其他两个查询方法需要在子类中实现：

abstract int[] getMidiFileTypes() 
abstract int[] getMidiFileTypes(Sequence sequence)

其中第一个返回通常支持的所有文件类型的数组。典型的实现可能会在文件编写器的构造函数中初始化数组，然后从此方法返回数组。从该组文件类型中，第二种方法找到文件编写器可以向其写入给定Sequence的子集。根据MIDI规范，并非所有类型的音序都可以写入所有类型的MIDI文件。

的write一个的方法MidiFileWriter子类执行给定数据的编码Sequence转换为所需类型的MIDI文件的正确数据格式，然后将编码流写入文件或输出流：

abstract int write(Sequence in, int fileType, 
                   java.io.File out) 
abstract int write(Sequence in, int fileType, 
                   java.io.OutputStream out)

为此， write方法必须解析Sequence通过遍历其轨道，构造适当的文件头，然后将头和轨道写入输出。MIDI文件的标题格式当然是由MIDI规范定义的。它包括诸如“幻数”之类的信息，以将其标识为MIDI文件，标题的长度，音轨数以及序列的定时信息（分割类型和分辨率）。MIDI文件的其余部分由音轨数据组成，格式为MIDI规范定义。

让我们简要地看一下应用程序，MIDI系统和服务提供商在编写MIDI文件时是如何配合的。在典型情况下，应用程序具有特定的MIDI Sequence保存到文件。程序查询MidiSystem对象以查看特定MIDI文件支持哪些MIDI文件格式（如果有） Sequence在尝试写入文件之前。的MidiSystem.getMidiFileTypes(Sequence)方法返回系统可以向其写入特定序列的所有MIDI文件类型的数组。它通过调用相应的getMidiFileTypes每个已安装方法MidiFileWriter服务，并以整数数组形式收集和返回结果，这些整数可以视为与给定兼容的所有文件类型的主列表Sequence 。说到写Sequence到文件，调用MidiSystem.write传递了代表文件类型的整数，以及Sequence待写入和输出文件； MidiSystem使用提供的类型来决定安装哪个MidiFileWriter应该处理写请求，并分派相应的write适当的MidiFileWriter 。

提供MIDI文件读取服务

的MidiFileReader抽象类直接类似于javax.sampled.spi.AudioFileReader类。两者都包含两个重载方法，每个方法都可以采用File ， URL ， 要么InputStream论点。第一个重载方法返回指定文件的文件格式。如果是MidiFileReader ，API为：

abstract MidiFileFormat getMidiFileFormat(java.io.File file) 
abstract MidiFileFormat getMidiFileFormat(
    java.io.InputStream stream) 
abstract MidiFileFormat getMidiFileFormat(java.net.URL url)

具体的子类必须实现这些方法以返回填写的内容MidiFileFormat一个对象，它描述指定的MIDI文件（或流或URL）的格式，假设该文件是文件读取器支持的类型，并且包含有效的头信息。否则， InvalidMidiDataException应该扔。

另一种重载方法返回MIDI Sequence来自给定的文件，流或URL：

abstract Sequence getSequence(java.io.File file) 
abstract Sequence getSequence(java.io.InputStream stream) 
abstract Sequence getSequence(java.net.URL url)

的getSequence方法执行解析MIDI输入文件中的字节并构造对应的字节的实际工作Sequence宾语。从本质上讲，这与MidiFileWriter.write 。因为在MIDI规范定义的MIDI文件的内容与一个MIDI文件之间存在一一对应的关系Sequence由Java Sound API定义的对象，解析的细节很简单。如果文件传递到getSequence包含文件阅读器无法解析的数据（例如，由于文件已损坏或不符合MIDI规范）， InvalidMidiDataException应该扔。

提供特定的MIDI设备

一种MidiDeviceProvider可以被视为提供一种或多种特定类型的MIDI设备的工厂。该类包括一个返回MIDI设备实例的方法，以及查询该提供程序可以提供哪些设备的查询方法。

和其他一样javax.sound.midi.spi服务，应用程序开发人员可以间接访问MidiDeviceProvider通过致电服务MidiSystem方法，在这种情况下MidiSystem.getMidiDevice和MidiSystem.getMidiDeviceInfo 。子类化的目的MidiDeviceProvider是为了提供一种新型设备，因此服务开发人员还必须为返回的设备创建一个伴随类，就像我们看到的那样MixerProvider在里面javax.sound.sampled.spi包。在那里，返回的设备的类实现了javax.sound.sampled.Mixer接口;在这里它实现了javax.sound.midi.MidiDevice接口。它还可能实现以下子接口MidiDevice ， 如Synthesizer要么Sequencer 。

因为一个子类MidiDeviceProvider可以提供多种类型的MidiDevice ， getDeviceInfo类的方法返回一个数组MidiDevice.Info枚举不同的对象MidiDevices可用的：

abstract MidiDevice.Info[] getDeviceInfo()

当然，返回的数组可以包含一个元素。提供程序的典型实现可能会在其构造函数中初始化一个数组，然后在此处将其返回。这允许MidiSystem遍历所有已安装的MidiDeviceProviders构造所有已安装设备的列表。 MidiSystem然后可以返回此列表（ MidiDevice.Info[]数组）到应用程序。

MidiDeviceProvider还包括一个具体的查询方法：

boolean isDeviceSupported(MidiDevice.Info info)

此方法允许系统向提供者查询特定类型的设备。通常，您不需要重写此便捷方法。默认实现对getDeviceInfo返回的数组进行迭代，并将参数与每个元素进行比较。

第三也是最后MidiDeviceProvider方法返回请求的设备：

abstract MidiDevice getDevice(MidiDevice.Info info)

此方法应首先测试该参数，以确保它描述了此提供程序可以提供的设备。如果没有，它应该抛出一个IllegalArgumentException 。否则，它将返回设备。

提供Soundbank文件阅读服务

一种SoundBank是一套Instruments可以加载到Synthesizer 。一个Instrument是声音合成算法的一种实现，可以产生特定种类的声音，并包括附带的名称和信息字符串。一种SoundBank大致对应于MIDI规范中的库，但这是一个更广泛且可寻址的集合；最好将其视为MIDI库的集合。

SoundbankReader由一个重载方法组成，系统调用该方法读取一个Soundbank声音库文件中的对象：

abstract Soundbank getSoundbank(java.io.File file) 
abstract Soundbank getSoundbank(java.io.InputStream stream) 
abstract Soundbank getSoundbank(java.net.URL url)

的具体子类SoundbankReader将与特定的提供商定义的实现协同工作SoundBank ， Instrument和Synthesizer允许系统加载SoundBank从文件到特定实例Synthesizer类。合成技术可能与一种完全不同Synthesizer另一个，因此，存储在一个Instrument要么SoundBank提供控制或规格数据以用于合成Synthesizer可以采取多种形式。一种综合技术可能只需要几个字节的参数数据；另一个可能基于大量的声音样本。当前资源SoundBank将取决于Synthesizer将它们加载到其中，并因此实现getSoundbank方法SoundbankReader子类可以访问特定类型的知识SoundBank 。此外， SoundbankReader了解用于存储文件的特定文件格式SoundBank数据。该文件格式可以是特定于供应商的专有文件。

SoundBank只是一个界面，对内容的限制很小SoundBank宾语。对象实现该接口必须支持的方法（ getResources ， getInstruments ， getVendor ， getName等）对对象包含的数据施加宽松的要求。例如， getResources和getInstruments可以返回空数组。子类的实际内容SoundBank对象，特别是其工具和非仪器资源，由服务提供商定义。因此，解析音库文件的机制完全取决于该特定种类的音库文件的规范。

Soundbank文件是在Java Sound API外部创建的，通常是由可以加载这种Soundbank的合成器供应商提供的。一些供应商可能会提供用于创建此类文件的最终用户工具。
