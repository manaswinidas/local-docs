使用自定义类型映射

注意 ：MySQL当前不支持用户定义的类型。MySQL和Java DB当前不支持结构化类型，或者DISTINCT SQL数据类型。没有JDBC教程示例可用来演示本节中描述的功能。

随着业务的蓬勃发展，The Coffee Break的所有者会定期添加新商店，并对数据库进行更改。所有者决定对结构化类型使用自定义映射ADDRESS 。这使所有者可以更改映射到Java类的Java类。 ADDRESS类型。Java类的每个属性都有一个字段ADDRESS 。类的名称及其字段的名称可以是任何有效的Java标识符。

涵盖以下主题：

  • 实现SQLData
  • 使用连接的类型图
  • 使用您自己的类型图

实现SQLData

自定义映射的第一件事是创建一个实现接口的类SQLData 。

结构化类型的SQL定义ADDRESS看起来像这样：


  CREATE TYPE ADDRESS
  (
    NUM INTEGER,
    STREET VARCHAR(40),
    CITY VARCHAR(40),
    STATE CHAR(2),
    ZIP CHAR(5)
  );

实现SQLData接口的自定义映射ADDRESS类型可能如下所示：

public class Address implements SQLData {
    public int num;
    public String street;
    public String city;
    public String state;
    public String zip;
    private String sql_type;
    
    public String getSQLTypeName() {
        return sql_type;
    }

    public void readSQL(SQLInput stream, String type)
        throws SQLException {
        sql_type = type;
        num = stream.readInt();
        street = stream.readString();
        city = stream.readString();
        state = stream.readString();
        zip = stream.readString();
    }

    public void writeSQL(SQLOutput stream)
        throws SQLException {
        stream.writeInt(num);
        stream.writeString(street);
        stream.writeString(city);
        stream.writeString(state);
        stream.writeString(zip);
    }
}

使用连接的类型图

编写实现接口的类之后SQLData ，设置自定义映射的唯一另一件事就是在类型映射中创建一个条目。对于该示例，这意味着输入以下内容的标准SQL名称： ADDRESS类型和Class类的对象Address 。一个类型映射，一个实例java.util.Map接口在创建时会与每个新连接相关联，因此您可以使用该接口。假如说con是活动连接，以下代码片段为UDT添加了一个条目ADDRESS到与con 。

java.util.Map map = con.getTypeMap();
map.put("SchemaName.ADDRESS", Class.forName("Address"));
con.setTypeMap(map);

每当您致电getObject检索实例的方法ADDRESS类型，驱动程序将检查与连接关联的类型映射，并查看它是否具有用于ADDRESS 。司机会注意Class的对象Address类，创建它的实例，并在后台执行许多其他操作以映射ADDRESS至Address 。除了为映射生成类，然后在类型映射中创建一个条目以使驱动程序知道存在自定义映射外，您无需执行任何其他操作。驾驶员将完成所有其余的工作。

对于存储具有自定义映射的结构化类型，情况也是如此。调用方法时setObject ，驱动程序将检查要设置的值是否是实现接口的类的实例SQLData 。如果是（意味着有一个自定义映射），驱动程序将使用该自定义映射将值转换为其对应的SQL，然后再将其返回数据库。再次，驱动程序在幕后进行自定义映射。您需要做的就是提供方法setObject带有自定义映射的参数。您将在本节的后面看到一个示例。

查看使用标准映射， Struct对象和自定义映射（Java编程语言中的类）。以下代码段显示了到Struct对象，即连接的类型映射中没有条目时驱动程序使用的映射。

ResultSet rs = stmt.executeQuery(
    "SELECT LOCATION " +
    "WHERE STORE_NO = 100003");
rs.next();
Struct address = (Struct)rs.getObject("LOCATION");

变量address包含以下属性值： 4344 ， "First_Street" ， "Verona" ， "CA" ， "94545" 。

以下代码片段显示了当存在结构化类型的条目时会发生什么ADDRESS在连接的类型图中。记住那一栏LOCATION存储类型的值ADDRESS 。

ResultSet rs = stmt.executeQuery(
    "SELECT LOCATION " +
    "WHERE STORE_NO = 100003");
rs.next();
Address store_3 = (Address)rs.getObject("LOCATION");

变量store_3现在是该类的一个实例Address ，每个属性值是的其中一个字段的当前值Address 。请注意，您必须记住要转换由getObject方法Address对象，然后将其分配给store_3 。另请注意store_3必须是Address宾语。

比较与Struct反对使用的实例Address类。假设商店搬到附近城镇的更好位置，因此您必须更新数据库。使用自定义映射，重置store_3 ，如以下代码片段所示：

ResultSet rs = stmt.executeQuery(
    "SELECT LOCATION " +
    "WHERE STORE_NO = 100003");
rs.next();
Address store_3 = (Address)rs.getObject("LOCATION");
store_3.num = 1800;
store_3.street = "Artsy_Alley";
store_3.city = "Arden";
store_3.state = "CA";
store_3.zip = "94546";
PreparedStatement pstmt = con.prepareStatement(
    "UPDATE STORES " +
    "SET LOCATION = ? " +
    "WHERE STORE_NO = 100003");
pstmt.setObject(1, store_3);
pstmt.executeUpdate();

列中的值LOCATION是的实例ADDRESS类型。驱动程序检查连接的类型映射，发现有一个条目链接ADDRESS与班级Address并因此使用Address 。当代码调用方法时setObject与变量store_3作为第二个参数，驱动程序检查并查看store_3代表该类的一个实例Address ，它实现了接口SQLData用于结构化类型ADDRESS ，并再次自动使用自定义映射。

没有针对的自定义映射ADDRESS ，更新看起来像这样：

PreparedStatement pstmt = con.prepareStatement(
    "UPDATE STORES " +
    "SET LOCATION.NUM = 1800, " +
    "LOCATION.STREET = 'Artsy_Alley', " + 
    "LOCATION.CITY = 'Arden', " +
    "LOCATION.STATE = 'CA', " +
    "LOCATION.ZIP = '94546' " +
    "WHERE STORE_NO = 100003");
pstmt.executeUpdate;

使用您自己的类型图

到目前为止，您仅使用了与连接关联的类型映射进行自定义映射。通常，这是大多数程序员都将使用的唯一类型映射。但是，也可以创建一个类型映射并将其传递给某些方法，以便驱动程序将使用该类型映射，而不是与连接关联的类型映射。这为相同的用户定义类型（UDT）允许两个不同的映射。实际上，对于同一个UDT，可以有多个自定义映射，只要每个映射都使用实现了UDT的类进行设置即可。 SQLData接口和类型映射中的条目。如果没有将类型映射传递给可以接受的方法，则驱动程序将默认使用与连接关联的类型映射。

除了与连接关联的类型映射外，很少有其他情况需要使用类型映射。例如，如果几个在JDBC应用程序上工作的程序员将他们的组件放在一起并使用相同的连接，则可能需要为该方法提供类型映射。如果两个或更多程序员为同一SQL UDT创建了自己的自定义映射，则每个程序员都需要提供自己的类型映射，从而覆盖连接的类型映射。
