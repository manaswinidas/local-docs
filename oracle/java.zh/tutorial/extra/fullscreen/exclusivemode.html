全屏独占模式

使用Microsoft DirectX API的程序员可能已经熟悉全屏独占模式。其他程序员可能对这个概念有些陌生。在这两种情况下，全屏独占模式都是J2SE™1.4版的强大功能，允许程序员暂停窗口系统，以便可以直接在屏幕上绘制图形。

从许多方面来看，这与通常的GUI程序相比都是微小的范例转变。在传统的Java GUI程序中，AWT负责通过事件分发线程从操作系统传播绘画事件 ，并在适当时调用AWT的Component.paint方法。在全屏独家应用程序中，绘画通常是由程序本身主动完成的。此外，传统的GUI应用程序仅限于用户选择的屏幕的位深度和大小。在全屏专有应用程序中，该程序可以控制屏幕的位深度和大小（ 显示模式 ）。最后，在某些平台上，许多更高级的技术，例如页面翻转 （在下面讨论）和立体声缓冲（利用每只眼睛使用一组独立帧的系统），要求应用程序首先处于全屏独占模式。

硬件加速图像基础

要了解全屏独占模式API，您需要了解硬件加速图像的一些基本原理。VolatileImage接口封装了可以利用或不利用硬件加速的表面。此类表面可能会按照操作系统的要求失去其硬件加速或内存（因此，其名称为“ volatile”）。见VolatileImage有关易失性图像的更多信息的教程 （即将推出）。

全屏独占模式通过java.awt处理。GraphicsDevice对象。有关所有可用屏幕图形设备（在单显示器或多显示器系统中）的列表，可以在本地java.awt上调用方法getScreenDevices 。图形环境 ;对于默认（主）屏幕（单显示器系统上的唯一屏幕），可以调用方法getDefaultScreenDevice 。

使用图形设备后，可以调用以下方法之一：

  • 公共布尔isFullScreenSupported（）
    此方法返回全屏独占模式是否可用。在没有全屏独占模式的系统上，最好以固定大小的窗口模式运行应用程序，而不是设置全屏窗口。
  • 公共无效setFullScreenWindow（Window w）
    给定一个窗口，此方法使用该窗口进入全屏排他模式。如果没有全屏独占模式，则将窗口置于（0,0）并调整大小以适合屏幕。将此方法与null参数一起使用可退出全屏独占模式。

编程技巧

以下是有关使用全屏独占模式进行编程的一些提示：

  • 进入全屏独占模式之前，请检查isFullScreenSupported 。如果不支持，则性能可能会降低。
  • 使用try ... finally子句时，进入和退出全屏模式更加健壮。这不仅是良好的编码习惯，而且还使您的程序在全屏独占模式下的停留时间比应保持的时间长：
    GraphicsDevice myDevice;
Window myWindow;

try {
    myDevice.setFullScreenWindow(myWindow);
    ...
} finally {
    myDevice.setFullScreenWindow(null);
}
  • 大多数全屏独占应用程序更适合使用未装饰的窗口。使用setUndecorated方法关闭框架或对话框中的装饰 。
  • 全屏独占应用程序不应调整大小，因为调整全屏应用程序的大小可能会导致不可预测（或可能危险）的行为。
  • 出于安全原因，在applet中使用全屏独占模式时，用户必须授予fullScreenExclusive权限。
