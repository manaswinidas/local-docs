通配符带来更多乐趣

在本节中，我们将考虑通配符的一些更高级的用法。我们已经看到了几个示例，其中从数据结构读取数据时，有界通配符很有用。现在考虑相反的只写数据结构。介面Sink是这种简单的例子。

interface Sink<T> {
    flush(T t);
}

我们可以想象使用它，如下面的代码所示。方法writeAll()旨在刷新集合中的所有元素coll到水槽snk ，并返回刷新的最后一个元素。

public static <T> T writeAll(Collection<T> coll, Sink<T> snk) {
    T last;
    for (T t : coll) {
        last = t;
        snk.flush(last);
    }
    return last;
}
...
Sink<Object> s;
Collection<String> cs;
String str = writeAll(cs, s); // Illegal call.

按照书面规定， writeAll()是非法的，因为无法推断出有效的类型参数；都不String也不Object是适合的类型T ，因为Collection元素和Sink元素必须具有相同的类型。

我们可以通过修改签名来解决此错误writeAll()如下所示，使用通配符。

public static <T> T writeAll(Collection<? extends T>, Sink<T>) {...}
...
// Call is OK, but wrong return type. 
String str = writeAll(cs, s);

调用现在合法，但是分配是错误的，因为推断的返回类型为Object因为T匹配的元素类型s ，这是Object 。

解决方案是使用一种尚未出现的有界通配符形式：具有下界的通配符。语法? super T表示未知类型，它是的超类型T （要么T本身;请记住，超类型关系是自反的。这是我们一直在使用的有界通配符的对偶? extends T表示未知类型，它是的子类型T 。

public static <T> T writeAll(Collection<T> coll, Sink<? super T> snk) {
    ...
}
String str = writeAll(cs, s); // Yes!

使用此语法，该调用是合法的，并且推断的类型是String ， 如预期的。
