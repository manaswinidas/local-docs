将旧版代码转换为使用泛型

之前，我们展示了新代码和旧代码如何互操作。现在，该看一下“生成”旧代码的难题了。

如果您决定将旧代码转换为使用泛型，则需要仔细考虑如何修改API。

您需要确保通用API不受过度限制；它必须继续支持API的原始合同。再次考虑一些例子java.util.Collection 。通用API如下所示：

interface Collection {
    public boolean containsAll(Collection c);
    public boolean addAll(Collection c);
}

天真的尝试将其夸大如下：

interface Collection<E> {

    public boolean containsAll(Collection<E> c);
    public boolean addAll(Collection<E> c);
}

尽管这肯定是类型安全的，但它不符合API的原始合同。的containsAll()方法适用于任何类型的传入集合。仅当传入集合实际上仅包含的实例时，它才会成功E ，但是：

  • 传入集合的静态类型可能有所不同，可能是因为调用者不知道所传入集合的确切类型，或者可能是因为它是一个Collection ，哪里S是...的子类型E 。
  • 打电话是完全合法的containsAll()与不同类型的集合。例程应该工作，返回false 。

如果是addAll() ，我们应该能够添加任何由以下子类型的实例组成的集合E 。我们在“ 通用方法”一节中了解了如何正确处理这种情况。

您还需要确保修订后的API保留与旧客户端的二进制兼容性。这意味着对API的擦除必须与原始的未经过增强的API相同。在大多数情况下，这很自然，但是有些微妙的情况。我们将研究遇到的最微妙的情况之一，即方法Collections.max() 。正如我们在“ 通过通配符获得更多乐趣”一节中所看到的那样， max()是：

public static <T extends Comparable<? super T>> 
        T max(Collection<T> coll)

可以，但是可以删除此签名：

public static Comparable max(Collection coll)

与原始签名不同max() ：

public static Object max(Collection coll)

一定可以为该签名指定此签名max() ，但未完成，所有旧的二进制文件都调用了Collections.max()取决于返回的签名Object 。

我们可以通过在形式类型参数的绑定中明确指定一个超类来强制删除操作不同T 。

public static <T extends Object & Comparable<? super T>> 
        T max(Collection<T> coll)

这是使用语法为类型参数赋予多个范围的示例T1 & T2 ... & Tn 。已知具有多个界限的类型变量是界限中列出的所有类型的子类型。使用多重界限时，界限中提到的第一种类型将用作类型变量的擦除。

最后，我们应该记得max仅从其输入集合中读取，因此适用于以下任何子类型的集合T 。

这使我们了解了JDK中使用的实际签名：

public static <T extends Object & Comparable<? super T>> 
        T max(Collection<? extends T> coll)

涉及到的任何事情在实践中都是很少见的，但是在转换现有的API时，专家库设计人员应该准备好仔细考虑。

需要注意的另一个问题是协变返回 ，即在子类中优化方法的返回类型。您不应该在旧的API中利用此功能。要了解原因，我们来看一个示例。

假设原始API的格式为：

public class Foo {
    // Factory. Should create an instance of 
    // whatever class it is declared in.
    public Foo create() {
        ...
    }
}

public class Bar extends Foo {
    // Actually creates a Bar.
    public Foo create() {
        ...
    }
}

利用协变收益，可以将其修改为：

public class Foo {
    // Factory. Should create an instance of 
    // whatever class it is declared in.
    public Foo create() {
        ...
    }
}

public class Bar extends Foo {
    // Actually creates a Bar.
    public Bar create() {
        ...
    }
}

现在，假设您的代码的第三方客户端编写了以下内容：

public class Baz extends Bar {
    // Actually creates a Baz.
    public Foo create() {
        ...
    }
}

Java虚拟机不直接支持具有不同返回类型的方法的覆盖。编译器支持此功能。因此，除非上课Baz重新编译后，将无法正确覆盖create()的方法Bar 。此外， Baz必须修改，因为代码将按书面形式被拒绝-的返回类型create()在Baz不是返回类型的子类型create()在Bar 。
