介绍

可扩展的应用程序是您可以扩展而无需修改其原始代码库的应用程序。您可以使用新的插件或模块来增强其功能。通过将新的Java Archive（JAR）文件添加到应用程序类路径或特定于应用程序的扩展目录中，开发人员，软件供应商和客户可以添加新功能或应用程序编程接口（API）。

本节介绍如何创建具有可扩展服务的应用程序，使您或其他人可以提供不需要修改原始应用程序的服务实现。通过设计可扩展的应用程序，您可以在不更改核心应用程序的情况下提供升级或增强产品特定部分的方法。

可扩展应用程序的一个示例是文字处理器，它允许最终用户添加新的词典或拼写检查器。在此示例中，文字处理器提供了词典或拼写功能，其他开发人员甚至客户可以通过提供自己的功能实现来扩展词典或拼写功能。

以下是对理解可扩展应用程序很重要的术语和定义：

服务
一组编程接口和类，提供对某些特定应用程序功能或特性的访问。服务可以定义功能的接口以及检索实现的方式。在字处理器示例中，词典服务可以定义一种检索词典和单词定义的方法，但是它不实现基础功能集。而是依靠服务提供商来实现该功能。
服务提供商接口（SPI）
服务定义的一组公共接口和抽象类。SPI定义了可用于您的应用程序的类和方法。
服务提供者
实现SPI。具有可扩展服务的应用程序使您，供应商和客户无需修改原始应用程序即可添加服务提供商。

了解DictionaryServiceDemo示例

以下步骤显示了如何重新创建文件的内容DictionaryServiceDemo.zip 。这些步骤向您展示了示例如何工作以及如何运行。

1。定义服务提供商接口

的DictionaryServiceDemo样本定义了一个SPI， Dictionary.java接口。它仅包含一种方法：


package dictionary.spi;

public interface Dictionary {
    public String getDefinition(String word);
}

该示例将已编译的类文件存储在目录中DictionaryServiceProvider/build 。

2。定义检索服务提供者实现的服务

的DictionaryService.java类加载和访问可用Dictionary代表词典服务客户的服务提供商：


package dictionary;

import dictionary.spi.Dictionary;
import java.util.Iterator;
import java.util.ServiceConfigurationError;
import java.util.ServiceLoader;

public class DictionaryService {

    private static DictionaryService service;
    private ServiceLoader<Dictionary> loader;

    private DictionaryService() {
        loader = ServiceLoader.load(Dictionary.class);
    }

    public static synchronized DictionaryService getInstance() {
        if (service == null) {
            service = new DictionaryService();
        }
        return service;
    }


    public String getDefinition(String word) {
        String definition = null;

        try {
            Iterator<Dictionary> dictionaries = loader.iterator();
            while (definition == null && dictionaries.hasNext()) {
                Dictionary d = dictionaries.next();
                definition = d.getDefinition(word);
            }
        } catch (ServiceConfigurationError serviceError) {
            definition = null;
            serviceError.printStackTrace();

        }
        return definition;
    }
}

该示例将已编译的类文件存储在目录中DictionaryServiceProvider/build 。

的DictionaryService类实现单例设计模式。这意味着只有一个实例DictionaryService类是曾经创建的。有关更多信息，请参见“单例设计模式 ”部分。

的DictionaryService class是字典服务客户端使用任何已安装的入口点Dictionary服务提供者。使用ServiceLoader.load检索私有静态成员的方法DictionaryService.service ，单例服务入口点。然后应用程序可以调用getDefinition遍历可用方法的方法Dictionary提供者，直到找到目标词。的getDefinition如果否，则方法返回null Dictionary实例包含单词的指定定义。

字典服务使用ServiceLoader.load查找目标类的方法。SPI由接口定义dictionary.spi.Dictionary ，因此该示例将此类用作加载方法的参数。默认加载方法使用默认类加载器搜索应用程序类路径。

但是，此方法的重载版本使您可以根据需要指定自定义类加载器。这使您可以进行更复杂的类搜索。例如，一个特别热心的程序员可能会创建一个ClassLoader实例，可以在特定于应用程序的子目录中进行搜索，该子目录包含在运行时添加的提供程序JAR。结果是不需要重新启动即可访问新提供程序类的应用程序。

此类的加载器存在后，您可以使用其迭代器方法访问和使用它找到的每个提供程序。的getDefinition方法使用Dictionary迭代器将遍历提供程序，直到找到指定单词的定义。迭代器方法缓存Dictionary实例，因此连续调用几乎不需要额外的处理时间。如果自上次调用以来已将新的提供程序投入使用，则迭代器方法会将它们添加到列表中。

的DictionaryDemo.java类使用此服务。要使用该服务，应用程序需要获取一个DictionaryService实例并调用getDefinition方法。如果定义可用，则应用程序将其打印。如果没有可用的定义，则应用程序将显示一条消息，指出没有可用的词典携带该单词。

单例设计模式

设计模式是软件设计中常见问题的一般解决方案。想法是将解决方案转换为代码，并且可以将代码应用于出现问题的不同情况。单例模式描述了一种确保仅创建类的单个实例的技术。本质上，该技术采用以下方法：不要让类外部的任何人创建对象的实例。

例如， DictionaryService类实现单例模式，如下所示：

  • 宣告DictionaryService构造函数为private ，可以阻止所有其他类，但DictionaryService ，从创建它的实例开始。
  • 定义DictionaryService成员变量service如static ，这只能确保DictionaryService存在。
  • 定义方法getInstance ，这使其他类可以控制对DictionaryService成员变量service 。

3。实施服务提供商

要提供此服务，您必须创建一个Dictionary.java实施。为了简单起见，请创建仅定义几个单词的通用词典。您可以使用数据库，一组属性文件或任何其他技术来实现字典。演示提供程序模式的最简单方法是在单个文件中包含所有单词和定义。

以下代码显示了Dictionary SPI GeneralDictionary.java类。请注意，它提供了无参数的构造函数并实现了getDefinition SPI定义的方法。


package dictionary;

import dictionary.spi.Dictionary;
import java.util.SortedMap;
import java.util.TreeMap;

public class GeneralDictionary implements Dictionary {

    private SortedMap<String, String> map;
    
    public GeneralDictionary() {
        map = new TreeMap<String, String>();
        map.put(
            "book",
            "a set of written or printed pages, usually bound with " +
                "a protective cover");
        map.put(
            "editor",
            "a person who edits");
    }

    @Override
    public String getDefinition(String word) {
        return map.get(word);
    }

}

该示例将已编译的类文件存储在目录中GeneralDictionary/build 。注意 ：您必须编译类dictionary.DictionaryService和dictionary.spi.Dictionary上课前GeneralDictionary 。

的GeneralDictionary此示例的provider仅定义了两个词： book和editor 。显然，更有用的词典将提供更广泛的常用词汇表。

4。注册服务提供商

要注册服务提供商，请创建一个提供商配置文件，该文件存储在META-INF/services服务提供商的JAR文件的目录。配置文件的名称是服务提供商的完全限定的类名称，其中名称的每个组成部分均以句点（ . ），并且嵌套类之间用美元符号（ $ ）。

提供程序配置文件包含服务提供程序的完全限定的类名称，每行一个名称。该文件必须为UTF-8编码。此外，您可以在注释行中以数字符号（ # ）。

例如，注册服务提供商GeneralDictionary创建一个名为的文本文件dictionary.spi.Dictionary 。该文件包含一行：

dictionary.GeneralDictionary

同样，注册服务提供商ExtendedDictionary创建一个名为的文本文件dictionary.spi.Dictionary 。该文件包含一行：

dictionary.ExtendedDictionary

5，创建使用服务和服务提供者的客户端

因为开发一个完整的文字处理器应用程序是一项艰巨的任务，所以本教程提供了一个使用DictionaryService和Dictionary SPI。的DictionaryDemo样本从以下任何位置搜索单词book ， editor ， xml和REST单词Dictionary提供程序在类路径上并检索其定义。

以下是DictionaryDemo样品。它要求从DictionaryService实例，它将请求传递给已知的Dictionary提供者。


package dictionary;

import dictionary.DictionaryService;

public class DictionaryDemo {

  public static void main(String[] args) {

    DictionaryService dictionary = DictionaryService.getInstance();
    System.out.println(DictionaryDemo.lookup(dictionary, "book"));
    System.out.println(DictionaryDemo.lookup(dictionary, "editor"));
    System.out.println(DictionaryDemo.lookup(dictionary, "xml"));
    System.out.println(DictionaryDemo.lookup(dictionary, "REST"));
  }

  public static String lookup(DictionaryService dictionary, String word) {
    String outputString = word + ": ";
    String definition = dictionary.getDefinition(word);
    if (definition == null) {
      return outputString + "Cannot find definition for this word.";
    } else {
      return outputString + definition;
    }
  }
}

该示例将已编译的类文件存储在目录中DictionaryDemo/build 。注意 ：您必须编译类dictionary.DictionaryService和dictionary.spi.Dictionary上课前DictionaryDemo 。

ServiceLoader类

的java.util.ServiceLoader类可帮助您查找，加载和使用服务提供商。它在应用程序的类路径或运行时环境的扩展目录中搜索服务提供者。它将加载它们，并使您的应用程序可以使用提供程序的API。如果将新的提供程序添加到类路径或运行时扩展目录，则ServiceLoader全班找到他们。如果您的应用程序知道提供者接口，那么它可以找到并使用该接口的不同实现。您可以使用该接口的第一个可加载实例，或遍历所有可用接口。

的ServiceLoader class是最终的，这意味着您不能使其成为子类或覆盖其加载算法。例如，您不能更改其算法以从其他位置搜索服务。

从的角度ServiceLoader类，所有服务都具有单个类型，通常是单个接口或抽象类。提供程序本身包含一个或多个具体类，这些类通过特定于其目的的实现来扩展服务类型。的ServiceLoader类要求单个公开的提供程序类型具有默认构造函数，该构造函数不需要任何参数。这使得ServiceLoader类以轻松实例化它找到的服务提供商。

定位提供者并按需实例化。服务加载程序维护已加载的提供程序的缓存。加载程序的每次调用iterator方法返回一个迭代器，该迭代器首先以实例化顺序生成高速缓存的所有元素。然后，服务加载器查找并实例化任何新的提供程序，然后将每个提供程序依次添加到缓存中。您可以使用reload方法。

要为特定类创建加载程序，请将类本身提供给load要么loadInstalled方法。您可以使用默认的类加载器，也可以提供自己的类加载器ClassLoader子类。

的loadInstalled方法搜索安装的运行时提供程序的运行时环境的扩展目录。默认扩展位置是您的运行时环境的jre/lib/ext目录。您应该仅将扩展位置用于知名的，受信任的提供程序，因为该位置成为所有应用程序的类路径的一部分。在本文中，提供程序不使用扩展目录，而是依赖于特定于应用程序的类路径。

ServiceLoader API的局限性

的ServiceLoader API很有用，但有局限性。例如，不可能从ServiceLoader类，因此您无法修改其行为。您可以使用自定义ClassLoader子类来更改类的查找方式，但是ServiceLoader本身不能扩展。另外，目前ServiceLoader类不能告诉您的应用程序何时新的提供程序在运行时可用。此外，您无法将更改侦听器添加到加载器，以查明是否将新的提供程序放置在特定于应用程序的扩展目录中。

公众ServiceLoader API在Java SE 6中可用。尽管加载程序服务早在JDK 1.3便已存在，但该API是私有的，并且仅可用于内部Java运行时代码。
