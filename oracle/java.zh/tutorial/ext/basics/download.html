下载扩展

下载扩展是JAR文件中的类集（和相关资源）。JAR文件的清单可以包含引用一个或多个下载扩展名的标头。可以通过以下两种方式之一来引用扩展：

  • 通过Class-Path标头
  • 通过扩展列表标题

请注意，清单中最多允许每个。由Class-Path标头指示的下载扩展仅在下载它们的应用程序的生命周期内下载，例如Web浏览器。它们的优点是客户端上没有安装任何东西。它们的缺点是每次需要时都下载它们。由扩展列表标头下载的下载扩展安装在下载它们的JRE的/ lib / ext目录中。它们的优点是可以在需要时下载它们。随后，无需下载即可使用它们。但是，如本教程后面所示，它们的部署更加复杂。

由于使用Class-Path标头的下载扩展更为简单，因此我们首先考虑一下。例如，假设a.jar和b.jar是同一目录中的两个JAR文件，并且a.jar的清单包含以下标头：

Class-Path: b.jar

然后， b.jar中的类用作扩展类，以实现a.jar中的类。在a.jar文件中的类可以调用b.jar类而不必在类路径上被命名为b.jar的类。 a.jar本身可能是也可能不是扩展名。如果b.jar与a.jar不在同一目录中，则应将Class-Path标头的值设置为b.jar的相对路径名。

扮演下载扩展角色的类没有什么特别的。仅将它们视为扩展是因为其他某些JAR文件的清单引用了它们。

为了更好地了解下载扩展的工作方式，让我们创建一个并使用它。

一个例子

假设您要创建一个利用RectangleArea上一节的类：

public final class RectangleArea {  
    public static int area(java.awt.Rectangle r) {
        return r.width * r.height;
    }
}

在上一节中，您通过将包含RectangleArea类的JAR文件放入JRE的lib / ext目录中，将RectangleArea类制成了已安装的扩展名。通过使其成为已安装的扩展，可以使任何应用程序都可以使用RectangleArea类，就好像它是Java平台的一部分一样。

如果您希望能够通过applet使用RectangleArea类，则情况有所不同。例如，假设您有一个applet， AreaApplet ，它使用了RectangleArea类：

import java.applet.Applet;
import java.awt.*;

public class AreaApplet extends Applet {
    Rectangle r;

    public void init() {    
        int width = 10;
        int height = 5;

        r = new Rectangle(width, height);
    }

    public void paint(Graphics g) {
        g.drawString("The rectangle's area is " 
                      + RectangleArea.area(r), 10, 10);
    }
}

此小程序实例化一个10 x 5矩形，然后使用RectangleArea.area方法显示矩形的区域。

但是，您不能假定下载和使用您的applet的每个人都可以在其系统上使用RectangleArea类，作为已安装的扩展名或其他方式。解决该问题的一种方法是使RectangleArea类可从服务器端使用，您可以通过将其用作下载扩展来实现。

要查看其完成方式，请假设您已经捆绑了AreaApplet在一个名为AreaApplet.jar的JAR文件中，并且RectangleArea类捆绑在RectangleArea.jar中 。为了将RectangleArea.jar视为下载扩展，必须在AreaApplet.jar清单的Class-Path标头中列出RectangleArea.jar 。 AreaApplet.jar的清单可能看起来像这样，例如：

Manifest-Version: 1.0
Class-Path: RectangleArea.jar

此清单中的Class-Path标头的值为RectangleArea.jar ，未指定路径，指示RectangleArea.jar与该applet的JAR文件位于同一目录中。

有关类路径标题的更多信息

如果小程序或应用程序使用多个扩展名，则可以在清单中列出多个URL。例如，以下是有效的标头：

Class-Path: area.jar servlet.jar images/

在Class-Path标头中，列出的所有不以' / '结尾的URL均假定为JAR文件。以' / '结尾的URL表示目录。在前面的示例中， images /可能是包含applet或应用程序所需资源的目录。

请注意，清单文件中只允许使用一个Class-Path标头，清单中的每一行的长度不得超过72个字符。如果需要指定的类路径条目多于一行，则可以将它们扩展到后续的延续行。每条连续线以两个空格开头。例如：

Class-Path: area.jar servlet.jar monitor.jar datasource.jar
  provider.jar gui.jar

将来的发行版可能会消除以下限制：每个标头仅具有一个实例，并且行数只能限制为72个字符。

下载扩展可以是“菊花链式”，这意味着一个下载扩展的清单可以具有一个Class-Path标头，该标头引用第二个扩展，而第二个扩展可以引用第三个扩展，依此类推。

安装下载扩展

在上面的示例中，仅当加载小程序的浏览器仍在运行时，小程序下载的扩展名才可用。但是，如果applet和扩展的清单中都包含其他信息，则applet可以触发扩展的安装。

由于此机制扩展了平台的核心API，因此应谨慎使用。它很少适合单个或一小组应用程序使用的接口。所有可见符号应遵循反向域名和类层次结构约定。

基本要求是，小程序及其使用的扩展名都必须在其清单中提供版本信息，并且必须对其进行签名。版本信息允许Java插件确保扩展代码具有applet期望的版本。例如， AreaApplet可以在其清单中指定areatest扩展：

Manifest-Version: 1.0
Extension-List: areatest
areatest-Extension-Name: area
areatest-Specification-Version: 1.1
areatest-Implementation-Version: 1.1.2
areatest-Implementation-Vendor-Id: com.example
areatest-Implementation-URL: http://www.example.com/test/area.jar

area.jar中的清单将提供相应的信息：

Manifest-Version: 1.0
Extension-Name: area
Specification-Vendor: Example Tech, Inc
Specification-Version: 1.1
Implementation-Vendor-Id: com.example
Implementation-Vendor: Example Tech, Inc
Implementation-Version: 1.1.2

小程序和扩展名都必须由同一签名者签名。对jar文件签名将对其进行原位修改，从而在清单文件中提供更多信息。签名有助于确保仅安装受信任的代码。对jar文件进行签名的一种简单方法是，首先创建一个密钥库，然后使用该密钥库来保存applet和扩展名的证书。例如：

keytool -genkey -dname "cn=Fred" -alias test  -validity 180

系统将提示您输入密钥库和密钥密码。生成密钥后，可以对jar文件进行签名：

jarsigner AreaApplet.jar test
jarsigner area.jar test

系统将提示您输入密钥库和密钥密码。有关keytool ， jarsigner和其他安全工具的更多信息，请参见 Java 2 Platform Security工具摘要 。

这是AreaDemo.html ，它将加载小程序并导致下载和安装扩展代码：

<html>
<body>
  <applet code="AreaApplet.class" archive="AreaApplet.jar"/>
</body>
</html>

首次加载页面时，系统会告知用户该applet需要安装扩展。随后的对话框通知用户有关已签名的小程序的信息。接受两者都会将扩展安装在JRE的lib / ext文件夹中并运行小程序。

重新启动Web浏览器并加载相同的网页后，由于已经安装了area.jar ，因此仅显示有关小程序签名者的对话框。如果在其他Web浏览器中打开AreaDemo.html （假设两个浏览器使用相同的JRE），也是如此。
