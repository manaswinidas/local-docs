可序列化的对象

序列化对象意味着将其状态转换为字节流，以便可以将字节流还原回该对象的副本。如果Java对象的类或其任何超类实现了java.io，则它是可序列化的。可序列化的接口或其子接口java.io。可外部化 。反序列化是将对象的序列化形式转换回该对象的副本的过程。

例如， java.awt。Button类实现了Serializable接口，因此您可以序列化java.awt。按钮对象并将该序列化状态存储在文件中。稍后，您可以读回序列化状态并将其反序列化为java.awt。按钮对象。

Java平台指定了序列化可序列化对象的默认方式。（Java）类可以覆盖此默认序列化，并定义其自己的序列化该类对象的方式。对象序列化规范详细描述了对象序列化。

序列化对象时，将标识其类的信息记录在序列化流中。但是，不会记录类的定义（“类文件”）本身。对对象进行反序列化的系统负责确定如何查找和加载必要的类文件。例如，Java应用程序可能会在其类路径中包含一个JAR文件，该文件包含序列化对象的类文件，或者通过使用存储在目录中的信息来加载类定义，如本课程后面部分所述。

绑定可序列化对象

如果基础服务提供者支持该操作，则可以在目录中存储可序列化的对象，Oracle的LDAP服务提供者也可以。

以下示例调用Context.bind将AWT按钮绑定到名称“ cn = Button” 。要将属性与新绑定关联，请使用DirContext.bind 。要覆盖现有绑定，请使用Context.rebind和DirContext.rebind 。

// Create the object to be bound
Button b = new Button("Push me");

// Perform the bind
ctx.bind("cn=Button", b);

然后，您可以使用读回对象Context.lookup ， 如下。

// Check that it is bound
Button b2 = (Button)ctx.lookup("cn=Button");
System.out.println(b2);

跑步this example产生以下输出。

# java SerObj
java.awt.Button[button0,0,0,0x0,invalid,label=Push me]

指定代码库

注意：此处描述的过程用于将目录服务中的可序列化对象绑定到遵循RFC 2713定义的模式的目录服务中。这些过程可能通常不适用于支持将可序列化对象与指定代码库绑定的其他命名和目录服务。

如上一示例所示，当将序列化的对象绑定到目录中时，从目录读取序列化的对象的应用程序必须有权访问反序列化该对象所必需的类定义。

或者，可以在绑定对象或随后通过使用以下方式添加属性时在目录中记录带有序列化对象的代码库： DirContext.modifyAttributes 。您可以使用任何属性来记录此代码库，并让您的应用程序从目录中读取该属性并适当地使用它。或者，您可以使用中指定的“ javaCodebase”属性。在后一种情况下，Oracle的LDAP服务提供者将根据需要自动使用该属性来加载类定义。 “ javaCodebase”应包含代码库目录或JAR文件的URL。如果代码库包含多个URL，则每个URL必须用空格字符分隔。

以下示例类似于用于绑定java.awt的示例。按钮 。区别在于它使用了用户定义的Serializable类， Flower ，并提供一个“ javaCodebase”属性，其中包含Flower的类定义的位置。这是执行绑定的代码。

String codebase = ...;

// Create the object to be bound
Flower f = new Flower("rose", "pink");

// Perform the bind and specify the codebase
ctx.bind("cn=Flower", f, new BasicAttributes("javaCodebase", codebase));

当你跑步this example ，您必须提供安装类文件Flower.class的位置的URL。例如，如果Flower.class安装在Web服务器web1的example / classes目录中，则可以按以下方式运行此示例。

# java SerObjWithCodebase http://web1/example/classes/
pink rose

之后，您可以从类路径中删除Flower.class并运行任何查找或列出该对象的程序，而无需直接引用Flower类。如果程序直接引用Flower ，则必须使其类文件可用于编译和执行。
