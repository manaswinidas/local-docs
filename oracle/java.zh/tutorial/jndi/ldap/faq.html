1。上下文对于多线程访问是否安全，还是需要锁定/同步对上下文的访问？

答案取决于实现方式。这是因为Context和DirContext接口未指定同步要求。JDK中的LDAP实现针对单线程访问进行了优化。如果您有多个线程访问同一个Context实例，则每个线程在使用时都需要锁定Context实例。这也适用于从同一Context实例派生的任何NamingEnumeration 。但是，多个线程可以并发访问不同的 Context实例（甚至是从相同初始上下文派生的实例），而无需锁定。

2。如果不设置 上下文 ，LDAP提供程序为什么会忽略我的安全环境属性 。是SECURITY_CREDENTIALS （ “ java.naming.security.credentials” ）属性还是将其设置为空字符串？

如果提供一个空字符串，一个空字节 / 字符数组或Context的 null 。SECURITY_CREDENTIALS环境属性，那么即使上下文也将发生匿名绑定。SECURITY_AUTHENTICATION属性设置为“ simple” 。这是因为对于简单身份验证，LDAP要求密码为非空。如果未提供密码，则协议会自动将身份验证转换为“ none” 。

3。尝试创建初始上下文时， 为什么会不断收到 CommunicationException ？

您可能正在与仅支持LDAP v2的服务器通信。有关如何设置版本号的示例，请参见JNDI教程的其他课程。

4。如何跟踪LDAP消息？

尝试使用“ com.sun.jndi.ldap.trace.ber”环境属性。如果此属性的值是java.io的实例。OutputStream ，然后将有关LDAP提供程序发送和接收的BER缓冲区的跟踪信息写入该流。如果属性的值为null ，则不写入任何跟踪输出。

例如，以下代码会将跟踪输出发送到System.err 。

env.put("com.sun.jndi.ldap.trace.ber", System.err);

5，如何使用其他认证机制，例如Kerberos？

请遵循JNDI教程的GSS-API / Kerberos v5身份验证中的说明，以获取有关如何使用Kerberos身份验证的信息。要使用其他身份验证机制，请参阅JNDI教程的“使用任意SASL机制”部分。

6。更改密码时应该启用SSL吗？ /

这实际上取决于您使用的目录服务器。如果未启用SSL，某些目录服务器将不允许您更改密码，但某些服务器却允许。最好启用SSL，以在通信通道中保护您的密码。

7。当我要求一个属性时，我会得到另一个。为什么？

您使用的属性名称可能是另一个属性的同义词。在这种情况下，LDAP服务器可能返回规范属性名称，而不是您提供的名称。当查看服务器返回的属性时 ，需要使用规范名称而不是同义词。

例如， “ fax”可能是规范属性名称“ facsimiletelephonenumber”的同义词。如果要求输入“ fax” ，服务器将返回名为“ facsimiletelephonenumber”的属性。有关同义词和与属性名称有关的其他问题的详细信息，请参见“ 命名和目录操作”课程。

8。我如何知道属性值的类型？

属性的值可以是java.lang。字符串或字节[] 。有关将哪些属性值作为byte []返回的信息，请参见JNDI教程的“其他”部分。要以编程方式执行此操作，可以使用instanceof运算符检查从LDAP提供程序获取的属性值。

9。如何以非字符串或字节数组的形式获取属性值？

目前您还不能。LDAP提供程序仅返回为java.lang的属性值。字符串或字节[] 。请参阅“ JNDI教程”的“其他”部分。

10。为什么将“ *”作为属性值不能按我的搜索预期工作？

使用以下形式的search（）时 ，属性值被视为文字；也就是说，目录条目中的属性应包含该值： search（Name name，Attributes matchingAttrs）要使用通配符，应使用search（）的字符串过滤器形式，如下所示。 搜索（名称，字符串过滤器，SearchControls ctls）
搜索（名称，字符串filterExpr，对象[] filterArgs，SearchControls ctls）

对于最后一种形式，通配符必须出现在filterExpr参数中，而不是出现在filterArgs中 。filterArgs中的值也被视为文字。

11。为什么搜索过滤器中的通配符不总是有效？

在属性值之前或之后出现的通配符（例如在“ attr = * I *”中 ）表示服务器将通过使用属性的子字符串匹配规则来搜索匹配的属性值。如果属性的定义没有子字符串匹配规则，则服务器找不到该属性。您必须使用等式或“当前”过滤器进行搜索。

12当我知道目录中还有更多条目时，为什么只返回n个条目？ 某些服务器配置为限制可以返回的条目数。其他限制了在搜索过程中可以检查的条目数。检查您的服务器配置。

14。我如何知道我得到了多少搜索结果？

15为什么我在 SearchResult中 得到一个空字符串作为名称 ？

getName（）始终返回相对于搜索目标上下文的名称。因此，如果目标上下文满足搜索过滤条件，则返回的名称将为“”（空名称），因为这是相对于目标上下文的名称。有关详细信息，请参见“ 搜索结果”部分。

16。为什么我在 SearchResult中 得到一个URL字符串作为名称 ？

通过跟随别名或引用来检索LDAP条目，因此其名称为URL。有关详细信息，请参见“ 搜索结果”课程。

17。传递给Context和DirContext方法的 Name参数是什么类型 ？ -CompoundName或CompositeName ？

字符串形式接受复合名称的字符串表示形式。也就是说，使用字符串名称等效于调用new CompositeName（stringName）并将结果传递给Context / DirContext方法。Name参数可以是实现Name接口的任何对象。如果它是CompositeName的实例，那么该名称将被视为一个复合名称；否则，它将被视为复合名称。否则，将其视为复合名称。

18岁我可以将从 NameParser 返回的名称传递给Context方法吗？

这与上一个问题有关。是的你可以。 NameParser.parse（）返回实现Name接口的复合名称。该名称可以传递给Context方法，该方法会将其解释为复合名称。

19我用于 上下文 的名称之间有什么关系 。SECURITY_PRINCIPAL属性和目录？

您可以认为主体名称来自与目录不同的名称空间。有关LDAP身份验证机制的详细信息，请参见RFC 2829和“ 安全性”部分。JDK中的LDAP服务提供者接受一个字符串主体名称，它将直接传递给LDAP服务器。一些LDAP服务器接受DN，而其他一些则支持RFC 2829提出的方案。

20为什么我从目录中读取的名称中出现奇怪的引号和转义符？

JDK中的LDAP名称解析器在引用规则方面比较保守，但是仍然会生成“正确的”名称。另外，请记住NamingEnumeration返回的条目的名称是可以传递回Context和DirContext方法的复合名称。因此，如果名称包含与复合名称语法冲突的字符（例如正斜杠字符“ /”），则LDAP提供程序将提供编码以确保将斜杠字符视为LDAP名称的一部分而不是作为复合名称分隔符。
