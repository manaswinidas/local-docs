1概述

GNU Wget是一个免费实用程序，用于从Web非交互式下载文件。它支持HTTP ， HTTPS和FTP协议，以及通过HTTP代理进行检索。

本章是Wget功能的部分概述。

  • Wget是非交互式的，这意味着它可以在后台运行，而用户未登录。这使您可以开始检索并与系统断开连接，让Wget完成工作。相比之下，大多数Web浏览器都需要持续的用户状态，这在传输大量数据时可能是一个很大的障碍。
  • Wget可以跟踪HTML ， XHTML和CSS页面中的链接，以创建远程网站的本地版本，从而完全重新创建原始网站的目录结构。有时将其称为“递归下载”。在此过程中，Wget遵守机器人排除标准（ /robots.txt ）。可以指示Wget将下载文件中的链接转换为指向本地文件，以供离线查看。
  • 通过FTP检索时，可以使用文件名通配符匹配和目录的递归镜像。Wget可以读取HTTP和FTP服务器提供的时间戳信息，并将其存储在本地。这样，Wget可以查看自上次检索以来远程文件是否已更改，并自动检索新版本（如果有）。这使Wget适用于FTP站点以及主页的镜像。
  • Wget被设计为在缓慢或不稳定的网络连接上具有鲁棒性。如果下载由于网络问题而失败，它将继续重试，直到检索到整个文件为止。如果服务器支持重新获取，它将指示服务器从上次中断的地方继续下载。
  • Wget支持代理服务器，可以减轻网络负载，加快检索速度并提供防火墙后面的访问。Wget默认使用被动FTP下载，主动FTP是一个选项。
  • Wget支持IP版本6，即下一代IP。IPv6在编译时会自动检测到，并且可以在构建或运行时禁用。使用IPv6支持的二进制文件在仅IPv4的环境和双家庭环境中均能很好地工作。
  • 内置功能提供了一种机制，可以调整您希望关注的链接 （请参阅关注链接 ）。
  • 使用进度条跟踪单个下载的进度。交互式下载使用“温度计”样式的仪表进行跟踪，而非交互式下载则使用点进行跟踪，每个点表示接收到的固定数据量（默认为1KB）。两种量规均可根据您的喜好进行定制。
  • 通过命令行选项或初始化文件，大多数功能都是完全可配置的.wgetrc （请参阅启动文件 ）。Wget允许您定义全局启动文件（ /usr/local/etc/wgetrc默认情况下）进行网站设置。您也可以使用–config选项指定启动文件的位置。要禁用读取配置文件，请使用–no-config。如果同时给出了–config和–no-config，则将忽略–no-config。
  • 最后，GNU Wget是免费软件。这意味着每个人都可以按照自由软件基金会发布的GNU通用公共许可证的条款使用，重新分发和/或修改它（请参阅文件COPYING GNU Wget附带的详细信息）。

2.1 URL格式

URL是“统一资源定位器”的缩写。统一资源定位符是可通过Internet获得的资源的紧凑字符串表示形式。Wget根据RFC1738识别URL语法。这是使用最广泛的形式（方括号表示可选部分）：

http://host[:port]/directory/file
ftp://host[:port]/directory/file

您还可以在URL中对用户名和密码进行编码：

ftp://user:password@host/path
http://user:password@host/path

要么user要么password ，或两者都可能被忽略。如果您遗漏了HTTP用户名或密码，则不会发送任何身份验证。如果您忽略FTP用户名，则“ anonymous ' 将会被使用。如果您遗漏了FTP密码，那么您的电子邮件地址将作为默认密码提供。 1个

重要说明 ：如果您在命令行上指定了包含密码的URL ，则用户名和密码将通过以下方式对系统上的所有用户清晰可见： ps 。在多用户系统上，这是很大的安全风险。要解决此问题，请使用wget -i -并将URL馈送到Wget的标准输入，每个输入位于单独的行中，并以结尾C-d 。

您可以将网址中的不安全字符编码为“ %xy '， xy是字符ASCII值的十六进制表示形式。一些常见的不安全字符包括“ % ”（称为“ %25 '），' : ”（称为“ %3A '）和' @ ”（称为“ %40 '）。有关不安全字符的完整列表，请参阅RFC1738 。

Wget还支持type FTP URL的功能。默认情况下，以二进制模式检索FTP文档（类型为' i '），这意味着它们下载不变。另一个有用的模式是“ a '（ ASCII ）模式，可在不同操作系统之间转换行定界符，因此对于文本文件很有用。这是一个例子：

ftp://host/directory/file;type=a

由于历史原因（歇斯底里的？），还支持URL规范的两个备选变体。原因及其广泛使用。

仅FTP语法（受支持NcFTP ）：

host:/dir/file

仅HTTP语法（由Netscape ）：

host[:port]/dir/file

这两种替代形式已被弃用，将来可能会不再受支持。

如果您不了解这些表示法之间的区别，或者不知道使用哪种表示法，只需使用您喜欢的浏览器使用的普通普通格式，例如Lynx要么Netscape 。

2.2选项语法

由于Wget使用GNU getopt处理命令行参数，因此每个选项都有一个长格式和一个短格式。长选项更容易记住，但花一些时间输入。您可以自由地混合使用不同的选项样式，或在命令行参数后指定选项。因此，您可以这样写：

wget -r --tries=10 http://fly.srk.fer.hr/ -o log

接受参数的选项和参数之间的空格可以省略。代替 ' -o log ' 你可以写 ' -olog '。

您可以将几个不需要参数的选项放在一起，例如：

wget -drc URL

这完全等同于：

wget -d -r -c URL

由于可以在参数后指定选项，因此您可以使用' -- '。因此，以下将尝试下载URL ' -x '，向log ：

wget -o log -- -x

接受以逗号分隔的列表的所有选项均遵循约定，指定空白列表会清除其值。这对于清除.wgetrc设置。例如，如果您的.wgetrc套exclude_directories至/cgi-bin ，以下示例将首先重置它，然后将其设置为排除/~nobody和/~somebody 。您也可以清除中的列表.wgetrc （请参阅Wgetrc语法 ）。 

wget -X '' -X /~nobody,/~somebody

大多数不接受参数的选项都是布尔选项，之所以如此命名是因为它们的状态可以使用yes或no（“ boolean”）变量来捕获。例如， ' --follow-ftp '告诉Wget跟随HTML文件中的FTP链接，另一方面，' --no-glob '告诉它不要在FTP URL上执行文件遍历。布尔选项是肯定的还是否定的 （从'开始--no '）。所有这些选项都共享几个属性。

除非另有说明，否则假定默认行为与该选项完成的行为相反。例如，已记录的“ --follow-ftp假定默认为不遵循HTML页面中的FTP链接。

可以在前面加上“ --no- '选项名称；否定选项可以通过省略“ --no- ' 字首。这似乎是多余的-如果肯定性选项的默认设置是不执行任何操作，那么为什么要提供一种明确将其关闭的方法？但是启动文件实际上可能会更改默认值。例如，使用follow_ftp = on在.wgetrc使Wget默认情况下遵循 FTP链接，并使用' --no-follow-ftp '是从命令行恢复出厂默认设置的唯一方法。

' -q '

' -v '

打开所有所有可用数据的详细输出。默认输出是详细信息。

' -nv '
' --no-verbose '

' --save-cookies file '

包括“推荐人： url HTTP请求中的标题。对于通过服务器端处理来检索文档很有用，这些文档假定它们始终由交互式Web浏览器检索，并且仅当Referer设置为指向指向它们的页面之一时才可以正确显示。

识别为agent-string到HTTP服务器。

' --post-data=string '
' --post-file=file '

使用POST作为所有HTTP请求的方法，并在请求正文中发送指定的数据。 ' --post-data发送string作为数据，而“ --post-file '发送的内容file 。除此之外，它们以完全相同的方式工作。特别是，他们俩都希望表格的内容key1=value1&key2=value2 ，对特殊字符进行百分比编码；唯一的区别是，一个期望它的内容作为命令行参数，另一个期望它的内容来自文件。特别是， ' --post-file ' 不是用于将文件作为表单附件传输：这些文件必须显示为key=value数据（带有适当的百分比编码），就像其他所有内容一样。Wget当前不支持multipart/form-data用于发送POST数据；只要application/x-www-form-urlencoded 。只有一个--post-data '和' --post-file '应指定。

请注意，wget不需要内容的形式为key1=value1&key2=value2 ，也没有对其进行测试。Wget只会传输任何提供给它的数据。但是，大多数服务器期望在处理HTML表单时POST数据采用上述格式。

使用'发送POST请求时--post-file '选项，Wget将该文件视为二进制文件，并将发送POST请求中的每个字符，而不会去除尾随换行符或换页符。文本中的任何其他控制字符也将在POST请求中原样发送。

请注意，Wget需要提前知道POST数据的大小。因此， --post-file必须是常规文件；指定一个FIFO或类似的东西/dev/stdin将无法正常工作。尚不清楚如何解决HTTP / 1.0中固有的限制。尽管HTTP / 1.1引入了不需要事先知道请求长度的分块传输，但是客户端不能使用分块，除非它知道正在与HTTP / 1.1服务器通信。而且，在收到响应之前，它是不知道的，这反过来又要求请求已完成-一个鸡与蛋的问题。

注意：从1.15版开始，如果在POST请求完成后重定向Wget，则其行为将取决于服务器返回的响应代码。在301永久移动，302临时移动或307临时重定向的情况下，Wget将根据RFC2616继续发送POST请求。如果服务器希望客户端在重定向时更改Request方法，则应发送303 See Other响应代码。

本示例说明如何使用POST登录到服务器，然后继续下载所需的页面，该页面大概只能由授权用户访问：

# Log in to the server.  This can be done only once.
wget --save-cookies cookies.txt \
     --post-data 'user=foo&password=bar' \
     http://example.com/auth.php

# Now grab the page or pages we care about.
wget --load-cookies cookies.txt \
     -p http://example.com/interesting/article.php

如果服务器使用会话Cookie跟踪用户身份验证，则上述操作将无效，因为“ --save-cookies '不会保存它们（浏览器也不会保存），并且cookies.txt文件将为空。在这种情况下，请使用“ --keep-session-cookies ' 随着 ' --save-cookies以强制保存会话Cookie。

' --method=HTTP-Method '

为了实现RESTful脚本编制，Wget允许发送其他HTTP方法，而无需使用' --header=Header-Line '。Wget会在'之后使用传递给它的任何字符串--method '作为服务器的HTTP方法。

' --body-data=Data-String '
' --body-file=Data-File '

当需要将其他数据与使用'指定的方法一起发送到服务器时，必须设置--method '。 ' --body-data发送string作为数据，而“ --body-file '发送的内容file 。除此之外，它们以完全相同的方式工作。

目前， ' --body-file ' 不能整体传输文件。Wget当前不支持multipart/form-data用于传输数据；只要application/x-www-form-urlencoded 。将来可能会更改此设置，以便wget发送“ --body-file '作为完整文件，而不是将其内容发送到服务器。请注意，Wget需要事先了解BODY Data的内容，因此，“ --body-file '应该是常规文件。参见“ --post-file以获取更详细的说明。只有一个--body-data '和' --body-file '应指定。

如果在请求完成后重定向了Wget，则Wget将暂停当前方法并发送GET请求，直到重定向完成。这是它是用来明确指定请求方法不应该改变，除了307临时重定向所有重定向响应代码真。另一个例外是方法设置为POST ，在这种情况下，“ --post-data '跟随。

' --content-disposition '

如果将其设置为on，则实验性（非全功能）支持Content-Disposition标头已启用。当前，这可能会导致服务器额外往返服务器。 HEAD请求，并且已知存在一些错误，这就是为什么默认情况下当前未启用它的原因。

当与“ --metalink-over-http '和' --trust-server-names '， 一种 ' Content-Type: application/metalink4+xml '文件使用Content-Disposition文件名字段（如果有）。

如果将其设置为on，则当服务器使用指示错误的http状态代码响应时，wget将不会跳过内容。

' --trust-server-names '

如果设置了此选项，则在重定向时，本地文件名将基于重定向URL。默认情况下，本地文件名基于原始URL。在进行递归检索时，这可能会有所帮助，因为在许多网站中，重定向的URL对应于基础文件结构，而链接URL却不对应。

' --auth-no-challenge '

如果指定了此选项，则Wget将为所有请求发送基本HTTP身份验证信息（纯文本用户名和密码），就像Wget 1.10.2和以前的默认设置一样。

' --retry-on-host-error '

将主机错误（例如“名称解析中的临时故障”）视为非致命的临时错误。

' --retry-on-http-error=code[,code,...] '

使用此选项仅用于支持特殊用例，通常不建议使用，因为即使在服务器实际上正在尝试减少其负载的情况下，它也会强制重试。仅当您知道自己在做什么时，才应明智地使用它。

2.8 HTTPS（SSL / TLS）选项

为了支持加密的HTTP（HTTPS）下载，必须使用外部SSL库编译Wget。当前默认值为GnuTLS。此外，Wget还支持HSTS（HTTP严格传输安全性）。如果Wget在编译时没有SSL支持，则这些选项均不可用。

' --secure-protocol=protocol '

3递归下载

GNU Wget能够按照链接和目录结构遍历Web的各个部分（或单个HTTP或FTP服务器）。我们将其称为递归检索或递归 。

使用HTTP URL ，Wget通过给定的URL检索并解析HTML或CSS ，并通过标记等方式检索文档所引用的文件。 href要么src或CSS URI值使用指定的' url()功能标记。如果新下载的文件也属于text/html ， application/xhtml+xml ， 要么text/css ，它将被解析并进一步跟踪。

HTTP和HTML / CSS内容的递归检索是广度优先的 。这意味着Wget首先下载请求的文档，然后下载从该文档链接的文档，再下载由它们链接的文档，依此类推。换句话说，Wget首先下载深度为1的文档，然后下载深度为2的文档，依此类推，直到指定的最大深度。

检索可能下降的最大深度由' -l ' 选项。默认的最大深度为五层。

递归检索FTP URL时 ，Wget将从远程服务器上的给定目录树（包括指定深度的子目录）中检索所有数据，在本地创建其镜像。FTP检索也受到depth参数。与HTTP递归不同， FTP递归是深度优先的。

默认情况下，Wget将创建一个本地目录树，与远程服务器上的目录树相对应。

递归检索可以找到许多应用程序，其中最重要的是镜像。对于WWW演示文稿以及任何其他通过在本地存储文件来绕开慢速网络连接的机会，它也很有用。

应该警告您，递归下载可能会使远程服务器超载。因此，如果许多管理员检测到大量内容的非常快速的下载，他们可能会皱眉，并可能禁止访问您的站点。从Internet服务器下载时，请考虑使用“ -w '选项在访问服务器之间引入延迟。下载将花费更长的时间，但是服务器管理员不会因您的无礼而惊慌。

当然，递归下载可能会在您的计算机上引起问题。如果不进行检查而运行，则可以轻松填充磁盘。如果从本地网络下载，它还会占用系统带宽，并消耗内存和CPU。

尝试指定与您要实现的下载类型相匹配的条件。如果您只想下载一页，请使用“ --page-requisites无需任何其他递归。如果要在一个目录下下载内容，请使用“ -np以免从其他目录下载内容。如果要从一个目录下载所有文件，请使用“ -l 1以确保递归深度不会超过一。有关更多信息，请参见跟踪链接 。

递归检索应谨慎使用。不要说您没有被警告。

4.2文件类型

从网上下载资料时，您通常会希望将检索限制为仅某些文件类型。例如，如果您对下载GIF感兴趣，则不会因获得大量PostScript文档而大喜过望，反之亦然。

' -A acclist '

' accept = acclist '
' --accept-regex urlregex '
' accept-regex = urlregex '

“ --accept '选项是Wget在递归检索期间将下载的文件后缀或模式的列表。后缀是文件的结尾部分，由“普通”字母组成，例如“ gif ' 要么 ' .jpg '。匹配模式包含类似shell的通配符，例如' books* ' 要么 ' zelazny*196[0-9]* '。

因此，指定“ wget -A gif,jpg '将使Wget仅下载以'结尾的文件gif ' 要么 ' jpg '，即GIF和JPEG 。另一方面， ' wget -A "zelazny*196[0-9]*" “将仅下载以”开头的文件zelazny ”，其中包含1960年至1969年之间的数字。在您的外壳手册中查找有关模式匹配如何工作的描述。

当然，可以将任意数量的后缀和模式组合成一个逗号分隔的列表，并作为' -A '。

“ --accept-regex '选项是一个与完整URL匹配的正则表达式。

' -R rejlist '
' --reject rejlist '
' reject = rejlist '
' --reject-regex urlregex '
' reject-regex = urlregex '

' --reject '选项的作用方式与' --accept '，只有其逻辑是相反的； Wget将下载所有文件， 但不包括与列表中后缀（或模式）匹配的文件。

“ --accept-regex '选项是一个与完整URL匹配的正则表达式。

  • 如果本地文件已经存在，并且“ --no-directories已指定'，则数字后缀将附加到原始名称后。

这种行为也被认为是不希望的，并且可能在Wget的未来版本中更改。

什么时候 ' -L '开启，仅会跟踪相对链接。相对链接在这里定义为那些不引用Web服务器根目录的链接。例如，这些链接是相对的：

5时间戳记

从Internet镜像信息的最重要方面之一就是更新档案。

一遍又一遍地下载整个档案，只是为了替换一些更改的文件，这既浪费带宽和金钱，又花费时间进行更新，这是昂贵的。这就是为什么所有镜像工具都提供增量更新选项的原因。

这种更新机制意味着将在搜索新文件时扫描远程服务器。仅那些新文件将代替旧文件下载。

如果满足以下两个条件之一，则认为文件是新文件：

  1. 该名称的文件在本地尚不存在。
  2. 具有该名称的文件确实存在，但是远程文件的修改时间比本地文件的修改时间最近。

为了实现这一点，程序需要知道本地和远程文件的最后修改时间。我们将此信息称为文件的时间戳 。

GNU Wget中的时间戳已使用' --timestamping '（' -N '）选项，或通过timestamping = on指令中.wgetrc 。使用此选项，对于要下载的每个文件，Wget都会检查是否存在相同名称的本地文件。如果是这样，并且远程文件不是较新的，则Wget将不会下载它。

如果本地文件不存在，或者文件大小不匹配，则无论时间戳如何，Wget都会下载远程文件。

5.1时间戳用法

时间戳的用法很简单。假设您要下载文件，以便保留其修改日期。 

wget -S http://www.gnu.ai.mit.edu/

一个简单的ls -l显示本地文件上的时间戳等于Last-Modified标头，由服务器返回。如您所见，时间戳记信息在本地保留，即使没有“ -N '（至少对于HTTP而言 ）。

几天后，您希望Wget检查远程文件是否已更改，如果已下载，请下载。 

wget -N http://www.gnu.ai.mit.edu/

Wget将询问服务器最后修改的日期。如果本地文件具有与服务器相同的时间戳或较新的时间戳，则将不会重新获取远程文件。但是，如果远程文件是较新的文件，则Wget将继续获取它。

FTP也是如此 。例如：

wget "ftp://ftp.ifi.uio.no/pub/emacs/gnus/*"

（该URL周围的引号是为了防止shell尝试解释“ * '。）

下载后，本地目录列表将显示时间戳与远程服务器上的时间戳匹配。使用'重新发出命令-N ”将使Wget 仅重新获取自上次下载以来已修改的文件。

如果您希望每周镜像GNU归档文件，则可以每周使用以下命令：

wget --timestamping -r ftp://ftp.gnu.org/pub/gnu/

请注意，时间戳仅适用于服务器为其提供时间戳的文件。对于HTTP ，这取决于获取Last-Modified标头。对于FTP ，这取决于获取目录列表，该目录带有Wget可以解析的格式的日期（请参阅FTP时间戳内部 ）。

5.3 FTP时间戳内部

7.1简单用法

  • 假设您要下载一个URL 。只需输入：
    wget http://fly.srk.fer.hr/
  • 但是，如果连接速度慢且文件太长，会发生什么？在检索整个文件之前，连接可能会失败多次。在这种情况下，Wget将尝试获取文件，直到获取完整文件或超过默认重试次数（即20）为止。将尝试次数更改为45很容易，以确保整个文件都能安全到达：
    wget --tries=45 http://fly.srk.fer.hr/jpg/flyweb.jpg
  • 现在，让Wget在后台工作，并将其进度写入日志文件log 。键入' --tries '，因此我们将使用' -t '。 
    wget -t 45 -o log http://fly.srk.fer.hr/jpg/flyweb.jpg &

    该行末的“＆”号确保Wget在后台运行。要限制重试次数，请使用' -t inf '。

  • FTP的用法很简单。Wget将负责登录名和密码。 
    wget ftp://gnjilux.srk.fer.hr/welcome.msg

    wget ftp://ftp.gnu.org/pub/gnu/
links index.html

7.2高级用法

  • 您有一个包含要下载的URL的文件吗？使用 ' -i '开关：

    wget --convert-links -r https://www.gnu.org/ -o gnulog

    wget -p --convert-links http://www.example.com/dir/page.html

    wget -p --convert-links -nH -nd -Pdownload \
     http://www.example.com/dir/page.html

    wget -S http://www.lycos.com/

    wget --save-headers http://www.lycos.com/
more index.html
  • 检索“ wuarchive.wustl.edu '，将其保存到/tmp 。 
    wget -r -l2 -P/tmp ftp://wuarchive.wustl.edu/
  • 您要从HTTP服务器上的目录下载所有GIF 。您尝试过wget http://www.example.com/dir/*.gif ”，但这没有用，因为HTTP检索不支持globbing。在这种情况下，请使用：
    wget -r -l1 --no-parent -A.gif http://www.example.com/dir/

    更详细，但效果相同。 ' -r -l1 '表示以最大深度1进行递归检索（请参阅递归下载 ）。 --no-parent '表示对父目录的引用将被忽略（请参阅基于目录的限制 ），和' -A.gif '表示仅下载GIF文件。 ' -A "*.gif" '也可以。

    wget -nc -r https://www.gnu.org/

    wget ftp://hniksic:mypassword@unix.example.com/.emacs

8.1代理

代理是专用的HTTP服务器，旨在将数据从远程服务器传输到本地客户端。代理的一种典型用法是为缓慢连接后的用户减轻网络负载。这是通过将所有HTTP和FTP请求引导通过缓存传输数据的代理来实现的。当再次请求缓存的资源时，代理将从缓存中返回数据。代理的另一种用法是用于（出于安全原因）将其内部网络与Internet其余部分分开的公司。为了从Web获得信息，他们的用户使用授权的代理连接和检索远程数据。

Wget支持HTTP和FTP检索的代理。Wget可以识别的指定代理位置的标准方法是使用以下环境变量：

http_proxy
https_proxy

如果设置，则http_proxy和https_proxy变量应分别包含HTTP和HTTPS连接的代理的URL 。

ftp_proxy

此变量应包含FTP连接的代理的URL 。很常见的是http_proxy和ftp_proxy设置为相同的URL 。

no_proxy

此变量应包含逗号分隔的域扩展列表，不应将其用于代理。例如，如果no_proxy是' .mit.edu '，则不会使用代理从MIT检索文档。

除了环境变量之外，还可以从Wget本身中指定代理位置和设置。

' --no-proxy '
' proxy = on/off '

即使设置了适当的环境变量，此选项和相应的命令也可用于禁止使用代理。

' http_proxy = URL '
' https_proxy = URL '
' ftp_proxy = URL '
' no_proxy = string '

这些启动文件变量使您可以覆盖环境指定的代理设置。

某些代理服务器需要授权才能使用它们。授权由用户名和密码组成 ，必须由Wget发送。与HTTP授权一样，存在几种身份验证方案。对于代理授权，仅Basic当前已实施身份验证方案。

您可以通过代理URL或命令行选项指定用户名和密码。假设公司的代理位于“ proxy.company.com在端口8001上，包含授权数据的代理URL位置可能如下所示：

http://hniksic:mypassword@proxy.company.com:8001/

或者，您可以使用“ proxy-user '和' proxy-password '选项，以及等效项.wgetrc设定proxy_user和proxy_password设置代理用户名和密码。

8.6报告错误

欢迎您通过GNU Wget错误跟踪器（请参阅https://savannah.gnu.org/bugs/?func=additem&group=wget ）或我们的邮件列表bug-wget@gnu.org提交错误报告。

访问https://lists.gnu.org/mailman/listinfo/bug-wget以获取更多信息（如何订阅，列出档案等）。

在实际提交错误报告之前，请尝试遵循一些简单的准则。

  1. 请尝试确定您看到的行为确实是一个错误。如果Wget崩溃，那是一个错误。如果Wget的行为不像所记录的那样，那就是一个错误。如果一切正常，但是您不确定它们的工作方式，则可能是一个错误，但是您可能需要仔细检查文档和邮件列表（请参阅邮件列表 ）。
  2. 尝试在尽可能简单的情况下重复该错误。例如，如果Wget在下载“ wget -rl0 -kKE -t5 --no-proxy http://example.com -o /tmp/log '，则应尝试查看崩溃是否可重复，以及是否会发生一组更简单的选项。您甚至可以尝试在发生崩溃的页面上开始下载，以查看该页面是否以某种方式触发了崩溃。

8.8信号

由于Wget的目的是进行后台工作，因此它会捕获挂断信号（ SIGHUP ）并忽略它。如果输出在标准输出上，它将被重定向到名为wget-log 。除此以外， SIGHUP被忽略。当您希望在启动Wget后重定向其输出时，这很方便。 

$ wget http://www.gnus.org/dist/gnus.tar.gz &
...
$ kill -HUP %%
SIGHUP received, redirecting output to `wget-log'.

除此之外，Wget将不会尝试以任何方式干扰信号。 C-c ， kill -TERM和kill -KILL应该杀死它。

9.1机器人排除

使Wget在网站上漫无目的地漫游非常容易，它会吸收所有正在进行的可用数据。' wget -r site ”，您就定了。大？不适合服务器管理员。

只要Wget仅检索静态页面，并以合理的速度进行操作（请参见“ --wait '选项），没有太大的问题。问题在于Wget无法区分最小的静态页面和最苛刻的CGI。我知道的网站有一个由CGI Perl脚本处理的部分，该脚本可将Info文件即时转换为HTML 。该脚本很慢，但是对于人类用户偶尔查看Info文件来说效果很好。但是，当某人的递归Wget下载偶然通过索引链接到通过脚本链接到所有Info文件的索引页时，系统便陷于瘫痪，而没有向用户提供任何有用的信息（此转换Info文件的任务可以在本地完成并访问有关系统上所有已安装的GNU软件的“信息”文档，请访问info命令）。

为了避免此类事故，并保护需要保护免受行为正常的机器人侵害的文档的私密性，发明了机器人排除的概念。这个想法是服务器管理员和文档作者可以指定他们希望其免受机器人攻击的站点的哪些部分，以及他们将允许访问的部分。

最流行的机制，也是所有主要机器人都支持的事实上的标准，是Martijn Koster等人编写的“机器人排除标准”（RES）。在1994年。它指定包含指令的文本文件的格式，这些指令指示机械手应避免使用哪些URL路径。要由机器人找到，必须将规格放在/robots.txt在服务器根目录中，机器人应该下载并解析该根目录。

尽管从严格意义上讲，Wget并不是网络机器人，但它可以下载网站的大部分内容，而无需用户干预来下载单个页面。因此，Wget在递归下载时尊重RES。例如，当您发出：

wget -r http://www.example.com/

首先是“ www.example.com将会下载。如果Wget发现要从该服务器下载更多文档，它将请求“ http://www.example.com/robots.txt '，如果找到了它，则可以将其用于进一步的下载。 robots.txt每个服务器仅加载一次。

在1.8版之前，Wget支持Martijn Koster在1994年编写的该标准的第一个版本，该版本可从http://www.robotstxt.org/orig.html获得 。从1.8版开始，Wget支持Internet草案“ 标题为“ Web机器人控制方法”。据我所知，从来没有对RFC进行过草稿，请访问http://www.robotstxt.org/norobots-rfc.txt 。

本手册不再包含机器人排除标准的文本。

第二种鲜为人知的机制使单个文档的作者可以指定他们是否希望机械手跟踪文件中的链接。这是通过使用META标签，如下所示：

<meta name="robots" content="nofollow">

有关详细信息， 请参见http://www.robotstxt.org/meta.html 。除了通常的方式外，Wget还支持这种机器人排除方法/robots.txt排除。

如果您知道自己在做什么，并且确实希望关闭机器人排除功能，请设置robots变数为' off ' 在你的.wgetrc 。您可以使用以下命令从命令行获得相同的效果-e开关，例如'wget -e robots=off url... '。

9.2安全注意事项

使用Wget时，必须注意它会通过网络发送未加密的密码，这可能会带来安全问题。这里是主要问题和一些解决方案。

  1. 命令行上的密码可以使用来查看ps 。最好的解决方法是使用wget -i -并将URL馈送到Wget的标准输入，每个输入位于单独的行中，并以结尾C-d 。另一个解决方法是使用.netrc存储密码；但是，存储未加密的密码也被视为安全隐患。
  2. 使用不安全的基本身份验证方案，未加密的密码通过网络路由器和网关传输。
  3. FTP密码也绝不会加密。目前尚无好的解决方案。
  4. 尽管Wget的“正常”输出尝试隐藏密码，但调试日志以各种形式显示密码。发送调试日志时要格外小心，可以避免此问题（是的，即使将它们发送给我）。

    “标题为XYZ”一节是指文档的命名子单元，其标题恰好是XYZ或在括号中包含XYZ的文字，后面是将XYZ转换为另一种语言的文字。（此处XYZ代表以下提到的特定部分名称，例如“致谢”，“奉献”，“认可”或“历史记录”。）在修改文档时，要“保留此部分的标题”，则意味着根据此定义，该部分仍为“已命名的XYZ”部分。

    该文件的声明旁边可能包含“免责声明”，该声明指出本许可适用于该文件。这些保修免责声明被视为已包含在本许可中，但仅用于免责声明：这些保修免责声明可能具有的任何其他暗示都是无效的，并且对本许可的含义没有任何影响。

  3. VERBATIM复制

    您可以在任何媒介中以商业或非商业方式复制和分发本文档，但前提是本许可，版权声明和声明本许可适用于本文档的许可声明均以所有副本形式复制，并且您不添加任何其他条件符合本许可的规定。您不得使用技术措施来阻碍或控制您复制或分发的副本的阅读或进一步复制。但是，您可以接受赔偿以换取副本。如果分发足够多的副本，则还必须遵循第3节中的条件。

    您还可以在上述相同条件下出借副本，并可以公开展示副本。

  4. 大量复制

    如果您发布文档的印刷副本（或通常具有印刷封面的介质副本），且编号超过100，并且文档的许可通知书要求包含封面文字，则必须将这些副本随附在清晰可见的所有文档中封面文字：封面上的封面文字和封底上的封面文字。两种封面都必须清楚，清楚地标识您是这些副本的发行者。封面必须显示完整的标题，标题中的所有单词均应同样醒目且可见。您还可以在封面上添加其他材料。只要封面保留了文档标题并满足这些条件，则仅对封面进行更改即可在其他方面被视为逐字复印。

    如果任何一个封面所需的文本量太大，无法清晰地容纳，则应将列出的第一个（合理合理的数量）放在实际的封面上，然后将其余部分继续放在相邻的页面上。

    如果您发布或分发的Opaque文档副本数量超过100，则必须在每份Opaque副本中包含机器可读的透明副本，或者在每份Opaque副本中或其中声明一个计算机网络位置，通用网络将从该位置开始使用public可以使用公共标准网络协议下载完整的文档透明副本，而无需添加任何材料。如果使用后一种选项，则在开始批量分发不透明副本时，必须采取合理的审慎步骤，以确保该透明副本将在指定位置保持可访问的状态，直到您上一次分发不透明副本至少一年之后。该版本的不透明副本（直接或通过您的代理商或零售商）向公众发布。

    要求（但不是必须）在重新分发大量副本之前，与文档作者保持良好联系，以便他们有机会为您提供文档的更新版本。

  5. 修改方式

    您可以在上述第2节和第3节的条件下复制和分发本文档的修改版本，前提是您严格按照本许可协议发布修改版本，而修改版本则充当文档的角色，从而许可对本文档的分发和修改拥有副本的修改版本。此外，您必须在修改后的版本中执行以下操作：

    1. 在标题页（和封面，如果有的话）中使用与本文档以及先前版本（如果有的话，应在文档的“历史记录”部分中列出）不同的标题。如果该版本的原始发行者给予许可，则可以使用与先前版本相同的标题。
    2. 在标题页上列出作者，一个或多个负责修改版本著作权的作者或实体，以及至少五名文档的主要作者（所有主要作者，如果少于五），除非他们使您脱离此要求。
    3. 在“标题”页面上声明修改版本的发布者的名称，作为发布者。
    4. 保留文档的所有版权声明。
    5. 在其他版权声明旁边添加用于修改的适当版权声明。
    6. 在版权声明之后，立即包含一个许可声明，以下面的附录中所示的形式，向公众授予根据本许可条款使用修改版的许可。
    7. 在该许可证声明中保留不可变部分的完整列表以及文档的许可证声明中提供的必需封面文字。
    8. 随附本许可证的未更改副本。
    9. 保留标题为“历史”的部分，保留其标题，并在其上添加至少说明标题页上给出的修改版本的标题，年份，新作者和发行者的项目。如果文档中没有标题为“历史记录”的部分，请在其标题页上创建一个说明文档的标题，年份，作者和出版者的内容，然后添加一项描述上一句中所述的修改版本。
    10. 保留文档中给出的网络位置（如果有），以供公众访问文档的透明副本，同样，保留文档中给出的基于其以前版本的网络位置。这些可以放在“历史记录”部分中。对于在文档本身之前至少四年出版的作品，或者如果其所指版本的原始发行者给予许可，您可以省略其网络位置。
    11. 对于标题为“致谢”或“奉献”的任何部分，请保留本部分的标题，并在本部分中保留其中给出的每个贡献者致谢和/或奉献的所有内容和基调。
    12. 保留文档的所有不变部分，其文本和标题保持不变。章节编号或同等编号不视为章节标题的一部分。
    13. 删除任何标题为“背书”的部分。这样的部分可能不包含在修改版本中。
    14. 请勿将任何现有部分的标题重命名为“背书”，或与任何不变部分的标题冲突。
    15. 保留所有保修免责声明。

    如果修改后的版本包括新的前节或附录，这些节或附录符合第二节的规定，并且不包含从文档中复制的材料，则可以选择将其中的部分或全部指定为不变的。为此，请将其标题添加到“修改版本”许可声明中的“不变部分”列表中。这些标题必须与任何其他节标题不同。

    您可以添加标题为“背书”的部分，但其中仅包含各方对您的修改版本的背书，例如，同行评审声明或该文本已被组织批准为标准的权威定义。

    您可以在修改版本的封面文本列表的末尾添加最多5个单词的段落作为“封面文本”，以及最多25个单词的段落作为“封面文本”。任何一个实体（或通过它们的安排）都只能添加“封面文字”和“封面文字”中的一个段落。如果文档已经包含同一封面的封面文字，该文字是您先前添加的或由您代表的同一实体做出的安排而添加的，则不得添加其他文字；但是您可以在添加了旧版本的以前的发布者的明确许可下替换旧版本。

    本文档的作者和发行者未获得本许可证的许可，不得使用其名称进行宣传或主张或暗示认可任何修改版本。

  6. 合并文件

    您可以根据上文第4节中针对修改版本定义的条款，将文档与根据本许可发布的其他文档进行组合，但前提是您必须将所有未经修改的原始文档的所有不变部分都包含在其中，并列出所有这些文档作为您许可协议中合并工作的不变部分，并保留其所有保修免责声明。

    合并后的作品仅包含此许可的一个副本，并且多个相同的不变部分可用一个副本代替。如果存在多个具有相同名称但内容不同的不变部分，请在该部分的末尾加上括号，在该部分的末尾添加原始作者或发布者的名称（如果已知），以使每个此类的标题唯一。唯一编号。对合并作品的许可声明中不变部分列表中的部分标题进行相同的调整。

    在组合中，必须将各种原始文档中标题为“ History”的任何部分组合在一起，组成一个标题为“ History”的部分；同样，将标题为“致谢”的所有部分以及标题为“奉献”的任何部分组合在一起。您必须删除所有标题为“背书”的部分。

  7. 文件收集

    您可以收集由本许可协议下发布的文档和其他文档组成的集合，并在该文档中将其替换为各个文档中包含的单个副本，前提是您遵守本许可协议的规定。在所有其他方面逐字逐字复印。

    您可以从此类集合中提取单个文档，并根据本许可证单独分发它，前提是您将本许可证的副本插入提取的文档中，并且在与该文档的逐字复制有关的所有其他方面遵守本许可证。

  8. 独立工作的集合

    如果文件或衍生产品的版权不用于限制合法权利，则该文件或其衍生物与其他单独或独立的文件或作品在存储或分发介质中或在其上的汇编将被称为“汇总”。汇编用户的数量超出了个人作品的许可范围。当文档包含在汇总中时，本许可不适用于汇总中的其他作品，而这些作品本身不是文档的衍生作品。

    如果第3节的封面文字要求适用于本文档的这些副本，则如果该文档少于全部汇总的一半，则可以将文档的封面文字放在汇总中将文档括起来的封面上，或者如果文档为电子形式，则具有等同的封面电子形式。否则，它们必须出现在将整个集合括起来的印刷封面上。

  9. 翻译

    翻译被视为一种修改，因此您可以根据第4节的条款分发文档的翻译。用翻译替换不变部分需要其版权所有者的特殊许可，但是除了这些不变部分的原始版本之外，您还可以包括部分或全部不变部分的翻译。您可以在本文档以及所有保修免责声明中包括本许可，所有许可声明的翻译，但前提是您还包括本许可的原始英文版本以及这些声明和免责声明的原始版本。如果本许可的翻译版本与原始版本或声明或免责声明之间存在分歧，则以原始版本为准。
