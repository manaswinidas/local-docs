21.6睡觉

功能sleep提供了一种使程序等待较短时间的简单方法。如果您的程序不使用信号（终止除外），那么您可以期望sleep在整个指定间隔内可靠地等待。除此以外， sleep如果有信号到达可以早点返回；如果您想等待给定的时间间隔而不管信号如何，请使用select （请参见等待I / O ），并且不指定要等待的任何描述符。

函数： unsigned int 睡眠 （unsigned int seconds ）

初步： MT不安全信号：SIGCHLD / linux | AS不安全| AC不安全|请参阅POSIX安全概念 。

的sleep功能等待seconds秒或直到信号发出为止，以先到者为准。

如果sleep由于请求的间隔已结束而返回，因此返回零值。如果由于信号传递而返回，则其返回值是睡眠间隔中的剩余时间。

的sleep函数在unistd.h 。

通过使用的返回值，抵制在固定时间段内实施睡眠的诱惑sleep ，当非零时调用sleep再次。只要信号不频繁到达，这将以一定的精度工作。但是，每个信号都可能导致最终唤醒时间再增加一秒钟左右。假设一些信号碰巧由于运气而迅速接—而至，这对缩短或延长等待时间没有限制。

而是，计算程序应停止等待的日历时间，并继续尝试等待该日历时间。这不会超过一秒钟。只需多一点工作，您就可以使用select并让等待时间很准确。（当然，沉重的系统负载会导致不可避免的额外延迟-除非计算机专用于一种应用程序，否则您无法避免这种情况。）

在某些系统上sleep如果您的程序使用，可以做奇怪的事情SIGALRM明确地。即使SIGALRM信号在以下情况下被忽略或阻止sleep叫做， sleep可能会在交付商品时过早归还SIGALRM信号。如果您已经为SIGALRM信号和SIGALRM在进程处于休眠状态时传递了信号，所采取的措施可能仅仅是导致sleep返回而不是调用处理程序。而如果sleep被信号传递中断，该信号的处理程序请求警报或更改对信号的处理SIGALRM ，此处理程序和sleep会干扰。

在GNU系统上，可以安全使用sleep和SIGALRM在同一程序中，因为sleep不能通过SIGALRM 。

函数： int nanosleep （const struct timespec * requested_time ，struct timespec * remaining ）

初步： MT安全| AS安全交流安全请参阅POSIX安全概念 。

如果以秒为单位的分辨率还不够nanosleep功能可以使用。顾名思义，睡眠间隔可以指定为纳秒。睡眠间隔的实际经过时间可能会更长，因为系统会将您请求的经过时间四舍五入到系统可以提供的实际分辨率的下一个整数倍。

* requested_time是您要睡眠的间隔的经过时间。

该函数返回为* remaining您请求睡眠的时间间隔中剩余的经过时间。如果间隔完成而没有信号中断，则为零。

struct timespec请参见“ 经过时间”中的说明 。

如果函数由于间隔超过而返回，则返回值为零。如果函数返回-1，则全局变量errno设置为以下值：

EINTR

调用被中断，因为信号已传递到线程。如果remaining参数不是结构所指向的空指针remaining更新为包含剩余的经过时间。

EINVAL

的纳秒值requested_time参数包含非法值。该值是负数或大于或等于10亿。

此功能是多线程程序中的取消点。如果线程当时分配了一些资源（例如内存，文件描述符，信号量或其他内容），这将是一个问题nanosleep叫做。如果线程被取消，这些资源将保持分配状态，直到程序结束。为了避免这种情况nanosleep应该使用取消处理程序进行保护。

的nanosleep函数在time.h 。
