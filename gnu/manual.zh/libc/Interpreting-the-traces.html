3.2.4.4解释痕迹

如果您查看输出，它将类似于以下内容：

= Start
 [0x8048209] - 0x8064cc8
 [0x8048209] - 0x8064ce0
 [0x8048209] - 0x8064cf8
 [0x80481eb] + 0x8064c48 0x14
 [0x80481eb] + 0x8064c60 0x14
 [0x80481eb] + 0x8064c78 0x14
 [0x80481eb] + 0x8064c90 0x14
= End

这一切并不真正重要，因为跟踪文件并不意味着人类可以读取。因此，没有注意可读性。取而代之的是GNU C库附带的一个程序，该程序解释跟踪并以用户友好的方式输出摘要。该程序称为mtrace （实际上是Perl脚本），它需要一个或两个参数。无论如何，必须指定带有跟踪输出的文件名。如果在跟踪文件的名称之前有一个可选参数，则该参数必须是生成跟踪的程序的名称。 

drepper$ mtrace tst-mtrace log
No memory leaks.

在这种情况下，程序tst-mtrace运行并生成了跟踪文件log 。消息打印者mtrace显示代码没有问题，此后释放了所有分配的内存。

如果我们打电话mtrace在上面给出的示例跟踪中，我们将得到一个不同的结果：

drepper$ mtrace errlog
- 0x08064cc8 Free 2 was never alloc'd 0x8048209
- 0x08064ce0 Free 3 was never alloc'd 0x8048209
- 0x08064cf8 Free 4 was never alloc'd 0x8048209

Memory not freed:
-----------------
   Address     Size     Caller
0x08064c48     0x14  at 0x80481eb
0x08064c60     0x14  at 0x80481eb
0x08064c78     0x14  at 0x80481eb
0x08064c90     0x14  at 0x80481eb

我们已经打电话mtrace仅带有一个参数，因此脚本没有机会找出跟踪中给出的地址的含义。我们可以做得更好：

drepper$ mtrace tst errlog
- 0x08064cc8 Free 2 was never alloc'd /home/drepper/tst.c:39
- 0x08064ce0 Free 3 was never alloc'd /home/drepper/tst.c:39
- 0x08064cf8 Free 4 was never alloc'd /home/drepper/tst.c:39

Memory not freed:
-----------------
   Address     Size     Caller
0x08064c48     0x14  at /home/drepper/tst.c:33
0x08064c60     0x14  at /home/drepper/tst.c:33
0x08064c78     0x14  at /home/drepper/tst.c:33
0x08064c90     0x14  at /home/drepper/tst.c:33

突然之间，输出变得更加有意义，用户可以立即看到可以找到引起问题的函数调用的位置。

解释此输出并不复杂。最多检测到两种不同的情况。第一， free调用了从未由分配函数之一返回的指针。这通常是一个非常糟糕的问题，输出的前三行显示了该问题。像这样的情况非常罕见，如果出现的话，它们会急剧地出现：程序通常会崩溃。

另一个很难检测到的情况是内存泄漏。如您在输出中看到的mtrace函数收集所有这些信息，因此可以说该程序从源文件的第33行调用分配函数/home/drepper/tst-mtrace.c程序终止前四次没有释放此内存。这是否是一个真正的问题仍有待调查。
