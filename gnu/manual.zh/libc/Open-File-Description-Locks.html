13.17打开文件描述锁

与进程相关的记录锁（请参阅文件锁 ）相比，打开文件描述记录锁与打开的文件描述而不是进程相关联。

使用fcntl在已经具有通过同一文件描述符创建的现有打开文件描述锁的区域上应用打开文件描述锁将永远不会导致锁冲突。

子进程还继承了打开文件描述锁fork ， 要么clone与CLONE_FILES设置（请参阅创建流程 ），以及文件描述符。

区分打开的文件描述 （打开文件的实例，通常是通过调用open ）和打开文件描述符 ，该描述符是一个数字值，表示打开文件的描述。这里所描述的锁与打开的文件的描述 ，而不是打开文件描述符关联。

使用dup （请参阅“ 复制描述符” ）复制文件描述符并不会为您提供新的打开文件描述，而是会将引用复制到现有的打开文件描述并将其分配给新的文件描述符。因此，打开的文件描述锁设置在由克隆的文件描述符上dup永远不会与原始描述符上设置的打开文件描述锁发生冲突，因为它们引用了相同的打开文件描述。根据所涉及锁的范围和类型，原始锁可以通过以下方式修改： F_OFD_SETLK要么F_OFD_SETLKW在这种情况下指挥。

即使文件是由同一进程或在同一打开文件描述符上获取的，打开文件描述锁也总是与进程相关的锁冲突。

打开文件描述锁使用相同的struct flock作为与进程相关的锁（作为参数）（请参见File Locks ）以及command值也在头文件中声明fcntl.h 。要使用它们，宏_GNU_SOURCE必须在包含任何头文件之前定义。

与进程相关的锁相反，任何struct flock用作打开文件描述锁定命令的参数时，必须具有l_pid值设置为0 。另外，返回有关打开文件描述的信息时，请锁定F_GETLK要么F_OFD_GETLK要求l_pid在struct flock将被设置为-1，以指示该锁与进程无关。

当相同struct flock用作参数F_OFD_SETLK要么F_OFD_SETLKW用于后的请求F_OFD_GETLK要求，有必要检查并重置l_pid字段为0 。

巨集： int F_OFD_GETLK

此宏用作command争论fcntl ，以指定它应获取有关锁的信息。此命令需要类型的第三个参数struct flock * 传递给fcntl ，因此调用的形式为：

fcntl (filedes, F_OFD_GETLK, lockp)

如果已经有一个锁，它将阻塞lockp参数，有关该锁的信息被写入*lockp 。如果现有锁与按指定进行新锁兼容，则不会报告这些锁。因此，您应该指定一种锁定类型F_WRLCK如果您想了解读和写锁，或者F_RDLCK如果您只想了解写锁定。

可能有一个以上的锁影响到由lockp争论，但是fcntl仅返回有关其中之一的信息。在这种情况下返回的锁是不确定的。

的l_whence的成员lockp结构设置为SEEK_SET和l_start和l_len字段设置为标识锁定区域。

如果不存在冲突的锁，则唯一更改为lockp结构是更新l_type值字段F_UNLCK 。

的正常返回值fcntl使用此命令时，成功则为0或-1 ，表示错误。下列errno为该命令定义了错误条件：

EBADF

的filedes参数无效。

EINVAL

要么lockp参数未指定有效的锁定信息，操作系统内核不支持打开的文件描述锁定或与之关联的文件filedes不支持锁。

巨集： int F_OFD_SETLK

此宏用作command争论fcntl ，以指定应设置或清除锁定。此命令需要类型的第三个参数struct flock * 传递给fcntl ，因此调用的形式为：

fcntl (filedes, F_OFD_SETLK, lockp)

如果打开的文件已经在区域的任何部分上具有锁，则该部分上的旧锁将被新锁替换。您可以通过指定以下类型的锁来删除锁： F_UNLCK 。

如果无法设置锁定， fcntl立即返回值-1 。该命令不等待其他任务释放锁。如果fcntl成功，返回0 。

下列errno为该命令定义了错误条件：

EAGAIN

无法设置该锁定，因为该锁定已被文件上的现有锁定阻止。

EBADF

要么： filedes参数无效；您请求了读锁，但是filedes未开放读取访问权限；或者，您请求写锁定，但是filedes未打开以进行写访问。

EINVAL

要么lockp参数未指定有效的锁定信息，操作系统内核不支持打开的文件描述锁定或与之关联的文件filedes不支持锁。

ENOLCK

系统已用完文件锁定资源；已经有太多的文件锁。

设计良好的文件系统永远不会报告此错误，因为它们对锁的数量没有限制。但是，您仍然必须考虑此错误的可能性，因为它可能是由于网络访问另一台计算机上的文件系统而导致的。

巨集： int F_OFD_SETLKW

此宏用作command争论fcntl ，以指定应设置或清除锁定。就像F_OFD_SETLK命令，但使进程等待，直到请求可以完成。

此命令需要类型的第三个参数struct flock * ，至于F_OFD_SETLK命令。

的fcntl返回值和错误与F_OFD_SETLK命令，但这些额外errno为该命令定义了错误条件：

EINTR

该函数在等待时被信号中断。请参阅中断的基元 。

在与进程相关的锁相同的情况下，打开文件描述锁也很有用。通过让每个线程执行各自的操作，它们还可以用于在同一进程中的线程之间同步文件访问。 open文件，以获取其自己的打开文件描述。

由于打开的文件描述锁定仅在关闭引用打开文件描述的最后一个文件描述符时才自动释放，因此这种锁定机制避免了由于库例程打开和关闭文件而导致应用程序不知情而无意中释放了锁定的可能性。

与进程相关的锁一样，建议使用打开文件描述锁。
