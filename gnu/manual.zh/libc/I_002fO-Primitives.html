13.2输入和输出原语

本节描述了对文件描述符执行原始输入和输出操作的功能： read ， write和lseek 。这些函数在头文件中声明unistd.h 。

数据类型： ssize_t

此数据类型用于表示可以在单个操作中读取或写入的块的大小。它类似于size_t ，但必须是带符号的类型。

函数： ssize_t 读取 （int filedes ，无效* buffer ，size_t size ）

初步： MT安全| AS安全交流安全请参阅POSIX安全概念 。

的read函数最多读取size带描述符的文件中的字节filedes ，将结果存储在buffer 。（这不一定是字符串，并且不添加任何终止空字符。）

返回值是实际读取的字节数。这可能少于size ;例如，如果文件中没有多少字节，或者立即可用的字节不多。确切的行为取决于文件的类型。注意阅读少于size字节不是错误。

零值表示文件结束（除非size参数也为零）。这不被视为错误。如果你继续打电话read在文件结束时，它将继续返回零，并且什么也不做。

如果read返回至少一个字符，则无法判断是否已到达文件末尾。但是，如果您确实到达末尾，则下一次读取将返回零。

如有错误， read返回-1 。下列errno为此功能定义了错误条件：

EAGAIN

通常，当没有输入立即可用时， read等待一些输入。但是如果O_NONBLOCK为文件设置了标志（请参阅文件状态标志 ）， read立即返回而不读取任何数据，并报告此错误。

兼容性说明：大多数BSD Unix版本为此使用不同的错误代码： EWOULDBLOCK 。在GNU C库中， EWOULDBLOCK是的别名EAGAIN ，因此使用哪个名称都没有关系。

在某些系统上，从字符特殊文件读取大量数据也会因EAGAIN如果内核找不到足够的物理内存来锁定用户页面。这仅限于通过直接内存访问转移到用户内存的设备，这意味着它不包括终端，因为它们始终在内核内部使用单独的缓冲区。在GNU / Hurd系统上永远不会发生此问题。

任何可能导致EAGAIN相反可以导致成功read返回的字节数少于请求的字节数。呼唤read再次立即将导致EAGAIN 。

EBADF

的filedes参数不是有效的文件描述符，或者未打开以供读取。

EINTR

read等待输入时被信号中断。请参阅中断的基元 。信号不一定会导致read返回EINTR ;它可能会导致成功read返回的字节数少于请求的字节数。

EIO

对于许多设备和磁盘文件，此错误代码表示硬件错误。

EIO当后台进程试图从控制终端读取数据时，也会发生这种情况，并且通过向进程发送SIGTTIN信号不起作用。如果信号被阻止或忽略，或者因为进程组被孤立，则可能会发生这种情况。有关作业控制的更多信息，请参见作业控制 ；有关信号的信息，请参见信号处理 。

EINVAL

在某些系统中，从字符或块设备读取数据时，位置和大小偏移必须与特定的块大小对齐。此错误表明偏移量未正确对齐。

请注意，没有名为read64 。这是不必要的，因为此功能不会直接修改或处理可能很大的文件偏移量。由于内核在内部处理此状态，因此read功能可用于所有情况。

此功能是多线程程序中的取消点。如果线程当时分配了一些资源（例如内存，文件描述符，信号量或其他内容），这将是一个问题read叫做。如果线程被取消，这些资源将保持分配状态，直到程序结束。为避免这种情况，请致电read应该使用取消处理程序进行保护。

的read函数是从流中读取的所有函数的基础原语，例如fgetc 。

功能： ssize_t pread （int filedes ，无效* buffer ，size_t size ，off_t offset ）

初步： MT安全| AS安全交流安全请参阅POSIX安全概念 。

的pread功能类似于read功能。前三个参数相同，并且返回值和错误代码也对应。

区别在于第四个参数及其处理方式。没有从文件描述符的当前位置读取数据块filedes 。而是从位置开始的文件中读取数据offset 。文件描述符本身的位置不受操作影响。该值与调用之前的值相同。

使用以下命令编译源文件时_FILE_OFFSET_BITS == 64的pread功能其实pread64和类型off_t具有64位，这使得可以处理最大2 ^ 63字节的文件。

的返回值pread描述读取的字节数。在错误的情况下，它返回-1，例如read并添加了以下错误代码，它们也相同：

EINVAL

给定的值offset是负面的，因此是非法的。

ESPIPE

文件描述符filedes与管道或FIFO关联，并且该设备不允许定位文件指针。

该函数是Unix Single Specification版本2中定义的扩展。

功能： ssize_t pread64 （int filedes ，无效* buffer ，size_t size ，off64_t offset ）

初步： MT安全| AS安全交流安全请参阅POSIX安全概念 。

此功能类似于pread功能。区别在于offset参数的类型off64_t代替off_t这样就可以在32位计算机上寻址大于2 ^ 31字节且最大为2 ^ 63字节的文件。文件描述符filedes必须使用open64因为否则可能会产生较大的偏移量off64_t在小文件模式下将导致描述符错误。

使用以下命令编译源文件时_FILE_OFFSET_BITS == 64在32位计算机上，该功能实际上可以在名称下使用pread因此透明地替换了32位接口。

函数： ssize_t 写入 （int filedes ，const void * buffer ，size_t size ）

初步： MT安全| AS安全交流安全请参阅POSIX安全概念 。

的write函数写到size来自的字节buffer到带有描述符的文件filedes 。数据在buffer不一定是字符串，并且像其他任何字符一样输出空字符。

返回值是实际写入的字节数。这可能是size ，但始终可以较小。您的程序应始终调用write循环执行，直到所有数据都写入为止。

一旦write返回时，数据就排队待写，并且可以立即读回，但是不必立即写出到永久存储中。您可以使用fsync当您需要确保数据已永久存储后再继续。（对于系统而言，分批连续写入并在方便时一次完成所有写入操作效率更高。通常，它们总是会在一分钟或更短的时间内写入磁盘。）现代系统提供了另一个功能fdatasync这仅保证文件数据的完整性，因此速度更快。您可以使用O_FSYNC开放模式write返回之前始终将数据存储到磁盘；请参阅操作模式 。

如果有错误， write返回-1 。下列errno为此功能定义了错误条件：

EAGAIN

一般， write直到写操作完成为止。但是如果O_NONBLOCK为该文件设置了标志（请参见Control Operations ），它立即返回而不写入任何数据，并报告此错误。一种可能导致进程在输出时阻塞的情况的示例是，写入支持流控制的终端设备，在该设备上，由于接收到STOP字符而挂起了输出。

兼容性说明：大多数BSD Unix版本为此使用不同的错误代码： EWOULDBLOCK 。在GNU C库中， EWOULDBLOCK是的别名EAGAIN ，因此使用哪个名称都没有关系。

在某些系统上，从特殊字符文件写入大量数据也会因EAGAIN如果内核找不到足够的物理内存来锁定用户页面。这仅限于通过直接内存访问转移到用户内存的设备，这意味着它不包括终端，因为它们始终在内核内部使用单独的缓冲区。在GNU / Hurd系统上不会出现此问题。

EBADF

的filedes参数不是有效的文件描述符，或未打开以供写入。

EFBIG

文件的大小将变得大于实现可以支持的大小。

EINTR

的write在等待完成时被阻塞的信号中断了操作。信号不一定会导致write返回EINTR ;它可能会导致成功write它写入的字节数少于请求的字节数。请参阅中断的基元 。

EIO

对于许多设备和磁盘文件，此错误代码表示硬件错误。

ENOSPC

包含文件的设备已满。

EPIPE

当您尝试写入未打开供任何进程读取的管道或FIFO时，将返回此错误。发生这种情况时， SIGPIPE信号也发送到过程；请参阅信号处理 。

EINVAL

在某些系统中，当写入字符或块设备时，位置和大小偏移必须与特定的块大小对齐。此错误表明偏移量未正确对齐。

除非您已做好预防EINTR失败，你应该检查errno每次呼叫失败后write ，如果错误是EINTR ，您只需重复通话即可。请参阅中断的基元 。最简单的方法是使用宏TEMP_FAILURE_RETRY ， 如下：

nbytes = TEMP_FAILURE_RETRY (write (desc, buffer, count));

请注意，没有名为write64 。这是不必要的，因为此功能不会直接修改或处理可能很大的文件偏移量。由于内核内部处理此状态，因此write功能可用于所有情况。

此功能是多线程程序中的取消点。如果线程当时分配了一些资源（例如内存，文件描述符，信号量或其他内容），这将是一个问题write叫做。如果线程被取消，这些资源将保持分配状态，直到程序结束。为避免这种情况，请致电write应该使用取消处理程序进行保护。

的write function是写入流的所有函数的基础原语，例如fputc 。

函数： ssize_t pwrite （int filedes ，const void * buffer ，size_t size ，off_t offset ）

初步： MT安全| AS安全交流安全请参阅POSIX安全概念 。

的pwrite功能类似于write功能。前三个参数相同，并且返回值和错误代码也对应。

但是，在Linux上，如果使用O_APPEND ， pwrite不管数据的值如何，都将数据追加到文件末尾offset 。
