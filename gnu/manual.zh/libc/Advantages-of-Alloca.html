3.2.7.2的优点alloca

这是为什么的原因alloca可能比malloc ：

  • 使用alloca浪费很少的空间并且非常快。（它由GNU C编译器开放编码。）
  • 以来alloca对于不同大小的块，没有单独的池，可以将任何大小的块使用的空间重用于其他大小的块。 alloca不会导致内存碎片。
  • 非本地退出完成longjmp （请参阅非本地出口 ）自动释放分配给alloca当他们通过调用的函数退出时alloca 。这是最重要的使用理由alloca 。

    为了说明这一点，假设您有一个函数open_or_report_error它返回一个描述符，例如open ，如果成功，但是如果失败则不返回调用者。如果无法打开该文件，它将显示一条错误消息，并使用以下命令跳到程序的命令级别longjmp 。让我们改变open2 （请参阅Alloca示例 ）以使用此子例程：

    int
open2 (char *str1, char *str2, int flags, int mode)
{
  char *name = (char *) alloca (strlen (str1) + strlen (str2) + 1);
  stpcpy (stpcpy (name, str1), str2);
  return open_or_report_error (name, flags, mode);
}

    由于方式alloca可以正常工作，即使发生错误也可以释放它分配的内存，而无需特别的努力。

    相比之下，以前的定义open2 （使用malloc和free ）如果以这种方式进行了更改，则会产生内存泄漏。即使您愿意进行更多更改来修复它，也没有简单的方法可以解决。
