13.1打开和关闭文件

本节描述了使用文件描述符打开和关闭文件的原语。的open和creat函数在头文件中声明fcntl.h ，而close在中声明unistd.h 。

函数： int open （const char * filename ，int flags [，mode_t mode ]）

初步： MT安全| AS安全交流安全fd |请参阅POSIX安全概念 。

的open函数为名称为的文件创建并返回一个新的文件描述符filename 。最初，文件的文件位置指示符位于文件的开头。争论mode （请参阅Permission Bits ）仅在创建文件时使用，但在任何情况下提供参数都不会造成损害。

的flags参数控制如何打开文件。这是一个位掩码；您可以通过相应参数的按位或来创建值（使用“ | C中的'运算符）。有关可用参数，请参见文件状态标志 。

的正常返回值open是一个非负整数文件描述符。如果发生错误，则返回值-1 。除了常见的文件名错误（请参阅“ 文件名错误” ）之外，以下内容errno为此功能定义了错误条件：

EACCES

该文件存在，但无法按照文件的要求进行读写。 flags参数，或者文件不存在并且目录不可写，因此无法创建。

EEXIST

都O_CREAT和O_EXCL被设置，并且命名文件已经存在。

EINTR

的open操作被信号中断。请参阅中断的基元 。

EISDIR

的flags参数指定写访问权限，并且文件是目录。

EMFILE

该进程打开了太多文件。文件描述符的最大数量由RLIMIT_NOFILE资源限制；请参阅资源限制 。

ENFILE

整个系统或包含目录的文件系统目前无法支持任何其他打开的文件。（在GNU / Hurd系统上不会发生此问题。）

ENOENT

命名文件不存在，并且O_CREAT未指定。

ENOSPC

包含新文件的目录或文件系统无法扩展，因为没有剩余的磁盘空间。

ENXIO

O_NONBLOCK和O_WRONLY都设置在flags参数，文件名为filename是FIFO（请参见Pipes和FIFO ），并且没有进程打开该文件以供读取。

EROFS

该文件驻留在只读文件系统上，并且任何O_WRONLY ， O_RDWR和O_TRUNC设置在flags论点，或O_CREAT设置并且文件不存在。

如果在32位计算机上，则使用_FILE_OFFSET_BITS == 64功能open返回以大文件模式打开的文件描述符，该文件描述符使文件处理功能可以使用最大2 ^ 63字节的文件，并且偏移量从-2 ^ 63到2 ^ 63。由于所有低级文件处理功能都被同等替换，因此这对于用户而言是透明的。

此功能是多线程程序中的取消点。如果线程当时分配了一些资源（例如内存，文件描述符，信号量或其他内容），这将是一个问题open叫做。如果线程被取消，这些资源将保持分配状态，直到程序结束。为了避免这种情况open应该使用取消处理程序进行保护。

的open函数是fopen和freopen创建流的函数。

功能： int open64 （const char * filename ，int flags [，mode_t mode ]）

初步： MT安全| AS安全交流安全fd |请参阅POSIX安全概念 。

该功能类似于open 。它返回一个文件描述符，该描述符可用于访问由命名的文件filename 。唯一的区别是在32位系统上，该文件以大文件模式打开。即，文件长度和文件偏移量可以超过31位。

当源翻译成_FILE_OFFSET_BITS == 64该功能实际上在名称下可用open 。也就是说，使用64位文件大小和偏移量的新扩展API透明地替换了旧API。

过时的函数： int creat （const char * filename ，mode_t mode ）

初步： MT安全| AS安全交流安全fd |请参阅POSIX安全概念 。

此功能已过时。电话：

creat (filename, mode)

等效于：

open (filename, O_WRONLY | O_CREAT | O_TRUNC, mode)

如果在32位计算机上，则使用_FILE_OFFSET_BITS == 64功能creat返回以大文件模式打开的文件描述符，该文件描述符使文件处理功能可以使用最大2 ^ 63的文件以及从-2 ^ 63到2 ^ 63的偏移量。由于所有低级文件处理功能都被同等替换，因此这对于用户而言是透明的。

过时的函数： int creat64 （const char * filename ，mode_t mode ）

初步： MT安全| AS安全交流安全fd |请参阅POSIX安全概念 。

该功能类似于creat 。它返回一个文件描述符，该描述符可用于访问由命名的文件filename 。唯一的区别是在32位系统上，该文件以大文件模式打开。即，文件长度和文件偏移量可以超过31位。

要使用此文件描述符，一定不能使用常规操作，而要使用对应的名称*64 ，例如read64 。

当源翻译成_FILE_OFFSET_BITS == 64该功能实际上在名称下可用open 。也就是说，使用64位文件大小和偏移量的新扩展API透明地替换了旧API。

函数： int close （int filedes ）

初步： MT安全| AS安全交流安全fd |请参阅POSIX安全概念 。

功能close关闭文件描述符filedes 。关闭文件具有以下后果：

  • 文件描述符被释放。
  • 该文件上的进程所拥有的所有记录锁均被解锁。
  • 关闭与管道或FIFO相关联的所有文件描述符后，所有未读数据都将被丢弃。

此功能是多线程程序中的取消点。如果线程当时分配了一些资源（例如内存，文件描述符，信号量或其他内容），这将是一个问题close叫做。如果线程被取消，这些资源将保持分配状态，直到程序结束。为避免这种情况，请致电close应该使用取消处理程序进行保护。

的正常返回值close是0 ;如果失败，则返回-1 。下列errno为此功能定义了错误条件：

EBADF

的filedes参数不是有效的文件描述符。

EINTR

的close呼叫被信号中断。请参阅中断的基元 。这是一个如何处理的例子EINTR正确地：

TEMP_FAILURE_RETRY (close (desc));
ENOSPC
EIO
EDQUOT

NFS访问文件时，这些错误来自write有时直到close 。有关其含义的详细信息，请参见I / O基元 。

请注意， 没有单独的close64功能。这不是必需的，因为此功能既不确定也不依赖于文件的模式。执行close操作知道描述符使用的模式，并且可以处理这种情况。

要关闭流，请致电fclose （请参见关闭流 ），而不是尝试使用以下命令关闭其基础文件描述符close 。这将刷新所有缓冲的输出并更新流对象以指示它已关闭。
