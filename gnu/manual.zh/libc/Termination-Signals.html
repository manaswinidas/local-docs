24.2.2终止信号

这些信号全部用于告诉进程以一种或另一种方式终止。它们的名称不同，因为它们的用途略有不同，并且程序可能希望以不同的方式处理它们。

处理这些信号的原因通常是为了使您的程序可以在实际终止之前进行适当的整理。例如，您可能想保存状态信息，删除临时文件或恢复以前的终端模式。这样的处理程序应该通过为发生的信号指定默认操作然后重新引发来结束。这将导致程序以该信号终止，就好像它没有处理程序一样。（请参阅处理程序中的终止 。）

所有这些信号的（明显）默认操作是导致进程终止。

巨集： int SIGTERM

的SIGTERM signal是用于导致程序终止的通用信号。不像SIGKILL ，此信号可以被阻止，处理和忽略。这是礼貌地要求程序终止的正常方法。

shell命令kill产生SIGTERM默认。

巨集： int SIGINT

的SIGINT当用户键入INTR字符时（通常是“程序中断”）信号被发送C-c ）。有关支持终端驱动程序的信息，请参见特殊字符 。 C-c 。

巨集： int SIGQUIT

的SIGQUIT信号类似于SIGINT ，除了它由另一个键-QUIT字符控制外，通常C-\ —并在终止进程时产生核心转储，就像程序错误信号一样。您可以将其视为用户“检测到”的程序错误情况。

有关核心转储的信息，请参见“ 程序错误信号” 。有关终端驱动程序支持的信息，请参见特殊字符 。

某些清理最好在处理中省略SIGQUIT 。例如，如果程序创建了临时文件，则应通过删除临时文件来处理其他终止请求。但这对SIGQUIT不要删除它们，以便用户可以与核心转储一起检查它们。

巨集： int SIGKILL

的SIGKILL信号用于立即终止程序。它不能被处理或忽略，因此总是致命的。也不可能阻塞该信号。

该信号通常仅由显式请求生成。由于无法处理它，因此您应先尝试使用不太激烈的方法（例如， C-c要么SIGTERM 。如果某个进程未响应任何其他终止信号，请发送一个SIGKILL信号几乎总是会导致其消失。

实际上，如果SIGKILL无法终止进程，该进程本身就是您应报告的操作系统错误。

系统将生成SIGKILL在某些程序可能无法继续运行（甚至无法运行信号处理程序）的异常条件下处理进程本身。

巨集： int SIGHUP

的SIGHUP （“挂断”）信号用于报告用户终端已断开连接，可能是因为网络或电话连接断开了。有关此的更多信息，请参见控制模式 。

该信号还用于向与该会话关联的作业报告终端上控制过程的终止；该终止有效地断开了会话中的所有进程与控制终端的连接。有关更多信息，请参见《 内部端接》 。
