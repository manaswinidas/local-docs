前言

验证数据是一项常见任务，遍历从表示层到持久层的所有应用程序层。通常，在每个层中都实现相同的验证逻辑，这既耗时又容易出错。为了避免重复这些验证，开发人员通常将验证逻辑直接捆绑到域模型中，从而使域类与验证代码杂乱无章，这些代码实际上是有关类本身的元数据。

Jakarta Bean Validation 2.0-定义用于实体和方法验证的元数据模型和API。默认的元数据源是注释，能够通过使用XML覆盖和扩展元数据。 API不受特定应用程序层或编程模型的约束。它特别不与Web层或持久层绑定在一起，并且可用于服务器端应用程序编程以及富客户端Swing应用程序开发人员。

Hibernate Validator是Jakarta Bean Validation的参考实现。该实现本身以及Jakarta Bean验证API和TCK均在Apache Software License 2.0下提供和分发。

Hibernate Validator 6和Jakarta Bean Validation 2.0需要Java 8或更高版本。

1。入门

本章将向您展示如何开始使用Hibernate Validator，它是Jakarta Bean Validation的参考实现（RI）。对于以下快速入门，您需要：

  • 一个JDK 8

  • 阿帕奇Maven

  • Internet连接（Maven必须下载所有必需的库）

1.1。项目设立

为了在Maven项目中使用Hibernate Validator，只需将以下依赖项添加到pom.xml中 ：

例1.1：Hibernate Validator Maven依赖项

    org.hibernate.validator
    hibernate-validator
    6.1.0.Final

这将依赖关系传递给Jakarta Bean Validation API（ jakarta.validation:jakarta.validation-api:2.0.2 ）。

1.1.1。统一EL

Hibernate Validator需要Jakarta表达式语言的实现，用于评估约束违反消息中的动态表达式（请参见第4.1节“默认消息插值” ）。当您的应用程序在Java EE容器（例如JBoss AS）中运行时，该容器已经提供了EL实现。但是，在Java SE环境中，必须将实现添加为对POM文件的依赖。例如，您可以添加以下依赖项以使用Jakarta EL 参考实现 ：

示例1.2：Unified EL参考实现的Maven依赖关系

    org.glassfish
    jakarta.el
    3.0.3

对于无法提供EL实施的环境，Hibernate Validator提供了第12.9节“ ParameterMessageInterpolator ” 。但是，此插值器的使用不符合Jakarta Bean验证规范。

1.1.2。CDI

Jakarta Bean验证使用CDI（ Jakarta EE的上下文和依赖注入 ）定义集成点。如果您的应用程序在未提供集成的环境中运行，则可以通过将以下Maven依赖项添加到POM中来使用Hibernate Validator CDI可移植扩展：

例1.3：Hibernate Validator CDI可移植扩展Maven依赖项

    org.hibernate.validator
    hibernate-validator-cdi
    6.1.0.Final

请注意，对于在Java EE应用程序服务器上运行的应用程序，通常不需要添加此依赖关系。您可以在第11.3节“ CDI”中了解有关Jakarta Bean验证和CDI集成的更多信息。

1.1.3。与安全经理一起运行

Hibernate Validator支持在启用安全管理器的情况下运行。为此，您必须为Hibernate Validator的代码库，Jakarta Bean验证API，Classmate和JBoss Logging以及为调用Jakarta Bean验证的代码库分配多个权限。下面显示了如何通过Java默认策略实现处理的策略文件来执行此操作：

    permission java.util.PropertyPermission "mapAnyUriToUri", "read";

grant codeBase "file:path/to/jboss-logging-3.3.2.Final.jar" {

1.1.5。在Java 9上运行

从Hibernate Validator 6.1.0开始。最后，对Java 9和Java平台模块系统（JPMS）的支持尚处于试验阶段。尚未提供JPMS模块描述符，但是Hibernate Validator可用作自动模块。

这些是使用声明的模块名称Automatic-Module-Name标头：

  • Jakarta Bean验证API：java.validation

  • Hibernate Validator核心：org.hibernate.validator

  • Hibernate Validator CDI扩展：org.hibernate.validator.cdi

  • Hibernate Validator测试实用程序：org.hibernate.validator.testutils

  • Hibernate Validator批注处理器：org.hibernate.validator.annotationprocessor

这些模块名称是初步的，在将来的版本中提供实际的模块描述符时可能会更改。

将Hibernate Validator与CDI结合使用时，请注意不要启用java.xml.ws.annotation JDK的模块。该模块包含Jakarta批注的子集，但有些批注例如javax.annotation.Priority缺失。这导致未注册Hibernate Validator的方法验证拦截器，即方法验证将不起作用。

而是将完整的Jakarta Annotations API添加到未命名的模块（即类路径）中，例如通过拉入jakarta.annotation：jakarta.annotation-api依赖项（当依赖于组织时，已经存在对Jakarta Annotations API的传递性依赖项。 hibernate.validator：hibernate-validator-cdi ）。

如果您需要启用java.xml.ws.annotation出于某种原因，您应该通过附加完整的API内容对模块进行修补--patch-module java.xml.ws.annotation=/path/to/complete-jakarta.annotation-api.jar到您的Java调用。

1.2。应用约束

让我们直接看一个例子，看看如何应用约束。

例1.9：用约束注释的Class Car
package org.hibernate.validator.referenceguide.chapter01;

import javax.validation.constraints.Min;
import javax.validation.constraints.NotNull;
import javax.validation.constraints.Size;

public class Car {

    @NotNull
    private String manufacturer;

    @NotNull
    @Size(min = 2, max = 14)
    private String licensePlate;

    @Min(2)
    private int seatCount;

    public Car(String manufacturer, String licencePlate, int seatCount) {
        this.manufacturer = manufacturer;
        this.licensePlate = licencePlate;
        this.seatCount = seatCount;
    }

    //getters and setters ...
}

的@NotNull ， @Size和@Min批注用于声明应应用于Car实例的字段的约束：

  • manufacturer绝对不能null

  • licensePlate绝对不能null并且必须介于2到14个字符之间

  • seatCount必须至少为2

您可以在GitHub上的Hibernate Validator 源存储库中找到本参考指南中使用的所有示例的完整源代码。

1.3。验证约束

要验证这些约束，请使用Validator实例。让我们看一下Car ：

例1.10：类CarTest显示验证示例
package org.hibernate.validator.referenceguide.chapter01;

import java.util.Set;
import javax.validation.ConstraintViolation;
import javax.validation.Validation;
import javax.validation.Validator;
import javax.validation.ValidatorFactory;

import org.junit.BeforeClass;
import org.junit.Test;

import static org.junit.Assert.assertEquals;

public class CarTest {

    private static Validator validator;

    @BeforeClass
    public static void setUpValidator() {
        ValidatorFactory factory = Validation.buildDefaultValidatorFactory();
        validator = factory.getValidator();
    }

    @Test
    public void manufacturerIsNull() {
        Car car = new Car( null, "DD-AB-123", 4 );

        Set> constraintViolations =
                validator.validate( car );

        assertEquals( 1, constraintViolations.size() );
        assertEquals( "must not be null", constraintViolations.iterator().next().getMessage() );
    }

    @Test
    public void licensePlateTooShort() {
        Car car = new Car( "Morris", "D", 4 );

        Set> constraintViolations =
                validator.validate( car );

        assertEquals( 1, constraintViolations.size() );
        assertEquals(
                "size must be between 2 and 14",
                constraintViolations.iterator().next().getMessage()
        );
    }

    @Test
    public void seatCountTooLow() {
        Car car = new Car( "Morris", "DD-AB-123", 1 );

        Set> constraintViolations =
                validator.validate( car );

        assertEquals( 1, constraintViolations.size() );
        assertEquals(
                "must be greater than or equal to 2",
                constraintViolations.iterator().next().getMessage()
        );
    }

    @Test
    public void carIsValid() {
        Car car = new Car( "Morris", "DD-AB-123", 2 );

        Set> constraintViolations =
                validator.validate( car );

        assertEquals( 0, constraintViolations.size() );
    }
}

在里面setUp()方法一Validator对象是从ValidatorFactory 。一种Validator实例是线程安全的，并且可以多次重用。因此，它可以安全地存储在静态字段中，并用于测试方法中以验证不同Car实例。

的validate()方法返回一组ConstraintViolation实例，您可以对其进行迭代以查看发生了哪些验证错误。前三种测试方法显示了一些预期的约束违规：

  • 的@NotNull约束manufacturer被侵犯manufacturerIsNull()

  • 的@Size约束licensePlate被侵犯licensePlateTooShort()

  • 的@Min约束seatCount被侵犯seatCountTooLow()

如果对象验证成功， validate()返回一个空集，如您所见carIsValid() 。

请注意，仅包中的类javax.validation被使用。这些是从Bean验证API提供的。没有直接引用Hibernate Validator中的类，从而导致了可移植代码。

2.1。声明bean约束

Jakarta Bean验证中的约束通过Java注释表示。在本节中，您将学习如何使用这些注释来增强对象模型。bean约束有四种类型：

  • 场约束

  • 属性约束

  • 容器元素约束

  • 类约束

并非所有约束都可以置于所有这些级别上。实际上，Jakarta Bean Validation定义的默认约束都不能放在类级别。的java.lang.annotation.Target约束本身中的注释注释本身确定可以在哪些元素上放置约束。有关更多信息，请参见第6章， 创建自定义约束 。

2.1.1。现场级别的约束

可以通过注释类的字段来表达约束。例2.1“字段级约束”显示了一个字段级配置示例：

示例2.1：字段级约束
package org.hibernate.validator.referenceguide.chapter02.fieldlevel;

public class Car {

    @NotNull
    private String manufacturer;

    @AssertTrue
    private boolean isRegistered;

    public Car(String manufacturer, boolean isRegistered) {
        this.manufacturer = manufacturer;
        this.isRegistered = isRegistered;
    }

    //getters and setters...
}

使用字段级约束时，将使用字段访问策略来访问要验证的值。这意味着验证引擎将直接访问实例变量，并且即使存在这样的访问器，也不会调用属性访问器方法。

约束可以应用于任何访问类型（公共，私有等）的字段。但是，不支持对静态字段的约束。

验证字节码增强的对象时，应使用属性级别约束，因为字节码增强库将无法通过反射确定字段访问。

2.1.2。属性级别约束

如果您的模型类遵循JavaBeans标准，则还可以注释bean类的属性而不是其字段。 示例2.2“属性级别约束”使用与示例2.1“字段级别约束”相同的实体，但是，使用了属性级别约束。

示例2.2：属性级别的约束
package org.hibernate.validator.referenceguide.chapter02.propertylevel;

public class Car {

    private String manufacturer;

    private boolean isRegistered;

    public Car(String manufacturer, boolean isRegistered) {
        this.manufacturer = manufacturer;
        this.isRegistered = isRegistered;
    }

    @NotNull
    public String getManufacturer() {
        return manufacturer;
    }

    public void setManufacturer(String manufacturer) {
        this.manufacturer = manufacturer;
    }

    @AssertTrue
    public boolean isRegistered() {
        return isRegistered;
    }

    public void setRegistered(boolean isRegistered) {
        this.isRegistered = isRegistered;
    }
}

必须注释该属性的getter方法，而不是其setter。这样，还可以限制没有设置方法的只读属性。

使用属性级别约束时，将使用属性访问策略来访问要验证的值，即验证引擎通过属性访问器方法访问状态。

建议在一个类中坚持字段或属性注释。不建议对字段和随附的getter方法进行注释，因为这将导致对该字段进行两次验证。

2.1.6。对象图

Jakarta Bean验证API不仅允许验证单个类实例，而且还允许完整的对象图（级联验证）。为此，只需使用以下命令注释表示对另一个对象的引用的字段或属性@Valid如示例2.11“级联验证”所示 。

package org.hibernate.validator.referenceguide.chapter02.objectgraph;

    @NotNull
    @Valid
    private Person driver;

    //...
}
package org.hibernate.validator.referenceguide.chapter02.objectgraph;

    @NotNull
    private String name;

    //...
}

对象图的验证是递归的，即，如果标记为级联验证的引用指向本身具有用注释的属性的对象@Valid ，验证引擎也会跟进这些参考。验证引擎将确保在级联验证期间不会发生无限循环，例如，如果两个对象相互引用。
