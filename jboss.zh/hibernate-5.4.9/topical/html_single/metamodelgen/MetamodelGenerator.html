什么事啊

JPA 2定义了类型安全的Criteria API，该API允许Criteria利用所谓的静态元模型类以强类型方式构造查询。对于开发人员而言，元模型生成的任务可以自动化是很重要的。Hibernate静态元模型生成器是基于JSR_269的注释处理器，其任务是创建JPA 2静态元模型类。以下示例显示了两个JPA 2实体Order和Item ，以及元模型类Order_和类型安全查询。

范例1。JPA 2带注释的实体Order和Item
@Entity
public class Order {
    @Id
    @GeneratedValue
    Integer id;

    @ManyToOne
    Customer customer;

    @OneToMany
    Set<Item> items;
    BigDecimal totalCost;

    // standard setter/getter methods
    ...
}

@Entity
public class Item {
    @Id
    @GeneratedValue
    Integer id;

    int quantity;

    @ManyToOne
    Order order;

    // standard setter/getter methods
    ...
}
示例2元模型类Order_
@StaticMetamodel(Order.class)
public class Order_ {
    public static volatile SingularAttribute<Order, Integer> id;
    public static volatile SingularAttribute<Order, Customer> customer;
    public static volatile SetAttribute<Order, Item> items;
    public static volatile SingularAttribute<Order, BigDecimal> totalCost;
}
范例3。Typesafe citeria查询
CriteriaBuilder cb = entityManager.getCriteriaBuilder();

CriteriaQuery<Order> cq = cb.createQuery(Order.class);

SetJoin<Order, Item> itemNode = cq.from(Order.class).join(Order_.items);

cq.where( cb.equal(itemNode.get(Item_.id), 5 ) ).distinct(true);

Hibernate Static Metamodel Generator还考虑了在中指定的xml配置orm.xml或在中指定的映射文件persistence.xml 。但是，如果XML是唯一的配置源，则需要至少在映射文件中添加以下持久性单元元数据：

<persistence-unit-metadata>
  <xml-mapping-metadata-complete/>
</persistence-unit-metadata>

规范元模型

JPA 2（JSR 317） 规范中描述了元模型类的结构，但为完整起见，在以下段落中重复了该定义。如果您不对血腥细节感兴趣，请随时跳至用法一章 。

注释处理器根据以下规则为持久性单元中的每个受管实体生成一个元模型类：

  • 对于包p中的每个托管类X，将创建包p中的元模型类X_。

  • 通过将“ _”附加到托管类的名称，可以从托管类的名称派生元模型类的名称。

  • 元模型类X_必须使用注释javax.persistence.StaticMetamodel注解。

  • 如果类X扩展了另一个类S，其中S是X扩展的最派生托管类（即实体或映射超类），则类X_必须扩展类S_，其中S_是为S创建的元模型类。

  • 对于类X声明的每个持久性非集合值属性y（其中y的类型为Y），元模型类必须包含如下声明：

    public static volatile SingularAttribute<X, Y> y;

  • 对于类X声明的每个持久性具有集合值的属性z（其中z的元素类型为Z），元模型类必须包含如下声明：

    • 如果z的收集类型为java.util。然后收集

      public static volatile CollectionAttribute<X, Z> z;

    • 如果z的收集类型为java.util。设置，然后

      public static volatile SetAttribute<X, Z> z;

    • 如果z的收集类型为java.util。列出，然后

      public static volatile ListAttribute<X, Z> z;

    • 如果z的收集类型为java.util。地图，然后

      public static volatile MapAttribute<X, K, Z> z;

      其中K是X类中映射键的类型

必须根据需要包含导入语句javax.persistence.metamodel类型以及所有X，Y，Z和K类。

用法

注释处理器的jar文件可以在Maven Central存储库中的以下位置找到：

<dependency>
    <groupId>org.hibernate</groupId>
    <artifactId>hibernate-jpamodelgen</artifactId>
    <version>5.4.9.Final</version>
</dependency>

或者，可以在SourceForge的ORM分发包中找到它。

在大多数情况下，只要将处理器jar添加到构建类路径中，注释处理器就会自动运行。发生这种情况是由于Java的服务提供商合同以及Hibernate Static Metamodel Generator jar文件包含文件javax.annotation.processing的事实。META-INF / services目录中的处理器 。

处理器本身的标准名称是： org.hibernate.jpamodelgen.JPAMetaModelEntityProcessor 。

从命令行使用

与Ant结合使用

如上所述，只要jar文件位于类路径中，注释处理器将在每次调用Java编译器时自动运行。但是，有时候，更详细地控制注释处理很有用，例如，如果您只想运行处理器而不编译任何其他源文件。下面的Javac任务配置显示了如何将Ant配置为仅运行注释处理。

示例4Javac任务配置
<javac srcdir="${src.dir}"
    destdir="${target.dir}"
    failonerror="false"
    fork="true"
    classpath="${classpath}">
    <compilerarg value="-proc:only"/>
</javac>

选项-proc：only指示编译器仅运行注释处理。您还可以通过指定-proc：none完全禁用处理。

跑'javac -help'查看可以指定哪些其他注释处理器相关选项。

与Maven结合使用

作为Maven构建的一部分，有几种方法可以运行注释处理器。同样，如果您使用的JDK> 6进行编译，它将自动运行。如果您的类路径上有多个注释处理器，则可以将处理器选项显式传递给编译器插件：

示例5Maven编译器插件配置-直接执行
<plugin>
    <artifactId>maven-compiler-plugin</artifactId>
    <configuration>
        <source>1.8</source>
        <target>1.8</target>
        <compilerArguments>
            <processor>org.hibernate.jpamodelgen.JPAMetaModelEntityProcessor</processor>
        </compilerArguments>
    </configuration>
</plugin>

maven-compiler-plugin方法具有以下缺点：maven编译器插件当前不允许指定多个编译器参数（ MCOMPILER-62 ），并且来自Messenger API的消息被禁止（ MCOMPILER-66 ）。更好的方法是禁用编译器插件的注释处理，如下所示。

范例6。Maven编译器插件配置-间接执行
<plugin>
    <artifactId>maven-compiler-plugin</artifactId>
    <configuration>
        <source>1.8</source>
        <target>1.8</target>
        <compilerArgument>-proc:none</compilerArgument>
    </configuration>
</plugin>

禁用后，可以使用用于注释处理的maven-processor-plugin ：

范例7。使用Maven处理器插件进行配置
<plugin>
    <groupId>org.bsc.maven</groupId>
    <artifactId>maven-processor-plugin</artifactId>
    <version>2.0.5</version>
    <executions>
        <execution>
            <id>process</id>
            <goals>
                <goal>process</goal>
            </goals>
            <phase>generate-sources</phase>
            <configuration>
                <processors>
                    <processor>org.hibernate.jpamodelgen.JPAMetaModelEntityProcessor</processor>
                </processors>
            </configuration>
        </execution>
    </executions>
    <dependencies>
        <dependency>
            <groupId>org.hibernate</groupId>
            <artifactId>hibernate-jpamodelgen</artifactId>
            <version>5.4.9.Final</version>
        </dependency>
    </dependencies>
</plugin>

另一种可能性是将依赖项作为注释处理器路径提供给maven-compiler-plugin：

范例8。使用maven-compiler-plugin进行配置
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-compiler-plugin</artifactId>
    <version>3.7.0</version>
    <configuration>
        <annotationProcessorPaths>
            <path>
                <groupId>org.hibernate</groupId>
                <artifactId>hibernate-jpamodelgen</artifactId>
                <version>5.4.9.Final</version>
            </path>
        </annotationProcessorPaths>
    </configuration>
</plugin>

在IDE中的用法

当然，您还希望在您喜欢的IDE中提供注释处理。以下段落和屏幕截图显示了如何在IDE中启用Hibernate Static Metamodel Generator。

理念

Intellij Idea从9.x版本开始包含一个特定的配置部分，用于在项目设置窗口下进行注释处理。屏幕快照显示了如何配置Hibernate Static Metamodel Generator。

在注释处理器配置中，启用注释处理并选择从项目类路径获取。添加注释处理器名称org.hibernate.jpamodelgen.JPAMetaModelEntityProcessor （以及注释处理器选项）。选择包含您的实体的模块。如果按照建议配置了Maven，则最好为生成的类选择相同的输出目录。在撰写本文时，它是target / generation-sources / apt 。这样，生成的类将在IntelliJ Idea中可用。

日食

在Eclipse中，Java编译器下还有一个附加的配置部分。您可以在那里配置注释处理的所有方面。只需选中“启用注释处理”选项，为生成的源配置目录，最后将Hibernate Static Metamodel Generator和JPA 2 jar文件添加到工厂路径。

如果您使用JDK 11+，则还需要添加javax.xml.bind:jaxb-api和org.glassfish.jaxb:jaxb-runtime JAXB不再包含在JDK中了。

处理器特定选项

Hibernate静态元模型生成器接受一系列自定义选项，这些选项可以采用以下格式传递给处理器：-A[property]=[value]

支持的属性可以在下表中找到：

表格1。Hibernate静态元模型生成器选项
选项名称 期权价值和用途

调试

如果设置为true处理器将输出其他跟踪信息

持久性

默认情况下，处理器在/ META-INF中查找persistence.xml。指定此选项，可以指定其他位置的persitence.xml文件（必须位于类路径上）

ormXml

允许指定其他实体映射文件。此选项的指定值是映射文件名的逗号分隔字符串。即使指定了此选项，/ META-INF / orm.xml也是隐式的。

lazyXmlParsing

可能的值为true要么false 。如果设置为true注释处理器尝试确定两次调用之间是否已更改任何xml文件，如果未更改则跳过xml解析。此功能是实验性的，在混合模式配置的某些情况下，存在错误结果的风险。确定文件是否已被修改为临时文件Hibernate-Static-Metamodel-Generator.tmp用来。该文件是在java.io.tmpdir目录。

fullyAnnotationConfigured

如果设置为true处理器将忽略orm.xml和persistence.xml 。

addGeneratedAnnotation

如果设置为true处理器会将@Generated添加到生成的Java源文件中。使用JDK 5添加此注释将导致编译错误。在这种情况下，将标志设置为false。此选项的默认值为true

addGenerationDate

如果设置为true，则将元模型类的生成日期插入@Generated批注的date参数中。默认是false 。如果addGeneratedAnnotation设置为false，则忽略此参数。

addSuppressWarningsAnnotation

如果设置为true处理器将添加@SuppressWarnings("all")到生成的Java源文件。默认情况下，不会生成此注释。另请参阅METAGEN-50 。
