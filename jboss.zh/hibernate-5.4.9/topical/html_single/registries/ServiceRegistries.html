服务和注册是从4.0开始的正式概念 。但是，不同服务提供的功能实际上在Hibernate中已经存在了很长时间。新功能是通过一个轻巧的专用容器（称为ServiceRegistry）来管理它们，它们的生命周期和依赖性。本指南的目的是描述这些服务和注册中心的设计和目的，并在适当时查看其实现的详细信息。它还将研究第三方集成商和应用程序可以利用和定制服务和注册中心的方式。

什么是服务？

服务以可插入方式提供各种类型的功能。具体来说，它们是定义某些功能的接口，然后是那些服务合同接口的实现。该接口称为服务角色。实现类称为服务实现。可插入性来自以下事实：服务实现遵守由服务角色的接口定义的约定，并且服务程序的使用者依赖于服务角色，而不是实现。

所有服务均应实现org.hibernate.service。服务“标记”界面。Hibernate在内部出于某种基本类型安全性使用它。它没有定义任何方法（目前）。

让我们看一个更好地定义什么是服务的示例。Hibernate需要能够访问与数据库的JDBC连接。它获取和释放这些连接的方式是通过ConnectionProvider服务。该服务由接口（服务角色）org.hibernate.engine.jdbc.connections.spi定义。ConnectionProvider声明用于获取和释放Connections的方法。然后，该服务协定有多种实现方式，它们实际管理连接的方式各不相同：

  • org.hibernate.engine.jdbc.connections.internal。DatasourceConnectionProviderImpl用于使用javax.sql。数据源

  • org.hibernate.c3p0.internal。C3P0ConnectionProvider用于使用C3P0连接池

  • 等等

在内部，Hibernate始终引用org.hibernate.engine.jdbc.connections.spi。使用ConnectionProvider而不是使用服务时的特定实现（稍后我们将讨论注册表时，将开始提供服务）。因此，可以插入其他ConnectionProvider服务实现。

这里没有革命性的东西。对接口进行编程通常被认为是良好的编程习惯。有趣的是ServiceRegistry和不同实现者的可插拔交换。

什么是ServiceRegistry？

最基本的ServiceRegistry托管和管理服务。它的合同由org.hibernate.service定义。ServiceRegistry接口。

我们已经对服务进行了基本概述和定义。但是服务也具有其他有趣的特征。服务具有生命周期。他们有范围。服务可能依赖于其他服务。并且需要生产它们（选择使用一种实现而不是另一种实现）。ServiceRegistry满足所有这些需求。

简而言之，ServiceRegistry充当控制反转（IoC）容器。

尽管有一些近期的修正主义者历史，Spring还是没有发明IoC或依赖项注入，甚至没有第一个将其引入Java的人。诸如JBoss MicroContainer和Apache Avalon之类的项目比Spring早很多年，并且每个项目都进行了IoC和依赖注入。ServiceRegistry中的概念实际上与Apache Avalon非常相似。

为什么不只使用现有的IoC框架？主要原因是它必须重量轻且占用空间尽可能小。最初的设计还要求服务在运行时可交换，但是由于基于代理的交换解决方案中的性能问题，不幸的是必须将其删除（该计划是研究以另一种方式实现具有更好性能的交换能力的方法。以后的日期）。

服务与ServiceRegistry相关联。ServiceRegistry对服务进行范围划分。ServiceRegistry管理服务的生命周期。ServiceRegistry处理将依赖项注入到Service中（实际上既支持拉取，也支持推入/注入方法）。ServiceRegistries也是分层的，这意味着ServiceRegistry可以具有父ServiceRegistry。一个注册表中的服务可以依赖并利用同一注册表以及任何父注册表中的服务。

服务绑定

给定服务与给定ServiceRegistry的关联称为绑定，由org.hibernate.service.spi表示。ServiceBinding接口。此外，org.hibernate.service.spi表示ServiceBinding和ServiceRegistry之间的特定协定。ServiceBinding。ServiceLifecycleOwner接口。

有两种将服务与ServiceRegistry关联的方法。可以直接实例化服务，然后将其传递给ServiceRegistry，或者可以将ServiceInitiator赋予ServiceRegistry（如果需要服务的时间和时间，ServiceRegistry将使用该服务）。ServiceRegistry实现（使用org.hibernate.service.internal的实现。AbstractServiceRegistryImpl便利性基础实现）通过对接受Service实例或ServiceInitiator实例的重载AbstractServiceRegistryImpl＃createServiceBinding方法的调用来注册绑定。但是，每种特定的ServiceRegistry类型都有一个专用的构建器，通常通过该构建器定义和定制其服务。

服务注册表的类型

当前，Hibernate利用3个不同的ServiceRegistry实现来形成一个层次结构。

BootstrapServiceRegistry

根ServiceRegistry是org.hibernate.boot.registry。BootstrapServiceRegistry。BootstrapServiceRegistry是org.hibernate.service的专业化。ServiceRegistry。BootstrapServiceRegistry接口没有添加任何新行为，它只是出于类型安全目的的一种特殊化。在正常使用中，BootstrapServiceRegistry没有父级。

该注册表包含的服务对于Hibernate中的大多数功能来说绝对是可用的。

BootstrapServiceRegistry通常包含3个服务，通常是通过org.hibernate.boot.registry构建的。BootstrapServiceRegistryBuilder类。构建器使类型安全地访问以定制这3个服务。

ClassLoaderService

该服务提供了与ClassLoader进行交互的功能。Hibernate（或任何库）与ClassLoader交互的方式因托管应用程序的运行时环境而异。应用服务器，OSGi容器和其他模块化类加载系统提出了非常特定的类加载要求。该服务为Hibernate提供了这种环境复杂性的抽象。同样重要的是，它以集中式，可交换的方式进行操作。

此服务公开的特定功能包括：

  • 找到java.lang。类按名称引用。这包括应用程序类以及“集成”类。

  • 将资源（属性文件，xml文件等）定位为“类路径资源”

  • 与java.util交互。服务加载器

该服务的服务角色是org.hibernate.boot.registry.classloading.spi。ClassLoaderService。

集成商服务

应用程序，第三方集成商和其他所有人都需要与Hibernate集成。从历史上看，这曾经需要某些东西（通常是应用程序）来协调注册代表每个集成所需的每个集成的各个部分。org.hibernate.integrator.spi。集成商正式确定了该“集成SPI”。IntegratorService管理所有已知的集成商。

作为5.0代码库的一部分，“集成器”的概念仍在积极地定义和发展。期望这些SPI有所变化；实际上，这些更改已经在存储库分支5.0版开发中开始。

积分器有两种已知的方法。

  1. 可以通过调用BootstrapServiceRegistryBuilder＃with（Integrator）手动注册集成商

  2. 利用标准Java java.util，可以发现集成器。ClassLoaderService提供的ServiceLoader功能。集成商只需定义一个名为_ / META-INF / services / org.hibernate.integrator.spi的文件。Integrator_并将其在类路径上可用。java.util。ServiceLoader详细介绍了此文件的格式，但从本质上讲，它按FQN列出了实现org.hibernate.integrator.spi的类。每行一个积分器。

该服务的服务角色是org.hibernate.integrator.spi。IntegratorService。

策略选择器

将此视为“短命名”服务。从历史上看，配置Hibernate用户经常需要给FQN对内部Hibernate类的引用。

例如，要告诉Hibernate使用基于JDBC的事务，我们需要告诉它使用org.hibernate.engine.transaction.internal.jdbc。通过指定其FQN作为config的一部分来指定JdbcTransactionFactory类：

hibernate.transaction.factory_class=org.hibernate.engine.transaction.internal.jdbc.JdbcTransactionFactory

当然，这在我们重构内部代码并将这些类移入不同的包结构时引起了很多问题。输入简短命名的概念，为impl类使用定义明确且众所周知的“短名称”。例如，此JdbcTransactionFactory以短名称“ jdbc”注册，因此：

hibernate.transaction.factory_class=jdbc

在功能上等同于初始示例。第二种形式不仅更简洁，而且是升级证明。

即使是功能非常强大的应用程序和集成商，也可以管理此服务中的简称映射。有关此方面的更多信息，请参见：

  • BootstrapServiceRegistryBuilder＃withStrategySelector

  • BootstrapServiceRegistryBuilder＃withStrategySelectors

  • org.hibernate.boot.registry.selector。StrategyRegistrationProvider（通过ServiceLoader发现）

  • 'StrategySelector＃registerStrategyImplementor` /'StrategySelector＃unRegisterStrategyImplementor`

该服务的服务角色是org.hibernate.boot.registry.selector.spi。StrategySelector。

标准服务注册中心

org.hibernate.boot.registry。StandardServiceRegistry在BootstrapServiceRegistry（BootstrapServiceRegistry是其父项）的基础上定义了主要的Hibernate ServiceRegistry。通常使用org.hibernate.boot.registry构建此注册表。StandardServiceRegistryBuilder类。默认情况下，它包含Hibernate使用的大多数服务。有关StandardServiceRegistry中通常包含的服务的完整列表，请参见org.hibernate.service的源代码。StandardServiceInitiators。需要注意的一些特殊的StandardServiceRegistry服务包括：

ConnectionProvider / MultiTenantConnectionProvider

根据需要为Hibernate提供Connections的服务。有2个不同（且互斥）的角色：

  • org.hibernate.engine.jdbc.connections.spi。ConnectionProvider在正常环境中提供连接。

  • org.hibernate.engine.jdbc.connections.spi。MultiTenantConnectionProvider在多租户环境中提供（特定于租户的）连接。

Jdbc服务

org.hibernate.engine.jdbc.spi。JdbcServices是一个聚合服务（一个聚合其他服务的服务），它公开了有关JDBC可访问性的统一功能。

TransactionFactory

org.hibernate.engine.transaction.spi。TransactionFactory用于告诉Hibernate如何控制事务或与事务集成。

Jta平台

使用基于JTA的TransactionFactory时，请使用org.hibernate.engine.transaction.jta.platform.spi。JtaPlatform Service提供对JTA TransactionManager和UserTransaction的Hibernate访问，以及处理同步注册。

服务生命周期

管理服务的生命周期是ServiceRegistry作为这些服务的容器的重要角色。服务的整体生命周期为：

  1. 引发

  2. （可选） 配置

  3. （可选） 开始

  4. 使用中-直到注册表关闭

  5. （可选） 停止

启动（创建）

服务必须启动/创建。在讨论Building ServiceRegistry时，我们将进一步探讨细节。但总的来说，

  • 可以直接实例化服务并将其传递给ServiceRegistry

  • 一种ServiceInitiator可以交给ServiceRegistry来按需启动服务。

组态

服务可以选择实现org.hibernate.service.spi.Configurable交接的界面java.util.Map初始引导过程中交给Hibernate的配置设置。Configurable#configure在启动之后但使用之前被调用

开始

服务可以有选择地实施org.hibernate.service.spi.Startable在即将进入“使用中”之前接收回调。反身地说，对于需要Startable也需要Stoppable （ 停止 ）。

正在停止

服务可以有选择地实施org.hibernate.service.spi.Stoppable在ServiceRegistry关闭过程中，由于服务已退出“使用中”状态，因此将收到回调。

可管理（JMX）

服务可以有选择地实施org.hibernate.service.spi.Manageable可供JMX使用。

此特定功能仍在设计/开发中

服务依赖

服务有时依赖于其他服务。例如，DataSourceConnectionProvider服务实现通常需要访问JndiService才能执行JNDI查找。这有两个含义。首先，这意味着DataSourceConnectionProvider需要访问JndiService。其次，这意味着在从DataSourceConnectionProvider对其进行使用之前，必须完全“使用” JndiService。

有两种方法可以访问相关服务：

  1. 有服务实施org.hibernate.service.spi.ServiceRegistryAwareService ，这会将ServiceRegistry注入您的服务。然后，您可以查找需要访问的任何服务。您查找的返回服务将完全可以使用。

  2. 使用注入特定服务@org.hibernate.service.spi.InjectService 。

    1. 通常通过附加注释的方法的参数类型来推断要注入的Service角色。如果参数类型不同于“服务”角色，请使用InjectService#serviceRole明确命名角色。

    2. 默认情况下，要注入的服务被认为是必需的（如果找不到，则会抛出异常）。如果要注入的服务是可选的，请使用InjectService#required=false 。

屋宇装备注册处

构建后，通常认为ServiceRegistry是不可变的。服务本身可以接受重新配置，但是此处的不变性意味着添加/替换服务。因此，必须预先知道特定ServiceRegistry中托管的所有服务。为此，构建ServiceRegistry通常会雇用builder 。

构建BootstrapServiceRegistry

建立BootstrapServiceRegistry通常是通过'org.hibernate.boot.registry完成的。BootstrapServiceRegistryBuilder`类，用于公开定义方法ClassLoaders使用，不可发现Integrators合并等

默认情况下，Hibernate将使用Thread-context ClassLoader（TCCL）（如果有的话）以及其类的ClassLoader作为将在被要求加载类或资源或执行ServiceLoader解析时参考的ClassLoader。您可以通过重载告诉Hibernate考虑任何其他ClassLoader。 BootstrapServiceRegistryBuilder#with(ClassLoader)方法：

BootstrapServiceRegistry bootstrapServiceRegistry = new BootstrapServiceRegistryBuilder()
        .with( anAdditionalClassLoader )
        .with( anotherAdditionalClassLoader )
        .build();
您还可以通过以下方式告诉Hibernate使用完全不同的ClassLoaderService实现： BootstrapServiceRegistryBuilder#with(ClassLoaderService) 。

集成商通常是通过JDK发现的ServiceLoader机制。告诉Hibernate关于某个不会被发现的Integrator（无论出于何种原因），您可以使用BootstrapServiceRegistryBuilder#with(Integrator)方法：

BootstrapServiceRegistry bootstrapServiceRegistry = new BootstrapServiceRegistryBuilder()
        .with( new MyCustomIntegrator() )
        .build();

BootstrapServiceRegistryBuilder还公开了添加额外策略选择的方法。假设我们开发了一个自定义的基于CORBA的TransactionFactory，名为CORBATransactionFactory，我们希望通过短命名使它可用。一种选择是在BootstrapServiceRegistry构建期间显式设置简称：

BootstrapServiceRegistry bootstrapServiceRegistry = new BootstrapServiceRegistryBuilder()
        .withStrategySelector( TransactionFactory.class, "corba", CORBATransactionFactory.class )
        .build();

如果我们要分发CORBATransactionFactory，则可以开发一个org.hibernate.boot.registry.selector.StrategyRegistrationProvider ：

public class CORBATransactionFactoryStrategyRegistrationProvider implements StrategyRegistrationProvider {
    public Iterable<StrategyRegistration> getStrategyRegistrations() {
        return Collections.singletonList(
                (StrategyRegistration) new SimpleStrategyRegistrationImpl<TransactionFactory>(
                        TransactionFactory.class,
                        CORBATransactionFactory.class,
                        "corba"
                )
        );
    }
}

我们可以明确注册：

BootstrapServiceRegistry bootstrapServiceRegistry = new BootstrapServiceRegistryBuilder()
        .withStrategySelectors( new CORBATransactionFactoryStrategyRegistrationProvider() )
        .build();

或通过添加META-INF/services/org.hibernate.boot.registry.selector.StrategyRegistrationProvider归档到我们的工件命名CORBATransactionFactoryStrategyRegistrationProvider 。

我们可能会一次合并其中的几个：

BootstrapServiceRegistry bootstrapServiceRegistry = new BootstrapServiceRegistryBuilder()
        .with( anAdditionalClassLoader )
        .with( anotherAdditionalClassLoader )
        .with( new MyCustomIntegrator() )
        .withStrategySelector( ConnectionProvider.class, "custom", MyCustomConnectionProvider.class )
        .withStrategySelectors( new CORBATransactionFactoryStrategyRegistrationProvider() )
        .build();

建筑标准服务注册中心

建立StandardServiceRegistry通常是通过'org.hibernate.boot.registry完成的。StandardServiceRegistryBuilder，公开用于管理设置和控制由内置StandardServiceRegistry托管的服务的方法。

管理设置就像直接告诉构建者一个或多个设置一样简单：

StandardServiceRegistry serviceRegistry = new StandardServiceRegistryBuilder()
        .applySetting( "hibernate.hbm2ddl.auto", true )
        .applySettings( Collections.singletonMap( "hibernate.transaction.factory_class", "jdbc" ) )
        .build();

或者我们可以告诉它从各种文件中加载设置：

StandardServiceRegistry serviceRegistry = new StandardServiceRegistryBuilder()
        .configure()                                          (1)
        .configure( "com/acme/hibernate.cfg.xml" )            (2)
        .loadProperties( "com/acme/hibernate.properties" )    (3)
        .build();
1个 通过hibernate.cfg.xml的ClassLoader资源查找从XML文件（符合Hibernate cfg.xml DTD）加载设置。
2 通过com / acme / hibernate.cfg.xml的ClassLoader资源查找从XML文件（符合Hibernate cfg.xml DTD）加载设置。
3 通过ClassLoader资源查找从com / acme / hibernate.properties的属性中加载设置

感兴趣的其他方法StandardServiceRegistryBuilder与定制要使用的服务有关。正如我们已经讨论过的，我们可以传入要使用的Service实例，也可以传入ServiceInitiator。有两种不同的方式可以定制要使用的服务：

建立StandardServiceRegistry-覆盖

此处的目的是覆盖或替换服务提示。许多标准的ServiceInitiator会浏览设置，以确定要使用的适当服务。回到我们多次使用的示例：

hibernate.transaction.factory_class=jdbc

标准TransactionFactoryInitiator查找此设置并确定什么TransactionFactory服务实现使用。不管出于什么原因，我们总是希望它使用JdbcTransactionFactory：

StandardServiceRegistry serviceRegistry = new StandardServiceRegistryBuilder()
        .addService( TransactionFactory.class, new JdbcTransactionFactory() )
        .build();

或者说我们要解决服务实现以不同方式使用：

StandardServiceRegistry serviceRegistry = new StandardServiceRegistryBuilder()
        .addInitiator( new MyCustomTransactionFactoryInitiator() )
        .build();

建立StandardServiceRegistry-扩展

这里的目的是让ServiceRegistry托管自定义服务（全新的Service角色）。举例来说，假设我们的应用程序将Hibernate事件发布到JMS主题，并且我们想利用Hibernate ServiceRegistry托管一个代表TopicPublisher的服务。因此，我们将扩展ServiceRegistry来承载此全新的Service角色：

/**
 * The service role
 */
public interface EventPublishingService extends Service {
    public void publish(Event theEvent);
}

/**
 * A disabled (no-op) impl
 */
public class DisabledEventPublishingServiceImpl implements EventPublishingService {
    public static DisabledEventPublishingServiceImpl INSTANCE = new DisabledEventPublishingServiceImpl();

    private DisabledEventPublishingServiceImpl() {
    }

    @Override
    public void publish(Event theEvent) {
        // nothing to do...
    }
}

/**
 * A standard impl
 */
public class EventPublishingServiceImpl
        implements EventPublishingService, Configurable, Startable, Stoppable, ServiceRegistryAwareService {

    private ServiceRegistryImplementor serviceRegistry;
    private String jmsConnectionFactoryName;
    private String destinationName;

    private Connection jmsConnection;
    private Session jmsSession;
    private MessageProducer publisher;

    @Override
    public void injectServices(ServiceRegistryImplementor serviceRegistry) {
        this.serviceRegistry = serviceRegistry;
    }

	public void configure(Map configurationValues) {
	    this.jmsConnectionFactoryName = configurationValues.get( JMS_CONNECTION_FACTORY_NAME_SETTING );
	    this.destinationName = configurationValues.get( JMS_DESTINATION_NAME_SETTING );
	}

    @Override
	public void start() {
	    final JndiService jndiService = serviceRegistry.getService( JndiService.class );
	    final ConnectionFactory jmsConnectionFactory = jndiService.locate( jmsConnectionFactoryName );

        this.jmsConnection = jmsConnectionFactory.createConnection();
        this.jmsSession = jmsConnection.createSession( true,  Session.AUTO_ACKNOWLEDGE );

	    final Destination destination = jndiService.locate( destinationName );

	    this.publisher = jmsSession.createProducer( destination );
	}

    @Override
    public void publish(Event theEvent) {
        publisher.send( theEvent );
    }

    @Override
	public void stop() {
        publisher.close();
        jmsSession.close();
        jmsConnection.close();
	}
}

public class EventPublishingServiceInitiator implements StandardServiceInitiator<EventPublishingService> {
    public static EventPublishingServiceInitiator INSTANCE = new EventPublishingServiceInitiator();
    public static final String ENABLE_PUBLISHING_SETTING = "com.acme.EventPublishingService.enabled";

	public Class<R> getServiceInitiated() {
	    return EventPublishingService.class;
	}

    @Override
	public R initiateService(Map configurationValues, ServiceRegistryImplementor registry) {
	    final boolean enabled = extractBoolean( configurationValues, ENABLE_PUBLISHING_SETTING );
	    if ( enabled ) {
	        return new EventPublishingServiceImpl();
	    }
	    else {
	        return DisabledEventPublishingServiceImpl.INSTANCE;
	    }
	}
}

现在，让我们告诉Hibernate这个定制服务：

StandardServiceRegistry serviceRegistry = new StandardServiceRegistryBuilder()
        .addInitiator( EventPublishingServiceInitiator.INSTANCE )
        ...
