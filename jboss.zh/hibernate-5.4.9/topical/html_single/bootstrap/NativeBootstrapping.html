本指南讨论了引导Hibernate的过程org.hibernate.SessionFactory 。它还讨论了应用程序和集成商可以采用哪些方法来影响该过程。此引导过程分为2个不同的步骤。第一步是构建一个ServiceRegistry，其中包含Hibernate在引导和运行时所需的服务。第二步是构建一个Metadata对象，该对象表示应用程序模型的映射信息及其到数据库的映射。

在5.0版之前的应用程序启动了SessionFactory通过org.hibernate.cfg.Configuration宾语。仍然以某种有限的方式支持该方法。有关详细信息，请参见旧式引导指南。

建立服务注册表

实际上，我们关心建立两个不同的ServiceRegistries：

  • org.hibernate.boot.registry.BootstrapServiceRegistry

  • org.hibernate.boot.registry.StandardServiceRegistry

这些都是由构建器构建的， org.hibernate.boot.registry.BootstrapServiceRegistryBuilder和org.hibernate.boot.registry.StandardServiceRegistryBuilder分别。

有关一般情况下有关ServiceRegistries的更多信息，请参阅《 Services and Registries》指南。

BootstrapServiceRegistry

的BootstrapServiceRegistry用于保留Hibernate在引导和运行时所需的服务。这归结为3种服务：

  • ClassLoaderService -控制Hibernate与ClassLoaders交互的方式

  • IntegratorService -控制管理和发现org.hibernate.integrator.spi.Integrator实例。

  • StrategySelector -控制Hibernate如何解决各种战略合同的实现。这是一项非常强大的服务，但是对其的完整讨论不在本指南的范围之内。

如果您对这些BootstrapServiceRegistry服务的默认行为是Hibernate很好（这是常见的情况，尤其是在SE环境中），则可以跳过构建BootstrapServiceRegistry。如果您想更改BootstrapServiceRegistry是内置的，您将使用BootstrapServiceRegistryBuilder ：

范例1。建立一个BootstrapServiceRegistry
    BootstrapServiceRegistryBuilder bootstrapRegistryBuilder
            = new BootstrapServiceRegistryBuilder();
    // add a special ClassLoader
    bootstrapRegistryBuilder.applyClassLoader( mySpecialClassLoader );
    // manually add an Integrator
    bootstrapRegistryBuilder.applyIntegrator( mySpecialIntegrator );
    ...

    BootstrapServiceRegistry bootstrapRegistry = bootstrapRegistryBuilder.build();

的服务BootstrapServiceRegistry不能扩展（添加到）或覆盖（替换）。

标准服务注册中心

的服务StandardServiceRegistry可能被扩展和覆盖。

一种StandardServiceRegistry是通过StandardServiceRegistryBuilder可以采用以下两种方式之一进行构建：

示例2使用提供的BootstrapServiceRegistry构建StandardServiceRegistryBuilder
    BootstrapServiceRegistry bootstrapRegistry = ...;
    StandardServiceRegistryBuilder standardRegistryBuilder
            = new StandardServiceRegistryBuilder( bootstrapRegistry );
范例3。在不提供BootstrapServiceRegistry的情况下构建StandardServiceRegistryBuilder
    StandardServiceRegistryBuilder standardRegistryBuilder
            = new StandardServiceRegistryBuilder();

第二种形式将创建一个BootstrapServiceRegistry具有默认行为的即时运行。

一种StandardServiceRegistry也可以通过以下方式进行高度配置StandardServiceRegistryBuilder API。见StandardServiceRegistryBuilder有关详细信息，请参见javadocs。一些感兴趣的特定方法：

示例4配置StandardServiceRegistryBuilder
    StandardServiceRegistryBuilder standardRegistryBuilder = ...;
    // load some properties via resource lookup
    standardRegistryBuilder.loadProperties( "org/hibernate/example/MyProperties.properties" );
    // configure the registry from a resource lookup for a cfg.xml config file
    standardRegistryBuilder.configure( "org/hibernate/example/MyCfg.xml" );
    // apply a random setting
    standardRegistryBuilder.applySetting( "myProp", "some value" );
    // apply a service initiator
    standardRegistryBuilder.addInitiator( new CustomServiceInitiator() );
    // apply a service impl
    standardRegistryBuilder.addService( SomeCustomService.class, new SomeCustomServiceImpl() );

    // and finally build the StandardServiceRegistry
    StandardServiceRegistry standardRegistry = standardRegistryBuilder.build();

建立元数据

的org.hibernate.boot.Metadata对象包含应用程序域模型的解析表示形式及其到数据库的映射。我们显然需要构建解析表示形式的第一件事是要解析的源信息（带注释的类， hbm.xml文件， orm.xml文件）。这是目的org.hibernate.boot.MetadataSources 。

示例5配置元数据源
    MetadataSources sources = new MetadataSources( standardRegistry );
    // alternatively, we can build the MetadataSources without passing
    // a service registry, in which case it will build a default
    // BootstrapServiceRegistry to use
    // MetadataSources sources = new MetadataSources();

	// add a class using JPA/Hibernate annotations for mapping
    sources.addAnnotatedClass( MyEntity.class );

	// add the name of a class using JPA/Hibernate annotations for mapping.
	// differs from above in that accessing the Class is deferred which is
	// important if using runtime bytecode-enhancement
    sources.addAnnotatedClassName( "org.hibernate.example.Customer" );

    // Adds the named hbm.xml resource as a source: which performs the
    // classpath lookup and parses the XML
    sources.addResource( "org/hibernate/example/Order.hbm.xml" );

    // Adds the named JPA orm.xml resource as a source: which performs the
    // classpath lookup and parses the XML
    sources.addResource( "org/hibernate/example/Product.orm.xml" );

MetadataSources还有许多其他方法；探索其API和javadocs以获得更多信息。另外，所有方法MetadataSources如果您喜欢该样式，则允许链接。

范例6。使用方法链接配置MetadataSources
    MetadataSources sources = new MetadataSources( standardRegistry )
            .addAnnotatedClass( MyEntity.class )
            .addAnnotatedClassName( "org.hibernate.example.Customer" )
            .addResource( "org/hibernate/example/Order.hbm.xml" )
            .addResource( "org/hibernate/example/Product.orm.xml" );

定义了映射信息的来源后，我们需要构建Metadata宾语。如果您可以接受构建默认行为Metadata （或者如果依靠MetadataBuilderInitializer -见下文），那么您只需致电MetadataSources#buildMetadata 。

注意：请注意，在此引导过程中，可以在许多点传递ServiceRegistry。建议的方法是建立一个StandardServiceRegistry自己，并传递给MetadataSources构造函数。从那里， MetadataBuilder ， Metadata ， SessionFactoryBuilder和SessionFactory都一样StandardServiceRegistry 。

但是，如果您希望调整建造过程Metadata从MetadataSources您将需要使用MetadataBuilder通过获取MetadataSources#getMetadataBuilder 。 MetadataBuilder允许对Metadata建设过程。有关完整的详细信息，请参见其javadocs。

范例7。通过MetadataBuilder构建元数据
    MetadataBuilder metadataBuilder = sources.getMetadataBuilder();

    // Use the JPA-compliant implicit naming strategy
    metadataBuilder.applyImplicitNamingStrategy( ImplicitNamingStrategyJpaCompliantImpl.INSTANCE );

    // specify the schema name to use for tables, etc when none is explicitly specified
	metadataBuilder.applyImplicitSchemaName( "my_default_schema" );

	Metadata metadata = metadataBuilder.build();

建立SessionFactory

就像我们上面讨论的一样，如果您可以接受默认的行为SessionFactory从一个Metadata参考，您只需致电Metadata#buildSessionFactory 。但是，如果您要调整该构建过程，则需要使用SessionFactoryBuilder通过获取Metadata#getSessionFactoryBuilder 。见SessionFactoryBuilder javadocs，以了解它允许通过SessionFactory建设过程。

范例8。通过SessionFactoryBuilder构建SessionFactory
    SessionFactoryBuilder sessionFactoryBuilder = metadata.getSessionFactoryBuilder();

    // Supply a SessionFactory-level Interceptor
    sessionFactoryBuilder.applyInterceptor( new MySessionFactoryInterceptor() );

    // Add a custom observer
    sessionFactoryBuilder.addSessionFactoryObservers( new MySessionFactoryObserver() );

    // Apply a CDI BeanManager (for JPA event listeners)
    sessionFactoryBuilder.applyBeanManager( getBeanManagerFromSomewhere() );

    SessionFactory sessionFactory = sessionFactoryBuilder.build();

全部放在一起

自举API非常灵活，但是在大多数情况下，将其视为3个步骤的流程是最有意义的：

  1. 建立StandardServiceRegistry

  2. 建立Metadata

  3. 使用这两件事来构建SessionFactory

范例9。通过SessionFactoryBuilder构建SessionFactory
    StandardServiceRegistry standardRegistry = new StandardServiceRegistryBuilder()
            .configure( "org/hibernate/example/MyCfg.xml" )
            .build();

    Metadata metadata = new MetadataSources( standardRegistry )
            .addAnnotatedClass( MyEntity.class )
            .addAnnotatedClassName( "org.hibernate.example.Customer" )
            .addResource( "org/hibernate/example/Order.hbm.xml" )
            .addResource( "org/hibernate/example/Product.orm.xml" )
            .getMetadataBuilder()
            .applyImplicitNamingStrategy( ImplicitNamingStrategyJpaCompliantImpl.INSTANCE )
            .build();

    SessionFactory sessionFactory = metadata.getSessionFactoryBuilder()
            .applyBeanManager( getBeanManagerFromSomewhere() )
            .build();
