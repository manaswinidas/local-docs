前言

同时使用面向对象的软件和关系数据库既麻烦又费时。由于对象和关系数据库中的数据表示方式之间存在许多“范式不匹配”，因此开发成本明显更高。Hibernate是一种用于Java环境的对象/关系映射（ORM）解决方案。术语“对象/关系映射”是指在对象模型表示与关系数据模型表示之间映射数据的技术。有关良好的高级讨论，请参见http://en.wikipedia.org/wiki/Object-relational_mapping 。同样，Martin Fowler的OrmHate文章介绍了许多不匹配问题。

尽管使用Hibernate不需要具备扎实的SQL背景知识，但是对这些概念有基本的了解可以帮助您更快，更全面地理解Hibernate。对数据建模原理的理解特别重要。http://www.agiledata.org/essays/dataModeling101.html和http://en.wikipedia.org/wiki/Data_modeling都是理解这些数据建模原理的良好起点。如果您完全不熟悉Java中的数据库访问，则https://www.marcobehler.com/guides/a-guide-to-accessing-databases-in-java包含有关各个部分，部分和选项的良好概述。

Hibernate负责从Java类到数据库表以及从Java数据类型到SQL数据类型的映射。此外，它还提供数据查询和检索功能。它可以显着减少开发时间，否则将花费在SQL和JDBC中的手动数据处理上。Hibernate的设计目标是通过消除使用SQL和JDBC进行手工数据手工处理的需求，使开发人员摆脱95％的常见数据持久性相关编程任务。但是，与许多其他持久性解决方案不同，Hibernate不会向您隐藏SQL的强大功能，并保证您对关系技术和知识的投资一如既往地有效。

对于仅使用存储过程在数据库中实现业务逻辑的以数据为中心的应用程序，Hibernate可能不是最佳解决方案，它对于基于Java的中间层中的面向对象域模型和业务逻辑最有用。但是，Hibernate当然可以帮助您删除或封装特定于供应商的SQL代码，并简化将结果集从表格表示形式转换为对象图的常见任务。

有关参与的信息，请参见http://hibernate.org/orm/contribute/ 。

本指南中引用的教程的项目和代码可作为hibernate-tutorials.zip获得。

  • 的lib/required/目录包含hibernate-core jar及其所有依赖项。无论正在使用Hibernate的哪些功能，所有这些jar都必须在您的类路径中可用。

  • 的lib/spatial/目录包含hibernate-spatial jar及其所有依赖项（除了lib/required/ ）

  • 的lib/jpa-metamodel-generator/目录包含生成Criteria API类型安全的元模型所需的jar。

  • 的lib/optional/该目录包含Hibernate提供的各种连接池和二级缓存集成所需的jar，以及它们的依赖关系。

2。使用本机Hibernate API和hbm.xml映射的教程

本教程位于下载包中，位于basic/ 。
目标

  • 引导HibernateSessionFactory

  • 使用Hibernate映射（ hbm.xml ）文件以提供映射信息

2.1。Hibernate配置文件

对于本教程， hibernate.cfg.xml文件定义了Hibernate配置信息。

的connection.driver_class ， connection.url ， connection.username和connection.password元素定义JDBC连接信息。这些教程利用了H2内存数据库，因此这些属性的值都特定于在其内存模式下运行H2。 connection.pool_size用于配置Hibernate内置连接池中的连接数。

内置的Hibernate连接池绝不用于生产用途。它缺少可用于生产环境的连接池上发现的几个功能。

的dialect属性指定Hibernate进行对话的特定SQL变体。

在大多数情况下，Hibernate能够正确确定要使用的方言。如果您的应用程序针对多个数据库，则这特别有用。

的hbm2ddl.auto属性启用直接将数据库模式自动生成到数据库中的功能。

最后，将持久性类的映射文件添加到配置中。的resource的属性元素使Hibernate尝试使用a来将该映射定位为类路径资源java.lang.ClassLoader抬头。

有很多方法和选项可以引导Hibernate SessionFactory 。有关其他详细信息，请参见本机引导主题指南。

2.2。实体Java类

本教程的实体类是org.hibernate.tutorial.hbm.Event

关于实体的注意事项

  • 此类将标准JavaBean命名约定用于属性getter和setter方法，以及字段的私有可见性。尽管这是推荐的设计，但不是必需的。

  • 无参数构造函数（也是JavaBean约定）是所有持久类的必需项。Hibernate需要使用Java Reflection为您创建对象。构造函数可以是私有的。但是，在没有字节码检测的情况下，运行时代理生成和有效数据检索需要包或公共可见性。

2.3。映射文件

本教程的映射文件是classpath资源org/hibernate/tutorial/hbm/Event.hbm.xml （如上所述）。

Hibernate使用映射元数据来确定如何加载和存储持久类的对象。Hibernate映射文件是为Hibernate提供此元数据的一种选择。

范例1。类映射元素
<class name="Event" table="EVENTS">
	...
</class>
类映射元素的功能

  • 的name属性（此处与package来自的属性元素）命名要定义为实体的类的FQN。

  • 的table属性为包含该实体数据的数据库表命名。

的实例Event现在将类映射到EVENTS数据库表。

示例2id映射元素
<id name="id" column="EVENT_ID">
    ...
</id>

Hibernate使用由元素，以唯一标识表中的行。

id元素不需要映射到表的实际主键列，但这是常规约定。在Hibernate中映射的表甚至不需要定义主键。但是，强烈建议所有模式都定义适当的参照完整性。因此，id和主键在整个Hibernate文档中可互换使用。

的此处的元素将EVENT_ID列命名为EVENTS表的主键。它还可以识别id的属性Event类作为包含标识符值的属性。

的generator元素告知Hibernate使用哪种策略为该实体生成主键值。本示例使用简单的递增计数。

范例3。属性映射元素
<property name="date" type="timestamp" column="EVENT_DATE"/>
<property name="title"/>

他们俩元素声明的其余两个持久属性Event类： date和title 。的date属性映射包括column属性，但title才不是。在没有column属性，Hibernate使用属性名称作为列名称。这适合title ， 但是由于date是大多数数据库中的保留关键字，您需要为列名指定一个非保留字。

的title映射也缺少类型属性。映射文件中声明和使用的类型既不是Java数据类型也不是SQL数据库类型。相反，它们是Hibernate映射类型 ，它们是在Java和SQL数据类型之间转换的转换器。如果未在映射中指定type属性，则Hibernate尝试自动确定正确的转换和映射类型，方法是使用Java反射来确定已声明属性的Java类型，并使用该Java类型的默认映射类型。

在某些情况下，自动检测可能未选择您期望或需要的默认值，如date属性。Hibernate不知道该类型的属性是否java.util.Date ，应映射到SQL DATE ， TIME或TIMESTAMP数据类型。通过将属性映射到时间戳转换器来保留完整的日期和时间信息，该时间戳转换器标识转换器类org.hibernate.type.TimestampType 。

处理映射文件时，Hibernate使用反射来确定映射类型。此过程增加了时间和资源方面的开销。如果启动性能很重要，请考虑明确定义要使用的类型。

2.4。范例程式码

的org.hibernate.tutorial.hbm.NativeApiIllustrationTest该类说明了如何使用Hibernate本机API。

为了易于使用，这些教程中的示例以JUnit测试的形式呈现。这种方法的好处是setUp和tearDown大致说明org.hibernate.SessionFactory是在应用程序启动时创建的，并在应用程序生命周期结束时关闭。
示例4获得org.hibernate.SessionFactory
protected void setUp() throws Exception {
	// A SessionFactory is set up once for an application!
	final StandardServiceRegistry registry = new StandardServiceRegistryBuilder()
			.configure() // configures settings from hibernate.cfg.xml
			.build();
	try {
		sessionFactory = new MetadataSources( registry ).buildMetadata().buildSessionFactory();
	}
	catch (Exception e) {
		// The registry would be destroyed by the SessionFactory, but we had trouble building the SessionFactory
		// so destroy it manually.
		StandardServiceRegistryBuilder.destroy( registry );
	}
}

的setUp方法首先建立一个org.hibernate.boot.registry.StandardServiceRegistry实例，它将配置信息合并到服务的工作集中，以供SessionFactory使用。在本教程中，我们定义了所有配置信息hibernate.cfg.xml因此，在这里看到的内容没有太多有趣的地方。

使用StandardServiceRegistry我们创建org.hibernate.boot.MetadataSources这是告诉Hibernate您的域模型的起点。同样，由于我们在hibernate.cfg.xml因此，在这里看到的内容没有太多有趣的地方。

org.hibernate.boot.Metadata代表应用程序域模型的完整，部分验证的视图， SessionFactory将基于。

引导程序的最后一步是构建SessionFactory 。的SessionFactory是一个线程安全的对象，实例化一次即可服务整个应用程序。

的SessionFactory充当...的工厂org.hibernate.Session实例，应将其视为“工作单元”的必然结果。

示例5保存实体
Session session = sessionFactory.openSession();
session.beginTransaction();
session.save( new Event( "Our very first event!", new Date() ) );
session.save( new Event( "A follow up event", new Date() ) );
session.getTransaction().commit();
session.close();

testBasicUsage()首先创造一些新的Event对象，并将它们交给Hibernate进行管理，方法是使用save()方法。Hibernate现在负责对数据库中的每个数据库执行INSERT Event 。

范例6。获取实体列表
session = sessionFactory.openSession();
session.beginTransaction();
List result = session.createQuery( "from Event" ).list();
for ( Event event : (List<Event>) result ) {
    System.out.println( "Event (" + event.getDate() + ") : " + event.getTitle() );
}
session.getTransaction().commit();
session.close();

在这里，我们看到了一个Hibernate查询语言（HQL）的示例，该示例可以加载所有现有的Event通过生成适当的SELECT SQL，将其发送到数据库并填充数据库中的对象Event结果集数据的对象。

范例7。将类识别为实体
@Entity
@Table( name = "EVENTS" )
public class Event {
    ...
}

的@javax.persistence.Entity批注用于将类标记为实体。它的功能与 映射文件中讨论的映射元素。另外@javax.persistence.Table批注显式指定表名。没有此规范，默认表名将为EVENT 。

范例8。标识标识符属性
@Id
@GeneratedValue(generator="increment")
@GenericGenerator(name="increment", strategy = "increment")
public Long getId() {
    return id;
}

@javax.persistence.Id标记定义实体标识符的属性。

@javax.persistence.GeneratedValue和@org.hibernate.annotations.GenericGenerator协同工作以指示Hibernate应该使用Hibernate的increment该实体的标识符值的生成策略。

范例9。识别基本属性
public String getTitle() {
    return title;
}

@Temporal(TemporalType.TIMESTAMP)
@Column(name = "EVENT_DATE")
public Date getDate() {
    return date;
}

如在映射文件中 ， date属性需要特殊处理才能考虑其特殊命名和其SQL类型。

使用注释进行映射时，默认情况下，实体的属性被认为是持久性的，这就是为什么我们看不到任何与之相关的映射信息的原因title 。

4.1。 persistence.xml

以前的教程使用了特定于Hibernate的内容hibernate.cfg.xml配置文件。但是，JPA定义了一个不同的引导过程，该过程使用其自己的名为persistence.xml 。该引导过程由JPA规范定义。在Java™SE环境中，需要持久性提供程序（在这种情况下为Hibernate）通过对文件的类路径查找来找到所有JPA配置文件。 META-INF/persistence.xml资源名称。

例子10. persistence.xml
<persistence xmlns="http://java.sun.com/xml/ns/persistence"
        xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
        xsi:schemaLocation="http://java.sun.com/xml/ns/persistence http://java.sun.com/xml/ns/persistence/persistence_2_0.xsd"
        version="2.0">
    <persistence-unit name="org.hibernate.tutorial.jpa">
        ...
    </persistence-unit>
</persistence>

persistence.xml文件应为每个“持久性单元”提供唯一的名称。应用程序使用此名称在获取javax.persistence.EntityManagerFactory参考。

在 Hibernate配置文件中讨论了元素。在这里javax.persistence尽可能使用前缀的变体。注意，其余的特定于Hibernate的配置设置名称现在以前缀hibernate. 。

此外， 元素的功能与在Hibernate配置文件中看到的功能相同。

4.3。范例程式码

先前的教程使用了Hibernate本机API。本教程使用JPA API。

示例11获取javax.persistence。EntityManagerFactory
protected void setUp() throws Exception {
	sessionFactory = Persistence.createEntityManagerFactory( "org.hibernate.tutorial.jpa" );
}

再次注意，持久性单元名称是org.hibernate.tutorial.jpa ，它匹配persistence.xml 。

范例12。保存（持久）实体
EntityManager entityManager = sessionFactory.createEntityManager();
entityManager.getTransaction().begin();
entityManager.persist( new Event( "Our very first event!", new Date() ) );
entityManager.persist( new Event( "A follow up event", new Date() ) );
entityManager.getTransaction().commit();

该代码类似于保存实体 。的javax.persistence.EntityManager使用接口代替org.hibernate.Session接口。JPA将此操作称为“持久”而不是“保存”。

范例13。获取实体列表
entityManager = sessionFactory.createEntityManager();
entityManager.getTransaction().begin();
List<Event> result = entityManager.createQuery( "from Event", Event.class ).getResultList();
for ( Event event : result ) {
	System.out.println( "Event (" + event.getDate() + ") : " + event.getTitle() );
}
entityManager.getTransaction().commit();

同样，该代码与我们在获取实体列表中看到的代码非常相似。

5.3。范例程式码

public void testBasicUsage() {
    ...
    AuditReader reader = AuditReaderFactory.get( entityManager );

    ...

    ...
}

我们看到org.hibernate.envers.AuditReader从org.hibernate.envers.AuditReaderFactory包裹javax.persistence.EntityManager 。

接下来， find方法检索实体的特定修订版。第一个电话说要找到ID为2的事件的修订版本1。第二个呼叫说要找到ID为2的事件的修订版本2。
